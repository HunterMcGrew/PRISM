# Plan: THR-1654

## Ticket

https://linear.app/tractru/issue/THR-1654/devcontainer-entrypoint-creates-root-owned-wp-content-symlinks-on

## Goal

Eliminate the class of "devcontainer vs prod user drift" bugs by unifying both code paths on a single non-root user (`thrive`, renamed from `www-data` at image build time) baked into the base image rather than created at runtime. Closes the three-owner ownership chain that caused THR-1654 and removes the privilege-drop fragility that PR #1827 has been fighting.

---

## Design

Not applicable — infrastructure change, no UI.

## Decisions

- **Unify on a single non-root user (`thrive`, UID/GID 33) for both dev and prod.** Renamed in place from `www-data` at image build time via `usermod -l` + `groupmod -n` so Apache's UID expectation is preserved. Named `thrive` because this user is a dev+prod hybrid — Apache's runtime user AND the developer's login seat — and no purist name serves both. `thrive` (the project name) is self-documenting across both contexts: "the user that runs the Thrive platform." Closes the three-owner chain (root symlink → devuser /workspace → www-data reader) that caused THR-1654.
- **Rename updates Apache envvars too.** `/etc/apache2/envvars` has `APACHE_RUN_USER=www-data` and `APACHE_RUN_GROUP=www-data`; both become `thrive`. Apache reads these at startup — without the update Apache won't find the user and fails to start.
- **Playwright system deps move to Dockerfile.admin build time.** They were the only reason setup.sh needed root. Moving them removes the root-only runtime phase entirely. CI build gets marginally longer; accepted cost, optimize later if it matters.
- **Drop `user: root` in the devcontainer overlay.** With root-only work gone from setup.sh, the override has no remaining purpose, and keeping it re-invites privilege-drop bugs like THR-1654.
- **Keep the setup.sh entrypoint override.** Per-container-lifetime dev-user config (`corepack prepare`, `git core.editor`) still needs to run once per container; these run fine as `thrive`.
- **Give thrive a real shell (`/bin/bash`) and home (`/home/thrive`) in Dockerfile.admin.** Required for VS Code to attach and for developer ergonomics (bashrc, pnpm cache, gh auth). Apache reads its own config for User/Group — doesn't consult passwd's shell field.
- **Include `HOST_UID` / `HOST_GID` build args from the start (default 33).** Our Windows teammate likely runs with the workspace on the WSL2 filesystem (fast-pnpm path), where the workspace is owned by WSL UID 1000. Without a UID remap, `thrive` (UID 33) can't write to bind-mounted files — VS Code saves and `pnpm install` would fail. On OrbStack/Mac the default 33 works because the VM layer maps ownership transparently. A three-line `ARG` + conditional `usermod` handles both paths; documented in the architect doc.
- **install_wp.sh is unchanged.** It now runs as thrive on both code paths by default (base compose sets `user: thrive`, overlay no longer overrides). Symlinks inherit thrive ownership naturally.
- **PR #1827 is superseded.** It attempted `exec runuser -u www-data -- "$@"` in setup.sh. This plan removes the need for any priv drop by removing root from the dev path entirely. Close #1827 with a pointer to THR-1654.

### Walkthrough notes (for the record, not blocking)

- **Rename ordering is correct.** `groupmod -n` precedes `usermod -l` so the group exists under its new name by the time usermod tries to keep user and group names consistent. Files owned by UID 33 resolve to `app-user` after the rename without any file-level chown needed (scenario A). The remap case (scenario B) is handled by the subsequent `chown -R app-user:app-user /var/www` which correctly rewrites ownership to the remapped UID. ✓
- **`-m` omission is correct.** Verified the base image's `/etc/passwd` has www-data's home at `/var/www`. Captured in the lessons file; the Dockerfile comment also spells out the reasoning in place.
- **The `sed -i 's/www-data/app-user/g' /etc/apache2/envvars` is safe.** That file only references `www-data` in the `APACHE_RUN_USER` / `APACHE_RUN_GROUP` lines in the wordpress:php8.2-apache base image. A narrower anchored sed would be more robust against future unknown additions, but the global form is also more robust against those same additions being other user references that would also need remapping — call it a wash.
