## Decisions

- **Panel-level click delegation in `MegaMenuNavigationItem`.**
  - **Root cause:** `CardBlock` (RSC) → `Standard/Horizontal/FloatingCard` → `CardsButtonsList` → `Button` → `<Link>` is fully RSC. No client component in the chain can read `useMegaMenuContext` to bind `closeMenu` as `onClick`. The bug class is wider than cards — any RSC chain rendering a link inside a Mega Menu panel has the same gap.
  - **Alternatives considered:** (a) targeted `MegaMenuCTAButton` / `MegaMenuCTALink` inside the card path (the PR #1958 shape) — works but inverts dependencies in the card folder (generic Card imports from feature-specific mega-menu-editor across 4 files). (b) Promote `Button`'s link branch to a `"use client"` `ButtonLink` that consumes `useMegaMenuCloseOnClick` (Shape A from earlier in this plan's history) — universal coverage, but `Button → mega-menu-editor` makes a primitive aware of a feature, which violates the architectural rule "primitives don't know about features." (c) `BlocksRenderer` per-block-type prop overrides (THR-1813) — proper architectural fix, larger scope. (d) `childrenButtons` / slot prop on Card — adds API surface without removing the import inversion (something upstream still has to fill the slot).
  - **Chosen approach:** the mega menu panel owns its own close behavior. `MegaMenuNavigationItem` is already a `"use client"` component with direct access to the popover's `close` callback. Adding a click-delegation handler at the panel boundary closes the menu on any anchor click bubbled up from inside, with zero changes to Card, Button, or any other generic component.
  - **Implementation guidance:** the delegation matches via `Element.closest("a[href]")` so it covers anchor children at any depth (e.g. an `<img>` wrapped in `<Link>`, or the inner span of a Button-rendered Link). Buttons (`<button>`) and other non-anchor interactive elements inside panels are intentionally not affected — they are not navigation, and closing the panel on every interactive click would surprise users. Anchors without an `href` are also ignored. The delegation respects `event.stopPropagation()` on inner elements that explicitly opt out.

- **No new files in `frontend/components/button/`, `frontend/components/card/`, or `frontend/components/mega-menu-editor/`.** Card and Button stay completely unaware of the mega menu. The `useMegaMenuCloseOnClick` hook proposed in earlier shapes is not introduced — there's no consumer for it.

- **Image-link close-on-menu is automatic, not configurable.** Any anchor inside the panel — image link, CTA button rendered as a Link, raw text link — closes the panel on click. The earlier "keep vs drop" decision becomes irrelevant: the delegation covers all anchors uniformly. This matches PR #1958's UX without requiring per-surface wrappers.

- **Per-block `onClick={closeMenu}` wiring is redundant once delegation owns the close.** Four blocks (`MegaMenuFeaturedLinks`, `MegaMenuNavigationList`, `MegaMenuLinkWithIcon`, `MegaMenuLocations`) had been individually calling `closeMenu` on link click. Delegation makes every one of those calls a no-op double-trigger. Removed in this PR. `MegaMenuInfoBanner` and `MegaMenuFeaturedBanner` retained for now — their `onClose` prop has external semantics worth investigating separately (Linear follow-up worthwhile if the prop is dead).

## Acceptance Criteria

### Behavioral

- [ ] **Given** the Mega Menu feature flag is enabled and a Card block with a CTA button is placed inside a Mega Menu panel, **When** I click the card's CTA button on the frontend, **Then** the destination page loads and the Mega Menu panel closes.
- [ ] **Given** the Mega Menu feature flag is enabled and a Mega Menu panel contains an info-banner, featured-banner, or any block whose CTAs render through links, **When** I click any of those CTAs, **Then** the destination page loads and the Mega Menu panel closes.
- [ ] **Given** a Card block placed on a regular page (not inside a Mega Menu panel), **When** I click the card's CTA button, **Then** the destination loads with no console errors and no unexpected close behavior.
- [ ] **Given** the Mega Menu feature flag is enabled and a card has both an image link and a CTA button (same destination), **When** I click the card image, **Then** the destination loads and the Mega Menu panel closes.

- [ ] **Given** I navigate the Mega Menu using a keyboard (Tab to focus, Enter to activate), **When** I press Enter on a card's CTA button inside an open panel, **Then** the destination loads and the panel closes — Enter on a link fires a click event, which the delegation handles identically to mouse activation.

## Review Issues

### Modifier-click closes panel even though navigation goes to a new tab

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:69`
- **Problem:** `cmd+click` / `ctrl+click` / `shift+click` on an inner anchor opens the link in a new tab/window but the delegation still fires `close()` on the current page — a user opening multiple panel links in new tabs has to reopen the menu between each click.
- **Suggested fix:** UX judgment call. If we want to preserve the menu for new-tab navigation, guard the handler with `if (e.metaKey || e.ctrlKey || e.shiftKey) return;` before the `closest("a[href]")` check, and add a test asserting the chosen behavior (e.g. `fireEvent.click(link, { metaKey: true })`).
- **Resolution:** Guard applied in `handlePanelClick` — `if (e.metaKey || e.ctrlKey || e.shiftKey) return;`. `it.each` test added covering all three modifiers.

### Wrapper `<div>` may shift the panel layout

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:137`
- **Problem:** The new `<div onClick={handlePanelClick}>` had no className. `PopoverPanel` is `flex w-full` and pane content uses `container-xl` (= `mx-auto w-full max-w-screen-xl px-6` per `frontend/styles/globals.css:56-58`). Previously the pane was a direct flex child and `mx-auto` distributed auto margins to center it; the unstyled wrapper became a shrink-wrapping flex item and the pane's `mx-auto` no longer had space to distribute. Visible at viewports > 1280px.
- **Suggested fix:** `className="contents"` removes the wrapper from layout entirely (cleanest), or `className="w-full"` as a conservative fallback.
- **Resolution:** Added `className="w-full"`. Wrapper now expands to full panel width; pane's `mx-auto` centers within the wrapper up to `max-w-screen-xl`. Functionally identical layout, no browser-quirk concerns. Chose `w-full` over `display: contents` to avoid historical a11y-tree quirks (Chromium <86, Firefox <89). All 28 `MegaMenuNavigationItem` tests pass; check-types + prettier clean.

### Missing test pinning the documented `stopPropagation` opt-out

- **Severity:** `minor`
- **Status:** `fixed` (Eric re-review confirmed)
- **File:** `frontend/components/navigation-menu-item/__tests__/MegaMenuNavigationItem.test.tsx`
- **Problem:** Both the plan (`## Decisions`) and `mega-menu.md` describe `event.stopPropagation()` from an inner element as an intentional opt-out — a child can suppress the panel close by stopping the click before it bubbles to the wrapper div. No test pins the contract. A future refactor that swaps to capture-phase, replaces bubbling with `event.composedPath()`, or moves the delegation up the tree would silently flip the contract without breaking a test.
- **Suggested fix:** Add a test rendering an inner anchor whose `onClick` calls `e.stopPropagation()`, fire click on the anchor, assert `mockPopoverClose` was NOT called.
- **Resolution:** Added `it("does not close when an inner anchor stops propagation", ...)` in the "Panel click delegation" describe block. Renders an anchor with an `onClick` handler that calls `e.stopPropagation()`, fires click, asserts `mockPopoverClose` was not called. All 29 `MegaMenuNavigationItem` tests pass.
