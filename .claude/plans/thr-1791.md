# Plan: THR-1791 — Architect-Doc Verification Rule + Branch-Plan Depth Rule

## Ticket

[THR-1791](https://linear.app/tractru/issue/THR-1791/add-architect-doc-verification-rule-branch-plan-depth-rule-plus-paired)

## Goal

Codify source-verification as the default review standard for architect docs and paired dev docs, and codify decision-depth on plan entries that record verified fixes — across the rule corpus, the ADR record, and the three skills (Winston, Eric, Briar) that author or review these artifacts.

---

## Context

PR #1925 (THR-1775) shipped with five substantive accuracy gaps in a `.claude/architect/` doc — composer-install asymmetry rationale, missing `composer test:install` prereq, undocumented `THRIVE_TOKEN` dual purpose, undocumented `release/v*` skip, muddled HOST_UID phrasing — because the doc received a glance-mode review right-sized for the change _count_ but wrong-sized for the change _class_. A deeper Winston audit caught them post-merge. Architect docs are durable agent context and load via `manifest.json` at every relevant edit, so a confident-sounding doc that drifts from source actively misleads every future agent that reads it.

The same audit surfaced a second gap: branch plans record the _what_ of a verified decision but rarely the _why_. When Winston confirms a Sasha diagnosis or settles on a fix, the entry tends to capture the conclusion only. Clove then chooses between plausible interpretations, Briar self-reviews without documented reasoning, and Eric PR-reviews against the same gap.

This ticket codifies both fixes as durable spec — a new rule, two ADRs, a depth section in the existing branch-plan rule, and triage hooks in three skill files — so the bar is in effect when the next architect-doc PR opens. Pure `.claude/` work — no source code touched.

## Decisions

- **Eli executes all eight file edits** — every artifact is durable spec content under `.claude/`, governed by `writing-voice.md` and `spec-editing.md`. The ticket's "Winston (skill changes)" assignment refers to subject-matter ownership (Winston is the source of truth for what his startup flow does); Eli authors the spec text. Considered: routing skill patches to a separate lane. Rejected: no separate lane exists for skill spec edits, and splitting authorship across two personas on a single ticket would invite voice drift between the rule body, ADRs, branch-plan section, and skill files — they read together, so one author keeps voice consistent.
- **ADR-0023 (architect docs) and ADR-0024 (branch-plan depth) are paired but separate** — both stem from the same THR-1775 audit, but they govern different surfaces (review review-class vs. plan-entry depth). Considered: combining into one ADR. Rejected: each ADR backs a different downstream rule (`architect-doc-verification.md` and the `branch-plan.md` Depth section), and a single ADR backing two rules muddies the cross-reference graph.
- **THR-1775 incident is recounted inline in both ADRs, not cross-linked to a plan file** — no `.claude/plans/thr-1775.md` exists and no `lessons.md` entry captures the audit findings. Considered: creating a retroactive THR-1775 plan file. Rejected: ADR Context sections are the right home for incident narrative when no other durable record exists, and creating a plan file purely to host a back-reference is busy-work.
- **Depth section lands after `## 5. Keep the Plan Clean and Concise`, not inside the Decisions template** — the brevity rule and the depth nuance read as a pair when adjacent. Considered: embedding Depth guidance into the existing Decisions template at line 205+. Rejected: the template is for the file structure (what sections exist), not the writing rule (how to fill them); landing the rule alongside the existing brevity rule keeps "how to write" guidance together.
- **Three-bucket triage (verified / diverged / missing) is the canonical phrasing** across the rule, both ADRs, and all three skill hooks. Considered: a four-bucket variant separating "missing source" from "missing claim verification." Rejected: three buckets covers the actual review states; four adds nuance the reviewer rarely needs.
- **Eric's and Briar's hooks use shared phrasing**, not parallel-but-different prose — per the ticket's "shared phrasing" requirement and to keep the rule reading identically across author-side and reviewer-side surfaces.
