# Plan: THR-773

## Ticket

https://linear.app/tractru/issue/THR-773/feature-block-cta-can-handle-linkformcall-us

## Goal

Enable the Feature Block CTA button to handle link, form modal, and call-us modal actions — matching CTA options available in other blocks.

---

## Decisions

- Fresh branch from main — original PR #946 (now draft) is reference only, code is a year stale
- Use `ModalButton` (not `FormCTAButton`/`CallCTAButton`) for form/call CTA types — it accepts custom `className` so we keep the Feature Block's existing CTA styling, and it handles formId-only modal opening without needing a server-side form fetch
- No resolver needed — `FeatureModalCta` passes `formId` to `AppContext.openModal()` and the shared modal system fetches the form client-side
- Replaced `ModalButton` with `FeatureModalCta` — a plain `<button>` + `openModal()` leaf component. `ModalButton` wraps `Button` which applies CVA defaults (`rounded-full`, `px-8`, `py-3`, etc.) that collide with FeatureBlock's custom CTA classes. Without `tailwind-merge`, the conflict is unresolvable via className ordering
- FeatureBlock.tsx stays RSC — `ModalButton` is the only `"use client"` leaf, imported as a child component
- `ctaType` defaults to `"link"` for backwards compatibility with existing Feature Block instances
- Card-level link wrapping (`<Link>` around entire block when no ctaText) only applies when `ctaType === "link"` or is absent
- Scope limited to link/form/call — no "content" or "menu" CTA types
- Decouple component props from `BlockAttributes` as part of this work (THR-1452 migration) — component defines its own `FeatureBlockProps`, resolver not needed so schema→props mapping happens at the block renderer level
- Editor CTA type selector follows icon-button pattern: `SelectControl` + conditional `GravityFormSelect` from `../../utils/GravityFormSelect`
- `isFeatureCtaType` type guard lives in `type-guards.ts` (colocated with block), not in `schema.ts` — keeps schema focused on types and constants

### Task 1: Update schema.ts — add CTA type attributes

### Task 2: Decouple component props from BlockAttributes
**File:** `frontend/blocks/feature/FeatureBlock.tsx`
- Define `FeatureBlockProps` independently — do not extend or import `BlockAttributes`
- Mark fields required if they are always provided after defaults are applied
- Mark truly optional fields (colors, mediaId, formId, etc.) as optional
- Remove any `import type { BlockAttributes }` from the component
- Rename `BlockFrontendProps` → `FeatureBlockProps`
- Follow component-props-decoupling.md rules

### Task 3: Update FeatureBlock.tsx — conditional CTA rendering
**File:** `frontend/blocks/feature/FeatureBlock.tsx`
- Add `ctaType` and `formId` to `FeatureBlockProps`
- Keep component as RSC — no `"use client"`
- When `ctaType === "link"` (or absent): render existing `<Link>` CTA (no change)
- When `ctaType === "form"` or `ctaType === "call"`: render `<ModalButton>` with Feature Block's existing CTA className, passing `actionType={ctaType}` and `formId={formId}`
- Card-level link wrapping: only when `ctaType === "link"` or undefined
- Depends on: Task 1 (schema types), Task 2 (props)

### Task 4: Update backend index.tsx — register new attributes
**File:** `backend/plugins/gravity-platform-core/src/blocks/feature/index.tsx`
- Add `ctaType` attribute (type: "string", default: "link")
- Add `formId` attribute (type: "number")
- Depends on: Task 1 (schema types for import)

### Task 5: Update backend edit.tsx — editor CTA type controls
**File:** `backend/plugins/gravity-platform-core/src/blocks/feature/edit.tsx`
- Add `SelectControl` for CTA type (link/form/call) — follow icon-button edit.tsx pattern
- When type is "link": show existing internal/external toggle + URL controls (no change)
- When type is "form": show `GravityFormSelect` from `../../utils/GravityFormSelect`
- When type is "call": show only the CTA text field (label)
- Import `GravityFormSelect` and its `GravityFormOption` type
- Depends on: Task 4 (attributes registered)

### Task 6: Write tests
**File:** `frontend/blocks/feature/__tests__/FeatureBlock.test.tsx`
- Test `ctaType === "link"` renders `<Link>` (existing behavior, verify not broken)
- Test `ctaType === "form"` renders `ModalButton` with correct actionType and formId
- Test `ctaType === "call"` renders `ModalButton` with correct actionType
- Test default (no ctaType) renders as link for backwards compatibility
- Test card-level link wrapping only applies for link type
- Mock `ModalButton` in tests
- Update existing tests to use new `FeatureBlockProps` type
- Depends on: Task 3 (component changes)

## Review Issues

### `as FeatureCtaType` assertion in edit.tsx
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/feature/edit.tsx:173`
- **Problem:** Uses `as FeatureCtaType` cast after an `includes()` guard — safe but not idiomatic
- **Suggested fix:** Replace with a type guard function; low priority since the runtime check is correct
- **Fixed in:** Added `isFeatureCtaType` type guard in `schema.ts`, replaced `as` cast in `handleCtaTypeChange`

### Stale formId when switching CTA types
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/feature/edit.tsx:172`
- **Problem:** Switching from "form" to "link" or "call" leaves `formId` set but unused in saved attributes
- **Suggested fix:** Clear `formId` in `handleCtaTypeChange` when switching away from "form"
- **Fixed in:** `handleCtaTypeChange` now spreads `formId: undefined` when `ctaType !== "form"`

### Unused HeadingSize import
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/blocks/feature/FeatureBlock.tsx:8`
- **Problem:** `HeadingSize` is imported but not used — only `HeadingLevel` is referenced in `FeatureBlockProps`
- **Suggested fix:** Remove `HeadingSize` from the import
- **Fixed in:** Removed unused `HeadingSize` from the import statement

### FeatureCtaType→ModalType type mismatch on ModalButton
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/blocks/feature/FeatureBlock.tsx:207`
- **Problem:** `ctaType` (`FeatureCtaType`) passed to `ModalButton.actionType` (`ModalType`) — `"link"` is not a valid `ModalType`. TypeScript can't narrow through the `isLinkType` boolean variable.
- **Suggested fix:** Split into two inline conditionals narrowing to literal `"form"` and `"call"`
- **Fixed in:** Replaced single `!isLinkType` conditional with two `ctaType === "form"` / `ctaType === "call"` conditionals

### Missing aria-haspopup on FeatureModalCta button
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/blocks/feature/FeatureModalCta.tsx:23`
- **Problem:** Button opens a modal dialog but doesn't signal this to screen readers
- **Suggested fix:** Add `aria-haspopup="dialog"` to the `<button>`
- **Fixed in:** Added `aria-haspopup="dialog"` attribute

### No-op e.preventDefault() on button type="button"
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/blocks/feature/FeatureModalCta.tsx:27`
- **Problem:** `e.preventDefault()` is a no-op on `<button type="button">` — no default behavior to prevent
- **Suggested fix:** Remove the call
- **Fixed in:** Removed `e.preventDefault()` and simplified onClick signature

### CVA style collision on ModalButton className
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/blocks/feature/FeatureBlock.tsx:207`
- **Problem:** `ModalButton` → `Button` applies CVA base classes (`rounded-full`, `px-8 py-3`, `border-primary bg-primary`) before appending custom `className`. Without `tailwind-merge`, conflicting Tailwind utilities produce unpredictable visuals — form/call CTA will look different from link CTA.
- **Suggested fix:** Bypass Button's CVA — create a thin `FeatureModalCta` client component that renders a plain `<button>` with only the custom className and wires `openModal`, or add an `unstyled` prop to Button.
- **Fixed in:** Created `FeatureModalCta` — a plain `<button>` + `openModal()` client leaf, bypassing Button/CVA entirely
