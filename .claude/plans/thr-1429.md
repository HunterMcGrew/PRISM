# Plan: THR-1429

## Ticket

https://linear.app/tractru/issue/THR-1429/pull-new-mega-menu-into-the-new-purpose-built-headers

## Goal

Add a `menuType` toggle to the configurable header block so editors can switch between classic dropdown menus and the mega menu. When mega menu mode is active, the header synthesizes a `mega-menu-renderer` inner block and renders it in place of the classic `HeaderMenu`.

---

## User Stories

- As a dealer admin, I want to choose between a classic dropdown menu and the mega menu in my site's header, so that I can use whichever navigation style fits my site best
- As a dealer admin, I want the menu type toggle in the header's sidebar settings, so that I can switch without reconfiguring blocks on the page
- As a site visitor, I want the mega menu to work correctly within the header on both desktop and mobile, so that I can navigate the site regardless of which menu type the dealer chose

---

## Decisions

- `menuType: "classic" | "mega-menu"` attribute on the header block — determines which menu system to render. Default: `"classic"` (backwards compatible).
- Header resolver owns mega menu data fetching directly — imports shared logic from `mega-menu-renderer/resolver.props.ts` (`resolveChildren` for desktop blocks, `resolveProps` for mobile nav items). No need to register `mega-menu-renderer` as a block — the header IS the entry point. All child blocks (navigation items, featured links, etc.) are already registered.
- Header variants render `{children}` for mega menu mode — each variant conditionally hides `HeaderMenu` + `MobileMenu` and renders `{children}` (the resolved mega menu blocks) in the appropriate layout position. Mobile nav items passed as props for `MegaMenuMobileMenu`.
- Editor shows a placeholder bar in mega menu mode — the mega menu is authored in its own dedicated editor, so the header editor preview shows a styled placeholder ("Mega Menu — configure in Mega Menu Editor") rather than attempting to render actual mega menu content.
- THR-1449 is merged — mega-menu-renderer component and resolver files exist on main. Shared fetch logic in `resolver.props.ts` is importable. `mega-menu-renderer` itself is NOT registered in `block-registry.ts` — intentionally deferred until it's needed as a standalone page-level block.
- No classic menu fallback — when `menuType === "mega-menu"`, the classic `HeaderMenu` and `MobileMenu` do not render. If no mega menu content is configured, the header renders empty children (no content in the nav area).
- **Open question (discuss with Hunter):** The header resolver imports shared fetch logic from `blocks/mega-menu-renderer/resolver.props.ts` — another block's internals. `mega-menu-renderer` isn't registered in `block-registry.ts` yet. If/when the mega menu fetch logic is needed by multiple consumers, it should be promoted to a service or shared utility rather than living inside an unregistered block's folder. Current cross-block import works but is a coupling concern.

## Implementation Tasks

1. **Add `menuType` attribute to header schema** — In `frontend/blocks/header/schema.ts`, add `menuType: "classic" | "mega-menu"` to `BlockAttributes` with default `"classic"` in `DEFAULT_ATTRIBUTES`.

2. **Extend `HeaderProps`** — In `frontend/components/header/types.ts`, add `menuType?: "classic" | "mega-menu"`, `children?: ReactNode`, `megaMenuMobileNavItems?: MegaMenuMobileNavItem[]`. Import `ReactNode` and the mobile nav type from `mega-menu-renderer`.

3. **Add `resolveChildren` and update `resolveProps` in header resolver** — In `frontend/blocks/header/resolver.ts`:
   - Import shared logic from `mega-menu-renderer/resolver.props.ts`
   - `resolveChildren`: when `menuType === "mega-menu"`, call the shared `resolveChildren` to return the raw mega menu blocks. Otherwise return `[]`.
   - `resolveProps`: pass `menuType` through. When `menuType === "mega-menu"`, also resolve `megaMenuMobileNavItems` via the shared `resolveProps` logic.

4. **Update `HeaderBlock` to forward children** — In `frontend/blocks/header/HeaderBlock.tsx`, accept `children` from props and pass through to the variant component.

5. **Update all six header variants** — In each `HeaderVariantOne.tsx` through `HeaderVariantSix.tsx`:
   - Accept `children`, `menuType`, and `megaMenuMobileNavItems` from `HeaderProps`
   - When `menuType === "classic"` (or undefined): render as today
   - When `menuType === "mega-menu"`: hide desktop `HeaderMenu`, hide `MobileMenu`, render `{children}` for desktop mega menu content, render `MegaMenuMobileMenu` with `megaMenuMobileNavItems` for mobile
   - Keep changes minimal — just the conditional swap

6. **Add menu type toggle to header editor** — In `backend/.../blocks/header/edit.tsx`:
   - Add `SelectControl` for `menuType` in the Layout `PanelBody` (options: Classic, Mega Menu)
   - When mega menu selected, render a styled placeholder div in the preview where the menu bar would be
   - Optionally dim/hide classic menu-dependent controls when mega menu is active

7. **Storybook stories** — Add mega menu mode stories for at least two representative variants (e.g., VariantOne for kubota, VariantFour for john-deere). Use mock children to simulate mega menu content.

8. **Tests** — Add resolver test for `resolveChildren` returning mega menu blocks. Add variant rendering test confirming classic vs mega menu conditional rendering.

## Acceptance Criteria

### Behavioral
- [ ] Given the header block in the WordPress editor, When I open the Layout panel in the sidebar, Then I see a "Menu Type" dropdown with options "Classic" and "Mega Menu"
- [ ] Given "Classic" is selected (default), When I view the header on the frontend, Then the standard dropdown navigation menu appears as it does today
- [ ] Given "Mega Menu" is selected, When I view the header on the frontend, Then the mega menu navigation bar appears where the classic menu was, with full desktop pane navigation
- [ ] Given "Mega Menu" is selected, When I view the site on a mobile device and tap the hamburger menu, Then the mega menu mobile sliding panel opens (not the classic mobile menu)
- [ ] Given "Mega Menu" is selected but no mega menu content has been configured, When I view the header on the frontend, Then a "No Mega Menu Content" fallback message appears instead of an empty space
- [ ] Given "Mega Menu" is selected in the editor, When I view the editor preview, Then I see a placeholder bar indicating the mega menu will appear here
- [ ] Given any header variant (Styles 1-6), When I switch between Classic and Mega Menu, Then the header layout remains intact — only the navigation bar area changes

### Non-behavioral
- [ ] Menu type defaults to "Classic" — existing sites are unaffected without any editor action
- [ ] No new `"use client"` boundary introduced in the header resolver chain
- [ ] Mega menu keyboard navigation works correctly when embedded in the header (tab flow: header content → mega menu → page)
- [ ] Storybook stories exist for at least two header variants in mega menu mode

## Review Issues

### Desktop hamburger button is a no-op in mega-menu mode
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/header/variants/HeaderVariantTwo.tsx:81`, `HeaderVariantThree.tsx:71`, `HeaderVariantFive.tsx:139`, `HeaderVariantSix.tsx:107`
- **Problem:** In mega-menu mode, variants Two, Three, Five, and Six show a hamburger button on desktop that opens nothing — `MegaMenuMobileMenu` has `lg:hidden`, so the panel is invisible at that breakpoint. Variants One and Four are unaffected (their hamburger is already `lg:hidden`).
- **Suggested fix:** In the mega-menu branch for Two and Three, remove the desktop hamburger or make it `lg:hidden`. For Five and Six, keep the `xl:hidden` class in mega-menu mode since there's no inline menu to overflow.
- **Fixed in:** Removed desktop hamburger from mega-menu branch in Two and Three. Restored unconditional `xl:hidden` on hamburger in Five and Six.

### menuType not registered in WordPress block attributes
- **Severity:** `critical`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/header/index.ts:15-36`
- **Problem:** `menuType` is not declared in the `attributes` object passed to `registerBlockType()`. The editor can set the value via `setAttributes()` but it won't persist to the database — reverts to `"classic"` on reload.
- **Suggested fix:** Add `menuType: { type: "string", default: DEFAULT_ATTRIBUTES.menuType }` to the attributes object.
- **Fixed in:** Added `menuType: { type: "string", default: DEFAULT_ATTRIBUTES.menuType }` to `registerBlockType()` attributes in `index.ts`.

### Inverted import direction — types.ts imports from schema.ts
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/header/types.ts:3`
- **Problem:** `types.ts` imports `MenuType` from `schema.ts`, creating a component → block dependency (wrong direction). No runtime issue due to `import type` erasure.
- **Suggested fix:** Move `MenuType` definition to `types.ts`; `schema.ts` re-imports it.
- **Fixed in:** Moved `MenuType` to `types.ts`. `schema.ts` imports it (no re-export). `edit.tsx` imports `MenuType` directly from `types.ts`. One-way dependency: `schema.ts` → `types.ts`.

### Missing import type usage
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** Various (schema.ts, HeaderBlock.tsx, Header.stories.tsx, all variants)
- **Problem:** Type-only imports use value import syntax instead of `import type`. Not enforced by current ESLint config.
- **Suggested fix:** Use `import type` for imports used only as type annotations.
- **Fixed in:** Updated all type-only imports to use `import type` across schema.ts, types.ts, HeaderBlock.tsx, Header.stories.tsx, and all six variant files.
