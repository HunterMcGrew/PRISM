# Plan: thr-1787

## Ticket

## Goal

Fix the "Open in new tab" toggle in the info-banner and featured-banner mega menu blocks so editors can disable it once enabled. The bug is in the parent block's `handleLinkChange`, not in `LinkControl` itself.

## Implementation Tasks

To be added by Winston.

## Decisions

- Bug is in the parent block's `handleLinkChange`, not in `LinkControl` or the link helpers. `LinkControl.tsx` and `link-utilities.ts` are correct and must not be modified. Reason: `setLinkTarget(link, undefined)` is documented to delete keys (THR-1443), and `LinkControl` correctly forwards the new link.
- Affected blocks share the spread-merge anti-pattern: `setAttributes({ link: { ...attributes.link, ...normalized } })`. The fix is a direct write — `setAttributes({ link: ensureLinkObject(updatedLink) })` — matching the pattern in `mega-menu-navigation-item` and `mega-menu-link-with-icon`. `LinkControl` already preserves untouched fields internally via `setLinkX` helpers, so the parent merge is unnecessary and harmful.

## Debugged Issues

### "Open in new tab" toggle does nothing on info-banner / featured-banner blocks

- **Status:** `fixed`
- **Fixed in:** `mega-menu-info-banner/edit.tsx:43`, `mega-menu-featured-banner/edit.tsx:65` — replaced spread-merge with `setAttributes({ link: ensureLinkObject(updatedLink) })`
- **Severity:** High
- **Environment:** WordPress block editor (wp-admin) — affects `mega-menu-info-banner` and `mega-menu-featured-banner` blocks on every dealer site
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-info-banner/edit.tsx:43-51` and `backend/plugins/gravity-platform-core/src/blocks/mega-menu-featured-banner/edit.tsx:65-73`
- **Root cause:** `verified` — the `handleLinkChange` handlers in info-banner and featured-banner spread the previous `attributes.link` over the new link object. `setLinkTarget(link, undefined)` correctly *deletes* the `target` and `rel` keys from the new object (`frontend/lib/utilities/link-utilities.ts:22-25`), but spread merge does not override absent keys, so the old `target: "_blank"` and `rel: "noopener noreferrer"` from `attributes.link` survive the merge. The resurrected attributes get persisted, the toggle never sees its `checked` value flip, and the panel remains rendered. From the editor's perspective the click does nothing.
- **Steps to Reproduce:**
  1. In the WordPress block editor, insert a `mega-menu-info-banner` block (or `mega-menu-featured-banner`).
  2. Open the block's sidebar settings and enter a URL.
  3. Toggle "Open in new tab" on (or paste an external URL so it auto-toggles on).
  4. Observe the advanced-settings panel expanded with the toggle on.
  5. Click the toggle to turn it off.
  6. Toggle visibly stays in the on position; panel stays rendered; nothing changes on the link.
- **Expected behavior:** Clicking the toggle off clears `target` and `rel` on the saved `attributes.link`, and the toggle visibly transitions to the off state.
- **Actual behavior:** Toggle stays on; panel stays open; the saved `attributes.link.target` remains `"_blank"`. The frontend continues to render `target="_blank"` after save.
- **Recommended fix:** Replace the spread-merge in both handlers with a direct write that matches the pattern used in `mega-menu-navigation-item/edit.tsx`:
    ```tsx
    const handleLinkChange = (updatedLink: LinkValue) => {
        setAttributes({ link: ensureLinkObject(updatedLink) });
    };
    ```
    `LinkControl` always passes the full link object through the `setLinkX` helpers, which spread the prior `link` internally and preserve untouched fields. The parent merge is unnecessary, and is what resurrects the deleted `target`/`rel` keys.
- **Suggested tests:**
  - Storybook interaction test on `mega-menu-info-banner`: render with `link: { href, target: "_blank", rel: "noopener noreferrer" }`, fire the toggle-off interaction, assert the persisted `attributes.link` no longer has `target` or `rel`.
  - Same test on `mega-menu-featured-banner`.
  - Optional Jest unit test: render either block's `edit.tsx`, simulate `LinkControl` calling `onChange` with a link object that has neither `target` nor `rel`, assert `setAttributes` was called with a `link` value that has neither key.
  - Regression check: same test on `mega-menu-navigation-item` and `mega-menu-link-with-icon` (which use the correct full-replace pattern) to lock in the desired behavior.
- **Linear:** `synced` — ticket description updated by Sasha with verified root cause and fix

### Sortable links modal

`mega-menu-featured-links` and `mega-menu-navigation-list` use `SortableLinksModal`, which renders `LinkControl` via `link-modal-parts.tsx`. The modal's `handleUpdateLink` does a spread merge, but `link-modal-parts.handleLinkChange` destructures `target: normalizedLink.target` explicitly — so the spread overwrites the prior value with `undefined` (key present, value undefined) rather than preserving the old `"_blank"`. Toggle-off works correctly there.

A separate, lower-impact UX glitch may exist in the modal: the LinkControl panel could auto-collapse on toggle-off because `expandedByUser` isn't promoted on toggle interaction (the original disappearing-panel hypothesis from the first investigation). In modal context this is mild — user is in explicit Done/Cancel mode — but it's a real footgun. Out of scope for THR-1787; worth a separate ticket if anyone reports it.
