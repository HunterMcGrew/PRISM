# Plan: THR-1373

## Ticket

https://linear.app/tractru/issue/THR-1373/enable-support-for-wp-69-stretchy-text-in-heading-and-paragraph-blocks

## Goal

Enable WordPress 6.9 stretchy text (fitText) support for Heading and Paragraph blocks so text dynamically scales to fill its container width.

---

## Decisions

- `FitText` lives in `frontend/components/utility/FitText.tsx` as a `"use client"` component — keeps HeadingBlock/ParagraphBlock as RSCs
- When `fitText` is true, normal font-size classes/styles are suppressed — FitText owns font sizing exclusively
- WP attribute is `fitText`, not `fontSizeFitText` — confirmed via console.log of block props at runtime
- Uses binary search (5px–600px, max ~7 iterations) + ResizeObserver to match WordPress core's algorithm
- FitText fits to the block's container width (respects `container mx-auto` max-width), not the viewport
- Backend change is removal-only: delete the `unregisterBlockVariation` calls, let WP core handle the editor
- No theme.json changes needed — fitText operates at block attribute level, not theme configuration
- `FitText` renders the semantic element directly via `as` prop (`h1`-`h6`, `p`) — no extra wrapper div
- `has-fit-text` is Gutenberg's canonical CSS class for fitText blocks (editor and save output) — use it to target editor CSS overrides
- Editor overlap fix is CSS-only in `main.scss` — reset `line-height: normal` on `.has-fit-text` headings and paragraphs so the block wrapper grows with the Gutenberg-applied font-size
- Do not update bundled WP core `block.json` — Gutenberg plugin registers fitText support at runtime via `blocks.registerBlockType` filter

## Debugged Issues

### FitText clips text on page-level blocks (not in columns)
- **Status:** `fixed`
- **File:** `frontend/components/utility/FitText.tsx:51`
- **Root cause:** Binary search measures against `parentElement.clientWidth` (full viewport width), but the element itself is constrained by `container mx-auto` (Tailwind max-width ~1280px). Font is sized for the parent, then clipped by `overflow: hidden` on the narrower element.
- **Recommended fix:** Use `element.clientWidth` instead of `parentElement.clientWidth` for the container width measurement. The element's own `clientWidth` already reflects `max-width` constraints from `container mx-auto`. Remove the `parentElement` preference entirely.
- **Suggested tests:** Add test verifying FitText measures against its own width, not its parent's. Add visual regression test with `container mx-auto` class applied.

### Editor: stretchy heading/paragraph height collapses, overlapping blocks below
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/src/main.scss:5-9` (heading-style mixin) and `:92-110` (editor heading styles)
- **Root cause:** Our `heading-style` mixin sets fixed `line-height` values (e.g. `2.5rem` for h2) on `.wp-block-heading` elements. Gutenberg's fitText hook overrides `font-size` with `!important` via an injected `<style>` tag, but our fixed `line-height` doesn't scale with it. A 200px font with 40px line-height means the block wrapper stays 40px tall while text overflows. Same issue affects `.wp-block-paragraph` in the editor. Gutenberg adds the class `has-fit-text` to fitText blocks.
- **Recommended fix:** Add CSS overrides in `main.scss` near line 111: `.wp-block-heading.has-fit-text { line-height: normal; }` and `.wp-block-paragraph.has-fit-text { line-height: normal; }`. This lets the browser compute line-height from the Gutenberg-applied font-size so the block wrapper grows naturally. Do NOT update bundled block.json — Gutenberg plugin handles fitText registration at runtime.
- **Suggested tests:** Manual editor test — insert a stretchy heading and stretchy paragraph each followed by another block, confirm no overlap and correct vertical spacing.

## Review Issues

### Missing `id` prop on FitText paths
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/components/utility/FitText.tsx:16`, `frontend/blocks/heading/HeadingBlock.tsx:86`, `frontend/blocks/paragraph/ParagraphBlock.tsx:72`
- **Problem:** FitText doesn't accept or pass through `id`, breaking anchor links for stretchy headings/paragraphs
- **Suggested fix:** Add `id?: string` to `FitTextProps`, pass through in `createElement`. Pass `id` from both block components into `<FitText>`.
- **Fixed in:** Added `id` to FitTextProps and createElement call; passed `id` from HeadingBlock and ParagraphBlock FitText paths; added test coverage for id passthrough in all three test files.

### Test type errors fail the build
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/blocks/heading/__tests__/HeadingBlock.test.tsx`, `frontend/blocks/paragraph/__tests__/ParagraphBlock.test.tsx`, `frontend/components/utility/__tests__/FitText.test.tsx:64`

- **Fixed in:** Added `blockIndex` and `blockName` to HeadingBlock defaultProps; typed mock `as` as `keyof JSX.IntrinsicElements` in both block mocks; added `id` to mock passthrough; added null guard on ResizeObserver mock access.

### `as` prop type overly broad
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/utility/FitText.tsx:13`
