# Plan: SortableLinksModal trapped in InspectorControls

## Ticket

https://linear.app/tractru/issue/THR-1558/mega-menu-editor-manage-links-toolbar-button-does-nothing-when-sidebar

## Goal

Fix the "Manage Links" toolbar button in mega menu navigation-list and featured-links blocks so the SortableLinksModal opens regardless of sidebar state.

---

## User Stories

Not applicable — this is a bug fix.

---

## Decisions

- `SortableLinksModal` must render outside `<InspectorControls>` — the WordPress `Modal` component portals to `document.body`, so it doesn't need to be inside the sidebar slot chain. Placing it inside `InspectorControls` prevents it from mounting when the sidebar is closed.
- The sidebar "Manage Links" `Button` stays inside `<InspectorControls>` — only the modal rendering moves out.
- Both `mega-menu-navigation-list` and `mega-menu-featured-links` have the same bug and must be fixed together.

## Debugged Issues

### SortableLinksModal trapped inside InspectorControls — toolbar "Manage Links" does nothing
- **Status:** `fixed`
- **Fixed in:** `hmcgrew/thr-1558-mega-menu-editor-manage-links-toolbar-button-does-nothing`
- **Severity:** High
- **Environment:** WordPress admin — Mega Menu Editor custom app (`/wp-admin/admin.php?page=mega-menu-editor`)
- **Files:**
  - `backend/plugins/gravity-platform-core/src/blocks/mega-menu-navigation-list/edit.tsx:126-139` — modal inside `InspectorControls`
  - `backend/plugins/gravity-platform-core/src/blocks/mega-menu-featured-links/edit.tsx:101-120` — same pattern
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/editor.tsx:763` — sidebar conditionally unmounted
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/components/sidebar/Sidebar.tsx:6-8,24` — custom `InspectorSlot` with `bubblesVirtually`
- **Root cause:** `SortableLinksModal` is rendered inside `<InspectorControls>` in both blocks. In the mega menu editor's custom layout, `InspectorControls` is a Fill whose Slot (`InspectorSlot`) lives inside `<Sidebar />`. The sidebar is conditionally rendered — when `isSettingsOpen` is `false` (editor.tsx:763), the entire sidebar div and its Slot are unmounted. With no Slot mounted, the Fill's children never render to the DOM. The toolbar `BlockControls` button correctly sets `isOpen = true`, but the `Modal` component (which would portal to `document.body`) never mounts because its parent in the Fill chain doesn't render. When the user then clicks the sidebar settings toggle, the Slot mounts, the Fill renders, sees `isOpen` is already `true`, and the modal appears — explaining why the sidebar toggle "opens both."
- **Steps to Reproduce:**
  1. Navigate to the Mega Menu Editor admin page
  2. Add a Mega Menu Navigation List (or Featured Links) block
  3. With the sidebar closed, click "Manage Links" in the block toolbar
  4. Observe nothing happens — no modal appears
  5. Click the Settings gear icon in the header to open the sidebar
  6. Observe both the sidebar AND the SortableLinksModal appear simultaneously
- **Expected behavior:** Clicking "Manage Links" in the toolbar opens the SortableLinksModal regardless of sidebar state
- **Actual behavior:** "Manage Links" sets state silently. Modal only appears after sidebar is manually opened.
- **Recommended fix:** Move `{isOpen && <SortableLinksModal ... />}` outside of `<InspectorControls>` in both blocks — place it as a sibling after `<InspectorControls>` inside the root `<div {...blockProps}>`. The `Modal` component portals to `document.body` anyway, so it doesn't need to live inside the sidebar slot chain. Apply to both `mega-menu-navigation-list/edit.tsx` and `mega-menu-featured-links/edit.tsx`.
- **Suggested tests:** With sidebar closed, click toolbar "Manage Links" — modal should appear. With sidebar open, click sidebar "Manage Links" button — modal should appear. Close modal, reopen from toolbar — should work. Verify modal still closes correctly via X button and Escape key.
