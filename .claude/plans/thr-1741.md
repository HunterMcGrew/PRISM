# Plan: THR-1741

## Ticket

https://linear.app/tractru/issue/THR-1741/card-block-clickable-inside-mega-menu-editor-causes-data-loss-on

## Goal

Stop card blocks from being clickable inside the mega menu editor canvas — clicks on the image link or CTA must select the block for editing, not navigate. Fix must not affect card behavior on the frontend or in other editor contexts.

---

## Implementation Tasks

### Sasha (debug)

1. Diagnose why the existing `pointer-events: none` rule on `.wp-block-gravity-platform-core-card a, button` (added in THR-1606) takes effect in the regular gutenberg editor but not in the mega menu editor.

### Clove (implementation)

1. Apply the recommended fix from `## Debugged Issues` after Sasha's diagnosis is reviewed.

---

## Decisions

- The card block uses Gutenberg `apiVersion: 2`, so the mega menu editor canvas does NOT iframe. CSS scoping issues are about specificity and load order, not iframe boundaries.
- Per-block `editor.scss` rules are bundled into `gravity-platform-blocks` (`build/main.css`) via the `main` webpack entry's transitive imports. They reach the mega menu editor canvas when loaded — the bug is overriding, not non-loading.

## Debugged Issues

### Card pointer-events rule overridden in mega menu editor

- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/includes/Admin/MegaMenuEditor.php` — `enqueueEditorAssets()`. Moved `wp_enqueue_style('gravity-platform-blocks')` and `wp_enqueue_style('gravity-platform-blocks-theme')` to enqueue AFTER `thrive-mega-menu-styles`, making per-block content styles the last stylesheets loaded so they win equal-specificity ties.
- **Severity:** High
- **Environment:** WordPress admin — Mega Menu editor canvas. Affects all dealer sites using the mega menu with `gravity-platform-core/card` blocks inside it.
- **File:** `backend/plugins/gravity-platform-core/src/blocks/card/editor.scss:22-27`
- **Root cause:** The card's `editor.scss` rule selector `.wp-block-gravity-platform-core-card a` has CSS specificity **(0,1,1)**. The mega menu editor app's `apps/mega-menu-editor/styles.scss:49-51` calls `@include wp-admin-reset(".block-editor")`, which emits WP admin reset rules scoped to `.block-editor` — the mega menu editor's wrapper has that class (`MegaMenuEditor.php:333`). The reset output includes anchor/button rules at specificity **(0,1,1)** which tie with the card rule. Stylesheet load order in `MegaMenuEditor.php` enqueues `gravity-platform-blocks` (main.css, line 258) BEFORE `thrive-mega-menu-styles` (adminMain.css, line 263) — so on equal-specificity ties, the reset wins and `pointer-events: none` is reset to default. The featured-banner rule (`.editor-styles-wrapper .wp-block-...-mega-menu-featured-banner a`, specificity (0,2,1)) survives because higher specificity wins outright. The regular gutenberg editor isn't affected because (a) it doesn't load `apps/mega-menu-editor/styles.scss` at all, and (b) `main.scss:81` (`.edit-post-visual-editor a { pointer-events: none }`) provides a redundant catch-all that masks the per-block rule's failure mode.
- **Steps to Reproduce:**
  1. Enable mega menu (`platform_settings_enable_mega_menu`).
  2. Open Mega Menu in WP admin.
  3. Add a card block inside a mega menu column with a `ctaLink`.
  4. Make any unsaved attribute change (e.g. edit the headline).
  5. Click on the card image or CTA button in the editor canvas.
  6. Browser navigates to `ctaLink`, abandoning the unsaved draft.
- **Expected behavior:** Click on the card selects the block for editing; no navigation; draft preserved.
- **Actual behavior:** Click on the card follows the link, navigating away from the editor and discarding the unsaved draft.
- **Recommended fix — enqueue-order swap (single change, fixes all affected blocks):**
  - In `backend/plugins/gravity-platform-core/includes/Admin/MegaMenuEditor.php::enqueueEditorAssets()`, move `wp_enqueue_style('gravity-platform-blocks')` and `wp_enqueue_style('gravity-platform-blocks-theme')` (currently lines 258-259) to AFTER the `thrive-mega-menu-styles` enqueue (currently lines 262-267).
  - Resulting load order: WP core editor styles → mega menu shell (adminMain.css with `wp-admin-reset`) → per-block content styles (main.css). Per-block rules now win equal-specificity ties, which matches developer intent — block content styles describe block content and should beat shell resets.
  - Why this is preferred over the per-rule specificity-bump alternative: a single PHP change fixes the card AND every other per-block `editor.scss` rule at once, with no per-rule audit follow-up and no convention for future block authors to remember.
  - Regression risk: low. `apps/mega-menu-editor/styles.scss` and its imports target shell selectors (`.thrive-mega-menu-block-editor`, `.interface-interface-skeleton`, `.editor-styles-wrapper` chrome rules) — none collide with `wp-block-*` selectors. The `wp-admin-reset(".block-editor")` is the exact rule whose unintended bleed into block content is the bug; flipping order is what corrects that.
- **Alternate fix (rejected):** prefix each affected per-block `editor.scss` rule with `.editor-styles-wrapper` to bump specificity from (0,1,1) → (0,2,1). Works but requires touching ~8 files (card plus all the others currently masked by `main.scss:81`'s `.edit-post-visual-editor a` catch-all in the regular editor) and creates a convention every future block author has to remember.
- **Suggested tests:** Manual verification — open the mega menu editor in DevTools after a build, click on a card's `<a>`, check the Computed pane for `pointer-events`. Should be `none` after the fix. Confirm the regular post editor still works (no regression). No automated test — this is a CSS-only rule and the editor canvas isn't covered by Jest/Pest.

## Acceptance Criteria

### Behavioral

- [ ] The card block remains visually identical inside the mega menu editor — only click/navigation behavior changes
- [ ] Editors can still select and edit the card block via the block toolbar / block list view inside the mega menu
