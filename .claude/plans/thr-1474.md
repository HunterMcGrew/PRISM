# Plan: THR-1474 (Epic)

## Ticket

https://linear.app/tractru/issue/THR-1474/useeffect-fixes-anti-patterns-and-async-safety

## Goal

Eliminate useEffect anti-patterns and add async safety across frontend and backend editor code to prevent race conditions, stale closures, and visual flash.

---

## Decisions

- Each story is independently shippable — no cross-story dependencies
- Priority order: production risk first, visual polish last
- Backend editor async safety (THR-1479) ships first — highest risk, lowest effort
- SlidingPanel uses `openPanel` prop directly with Headless UI `Dialog` — no local state shadow
- Both SlidingPanel consumers (`MobileMenu`, `MobileMegaMenu`) control open state via `AppContext.modal` — confirmed clean controlled pattern, no consumer relies on SlidingPanel managing its own state
- SlidingPanel `closePanel()` simplifies to just `onClose()` — parent handles state change, Dialog reacts to prop
- SearchBox debounce cleanup is a separate effect from any input value management
- PillsMenu moves `refine()` into the click handler — no effect needed
- PillsMenu guard uses `props.attribute` dynamically instead of hardcoded `"searchType"` — makes the component actually reusable
- NavigationMenuItem is the only useLayoutEffect upgrade — it's the textbook case (getBoundingClientRect → state → render)
- ButtonGroup excluded from useLayoutEffect upgrade — effect manipulates DOM classes directly (no React state), and the 100ms setTimeout conflicts with synchronous useLayoutEffect
- Story 7 (isMounted → cancelled migration) deferred — `careers/edit.tsx` already uses a local `isMounted` variable scoped to the effect closure, which is functionally identical to the `cancelled` flag; verify `locations` and `employees` match before deciding whether to touch them
- Editor data-fetching effects use `cancelled` flag pattern, not `isMounted` ref
- carousel-item warning is a derived value computed during render, not in an effect
- `getImageWarning` defined outside the component — pure function with no component scope dependency
- Plain `const` during render for image warning, no `useMemo` — trivial computation
- Warning strings preserved exactly as-is including pre-existing typos — copy fixes are a separate concern

### THR-1479: Editor data-fetching — add cancellation flags

**Priority:** 1 (highest production risk, lowest effort)
**Risk:** Very low — adds safety with no behavior change

**Problem:**
8 editor `edit.tsx` files fetch data in `useEffect` with no cancellation. If the block is removed or attributes change mid-fetch, `setState` fires on an unmounted or stale component. This causes React warnings in the console and is a potential memory leak.

**Files to modify (Story 6 — missing cancellation):**
- `backend/plugins/gravity-platform-core/src/blocks/offers/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/events-list/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/footer-menu/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/offers-carousel/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/locations-map/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/site-footer/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/feature/edit.tsx`
- `backend/plugins/gravity-platform-core/src/blocks/regional-offers-carousel/edit.tsx`

**What the current code looks like (example from offers/edit.tsx):**
```tsx
useEffect(() => {
  const fetchAndSetOffers = async () => {
    if (!siteGlobals || !spcApiUrl || !spcApiKey) return;
    const offers = await ServiceFactory.createOfferService(spcApiUrl, spcApiKey, siteGlobals).getItems();
    setPreviewProps((prev) => ({ ...prev, offers }));
  };
  fetchAndSetOffers();
}, [siteGlobals, spcApiUrl, spcApiKey]);
```

**What the fix looks like:**
```tsx
useEffect(() => {
  let cancelled = false;
  const fetchAndSetOffers = async () => {
    if (!siteGlobals || !spcApiUrl || !spcApiKey) return;
    const offers = await ServiceFactory.createOfferService(spcApiUrl, spcApiKey, siteGlobals).getItems();
    if (!cancelled) setPreviewProps((prev) => ({ ...prev, offers }));
  };
  fetchAndSetOffers();
  return () => { cancelled = true; };
}, [siteGlobals, spcApiUrl, spcApiKey]);
```

Apply this pattern to every async `useEffect` in the 8 files above. Wrap every `setState` call after an `await` in an `if (!cancelled)` guard. Add a cleanup function that sets `cancelled = true`.

**Story 7 — isMounted migration (deferred):**
`careers/edit.tsx`, `locations/edit.tsx`, and `employees/edit.tsx` already use a local `isMounted` variable with cleanup. This is functionally identical to the `cancelled` flag. Before touching these, verify all three use the same local-variable pattern (not a `useRef`). If they do, leave them alone — renaming is cosmetic. If any use `useRef(true)`, migrate that specific file to the local-variable pattern.

**Test:** Rapidly add/remove blocks in the editor while fetch requests are in flight. Confirm no console warnings about state updates on unmounted components.

### THR-1475: SlidingPanel — remove prop-to-state sync

**Priority:** 2
**Risk:** Low

**Problem:**
`frontend/components/utility/SlidingPanel.tsx` syncs the `openPanel` prop to local `isOpen` state via `useEffect`. This creates two sources of truth — the prop and local state — which can desync. The parent already controls open/close via `openPanel` and the component already calls `onClose()` on dismiss.

**File:** `frontend/components/utility/SlidingPanel.tsx`

**What the current code looks like:**
```tsx
const [isOpen, setIsOpen] = useState(openPanel);

useEffect(() => {
  setIsOpen(openPanel);
}, [openPanel]);

const closePanel = () => {
  setIsOpen(false);
  onClose();
};
```
The `isOpen` state is then passed to `<Dialog open={isOpen}>`.

**What the fix looks like:**
1. Remove the `isOpen` state and the sync effect entirely
2. Pass `openPanel` directly to `<Dialog open={openPanel}>`
3. In the close handler, just call `onClose()` — the parent will set `openPanel` to `false`
4. Headless UI `Dialog` handles enter/leave animation via its `transition` prop, so no local state is needed for animation

**Important:** Do NOT use a key-reset pattern here — `Dialog` with `transition` manages animation state internally, and remounting would break transitions.

**Before submitting:** Search for all consumers of `SlidingPanel` and verify they all pass a controlled `openPanel` prop and handle `onClose`. Run: `grep -r "SlidingPanel" frontend/` to find all usage sites.

**Test:** Open and close a sliding panel. Confirm animation still works on open and close. Confirm the parent can control the open state externally. Confirm that closing via the overlay/X button dismisses the panel.

**Implementation Tasks:**
1. Remove `useState(openPanel)` and the `useEffect` that syncs `openPanel` → `isOpen`
2. Replace `<Dialog open={isOpen}>` with `<Dialog open={openPanel}>`
3. Simplify `closePanel()` — remove `setIsOpen(false)`, keep only `onClose()` call
4. Clean up imports — remove `useState` and `useEffect` from the React import (component uses neither after the fix)
5. Add unit tests for `SlidingPanel`:
   - Renders dialog when `openPanel` is `true`
   - Does not render dialog content when `openPanel` is `false`/`undefined`
   - Calls `onClose` when close button is clicked
   - Calls `onBackArrowClick` when back arrow is clicked (with `showBackArrow` enabled)

**Consumer audit (completed by Winston):**
- `MobileMenu.tsx` — passes `openPanel={modal.open && modal.type === "menu"}`, `onClose={handleOnClose}` which calls `closeModal()`. Clean.
- `MobileMegaMenu.tsx` — same pattern. Clean.
- No consumer relies on SlidingPanel managing its own open state.

### THR-1477: PillsMenu — move stale-state guard to event handler

**Priority:** 3
**Risk:** Low

**Problem:**
`frontend/components/search/PillsMenu.tsx` uses a `useEffect` to call `refine()` when local `searchType` state changes, with a guard comparing against `indexUiState.menu?.searchType`. This is event logic disguised as an effect — the user clicks a pill, which sets state, which triggers the effect, which calls `refine()`. The logic belongs in the click handler.

**File:** `frontend/components/search/PillsMenu.tsx`

**What the current code looks like:**
```tsx
const [searchType, setSearchType] = useState<string>(initialSearchType);

useEffect(() => {
  if (searchType && searchType !== indexUiState.menu?.searchType) {
    refine(searchType);
  }
}, [indexUiState, searchType, refine]);

// In the JSX:
onClick={() => setSearchType(item.value)}
```

**What the fix looks like:**
1. Remove the `searchType` state entirely
2. Remove the `useEffect` entirely
3. Move the `refine()` call into the `onClick` handler, reading `indexUiState` at call time:
```tsx
const handlePillClick = (value: string) => {
  if (value !== indexUiState.menu?.searchType) {
    refine(value);
  }
};

// In the JSX:
onClick={() => handlePillClick(item.value)}
```

The guard check (`value !== indexUiState.menu?.searchType`) prevents toggling a selected pill off, which is the original intent of the comparison in the effect.

**Important:** Check whether `searchType` state was also used to determine the visually "active" pill (e.g. for styling/className). If so, the active pill styling must now come from `indexUiState.menu?.searchType` instead of the removed local state. Search for references to `searchType` in the JSX to confirm.

**Test:** Click pills in the search UI. Confirm the correct Algolia filter is applied. Click the same pill twice — it should NOT toggle off (the guard prevents this). Rapid-click between different pills — each should apply its filter without race conditions or stale state.

**Decisions (THR-1477):**
- Guard uses `props.attribute` dynamically instead of hardcoded `"searchType"` — makes the component reusable without behavioral risk
- Active pill styling confirmed from `item.isRefined` (Algolia hook state), not local state — no styling changes needed
- Props type (`MenuProps`) not renamed in this ticket — separate concern
- No consumers found importing PillsMenu — Clove should verify via lazy/dynamic import search before writing tests; if dead code, recommend deletion as a follow-up

**Implementation Tasks (THR-1477):**
1. **Verify PillsMenu is not dead code** — search for lazy imports, dynamic imports, barrel exports, or string references to `PillsMenu` across the full codebase. If truly unused, flag to user before proceeding — deletion may be the right call instead of refactoring.
2. **Remove state and effect** — delete `searchType` state (`useState`), the `useEffect` block, and clean up unused imports (`useState`, `useEffect` from React import)
3. **Add click handler** — create `handlePillClick(value: string)` that reads `indexUiState.menu?.[props.attribute]` at call time and calls `refine(value)` when the value differs
4. **Update JSX** — change `onClick` from `() => setSearchType(item.value)` to `() => handlePillClick(item.value)`
5. **Add unit tests** — create `frontend/components/search/__tests__/PillsMenu.test.tsx`:
   - Pill click calls `refine` with the correct value
   - Clicking a pill that matches current `indexUiState` does NOT call `refine`
   - Clicking a different pill calls `refine` with the new value

### THR-1476: SearchBox — untangle state sync from debounce cleanup

**Priority:** 4
**Risk:** Medium — search behavior is user-facing, test thoroughly

**Problem:**
`frontend/components/search/SearchBox.tsx` has a single `useEffect` that mixes two concerns: (1) syncing `query` from `useSearchBox()` to local `inputValue` state, and (2) cancelling the debounced refine on cleanup. The sync is the anti-pattern; the cleanup is legitimate.

**File:** `frontend/components/search/SearchBox.tsx`

**What the current code looks like:**
```tsx
const { query, refine } = useSearchBox();
const [inputValue, setInputValue] = useState(query);
const debouncedRefine = useMemo(() => debounce(refine, 300), [refine]);

useEffect(() => {
  setInputValue(query);
  return () => debouncedRefine.cancel();
}, [debouncedRefine, query]);
```

**What the fix looks like:**
1. Separate the debounce cleanup into its own effect:
   ```tsx
   useEffect(() => () => debouncedRefine.cancel(), [debouncedRefine]);
   ```
2. Keep local `inputValue` state — it IS needed because the user types ahead of the debounced query (the input should reflect what the user typed, not the last confirmed query). The component also uses the `inputValue !== query` comparison to show a pending spinner.
3. **Remove the sync effect entirely.** The input only needs to reset on explicit user actions (clear button), not on every external `query` change. Handle resets directly in the clear handler:
   ```tsx
   const handleClear = () => {
     setInputValue("");
     refine("");
   };
   ```
4. **Fallback only if needed:** If testing reveals that `query` can change from an external source other than clear (e.g., URL-driven search) and the input must reflect it, use `query` as the `key` on the input: `<input key={query} defaultValue={query} />`. But start without this — it's likely unnecessary.

**Important context:** The `debouncedRefine` dependency in the current effect means that if `refine`'s identity changes (which Algolia can cause), the effect fires and resets `inputValue` mid-typing. The fix must not introduce this regression.

**Test:**
- Type in the search box — confirm debounce fires after pause
- Type and immediately navigate away — confirm no stale search fires
- Click clear — confirm input resets to empty and results clear
- Type, wait for results, type again — confirm no mid-typing input resets
- Refresh the page with a query in the URL (if applicable) — confirm input shows the correct query

**Decisions (THR-1476):**
- SearchBox keeps local `inputValue` state — needed for type-ahead display and `inputValue !== query` spinner comparison
- Debounce cleanup is a standalone effect with `[debouncedRefine]` dependency — no sync concern mixed in
- `onClear` uses Algolia's `clear()` (not `refine("")`) to reset full search state, plus `setInputValue("")` for local input — `clear()` resets the entire search uiState, `refine("")` only sets the query
- External query sync removed — current consumers (DesktopFilters, RefinementsBar, EquipmentListBlock) don't have external query sources. If a future consumer needs it, add key-reset pattern at that time
- Rename `onKeyUp` → `handleInputChange` — the handler is an `onChange`, not a `keyup`; the current name is misleading
- Props type rename (`Props` → `SearchBoxCustomProps` or similar) deferred — separate concern, don't mix with useEffect fix

**Implementation Tasks (THR-1476):**
1. **Delete the mixed-concern effect** — remove the entire `useEffect(() => { setInputValue(query); return () => debouncedRefine.cancel(); }, [debouncedRefine, query])` block
2. **Add standalone debounce cleanup effect** — `useEffect(() => () => debouncedRefine.cancel(), [debouncedRefine])` — this cancels pending debounced calls when `refine` identity changes or on unmount
3. **Verify `onClear` handler** — confirm it already calls `setInputValue("")` and `clear()`. The existing code does this correctly; no change needed
4. **Rename `onKeyUp` to `handleInputChange`** — the function handles `onChange`, not `onKeyUp`. Update the function name and the JSX reference
5. **Write tests** — no tests exist for SearchBox. Add tests covering:
   - Typing triggers debounced refine (mock `useSearchBox`, assert `refine` called after debounce)
   - Clear resets input value and calls `clear()`
   - Debounce is cancelled on unmount (verify `.cancel()` called in cleanup)
   - Input value reflects what the user typed, not the Algolia `query` (type "abc", assert input shows "abc" even before debounce fires)
   - Spinner shows when `inputValue !== query`, clear button shows when `inputValue === query`

### THR-1478: NavigationMenuItem — switch to useLayoutEffect

**Priority:** 5
**Risk:** Very low

**Problem:**
`frontend/components/navigation-menu-item/NavigationMenuItem.tsx` uses `useEffect` to measure button position via `getBoundingClientRect()` and set portal position via `setButtonBottom()`. Because `useEffect` runs after paint, there's a visible flash where the portal renders at `top: 0px` before correcting to the measured position.

**File:** `frontend/components/navigation-menu-item/NavigationMenuItem.tsx`

**What the current code looks like:**
```tsx
useEffect(() => {
  if (buttonRef.current) {
    const rect = buttonRef.current.getBoundingClientRect();
    setButtonBottom(rect.bottom + window.scrollY);
  }
}, [buttonRef, setButtonBottom]);
```
The `buttonBottom` state gates portal creation (`buttonBottom > 0`) and positions it (`style={{ top: `${buttonBottom}px` }}`).

**What the fix looks like:**
Change `useEffect` to `useLayoutEffect`. No other changes needed. Import `useLayoutEffect` from React if not already imported.

```tsx
useLayoutEffect(() => {
  if (buttonRef.current) {
    const rect = buttonRef.current.getBoundingClientRect();
    setButtonBottom(rect.bottom + window.scrollY);
  }
}, [buttonRef, setButtonBottom]);
```

**Note on ButtonGroup:** The original plan included ButtonGroup in this ticket, but Winston's review found that ButtonGroup's effect manipulates DOM classes directly (no React state update) and uses an intentional 100ms setTimeout for hydration. `useLayoutEffect` is not the right fix there. **Only modify NavigationMenuItem in this ticket.**

**Test:** Open a navigation menu dropdown. Confirm the dropdown positions correctly below the trigger button without a visible flash or jump.

### THR-1480: carousel-item — move derived value out of effect

**Priority:** 6
**Risk:** Very low

**Problem:**
`backend/plugins/gravity-platform-core/src/blocks/carousel-item/edit.tsx` computes an `imageWarning` string in a `useEffect` based on image dimensions and `parentAttrs.autoHeight`. This is a derived value — it can be computed during render without state or effects.

**File:** `backend/plugins/gravity-platform-core/src/blocks/carousel-item/edit.tsx`

**What the current code looks like:**
```tsx
const [imageWarning, setImageWarning] = useState("");

useEffect(() => {
  if (!attributes.image) {
    setImageWarning("");
    return;
  }
  updateImageWarning(attributes.image);
}, [attributes.image, parentAttrs.autoHeight]);

const updateImageWarning = (image: CarouselMedia) => {
  if (image.width && image.height) {
    const aspectRatio = image.height / image.width;
    if (aspectRatio < 0.3) {
      setImageWarning(parentAttrs.autoHeight
        ? "This image is very wide. this may result in very short slides."
        : "This image very wide. This may result in unwanted cropping");
    } else if (aspectRatio > 1) {
      setImageWarning(parentAttrs.autoHeight
        ? "This image is rather tall. This may result in very tall slides."
        : "This image is rather tall. This may result in unwanted cropping");
    } else {
      setImageWarning("");
    }
  }
};
```

**What the fix looks like:**
1. Remove the `imageWarning` state
2. Remove the `useEffect`
3. Remove the `updateImageWarning` function
4. Compute the warning inline during render with a `useMemo` or plain function:
```tsx
const getImageWarning = (image: CarouselMedia | undefined, autoHeight: boolean): string => {
  if (!image?.width || !image?.height) return "";
  const aspectRatio = image.height / image.width;
  if (aspectRatio < 0.3) {
    return autoHeight
      ? "This image is very wide. this may result in very short slides."
      : "This image very wide. This may result in unwanted cropping";
  }
  if (aspectRatio > 1) {
    return autoHeight
      ? "This image is rather tall. This may result in very tall slides."
      : "This image is rather tall. This may result in unwanted cropping";
  }
  return "";
};

// In the component body:
const imageWarning = getImageWarning(attributes.image, parentAttrs.autoHeight);
```

This is a trivial computation — use a plain `const` computed during render. No `useMemo` needed; the aspect ratio math is negligible cost and `useMemo` would add ceremony for zero benefit.

**Test:** In the block editor, add a carousel item. Set an image with extreme aspect ratio (very wide or very tall). Confirm the warning message appears immediately. Change to a normal aspect ratio image. Confirm the warning disappears. Remove the image entirely. Confirm no warning is shown.

**Implementation Tasks (THR-1480):**
1. **Define `getImageWarning` as a pure function outside the component** — place it above the `Edit` function, after imports/constants. Parameters: `image: CarouselMedia | undefined`, `autoHeight: boolean`. Returns `string`. Preserve exact warning strings (including pre-existing typos).
2. **Replace state + effect with inline computation** — remove `useState<string>("")` for `imageWarning`, remove `updateImageWarning` function, remove the `useEffect` block. Add `const imageWarning = getImageWarning(attributes.image, parentAttrs.autoHeight);` in the component body.
3. **Clean up imports** — remove `useEffect` from the React import. Keep `useState` (still used for `showUrlPopover`).
4. **Remove stale comments** — delete the `// Image Warnings` and `// Image Warning` and `// Update Image Warning when image or autoHeight changes` comments that reference the removed code.

## Acceptance Criteria

### Behavioral
- [ ] Given a SlidingPanel is open, When the parent sets `openPanel` to false, Then the panel closes with animation intact
- [ ] Given a SlidingPanel is open, When the user clicks the close button or overlay, Then the panel closes with animation and the parent is notified
- [ ] Given a user types in the SearchBox, When they pause typing, Then the debounced search fires correctly
