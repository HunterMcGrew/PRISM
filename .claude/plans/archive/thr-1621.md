## Goal

Populate the full `ILink` (`href` + auto-derived or legacy-preserved `target`/`rel`) on migrated mega-menu top-level nav items server-side, so external links open in a new tab immediately after migration without an editor re-typing the URL in `LinkControl`.

## Implementation Tasks

### Clove (implementation)

1. Add a private helper `deriveNavItemLinkExtras(string $url): array` on `MegaMenuMigrationService` (`backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php`). Port `isExternalUrl` semantics from [url-utilities.ts:60-85](frontend/lib/utilities/url-utilities.ts#L60-L85) to PHP:
   - Empty URL → return `[]`.
   - Read `platform_settings_frontend_domain` via `get_option` (same direct call pattern as `FrontendLinkService::getFrontendUrl`). Empty domain → return `[]` (internal).
   - Normalize bare-domain / protocol-relative / scheme-prefixed URLs to a parseable form, compare host with `www.`-stripped / lowercased frontend host.
   - External → `['target' => '_blank', 'rel' => 'noopener noreferrer']`. Internal or unparseable → `[]`.

2. Update `transformNavigationItem` ([MegaMenuMigrationService.php:178-198](backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php#L178-L198)) to read legacy `target` and `rel` from `attrs` when `type === 'link'` and pass them through to `buildNavItemBlock`.

3. Update `buildNavItemBlock` ([MegaMenuMigrationService.php:460-481](backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php#L460-L481)) to accept optional `$target` and `$rel` params. When `type === 'link'` and `$url !== ''`:
   - If legacy `$target` is a non-empty string → use it as-is (legacy intent wins).
   - If legacy `$rel` is a non-empty string → use it as-is.
   - Else when legacy target is empty → call `deriveNavItemLinkExtras($url)` to auto-derive.
   - Write `attrs.link` as `['href' => $url, 'target' => ..., 'rel' => ...]` with only the keys that have values.
   - Dropdown-type items unchanged — no `link` attribute.

4. Create `backend/plugins/gravity-platform-core/tests/Unit/Services/MegaMenuMigrationServiceTest.php` (Pest, unit). Cover:
   - External URL, no legacy target, `frontend_domain` set → `link.target === '_blank'` and `link.rel === 'noopener noreferrer'`.
   - External URL with `www.` vs non-`www.` frontend_domain → treated as same-site (internal).
   - Relative URL (`/about`) → no `target`, no `rel`.
   - Bare-domain URL (`example.com`) when `frontend_domain` differs → external handling applies.
   - Legacy `target='_blank'` with internal URL → legacy value preserved (target stays `_blank`).
   - Legacy `target='_self'` with external URL → legacy value preserved (no auto-override).
   - Legacy `rel='nofollow'` → preserved regardless of target.
   - Empty URL → no `link` attribute on the block (existing behavior).
   - Dropdown-type nav item → no `link` attribute regardless of URL presence.
   - `frontend_domain` option unset / empty → no auto-target (treated as internal).

   Mock `get_option` via `Brain\Monkey\Functions\expect`. Follow the pattern in [FrontendLinkServiceTest.php](backend/plugins/gravity-platform-core/tests/Unit/Services/FrontendLinkServiceTest.php).

## Decisions

- Auto-derived `target`/`rel` for migrated nav items is **not** a violation of the "link target is editor-controlled only" rule from [mega-menu.md Decisions](.claude/architect/mega-menu.md). The renderer stays editor-controlled-only. Migration is emulating `LinkControl.handleUrlChange`'s create-time behavior — seeding the editor-controlled value with what a human editing the link in `LinkControl` would have produced.
- Legacy `target`/`rel` always win over auto-derivation. Rationale: the legacy data represents explicit editor intent; auto-derivation is only a fallback for the "never touched in editor" case.
- `rel` is extracted from legacy `attrs` even though no other builder in the service does this. Nav items are the only migration target where editors control `rel` via `LinkControl`'s advanced settings. Expanding `rel` extraction to other builders (links-with-icon, featured-banner, info-banner, normalizeLinks) is out of scope — no evidence legacy data carries `rel` there, and the renderer doesn't surface rel editing for those blocks.
- Helper lives as a private method on `MegaMenuMigrationService`, not in `FrontendLinkService` or a new utility. Rationale: single consumer, narrow migration-specific concern. Promote if a second consumer appears.
- `get_option('platform_settings_frontend_domain')` is called directly (not injected through the constructor), matching `FrontendLinkService::getFrontendUrl`. Brain\Monkey handles mocking at the test boundary — no need to expand the DI surface.
- PHP `isExternalUrl` port must match the TS behavior in [url-utilities.ts:60-85](frontend/lib/utilities/url-utilities.ts#L60-L85): `www.`-stripping, lowercase host compare, protocol-relative (`//example.com`) normalization, bare-domain heuristic.

## Review Issues

### Missing test: case-insensitive host normalization
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/tests/Unit/Services/MegaMenuMigration/NavigationItemLinkExtrasTest.php`
- **Problem:** No test exercises a mixed-case host (`https://WWW.example.com` vs `frontend_domain=example.com`). The `i` flag on the `^www\.` regex in `normalizeHost` is load-bearing because PHP's `parse_url` preserves host case (unlike the JS URL API, which lowercases). Without a regression test, a future edit that drops the `i` flag would silently misclassify `WWW.example.com` as external.
- **Suggested fix:** Add one test: `stubFrontendDomain('example.com')` + url `https://WWW.example.com/page` → `link` has no target/rel (same-site).
- **Fixed in:** `NavigationItemLinkExtrasTest.php` — added `it('treats mixed-case same-site hosts as internal', ...)` asserting `https://WWW.Example.COM/page` with `frontend_domain='example.com'` resolves to a link with no target/rel.

### Missing test: http:// URLs
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/tests/Unit/Services/MegaMenuMigration/NavigationItemLinkExtrasTest.php`
- **Problem:** All external-URL tests use `https://`. The `normalizeExternalUrl` regex uses `/^https?:\/\//i` so `http://` should work, but there's no explicit coverage.
- **Suggested fix:** Add one test: `stubFrontendDomain('example.com')` + url `http://google.com` → `link.target === '_blank'`, `link.rel === 'noopener noreferrer'`.
- **Fixed in:** `NavigationItemLinkExtrasTest.php` — added `it('auto-derives target=_blank and rel for http:// URLs', ...)` asserting `http://google.com` auto-derives `target=_blank` + `rel=noopener noreferrer`.

### Whitespace-only legacy target/rel bypasses auto-derivation
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:185-186`
- **Problem:** `transformNavigationItem` reads `target`/`rel` as strings; `buildNavItemBlock` only compares against the empty string (`$target === ''`). A legacy attr of `target: ' '` would be non-empty, skip the auto-derivation branch, and persist the whitespace into `link.target`. Low practical likelihood since `LinkControl` doesn't emit whitespace.
- **Suggested fix:** `trim()` legacy `$target` and `$rel` in `transformNavigationItem` before passing to `buildNavItemBlock`. Overlaps with qodo bot's thread on PR #1801 — Eric replied there agreeing on the trim and declining the full allowlist scope.
- **Raised by:** Eric (PR review, 2026-04-18).
- **Fixed in:** `MegaMenuMigrationService.php` — `transformNavigationItem` now `trim()`s legacy `target`/`rel` before passing to `buildNavItemBlock`, so whitespace-only values fall through to auto-derivation. Added regression test `it('treats whitespace-only legacy target/rel as absent and falls through to auto-derivation', ...)` to `NavigationItemTest.php`.

## Acceptance Criteria

### Behavioral

- [ ] Given a dealer site with legacy mega menu content where a top-level nav item link points to an external URL, When the migration runs, Then the resulting `mega-menu-navigation-item` opens in a new tab on the frontend without an editor opening it.
- [ ] Given a dealer site with legacy mega menu content where a nav item link is same-site or relative, When the migration runs, Then the resulting nav item opens in the same tab on the frontend.
- [ ] Given legacy nav item data that already carries a `target` value, When the migration runs, Then the saved nav item's `link.target` matches the legacy value (legacy intent wins over auto-inference).

### Non-behavioral

- [ ] `MegaMenuMigrationService::buildNavItemBlock` writes a complete `ILink` array (`href`, optional `target`, optional `rel`) for `type="link"` nav items.
- [ ] Site-aware external check uses `platform_settings_frontend_domain` (mirrors the `LinkControl` editor-side behavior).
- [ ] Dropdown-type nav items still carry no `link` attribute.
- [ ] Pest covers: external URL → `target="_blank"` + `rel="noopener noreferrer"`; relative URL → no target/rel; legacy `target` preserved when present; legacy `rel` preserved when present; dropdown unaffected; `frontend_domain` unset → no auto-target.
