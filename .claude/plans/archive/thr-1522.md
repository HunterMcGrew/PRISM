# Plan: thr-1522

## Ticket

https://linear.app/tractru/issue/THR-1522/accessibility-audit-close-wcag-21-aa-22-compliance-gaps

## Goal

Bring the codebase into documented, enforced WCAG 2.1 AA compliance — with awareness of WCAG 2.2 additions — across all frontend components and WordPress blocks.

---

## Decisions

- WCAG 2.1 AA is the enforced baseline; WCAG 2.2 AA additions are flagged and required where applicable.
- The color contrast utility for Q2 theming is a separate ticket — see `.claude/plans/thr-1523.md`. Out of scope here.
- `eslint-plugin-jsx-a11y` is the write-time enforcement mechanism — must be activated in `eslint.config.mjs`. **Why:** mechanical violations (missing `alt`, invalid ARIA, unlabeled form controls) caught at write time prevent the audit from re-accumulating debt as new components land.
- `aria-live` containers must be present on initial render rather than injected dynamically. **Why:** screen readers track live regions from the moment they enter the DOM — a container that arrives with content already inside is treated as a normal element and the text is never announced. This is the most commonly broken pattern in custom notification components.
- Every Next.js route must have a meaningful `<h1>` for the built-in route announcer.

## Implementation Tasks

### Clove

1. **Merge `origin/main` into the branch.** It's ~130 commits behind. Resolve any conflicts in `.claude/rules/accessibility.md` against the rewritten version on this branch — keep the rewritten content. Run formatters and confirm build before continuing.

2. **Verify `eslint-plugin-jsx-a11y` is activated in `eslint.config.mjs`.** Confirm `eslint-config-next/core-web-vitals` is wired in; if not, add it. Run lint after activation and capture the violation count — this becomes the realistic implementation scope for the rest of the audit before sprint estimation.

3. **Audit all routes/pages for a meaningful `<h1>`.** Any route without one is a screen reader dead zone on client-side navigation. Add or correct as needed.

4. **Audit toast, notification, and alert components for `role="alert"` or `aria-live`.** Verify (a) the live region container exists on initial render, (b) the live attributes are correct (`polite` vs `assertive`).

5. **Audit `<form>` components for the error pattern.** Every input with validation links to an `aria-live` error container that exists on initial render via `aria-describedby`.

6. **Audit modals, sliding panels, and mega menus for focus management.** Trap on open, restore on close, Escape to dismiss; sticky headers do not fully obscure focused elements (WCAG 2.4.11).

7. **Audit icon-only buttons and small interactive controls for 24×24px target size** (WCAG 2.5.8). Close buttons, icon toggles, pagination arrows.

8. **Audit DnD and sortable interfaces for keyboard alternatives** (WCAG 2.5.7). Block editor sortable lists, carousels, drag-to-reorder UIs.

9. **Audit tooltips and popovers against the three hover/focus content rules** (WCAG 1.4.13): Escape-dismissible without moving focus, hoverable, persistent until dismissed.

10. **Audit form fields collecting personal data for `autocomplete` attributes** (WCAG 1.3.5). Name, email, address, phone.

11. **Audit multi-step forms for redundant entry** (WCAG 3.3.7). Don't re-ask for data already collected in the same session.

12. **Add accessibility assertions to component tests** — one test per component per criterion where the behavior is testable.

### Eli

## Acceptance Criteria

### Behavioral

- Given a user navigates between pages using a keyboard, when focus moves to the new page, then a screen reader announces the new page title without requiring manual navigation.
- Given a user tabs through any interactive element, when the element receives focus, then a visible focus indicator is shown and no sticky header or overlay fully covers it.
- Given a user fills out a form with an error, when the error appears, then a screen reader announces the specific error message without the user needing to navigate to it.
- Given a user submits a form successfully, when the success message appears, then a screen reader announces it without the user needing to navigate to it.
- Given a user encounters a sortable list, when they use the keyboard only, then they can reorder items without using drag and drop.
- Given a user encounters a tooltip or popover, when they press Escape, then the tooltip closes without moving their focus.
- Given a user encounters a tooltip or popover, when they move their pointer toward the tooltip, then it does not close before they reach it.
- Given a user encounters any icon-only button, when they inspect it, then its purpose is communicated to a screen reader via label or visually hidden text.
- Given a user encounters any interactive element, when they tab to it, then it is visually focusable with a contrast ratio of at least 3:1 for the focus indicator.

### Non-behavioral

- [ ] `eslint-plugin-jsx-a11y` is activated in `eslint.config.mjs`.
- [ ] All routes have a meaningful `<h1>` element.
- [ ] All interactive targets are at least 24×24 CSS pixels.
- [ ] All form fields collecting personal data have correct `autocomplete` attributes.
- [ ] All `aria-live` regions exist in the DOM on initial render.
