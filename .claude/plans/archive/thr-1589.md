# Plan: THR-1589

## Ticket

https://linear.app/tractru/issue/THR-1589/mega-menu-migration-equipment-pane-card-not-migrated-shows-placeholder

## Goal

Fix `buildCard()` in the mega menu migration service so that card content (image, link) is preserved during migration instead of falling back to placeholder defaults.

---

## User Stories

Not applicable — this is a bug fix, not a new capability.

---

## Design

Not applicable — no UI changes. Fix is migration-only.

---

## Decisions

- Fix is isolated to `buildCard()` attribute mapping — no frontend rendering changes needed
- `url` maps to `ctaLink` — the card block's registered attribute name for the link URL
- `image.sourceUrl` maps to `imageUrl` as a flat string via existing `normalizeImageUrl()` — the card block expects a string, not the `{id, sourceUrl, altText}` object used by other blocks like `core/image` and `mega-menu-info-banner`
- `target` is dropped — the card block does not register a `target` attribute. No meaningful data loss; legacy cards rarely set target.
- `ctaText` is NOT explicitly set — left to default ("Call To Action"). Rationale: `CardBlock.tsx:41` renders a CTA button whenever `ctaLink` is truthy. There is no "linked card without button" mode. Setting `ctaText: ''` produces a visually broken empty-text button. The default label matches what every new card gets in the block editor. Legacy cards had a URL (data must be preserved) but no CTA button (a consequence of the legacy rendering, not the data). The migration's job is faithful data mapping; the CTA button appearing is the card block's standard behavior.
- Both equipment pane and service pane cards are affected since both call `buildCard()`
- Tests must assert ALL card attributes against the registered schema — the original tests only checked `blockName` and `headline`, which is how this bug shipped

## Debugged Issues

### Card attributes silently discarded by WordPress
- **Status:** `fixed`
- **Fixed in:** `buildCard()` — `url` → `ctaLink`, `image` object → `imageUrl` flat string via `normalizeImageUrl()`, `target` dropped. 77 Pest tests pass (370 assertions).
- **Severity:** High
- **Environment:** WordPress admin — mega menu migration tool
- **File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:773`
- **Root cause:** `buildCard()` produces attributes that don't match the card block's `registerBlockType()` schema. Three mismatches: (1) `url` should be `ctaLink`, (2) `image` (object `{id, sourceUrl, altText}`) should be `imageUrl` (flat string), (3) `target` is not a registered attribute. WordPress silently discards unregistered attributes. The card block's attribute schema (`imageUrl`, `ctaLink`) diverges from the image-as-object pattern used by other migration methods (`normalizeImageData`), and `buildCard()` was written against the assumed pattern rather than the actual schema.
- **Steps to Reproduce:**
  1. Have a legacy mega menu with a card in the equipment pane (with image, headline, URL)
  2. Run the Import / migration tool
  3. Observe the migrated card in the editor or frontend
- **Expected behavior:** Card image preserved, link URL preserved
- **Actual behavior:** 720×540 placeholder image (from `DEFAULT_PLACEHOLDER_IMAGE`), link falls to `#` (default `ctaLink`)
- **Recommended fix:** In `buildCard()`: (1) map `url` → `ctaLink`, (2) extract `image.sourceUrl` → `imageUrl` as flat string via `normalizeImageUrl()`, (3) drop `target` (not registered), (4) don't set `ctaText` (let block default apply)
- **Suggested tests:** See `## Implementation Tasks` — 3 new tests + assertions on 4 existing tests
- **Linear:** `synced`

- [ ] Given a legacy mega menu with a card in the equipment pane that has an image, When the migration runs, Then the migrated card displays the original image (not a placeholder)
- [ ] Given a legacy mega menu with a card in the equipment pane that has a link URL, When the migration runs, Then the migrated card's CTA link points to the original URL

- [ ] Pest PHP tests assert every card attribute against the registered schema (`headline`, `description`, `ctaLink`, `imageUrl`) — not just `blockName` and `headline`
- [ ] Pest PHP tests verify that unregistered attributes (`url`, `image`, `target`) are NOT present in the output
- [ ] Existing migration tests for equipment and service panes continue to pass
- [ ] No changes to frontend rendering code — fix is migration-only

### AC Adjustments

### AC Adjustment: CTA button behavior
- **Original:** Given a legacy mega menu with a card in the equipment pane, When the migration runs, Then no "Call To Action" button appears unless CTA text was explicitly set in the legacy card
- **Proposed:** Remove this AC item. The CTA button renders whenever `ctaLink` is truthy — this is the card block's standard rendering behavior, not a migration defect. The migration correctly maps the legacy URL to `ctaLink`; the button appearance is a consequence of the new card block's design. Dealers can customize or remove the CTA text in the editor.
- **Reason:** `CardBlock.tsx:41` has `buttons={ctaLink ? [{label: ctaText, link: ctaLink}] : undefined}`. There is no "linked card without CTA button" mode. Suppressing the button would require either losing the URL data or producing a broken empty-text button.
- **Status:** `accepted`

## Implementation Tasks

### Task 1: Fix `buildCard()` attribute mapping

**File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:773-800`

Change `buildCard()` to produce attributes matching the card block's `registerBlockType()` schema:

1. Rename `$url` assignment to map to `ctaLink`:
   - Line 784: `'url' => $url` → `'ctaLink' => $url`
2. Change image handling from object to flat string:
   - Line 789-791: Replace `$attrs['image'] = $this->normalizeImageData($imageData)` with `$attrs['imageUrl'] = $this->normalizeImageUrl($imageData)`
   - `normalizeImageUrl()` already exists (line 921) and returns a string — it handles both `sourceUrl` and `mediaUrl` fallback
3. Remove `target` handling entirely:
   - Delete lines 786-788 (the `if ($target !== '') { $attrs['target'] = $target; }` block)
   - Delete line 778 (`$target = is_string(...)`)

**After the fix, `buildCard()` should produce:**
```php
$attrs = [
    'headline' => $headline,
    'description' => $description,
    'ctaLink' => $url,
];
// Only add imageUrl when image data exists and has a URL
if ($imageData !== null) {
    $imageUrl = $this->normalizeImageUrl($imageData);
    if ($imageUrl !== '') {
        $attrs['imageUrl'] = $imageUrl;
    }
}
```

Note: only set `imageUrl` when the URL is non-empty — an empty string would override the block's undefined default and could produce a broken `<img src="">`.
