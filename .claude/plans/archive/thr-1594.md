# Plan: THR-1594

## Ticket

[THR-1594: Document third-party plugin ecosystem and supported integrations](https://linear.app/tractru/issue/THR-1594/document-third-party-plugin-ecosystem-and-supported-integrations)

## Goal

Document the curated third-party WordPress plugin ecosystem — why each category exists, the architectural implications of headless plugin support, and provide user-facing integration guides starting with HubSpot.

---

## Decisions

- Plugin ecosystem docs split across two audiences: architecture (dev) and integrations (user)
- **plugin-management.md owns the "how"** (sources, Azure storage, CI pipeline, `plugins-from-storage.txt` format); **plugin-ecosystem.md owns the "what and why"** (which plugins, why curated, evaluation criteria, frontend reimplementation commitment). Both pages cross-reference each other.
- Frontend reimplementation principle lives in headless-architecture.md (architectural boundary rule); plugin-ecosystem.md links there for the "why" and focuses on the inventory and evaluation criteria
- WooCommerce is documented as the product data layer, not an e-commerce storefront — this framing goes in the ecosystem page
- Plugin categories: Forms & CRM (Gravity Forms ecosystem), Inventory & Products (WooCommerce + import pipeline), Data Import & Sync (WPAI suite), SEO & Content (Yoast), Content Modeling (ACF), Events (The Events Calendar), AI Content (AI Content Generation, WP Wand Pro), Platform Operations (Wordfence [Composer], MainWP, Simple History, WP Mail SMTP, FileBird)
- Salesforce CRM Perks (`gf-salesforce-crm-perks-pro`) included in Gravity Forms add-on list
- User-facing integrations section (`user/integrations/`) created from the start for URL stability — links shared in Slack must remain valid as more pages are added
- The GraphQL bridge pattern (WP plugin → wp-graphql-* bridge → ServiceFactory → Next.js) is called out as the architectural principle enabling headless plugin support
- HubSpot page documents Thrive-specific setup (UTK cookie pipeline, item URL, dealer role capabilities) alongside links to official GF HubSpot docs — not just a link dump
- **HubSpot tracking has three layers, not one.** The original docs claimed "works without any configuration" — this is only true for the code-level automation. The full picture:
  - **Platform-level (site setup):** HubSpot tracking snippet (`<script ... src="//js.hs-scripts.com/XXXXXXXX.js">`) is added to Thrive Settings script fields. Handled during site setup — dealer doesn't do this. The script sets the `hubspotutk` cookie on visitors' browsers.
  - **Form-level (per form):** Each form that should pass HubSpot data needs hidden fields with "Allow field to be populated dynamically" enabled and parameter names set to `hubspotutk`, `item_url`, and optionally `form_submission_url`. Without these fields, the code has nothing to populate.
  - **Code-level (truly automatic):** Frontend reads the cookie and populates matching fields (`populate-form-fields.ts`). Backend copies `hubspotutk` to entry meta and `item_url` to `source_url` (`FormsSetup.php`). HubSpot permissions granted to dealer role (`RolesBuilder.php`). No configuration needed.
- HubSpot docs should distinguish what's done for the dealer (tracking snippet) from what the form builder needs to do (hidden fields) from what the code handles silently (cookie capture, relay, permissions)

- **File:** `docs/content/dev/architecture/plugin-ecosystem.md`
- **Content:**
  - Why Thrive supports a curated plugin set (not arbitrary WP plugins)
  - Evaluation criteria for new plugins: frontend surface area, block attribute complexity, update frequency, GraphQL bridge availability
  - The GraphQL bridge pattern as the universal integration mechanism
  - Backend-only vs frontend-impacting distinction: backend-only plugins (WPAI, Yoast, ACF) just need a GraphQL bridge; frontend plugins (Gravity Forms) require full UI reimplementation — links to headless-architecture.md for the architectural principle

    - Inventory & Products: WooCommerce core + extensions (Composite Products, Bundles, Feeds) — WooCommerce as product data layer
    - Data Import & Sync: WPAI Pro + add-ons, WP All Export Pro + add-ons, Space Station Sync (thrive-sss), MFP and data scraper integrations
    - SEO & Content: Yoast SEO/Premium
    - Content Modeling: ACF Pro + Extended
    - Events: The Events Calendar + ql-events
    - AI Content: AI Content Generation, WP Wand Pro
    - Platform Operations: Wordfence (Composer), MainWP, Simple History, WP Mail SMTP, FileBird (brief mention)
  - Cross-references plugin-management.md for the "how" (sources, storage, CI)

### 5. New: HubSpot integration guide
- **File:** `docs/content/user/integrations/hubspot.md`
- **Content:**
  - Overview: any Gravity Form on a dealer site can be connected to HubSpot
  - What Thrive handles automatically (no dealer action needed):
    - HubSpot tracking cookie (`hubspotutk`) is captured from the visitor's browser and passed through form submissions — enables HubSpot to associate form entries with tracked contacts
    - The page URL where the form was submitted is included automatically (`item_url` → `source_url`)
    - Dealer accounts have HubSpot integration permissions enabled by default (`gravityforms_hubspot` capability)
  - What the dealer configures: creating a HubSpot feed in Gravity Forms (link to official GF docs for step-by-step)
  - Tips: which forms to connect, field mapping considerations
