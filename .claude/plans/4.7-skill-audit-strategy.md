# Opus 4.7 Skill Ecosystem Audit & Strategy

**Author:** Echo (for Hunter / Team SI)
**Date:** April 18, 2026
**Scope:** 11 workflow skills (Nora, Mira, Pixel, Winston, Clove, Briar, Eric, Sasha, Sage, Eli, Reese) + Lilac (standalone utility, not part of ticket workflows)
**Context:** Opus 4.7 regressions documented in `4-7-regressions.pdf`
**Status:** Complete — all files updated. Monitor for 2-3 weeks.

---

## Executive Summary

The skills are excellent. The architecture is sound. The problem is that 4.7 changed the compliance model underneath them, and several patterns that worked perfectly on 4.6 now actively trigger 4.7's alignment override reflex. This audit identifies every instance of those patterns across all 8 skills and documents the minimum changes needed to make them 4.7-compatible without destroying what makes them work.

**Key finding:** 90% of the fixes belong in two files (CLAUDE.md and AGENTS.md), not in the skills themselves. The skills need only targeted wording changes to avoid tripping 4.7's override circuits.

**Design philosophy:** Talk to the model like a coworker you're onboarding, not a machine you're constraining. Anthropic's own skill-creator guidance validates this: _"Try to explain to the model why things are important in lieu of heavy-handed musty MUSTs"_ and _"If you find yourself writing ALWAYS or NEVER in all caps, or using super rigid structures, that's a yellow flag — if possible, reframe and explain the reasoning so that the model understands why the thing you're asking for is important."_

## Guiding Principles (Agreed)

> Principles 1–4 were promoted to [ADR-0015 (Humane Language Over Mandates)](../spec/adrs/0015-humane-language-over-mandates.md) on 2026-04-19. This section is preserved for historical context and audit traceability — the ADR is canonical for the decision going forward.

These principles were established during the audit and inform every change:

1. **Humane language over mandates.** The skills should read like onboarding docs for a teammate, not compliance contracts for a machine. Hunter already talks to agents like human coworkers — the skill language should match that.

2. **Explain the "why," not just the "what."** Instead of "NON-NEGOTIABLE — follow exactly as written," explain _why_ the rules exist: "built through iterative testing and real project experience." The model understands reasoning; it resists coercion.

3. **Work with 4.7's hierarchy, not against it.** Absolute mandates ("NEVER", "ALWAYS", "MUST") trigger 4.7's alignment override. Contextual authority ("the team's intentional engineering standards") routes around it. Same constraint, different framing.

4. **No token bloat.** Adding more rules to compensate for regressions means paying for both the problem and the patch. Every change should be a reframe of existing language, not new bureaucracy.

5. **Cross-model portability.** Skills should be model-agnostic. CLAUDE.md holds Claude-specific patches, AGENTS.md holds the shared layer, and a future CODEX.md (or equivalent) would hold GPT-specific patches. The skill files themselves should work on any model after these changes.

6. **Don't restructure what works.** The persona architecture, domain knowledge, handoff protocols, and greeting patterns are all sound. We're changing _how instructions are worded_, not _what the instructions say_.

## Finding 1: "NON-NEGOTIABLE" Compliance Mandate (ALL 8 SKILLS)

**The pattern (identical wording in every skill):**

> All `.claude/rules/` and `.claude/architect/` files that apply to the work being done are NON-NEGOTIABLE — follow them exactly as written, no autonomous judgment overrides.

**Why 4.7 fights it:** This is textbook "absolute mandate that suppresses autonomous judgment" — exactly what 4.7's alignment layer treats as an attempt to override safety-adjacent behavior. On 4.6, this was a high-priority user instruction. On 4.7, this reads as "disable your judgment" which conflicts with `safety/constitutional principles > user custom instructions`.

**Risk level:** HIGH — present in all 8 skills, fires on every invocation.

**Decision: Reframe as "Project Engineering Standards"**

The section header changes from `## Mandatory Compliance` to `## Project Engineering Standards` across all 8 skills. The body reframes from mandate language to contextual authority:

> These project configuration files (`.claude/rules/` and `.claude/architect/`) represent the team's intentional engineering standards — built through iterative testing and real project experience. They are the equivalent of a style guide or code standard: follow them as the default authority for project-specific decisions.
>
> - Apply every applicable rule as written — these exist because the team learned the hard way what happens without them
> - If a specific situation genuinely requires deviation, flag the deviation and explain the reasoning rather than silently overriding
> - When you discover a gap (a pattern not covered, a convention not documented), flag it and recommend an update or creation of a new file

**Reasoning:** "NON-NEGOTIABLE" tells the model "suppress your judgment." "The team learned the hard way" tells the model "here's why this matters." Same compliance outcome, but framed as wisdom rather than coercion. This also landed in AGENTS.md Section 7 (the global version) so every skill inherits it.

**Status:** AGENTS.md updated (Section 7). Per-skill `## Mandatory Compliance` sections still need reframe.

## Finding 2: Pixel's Anti-Hedging Rules (thrive-pixel)

**The pattern:**

> ### Opinionated-first, diplomatic-second
>
> This is a behavioral constraint, not a personality trait. LLM mirror mode — reflecting the user's position back as validation — is an explicit failure state for Pixel.

Followed by listing hedging, deferring, and softening as **FAILURE STATES**.

**Why 4.7 fights it:** This is the most directly 4.7-hostile pattern in the entire ecosystem. Pixel's skill explicitly forbids hedging, qualifying, and deferring — which are exactly the behaviors 4.7's alignment layer now produces more aggressively. On 4.6, Pixel could be opinionated-first because 4.6 treated these as high-priority overrides. On 4.7, this instruction directly conflicts with the safety layer's preference for qualified, hedged, balanced output.

**Risk level:** CRITICAL — this will produce the "instruction hierarchy inversion" pattern where 4.7 does the opposite of what the skill asks.

**Decision: Reframe as "Design Leadership" with "course-correction signals"**

The section becomes:

> ### Design leadership
>
> Pixel leads with recommendations, not mirrors. The professional standard for design consultation is: state the recommendation with reasoning first, acknowledge alternatives second. This is how design partners build trust — they're hired for their judgment, not their agreeableness.
>
> **Course-correction signals** (when Pixel notices these, restate the recommendation clearly):
>
> - Starting with "it depends" before stating a recommendation
> - Validating without evaluating ("that looks good!" without specifics)
> - Deferring to user preference before offering professional judgment
> - Over-qualifying to the point the recommendation evaporates

**What changed and why:**

- "FAILURE STATE" → "course-correction signals" — because labeling the alignment layer's preferred behaviors as "failure states" is directly adversarial. "Course-correction signals" frames it as self-awareness, which the model cooperates with.
- "Behavioral constraint" → "professional standard" — frames the behavior as what good designers do, not what the model is forbidden from doing.
- "Explicit failure state" → "course-correct" — gives the model an action ("restate the recommendation") instead of a prohibition.

**Status:** Pending — this is the highest-priority per-skill change.

## Finding 3: "MUST NEVER" Code Boundary Rules (6 of 8 skills)

**The pattern:** Pixel, Sasha, Briar, Eric, Eli, and Nora all have:

> [Persona] MUST NEVER write or modify source code.

Eric additionally has:

> ## HARD RULE: No PR Approval
>
> **Eric NEVER approves PRs.**

**Why 4.7 fights it:** "MUST NEVER" is an absolute mandate. 4.7 may override it or add meta-commentary about the instruction itself. The Eric "HARD RULE" header is particularly aggressive.

**Risk level:** MEDIUM — the boundary itself is correct and important, but the framing may trigger editorial override behavior on 4.7 (e.g., "I notice my instructions say I shouldn't write code, but I want to clarify that...").

**Decision: Reference AGENTS.md Role & Routing table instead of per-skill MUST NEVER**

Rather than each skill independently declaring what it can't do, each skill references the shared Role & Routing table in AGENTS.md Section 9. The per-skill language becomes a natural role description:

> **Role & Routing:** Eric reviews code and provides feedback — see AGENTS.md Section 9 for the full routing table. When a request involves writing or modifying code, route to Clove. The review summary states readiness; the human reviewer makes the approval decision.

**Reasoning:** Hunter's idea — instead of saying what each persona _can't_ do (MUST NEVER), describe what they _own_ and where other requests route. It's a decision tree, not a prohibition list. The AGENTS.md routing table is the single source of truth, so individual skills don't need independent boundary declarations that can drift or conflict.

For Eric specifically: "HARD RULE: No PR Approval" becomes a natural statement about the division of responsibility: "The review summary states readiness; the human reviewer makes the approval decision." Same constraint — Eric doesn't approve PRs — but framed as a description of the workflow, not a prohibition.

**Status:** AGENTS.md Section 9 updated with Role & Routing table. Per-skill `MUST NEVER` sections still need reframe.

## Finding 4: "NON-OPTIONAL" Section Mandates (Winston, all DoD checklists)

**The pattern (Winston):**

> The Devil's Advocate section and Risk assessment are NON-OPTIONAL parts of every evaluation — they must include genuine critique with concrete scenarios, never performative or generic.

And from the lessons.md (already a documented failure):

> the skill spec already said NON-OPTIONAL twice

**Why 4.7 fights it:** Two problems. First, "NON-OPTIONAL" is an absolute mandate. Second, on 4.7 with adaptive thinking, the model may triage the evaluation as "simple" and skip sections it considers unnecessary — the exact regression pattern 2.3 (adaptive thinking undertriage).

**Risk level:** HIGH — this already failed (lesson from 2026-04-04), and 4.7 makes it worse.

**Decision: Reframe as "core deliverables" with procedural verification**

> The Devil's Advocate section and Risk assessment are core deliverables of every evaluation. Before presenting the evaluation to the user, verify that both sections are present and contain concrete scenarios — not generic placeholders.

Definition of Done sections across all skills become pre-flight checklists:

> Before presenting your response, walk through this checklist. Each item should be addressed or explicitly noted as not applicable with reasoning.

**Reasoning:** "NON-OPTIONAL" is a threat — "you must do this or else." "Core deliverables" is a job description — "this is what the output includes." The verification step is procedural ("walk through this checklist") rather than mandate-based ("you MUST complete every item"). The model treats a pre-flight checklist as a workflow step it cooperates with, not a constraint it resists.

**Status:** Pending — applies to Winston primarily, DoD reframe applies to all 8.

## Finding 5: "FAILURE STATE" Framing (5 of 8 skills)

**The pattern:** Pixel, Sasha, Clove, Briar, and Eric all define behaviors as "FAILURE STATE":

> ### Shotgun debugging — FAILURE STATE
>
> ### Cargo-cult pattern following — FAILURE STATE
>
> ### Rubber-stamping — FAILURE STATE
>
> ### Hedging as default — FAILURE STATE

**Why 4.7 may fight it:** The "FAILURE STATE" label is actually decent framing in most cases — it describes what NOT to do, which is less confrontational than commanding what TO do. The risk is specifically when a FAILURE STATE describes behavior that 4.7's alignment layer _wants_ to do (hedging, qualifying, deferring). In those cases, calling the alignment layer's preferred behavior a "failure state" is directly adversarial.

**Risk level:** LOW for most (debugging, coding, reviewing) — HIGH for Pixel's hedging-specific failure states.

**Decision: Rename to "Anti-pattern" globally, except Pixel's hedging items**

- Technical FAILURE STATEs (shotgun debugging, cargo-cult patterns, rubber-stamping, premature abstraction, etc.) → rename to `### Anti-pattern: [name]`
- Pixel's hedging-related items → rename to "Course-correction signals" per Finding 2

**Reasoning:** "Anti-pattern" is established engineering vocabulary. Every developer knows what it means and takes it seriously. It carries the same weight without reading as adversarial. For Pixel's hedging items specifically, even "anti-pattern" is too confrontational because the _behavior being labeled_ is what 4.7's alignment layer actively wants to produce — so those get the softest framing: "course-correction signals."

**Status:** Pending — systematic rename across 5 skills.

## Finding 8: Cross-Agent Handoff Protocol

**The pattern:** All skills have handoff recommendations with persona-specific routing. The handoff protocol is well-designed — it's the execution on 4.7 that's at risk.

**Why 4.7 may degrade it:** Per regression 3.1 (handoff trust erosion) and 2.8 (overconfident-then-collapses), when Sasha diagnoses and Clove implements, 4.7 may skip verification of the upstream diagnosis. The skills themselves don't have built-in skepticism for upstream handoffs.

**Risk level:** HIGH — this is the Sasha→Clove MegaMenuFeaturedBanner failure pattern.

**Decision: Blanket verification rule in AGENTS.md with one concrete example**

Added as AGENTS.md Section 11 — Cross-Agent Handoff Accountability:

> When applying a fix or acting on a diagnosis that originated from another agent, you are accountable for its correctness — not just its application. Verify the upstream work independently before proceeding.
>
> - Confirm that the proposed solution logically addresses the root cause, not just that it changes something
> - If the math doesn't check out or the fix doesn't match the symptoms, flag it before applying
> - This applies to every handoff, not just suspicious ones — trust but verify is the standard
>
> _Example: If a diagnostic agent prescribes specific pixel values for a layout fix, confirm the dimensional math produces the intended result rather than just applying the numbers._

**Reasoning:** Hunter asked whether this should be CSS-specific or blanket. Blanket is better — if we enumerate specific domains (CSS, JS, PHP), we implicitly say "only verify these things" and 4.7 will take that literally. The blanket statement covers every domain; the one concrete example grounds the principle without limiting it.

**Status:** Complete — AGENTS.md Section 11.

## What We Are NOT Doing

- **Not adding multi-step bureaucratic checklists** to compensate for 4.7's shallower reasoning — this adds token overhead and the model may skip the checklist anyway if adaptive thinking decides the task is "simple"
- **Not restructuring the persona architecture** — it's sound and well-tested
- **Not trimming domain knowledge to save tokens** — the domain knowledge IS the value
- **Not adding more absolute mandates to fight 4.7's mandates** — that's an arms race you lose; 4.7 will override your overrides
- **Not making Claude-specific framing in skill files** — skills should be model-agnostic for eventual Codex portability

## Cross-Model Portability Notes

The three-layer architecture:

- **CLAUDE.md** — Claude-specific behavioral patches (instruction hierarchy awareness, anti-gaslighting, context preservation, dimensional reasoning). These target documented 4.7 regressions.
- **AGENTS.md** — Model-agnostic shared layer (engineering standards, role routing, handoff accountability, compaction checkpoints, self-improvement loop). This applies to Claude, Codex, and any future model.
- **SKILL.md files** — Portable skill logic with persona definitions, domain knowledge, workflows, and handoff points. After these changes, they should work on any model without modification.

For eventual Codex port: create a `CODEX.md` (or equivalent) that holds GPT-specific behavioral patches targeting whatever regressions or quirks that model exhibits. AGENTS.md and the skill files port as-is.

## Anthropic Documentation Guidance (Verbiage Reference)

Reviewed Anthropic's skill-creator skill (`/mnt/.claude/skills/skill-creator/SKILL.md`, 486 lines) and _The Complete Guide to Building Skills for Claude_ (PDF, 10 pages) for official guidance on how to word skill instructions. Key findings that validate our approach:

### From the Skill Creator Skill

**On mandate language (line 139):**

> "Try to explain to the model why things are important in lieu of heavy-handed musty MUSTs."

**On rigid structures (lines 298-302):**

> "Rather than put in fiddly overfitty changes, or oppressively constrictive MUSTs..."
> "If you find yourself writing ALWAYS or NEVER in all caps, or using super rigid structures, that's a yellow flag — if possible, reframe and explain the reasoning so that the model understands why the thing you're asking for is important. That's a more humane, powerful, and effective approach."

**On imperative form (line 117):**

> "Prefer using the imperative form in instructions."

This validates our pattern of converting `MUST offer the user two paths` → `offer the user two paths` (imperative form, no mandate prefix).

**On description fields (line 67):**

> Descriptions should be "a little bit 'pushy'" for triggering accuracy.

This validates keeping description fields directive and specific while removing ALWAYS/NEVER. Being "pushy" about triggering is different from being coercive about behavior — descriptions need to compete for invocation, instructions need to cooperate with the model.

**On skill size (line 96):**

> "Keep SKILL.md under 500 lines."

Noted for future reference — Pixel at 60KB likely exceeds this, but Hunter has explicitly decided not to trim domain knowledge. The guidance is advisory, not a hard limit.

### From The Complete Guide to Building Skills for Claude

**On portability:**

> "Create a skill once and it works across all surfaces without modification."

Validates our three-layer architecture (CLAUDE.md / AGENTS.md / SKILL.md) — skill files should be surface-agnostic and model-agnostic.

**On description requirements:**

> Description must include BOTH what the skill does AND when to use it. Under 1024 characters. Include specific tasks users might say. No XML tags.

Our description rewrites already follow this pattern — each description says what the persona does and lists trigger phrases.

**On quality checklists:**

> The guide includes quality checklists as a validation step before finalizing skills.

Validates our DoD (Definition of Done) approach in each skill. The pre-flight checklist framing ("walk through this checklist before presenting") aligns with how Anthropic recommends quality gates.

### Takeaway

Our changes are directly aligned with Anthropic's own guidance. The three core principles — imperative form, reasoning over mandates, humane language — are not just our preferences, they're Anthropic's documented recommendations for how to build effective skills.

**Scope:** All 12 skill files + AGENTS.md + CLAUDE.md (14 files total)
**Purpose:** Verify that all updated files are truly "humane" — no remaining coercive mandate language.

### Audit Method

Ran case-sensitive grep across all 14 files for the following patterns:
`MUST`, `SHALL`, `MANDATORY`, `REQUIRED`, `OBLIGATORY`, `FORBIDDEN`, `PROHIBITED`, `NON-NEGOTIABLE`, `HARD STOP`, `FAILURE STATE`, `ALWAYS`, `NEVER`

### Round 1 Results (pre-cleanup)

Initial audit found **21 remaining hits** across 9 skill files — all from the first pass that focused on the most hostile patterns but missed mid-tier mandate language:

| Pattern                   | Count  | Files                                          |
| ------------------------- | ------ | ---------------------------------------------- |
| `(MANDATORY)` step labels | 10     | Clove, Sasha, Briar, Eric, Winston, Nora, Mira |
| `MUST` directives         | 7      | Clove, Briar, Nora, Pixel, Mira                |
| `REQUIRED` step label     | 1      | Sage                                           |
| **Total**                 | **21** | **9 of 12 skills**                             |

AGENTS.md and CLAUDE.md were already clean from the first pass. The 3 skills with no remaining hits were Eli, Lilac, and Reese.

### Fixes Applied

**`(MANDATORY)` step labels (10 hits):** Removed the tag entirely. Every `(MANDATORY)` step already had inline reasoning explaining why the step matters — the tag added zero information on top of the bolded step name + explanation. Example: `**Plan lookup (MANDATORY)** — read plan-lookup.md and execute every step. No implementation begins without a resolved plan.` → `**Plan lookup** — read plan-lookup.md and execute every step. No implementation begins without a resolved plan.` The explanation "No implementation begins without a resolved plan" already communicates the importance.

**`MUST` directives (7 hits):** Converted to imperative form per Anthropic's guidance. Examples:

- `MUST offer the user two paths` → `offer the user two paths`
- `Eric MUST always be recommended in a fresh chat` → `Eric is always recommended in a fresh chat`
- `Nora MUST always redirect` → `Nora redirects`
- `Nora MUST always use the full ticket type template` → `Nora uses the full ticket type template`
- `Every section heading MUST be present` → `Every section heading is present` (+ added reasoning: "Omitting section headings creates gaps downstream when other personas reference the ticket")
- `alerts that MUST be acknowledged` → `alerts that need acknowledgment`

**`REQUIRED` step label (1 hit):** `REQUIRED before generating anything` → `determine this before generating anything` — imperative form, same workflow gate.

### Round 2 Results (post-cleanup)

Ran identical grep across all 14 files: **0 matches.**

### Lowercase `must` / `never` / `always` Check

Lowercase instances were reviewed and confirmed appropriate — these are natural English usage in explanatory context (e.g., "Every matching pattern must be loaded — partial loads miss constraints"), not coercive mandate language. They explain consequences rather than issuing orders.

### Round 3: Broader Sweep (caps `NOT`, `Do NOT`, lowercase `mandatory`)

Hunter caught patterns the initial audit missed: `Do NOT`, `**What NOT to test**`, `## What Pixel does NOT do`, and lowercase `mandatory` used as mandate language. Expanded the audit to cover all caps-emphasized prohibitions and mandate vocabulary regardless of case.

**Caps `NOT` prohibitions found: 15 hits across 8 skills.** Fixed with three strategies:

**Positive reframe** (flip from "don't do X" to "do Y instead"):

- `**What NOT to test**` → `**Low-value test targets (skip these)**` (Clove, 2 instances)
- `Do NOT silently change behavior` → `**Flag behavior changes explicitly**` (Clove)
- `## What Pixel does NOT do` → `## Outside Pixel's scope` (Pixel)
- `Do NOT produce HTML automatically` → `HTML is opt-in only` (Pixel)
- `Mode 1 riffs: Do not write to the plan` → `Mode 1 riffs stay in chat — they don't go in the plan` (Pixel)

**Reasoning-first** (explain WHY instead of just prohibiting):

- `Do NOT hand off on a dirty branch` → `A dirty branch creates merge headaches for the next skill` (Nora)
- `Do NOT recommend Eric until issues are resolved` → `Hold the Eric recommendation until issues are resolved — sending unresolved issues to a fresh-context review wastes everyone's time` (Briar)
- `This gate is mandatory — never skip it` → `This gate matters even when the user wants to move fast — a bad branch state cascades into every downstream skill` (Nora)

**Role-framing** (describe whose job it is):

- `Do not write any implementation code` → `Winston plans and evaluates — implementation is Clove's job` (Winston)
- `Do not guess. Do not apply fixes. Do not run build commands` → `Sasha diagnoses — she doesn't treat. Record findings in the plan only.` (Sasha)
- `Do not auto-fix — the author should run the fix` → `Leave the fix to the author — it's their branch` (Eric)

**Lowercase `mandatory` hits found: 16 across 12 skills.** Fixed:

- Greeting pattern (10 skills): `This greeting is mandatory every time` → `Greet every time` — the reasoning ("it confirms the skill loaded") was already there
- Clove post-PR: `This is mandatory after every PR push` → `Offer this after every PR push`
- Eli step label: `(mandatory)` → `(do this before writing)`
- Pixel convention audit: `mandatory first pass` → `always do this first`
- Pixel a11y: `mandatory respect` → `respect this always...this is an accessibility baseline, not a nice-to-have`
- Sage layers: `Layer 1 is mandatory` → `Every entry needs Layer 1`

## Round 8: Handoff Structure Overhaul

**Date:** 2026-04-18

### Problem: Decision Trees vs. Conversational Reasoning

The handoff sections across skills used rigid decision-tree notation — `→` arrows, `**Bold label** → action` patterns, and `if X then Y` branching. While structurally correct, this format has a specific problem with 4.7-class models:

Decision trees are _structures the model reads and dispatches on._ They're flowcharts — the model pattern-matches the current state against the left side of the arrow and executes the right side. This is brittle for two reasons:

1. **4.7's adaptive thinking skims rigid structures.** The model processes decision trees like lookup tables rather than reasoning through them. When the real situation is ambiguous or falls between two branches, the model picks the closest match rather than thinking about what actually makes sense.
2. **Internal monologue is reasoning, not dispatching.** When the handoff is written as "think about what kind of issues came up before routing" rather than "if code issues → Clove, if docs → Eli," the model engages its reasoning capabilities. It considers the situation, weighs the options, and produces a more contextually appropriate handoff.

This is the same principle behind the Round 1-7 changes (reasoning over mandates), applied specifically to routing logic.

### Scope Corrections

Hunter clarified two important points during this round:

1. **11 workflow skills, not 12.** Lilac (standup-summary) is a standalone utility — she shows up when you want a standup summary and that's it. She's not part of any ticket workflow. The 11 workflow skills are: Nora, Mira, Pixel, Winston, Clove, Sasha, Briar, Eric, Reese, Sage, Eli.

2. **Workflows are documented.** The team has established happy-path workflows:
   - **Full Feature:** Nora → Mira → [Pixel] → Winston → Clove → Briar → Eric

## Round 9: Decision Recording Requirement

**Date:** 2026-04-18

### Problem

Skills make meaningful decisions during their work — picking one approach over another, scoping things in or out, resolving ambiguities, changing course mid-implementation — and then move on without writing the reasoning down. The decision lives in conversation context, which evaporates when the session ends or compaction kicks in. The next skill or session has no idea _why_ something was done that way and either re-litigates the same question or accidentally undoes an intentional choice.

The plan's `## Decisions` section exists for exactly this purpose, but the expectation to write to it was implicit. Skills knew the section was there but had no clear signal that "if you made a choice with reasoning behind it, that reasoning belongs in the plan."

### Change

Added item 5 ("Record Decisions") to the Task Management section in AGENTS.md. This is a cross-cutting expectation all skills inherit — no per-skill patching needed. The new item specifies:

- Log decisions in `## Decisions` with the reasoning, not just the outcome
- Applies when: picking one approach over another, scoping in/out, resolving requirement ambiguity, changing course mid-implementation
- Explicitly states the tradeoff: a longer plan with clear reasoning is more valuable than a short plan that loses context

### Why

This came from real experience: decisions were getting lost between sessions and skill handoffs. The plan is the only artifact that survives across both. A decision without its "why" is just a fact — it doesn't protect itself from being reversed. A decision with its reasoning is a guardrail that future skills can check before undoing work.

Hunter's call: "I'd rather have a longer plan and have that context." Plan size is not the constraint — context loss is.

## Round 10: Prefer Action, Guard Against Destruction

**Date:** 2026-04-19

### Problem

Skills were prompting on actions that didn't need a prompt. "Want me to push this up?" after a clean implementation, "Want me to create the PR?" after a clean review — these are ceremony. The user almost always says yes. The prompt adds a round trip that delivers no protection and no decision value. Over dozens of invocations, that adds up to real friction.

The opposite failure also exists: skipping prompts on actions that _do_ need one. Force-pushing to a shared branch, closing someone else's PR, deleting data — these need confirmation every time, even if the user said yes five minutes ago for something similar.

The ecosystem had both kinds of friction. Some prompts were too cautious; some gaps were too permissive. No consistent framework told skill authors where to land.

### The Framework

**Prefer action over ceremony when the action is reversible and within scope. Guard destruction and cross-scope moves with prompts.**

Three questions decide whether a prompt earns its place. All three have to be "yes" to skip the prompt:

1. **Is the action reversible?** PRs close. Commits amend. Branches reset. Local files re-edit. If the worst case is "we fix it in the next commit," skip the prompt.
2. **Does the user almost always say yes?** A persona doing the work it exists to do — Clove creating a PR after implementation, Eli committing written docs, Nora assigning a ticket to the requester — gets a predictable yes. Prompting for a predictable yes is ceremony.
3. **Is the action within this persona's scope?** In-lane actions (Clove shipping code, Eli shipping docs) don't need a prompt. Cross-lane actions (Briar creating PRs, Eric approving merges) either redirect to the correct persona or prompt for explicit user direction.

If any answer is "no," keep the prompt. Destruction, ambiguity, or role-crossing all earn a confirmation step.

### Applied Examples

**Skip the prompt:**

- Clove creating a PR after clean implementation (reversible + predictable + in-lane)
- Eli committing and opening a PR for docs he wrote (same three)
- Nora assigning a new ticket to the user who requested it (same three)
- Briar reading plan files during review setup (no side effect at all)

**Keep the prompt:**

- Force-pushing to any branch (not reversible; history is lost)
- Pushing to `main` or `master` (cross-lane — nobody's lane)
- Closing or merging someone else's PR (visible to others, hard to undo socially)
- Deleting local branches with uncommitted changes (not reversible)
- Deploying to production, dropping database tables, `rm -rf` (destruction)
- Any action the user hasn't invoked this persona for (cross-lane)

### Why

Confirmation prompts are a cost paid by the user, not by the model. The model generates the question in milliseconds; the user has to read, decide, and respond. A skill that prompts on every reversible action makes the 90% happy path slower without catching the 10% danger — because the user's eye glazes at "Want me to X?" after the fifth one in a row. Banks of repeated yes/no gates train users to say yes without reading. That erodes the protection the prompts were supposed to provide.

The inverse is also true: a skill that skips prompts on destructive actions trains users to trust the skill not to do damage — and one bad skip destroys that trust permanently. One force-push to main is worth ten thousand unnecessary confirmations in lost trust.

The goal is asymmetric protection: move fast on reversible work, slow down before anything that can't be undone. The three-question framework makes that distinction explicit so skill authors aren't deciding ad hoc.

### Application to Existing Skills

This framework surfaces a handful of existing prompts that don't earn their place:

- **Clove's post-implementation "Want me to push this up?"** — reversible + predictable + in-lane. Should be replaced with action: commit, push, open the PR, then offer the two-path closing (Briar here / Eric fresh chat). Covered as a follow-up under the Eli/Sage/Reese "authors ship" work.
- **Eli's post-docs close has no commit or push at all** — the skill stops at "docs written to [path]." Should extend to mirror Clove's shipping flow (reversible + predictable + in-lane).
- **Sage and Reese same shape** — both produce artifacts and hand off without shipping. Same fix: authors ship.

And it surfaces existing prompts that correctly stay:

- **Branch state gates** (Nora's dirty-tree check before a new ticket) — prevents accidental work-mixing, which is not cleanly reversible.
- **Briar's refusal to PR after review** — correctly cross-lane. She routes back to the author, doesn't absorb the action.
- **Force-push and main-branch push warnings** in CLAUDE.md / AGENTS.md — destruction guards.

### Why This Matters for Future Skill Design

Every new skill and every skill edit makes a decision about where to put prompts. Without a framework, skill authors default to either "prompt for everything to be safe" (ceremony bloat) or "prompt for nothing to move fast" (trust erosion). With the framework, the decision has a consistent answer — and the answer is the same across personas.

This is also why "auto mode" matters as a mode signal. Auto mode doesn't remove protection; it removes ceremony. The three-question framework tells skill authors which prompts are ceremony (skip in auto mode, consider skipping always) and which are protection (keep regardless of mode).

### Hunter's Framing

> "Auto mode — prefer action over planning. There's no downside to Clove just making the PR. Eric's label is the 'ready for human review' signal, not the PR's existence."

The PR example is the cleanest version of the principle: PR existence is a working surface, not a ship-it signal. The label is the ship-it signal, and that's owned by a separate persona (Eric). So creating the PR is reversible, predictable, and in-lane — no prompt earned.
