# Plan: THR-1585

## Ticket

https://linear.app/tractru/issue/THR-1585/mega-menu-migration-produces-broken-layouts-missing-colors-and-empty

## Goal

Fix 6 visual bugs in the mega menu migration that produce broken layouts, missing colors, and empty columns on migrated dealer sites.

---

## Decisions

- Single ticket for all 6 issues — they share root cause (migration event), same component area, and same developer context
- Migration service fixes (PHP) and frontend component fixes (React) are both in scope
- Column count for nav lists should be derived from link count (>5 = 2 columns) since legacy data doesn't store column count explicitly — original `buildNavList()` had a `$context` param for this but it was never wired to output; removed as dead code before merge
- Empty columns should be omitted entirely rather than hidden — cleaner block structure
- Existing Pest test suite at `tests/Unit/Services/MegaMenuMigration/` covers all panes — tests need UPDATING to assert new behavior (dynamic columns, conditional layouts), not creating from scratch
- For Issue 5 (featured banner color): resolver must NOT produce explicit `styles.color` when `attrs?.textColor` is absent — let the component fall back to mega menu context color. This preserves the user's ability to set a banner-specific color while defaulting to the menu's global color.
- For Issue 6 (info banner image): use `image.mediaDetails` dimensions when available, keep 400×400 as fallback — don't add `object-contain` (that changes rendering semantics for correctly-sized images)

## Debugged Issues

### Issue 1: Navigation list shows 1 column instead of 2
- **Status:** `fixed`
- **Fixed in:** `buildNavList()` now takes `int $columns = 1` parameter; `transformEquipmentPane()` passes `columns: 2` for firstSection/secondSection
- **Severity:** High
- **Environment:** Frontend (Next.js) — migrated dealer sites
- **File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:530`
- **Root cause:** `buildNavList()` emits only `heading` and `links` attributes — no `columns` key. The frontend schema defaults `columns` to `1`. The legacy `EquipmentNavigationPane` hardcoded `columnCount: 2` for the first section list. The migration needs to match this behavior.
- **Steps to Reproduce:**
  1. Enable mega menu V2 feature flag on a dealer site
  2. Run legacy mega menu migration
  3. View the Equipment pane on the frontend
- **Expected behavior:** Equipment navigation list displays in 2 columns matching the legacy layout
- **Actual behavior:** Navigation list displays in 1 column — all links stacked vertically
- **Recommended fix:** Add a `columns` parameter to `buildNavList()`. For equipment pane first-section nav lists, set `columns: 2` (matching legacy hardcoded behavior). Consider deriving from link count (>5 links = 2 columns) for general use.
- **Suggested tests:** Pest PHP test asserting `buildNavList()` output includes `columns` attribute; test migration of equipment pane produces 2-column nav list
- **Linear:** `not synced`

### Issue 2: Service pane links-with-icon squished into half width
- **Status:** `fixed`
- **Fixed in:** `transformServicePane()` now emits single-column content row when only one side has data
- **Severity:** High
- **Environment:** Frontend (Next.js) — migrated dealer sites
- **File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:282`
- **Root cause:** `transformServicePane()` always builds a 2-column content row even when there are no cards (`$colBBlocks` is empty). The empty Column B renders a `<div>` with `flexBasis: 100%` via `ColumnBlock`, taking 50% of the flex row. This compresses links-with-icon into the other 50%, causing icon shrinking and text wrapping. The legacy service pane used CSS Grid (`lg:grid-cols-2`), where an empty second column collapses naturally.
- **Steps to Reproduce:**
  1. Migrate a dealer site with a Service pane that has links but no cards
  2. View the migrated Service pane on the frontend
- **Expected behavior:** Links-with-icon section spans the full pane width
- **Actual behavior:** Links-with-icon compressed into ~50% width, icons smaller, text like "K-Maintenance" wraps mid-word
- **Recommended fix:** In `transformServicePane()`, only include Column B when `$colBBlocks` is non-empty. When no cards exist, emit a single full-width column for the links.
- **Suggested tests:** Pest PHP test asserting service pane migration with no cards produces single-column layout
- **Linear:** `not synced`

### Issue 3: About pane has empty column when no image
- **Status:** `fixed`
- **Fixed in:** `transformAboutPane()` now builds columns dynamically — only includes columns with content, about-us always present
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — migrated dealer sites
- **File:** `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:374`
- **Root cause:** `transformAboutPane()` always creates 3 columns even when Column 2 (image) is empty. Same flex compression effect as Issue 2 — empty columns with `flexBasis: 100%` consume equal space in the flex row.
- **Steps to Reproduce:**
  1. Migrate a dealer site with an About pane that has no image
  2. View the migrated About pane on the frontend
- **Expected behavior:** About pane shows only columns with content — no empty column
- **Actual behavior:** Empty image column takes up space, compressing text columns
- **Recommended fix:** In `transformAboutPane()`, conditionally omit the image column when no image exists. Only include columns that have content.
- **Suggested tests:** Pest PHP test asserting about pane migration without image produces 2-column layout (not 3)
- **Linear:** `not synced`

### Issue 4: "Our Mission" text appears gray instead of configured text color
- **Status:** `fixed`
- **Fixed in:** `MegaMenuAboutUs.tsx` now has `"use client"`, imports `useMegaMenuContext`, applies textColor to Heading and Paragraph via `inlineStyle`
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — migrated dealer sites
- **File:** `frontend/components/mega-menu-about-us/MegaMenuAboutUs.tsx:19`
- **Root cause:** `MegaMenuAboutUs` was built as a stateless server component in THR-1133 (commit `a3b49bf2d8`, PR #1620) without context color integration. Every other mega menu text component (`MegaMenuNavigationList`, `MegaMenuLinkWithIcon`, `MegaMenuFeaturedLinks`) reads `megaMenu?.defaultColors?.textColor` from `useMegaMenuContext()` and applies it as an inline style. `MegaMenuAboutUs` lacks `"use client"`, doesn't import `useMegaMenuContext`, and doesn't apply any text color — so text inherits gray from the CSS cascade instead of the dealer's configured color.
- **Steps to Reproduce:**
  1. Configure a mega menu with text color set to black (or any non-gray color)
  2. Migrate a dealer site with an About pane containing "Our Mission" text
  3. View the About pane on the frontend
- **Expected behavior:** Heading and description text appears in the mega menu's configured text color
- **Actual behavior:** Text appears gray (inherited from CSS cascade)
- **Recommended fix:** Add `"use client"` directive; import `useMegaMenuContext`; read `megaMenu?.defaultColors?.textColor` with `MEGA_MENU_EDITOR_PANE_TEXT_COLOR` fallback; apply via `inlineStyle={{ color: textColor }}` on both `<Heading>` and `<Paragraph>`.
- **Suggested tests:** Storybook story with `MegaMenuProvider` wrapping `MegaMenuAboutUs` and asserting text color matches context value; test without provider to confirm fallback
- **Linear:** `not synced`

### Issue 5: Featured banner text color overridden by resolver default
- **Status:** `fixed`
- **Fixed in:** Resolver skips `buildBlockStyles` when `attrs?.textColor` is absent; component falls back to `megaMenu?.defaultColors?.textColor`
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — migrated dealer sites with non-black mega menu text color
- **File:** `frontend/blocks/mega-menu-featured-banner/resolver.props.ts:17` and `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx:69`
- **Root cause:** Two-layer problem. (1) The featured banner resolver defaults `textColor` to `"black"` when the attribute is absent (`DEFAULT_ATTRIBUTES.textColor`), producing an explicit inline style `color: var(--wp--preset--color--black)` via `buildBlockStyles()`. This overrides the mega menu's context text color. (2) The `MegaMenuFeaturedBanner` component doesn't fall back to `megaMenu?.defaultColors?.textColor` from context — unlike every other mega menu text component. In the legacy system, the banner inherited the mega menu's configured text color via CSS cascade. After migration, it always gets explicit black, regardless of the menu's configured color.
- **Steps to Reproduce:**
  1. Configure a mega menu with white text color (dark theme)
  2. Add a Parts pane with a featured banner
  3. Run migration
  4. View the featured banner on the frontend
- **Expected behavior:** Featured banner text color matches the mega menu's configured text color (e.g., white for dark themes)
- **Actual behavior:** Banner text is always black — the resolver's default overrides the context color
- **Recommended fix:** In `MegaMenuFeaturedBanner`, fall back to `megaMenu?.defaultColors?.textColor` when `styles?.color` is undefined — matching the pattern used by `MegaMenuNavigationList`, `MegaMenuLinkWithIcon`, and `MegaMenuFeaturedLinks`. The component already imports `useMegaMenuContext` (line 8) but only uses `megaMenu?.closeMenu()`.
- **Suggested tests:** Storybook story with `MegaMenuProvider` wrapping `MegaMenuFeaturedBanner` with white textColor context; verify text renders white when no explicit textColor attribute is set
- **Linear:** `not synced`

### Issue 6: Info banner image squished / distorted
- **Status:** `fixed`
- **Fixed in:** `MegaMenuInfoBanner.tsx` uses `image.mediaDetails` dimensions with 400×400 fallback; migration removes `buildNestedColumns` wrapper around info banner
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — migrated dealer sites with info banner in Service pane
- **File:** `frontend/components/info-banner/MegaMenuInfoBanner.tsx:44-53` and `backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php:325-327`
- **Root cause:** Two-part problem. (1) **Component:** The image container uses `w-2/12 flex-shrink-0 flex-grow-0` (16.67% of parent width), and `WithSkeletonImage` receives hardcoded `height={400} width={400}` which forces a 1:1 aspect ratio on the `<img>` element. Non-square banner images are distorted. The actual image dimensions from `IImage.mediaDetails` are available but never used. (2) **Migration:** `transformServicePane()` wraps the info banner in an unnecessary `buildNestedColumns([buildColumnBlock([...])])` (lines 325-327). This single-column-in-a-columns-wrapper adds a `ColumnsBlock` (with `container` class and default margins) that further constrains the banner's available width. The legacy `ServiceNavigationPane` renders the info banner directly in the pane's flex-col container without any column wrapper.
- **Steps to Reproduce:**
  1. Migrate a dealer site with a Service pane containing an info banner with a non-square image
  2. View the migrated Service pane on the frontend
- **Expected behavior:** Banner image displays at its correct aspect ratio, filling the allocated space without distortion
- **Actual behavior:** Image is squished into a square (1:1) within a narrow container
- **Recommended fix:** (1) **Component:** Use `image.mediaDetails?.width` and `image.mediaDetails?.height` instead of hardcoded 400×400. Add `object-contain` class to prevent distortion as fallback. (2) **Migration:** Remove the `buildNestedColumns` wrapper — place `buildInfoBanner($banner)` directly in `$singleColumnBlocks` without the extra columns/column nesting.
- **Suggested tests:** Storybook story with landscape-ratio image in info banner; Pest PHP test asserting service pane migration with banner produces info-banner block as direct child (no columns wrapper)
- **Linear:** `not synced`
