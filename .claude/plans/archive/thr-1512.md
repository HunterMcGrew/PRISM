# Plan: THR-1512

## Ticket

https://linear.app/tractru/issue/THR-1512/panel-hero-editors-replace-textcontrol-links-with-linkcontrol

## Goal

Replace manual TextControl/ToggleControl link inputs in Panel Hero editor blocks with the shared LinkControl component for consistent editor UX and built-in link validation.

---

## User Stories

- As a content author, I want the Panel Hero link editor to use the same link control as other blocks, so that I have a consistent editing experience across the site.

---

## Decisions

- **Adapter pattern over schema migration** — panel-hero keeps flat string attributes (`ctaLink`, `ctaTarget`, `ctaAlt`, `ctaRel`). The edit handler converts between flat attrs and `LinkValue`. Avoids breaking existing content and frontend changes.
- **`ctaAlt` maps to LinkControl's `label` field** — both serve the same purpose (accessible link description). No rename needed in the schema.
- **`ctaRel` editing is a free addition** — LinkControl's advanced settings expose rel editing that the panel-hero blocks didn't previously offer. Schema already supports `ctaRel`.
- **No frontend changes** — schema, resolvers, and view components are untouched. This is purely an editor-side swap.
- **`undefined` target/alt is safe** — frontend components already default `ctaTarget` via `|| "_self"` / `?? "_self"` and `ctaAlt` via `|| "Hero Main Panel"` / `|| "Hero inner panel"`. No normalization needed in the adapter.
- **`isLinkTarget` type guard over `as` cast** — `ctaTarget` from WordPress attrs is typed `string`. Use `isLinkTarget()` from `@frontend/lib/type-guards/link` to narrow at runtime instead of `as LinkTarget | undefined`.
- **Auto-external URL detection is intentional** — LinkControl auto-sets `_blank` for URLs starting with `http`. Users can override via the advanced settings toggle. Call out in PR description as a behavior change.
- **`noopener noreferrer` auto-set on `_blank` is desired** — `setLinkTarget` enforces this for security (prevents reverse tabnapping). This is new behavior vs the old manual toggle but is the correct default.

## Acceptance Criteria

### Behavioral
- [ ] Given a Panel Hero Main Panel block in the editor, When I open the sidebar "Panel Link Settings", Then I see the LinkControl component with a URL input and expandable advanced settings (open-in-new-tab toggle, label, link rel)
- [ ] Given a Panel Hero Inner Panel block in the editor, When I open the sidebar "Panel Link Settings", Then I see the same LinkControl component
- [ ] Given I enter a URL in the LinkControl, When I save the block, Then the URL is stored and renders correctly on the frontend
- [ ] Given I enter an external URL (e.g. https://google.com), When I save, Then "Open in new tab" is automatically enabled
- [ ] Given I expand advanced settings and set a label, When I view the frontend, Then the link's aria-label reflects the label text
- [ ] Given I expand advanced settings and set a link rel value, When I save, Then the rel attribute is rendered on the frontend link
- [ ] Given a Panel Hero block with existing link data saved before this change, When I open it in the editor, Then the existing URL, target, and alt text appear correctly in the LinkControl
- [ ] Given the help text "Note: This URL replaces any links set in child blocks.", Then it remains visible in the Panel Link Settings section

### Non-behavioral
- [ ] No changes to frontend schema, resolvers, or view components
- [ ] No WordPress data migration required — existing block content loads without modification

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `panel-hero-main-panel/edit.tsx:332` and `panel-hero-inner-panel/edit.tsx:304`
- **Problem:** `ctaAlt`, `ctaLink`, `ctaTarget` are destructured at the top of the component, but `ctaRel` is accessed as `attributes.ctaRel` in the JSX link object. Inconsistent access pattern.
- **Suggested fix:** Add `ctaRel` to the destructuring: `const { ctaAlt, ctaLink, ctaRel, ctaTarget } = attributes;` and use `ctaRel` directly in the link object.
