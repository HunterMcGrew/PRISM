# Plan: THR-1472

## Ticket

https://linear.app/tractru/issue/THR-1472/feature-block-cta-fix-external-link-normalization-and-form-modal

## Goal

Fix two QA-reported bugs in the Feature block CTA: external links without a protocol render as relative paths, and the form modal flickers a loading spinner before showing the correct form.

---

## Decisions

- Reverses THR-773 decision "no resolver needed" — a resolver is now required to fetch form data server-side and normalize legacy URLs
- Form data fetched server-side in resolver when `ctaType === "form"` — matches `FormCTAButton` pattern, eliminates client-side fetch flicker in `SharedModals`
- `formId` still passed alongside `form` in `openModal()` call — preserves `SharedModals` client-side fallback if server fetch fails
- `normalizeExternalUrl` applied in resolver (not component) for legacy data — component stays dumb, resolver handles data transformation
- Editor swaps bare `TextControl` for `ControlledUrlInput` on external URL field — normalizes on save, provides page suggestions. Eliminates need for manual `normalizeExternalUrl` in editor handler
- Internal/external toggle and page Select kept as-is — replacing them with `ControlledUrlInput`'s built-in detection is a bigger UX change, out of scope

---

## Implementation Tasks

### Task 1: Add resolver with form fetch and URL normalization
- Create `frontend/blocks/feature/resolver.ts`
- Import `"server-only"`, implement `resolveProps`
- When `ctaType === "form"` and `formId` is set, fetch the form via the form service (follow pattern from other resolvers that fetch GravityForms)
- When `isExternal` is true, normalize `ctaLink` via `normalizeExternalUrl()`
- Pass all other attributes through with defaults merged
- Register resolver in `block-registry.ts` for the feature block

### Task 2: Wire form prop through FeatureBlock → FeatureModalCta
- Add `form?: GfForm` to `FeatureBlockProps`
- Pass `form` from `FeatureBlock` → `FeatureModalCta`
- Add `form?: GfForm` to `FeatureModalCtaProps`
- Update `openModal()` call: `openModal({ type: "form", form, formId })` — form for instant render, formId as fallback
- Depends on: Task 1

### Task 3: Swap editor external URL input to ControlledUrlInput
- In `edit.tsx`, replace the `TextControl` for external URL with `ControlledUrlInput`
- Update `onChangeCtaLinkField` — `ControlledUrlInput` returns a normalized URL, so the handler just needs to call `setAttributes({ ctaLink: url, selectedPageLink: "" })`
- Keep the `ToggleControl` (internal/external) and `Select` (page picker) unchanged

### Task 4: Tests

#### Resolver tests (`frontend/blocks/feature/__tests__/resolver.test.ts`)

**URL normalization (isExternal path):**
- Bare domain (`google.com`) → normalized to `https://google.com`
- Protocol-relative (`//example.com`) → normalized to `https://example.com`
- Full URL (`https://google.com`) → passed through unchanged
- Domain with path (`google.com/search?q=test`) → normalized to `https://google.com/search?q=test`
- Empty `ctaLink` with `isExternal: true` → left as empty string (no crash)

**URL normalization (internal path):**
- `isExternal: false` → `ctaLink` NOT normalized (left as-is, internal URLs use `selectedPageLink`)

**Form fetch:**
- `ctaType: "form"` with `formId: 42` → calls `fetchForm(wpUrl, 42)`, returns form in props
- `ctaType: "form"` with no `formId` → returns `form: null` (no fetch attempted)
- `ctaType: "form"` with `formId` but `fetchForm` returns null (fetch fails) → returns `form: null` gracefully
- `ctaType: "link"` with `formId: 42` → does NOT fetch form (unnecessary work)
- `ctaType: "call"` → does NOT fetch form

**Attribute passthrough:**
- All other attributes merge with `DEFAULT_ATTRIBUTES` and pass through unchanged
- `attrs` is partial (WordPress omits defaults) → defaults fill in missing fields

#### FeatureModalCta test updates (`frontend/blocks/feature/__tests__/FeatureModalCta.test.tsx`)

- Form CTA with `form` prop → `openModal` called with `{ type: "form", form, formId }` (form object passed through)
- Form CTA with `form: undefined` → `openModal` called with `{ type: "form", form: undefined, formId }` (fallback to client fetch)
- Call CTA → unchanged behavior, `form` prop ignored

#### FeatureBlock test updates (`frontend/blocks/feature/__tests__/FeatureBlock.test.tsx`)

- Update mock for `FeatureModalCta` to accept `form` prop and render it as `data-form` attribute
- Form CTA with `form` prop → mock receives form object
- Existing tests must continue to pass with no changes (backwards compatibility)

#### Edge cases to verify manually

- Existing Feature blocks saved before this change render correctly without re-saving (backwards compatibility — resolver handles legacy data)
- Feature block with `ctaType: "form"` and valid `formId` → clicking CTA opens form immediately (no spinner)
- Feature block with `ctaType: "form"` and invalid `formId` → modal falls back to client-side fetch via `SharedModals` (graceful degradation)
- Feature block with `ctaType: "link"`, `isExternal: true`, and a bare domain → link navigates to the correct external URL
