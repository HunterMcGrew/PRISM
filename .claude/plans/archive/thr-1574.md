# Plan: THR-1574

## Ticket

https://linear.app/tractru/issue/THR-1574/sortablelinksmodal-unsaved-changes-persist-after-closing-without

## Goal

Add working-copy state management and Cancel/Save UX to the SortableLinksModal so unsaved changes are discarded on close and committed only on explicit Save.

---

## User Stories

Not applicable — this is a bug ticket.

---

## Decisions

- **Modal owns the working copy** — the transient editing state is a UI concern (per backend-editor-components.md). The modal initializes local state from the `links` prop on mount. All mutations operate on local state. The parent only receives committed data via `onSave`. Component unmount on close naturally discards the working copy — no explicit revert logic needed.
- **Cancel = discard, Save = commit, X/Escape = Cancel** — follows standard modal conventions. The current "instant commit" behavior is the bug being fixed.
- **Add Link moves to an action bar above the scrollable list** — separates content actions (Add Link) from modal lifecycle actions (Cancel/Save). No design mock exists; layout follows standard list-editor conventions and the `EquipmentLinksModal` precedent.
- **Action bar and footer are outside the scrollable area** — the scrollable div only wraps the link list. Add Link (top) and Cancel/Save (bottom) are always visible regardless of scroll position.
- **BottomButtonsBar replaced with separate ActionBar and ModalFooter subcomponents** — clearer separation of concerns in link-modal-parts.tsx.
- **Button variants: Save=primary, Cancel=tertiary, Add Link=secondary** — visual hierarchy guides the user toward the commitment action. Save is the strongest CTA.
- **Edit-mode buttons: Done (primary) + Cancel (tertiary)** — Done commits the edit-level changes to the working copy and collapses the form. Cancel discards edit-level changes: for existing links, reverts to the pre-edit snapshot; for new links (just added), removes the link entirely from the working copy. Replaces the previous "Remove" button — the semantic shift is from "delete this link" to "discard this edit session."
- **Two-level working copy** — the modal working copy is the scratch pad for the Save/Cancel lifecycle. Within that, each edit session has its own snapshot. When editing begins, the current link state is captured in `editingSnapshot`. Done clears the snapshot (changes stay). Cancel restores the snapshot (changes discarded). This keeps `workingLinks` as the single source of truth throughout.
- **Clicking away from edit without Done = Cancel** — if the user leaves an edit session by any means other than Done (clicking edit on another link, or any future UI path that closes the edit form), the edit is discarded and the snapshot is restored. Explicit commitment via Done is required.
- **Add Link button gets `font-semibold`** — visual weight increase to make it stand out from surrounding content.
- **Drag handle moves to the left of the link label** — standard sortable list convention. Positional controls (drag) on the leading edge, action controls (edit, trash) on the trailing edge. Creates three clear zones: grab → read → act.
- **Focus returns to trigger on edit form close** — when Done or Cancel closes the edit form, focus moves back to the ✎ button that opened it (or Add Link if the link was removed). Prevents keyboard users from losing their place.
- **Overlay click blocked when unsaved changes exist** — uses WordPress Modal's `shouldCloseOnClickOutside` prop set dynamically based on whether the working copy differs from the initial snapshot. No confirmation dialog needed — Escape, X, and Cancel remain available as deliberate exit paths.
- **Props API simplifies from 11 to 5** — `title`, `links`, `listClassName`, `onSave`, `onCancel`. The modal manages `workingLinks`, `editingId`, and all mutation handlers internally.
- **Empty state shows a message when the list is empty** — "No links added yet." displayed in the list area with Add Link still available above.

## Acceptance Criteria

### Behavioral

- [ ] Given the Sortable Links Modal is open, When the user adds, removes, reorders, or edits links and clicks Cancel, Then the links revert to the state they were in when the modal was opened
- [ ] Given the Sortable Links Modal is open, When the user makes changes and clicks Save, Then the changes are committed to the block
- [ ] Given the Sortable Links Modal is open with unsaved changes, When the user clicks the X button or presses Escape, Then the changes are discarded and the modal closes
- [ ] Given the Sortable Links Modal is open, When the user clicks outside the modal, Then the modal remains open (overlay click is always blocked)
- [ ] Given the user is editing an individual link and clicks Done, When they have not yet clicked Save, Then the edit is visible in the modal list but not yet saved to the block
- [ ] Given the user clicks Add Link and then clicks Cancel in the edit form, Then the new link is removed from the list entirely as if it was never added
- [ ] Given the user is editing an existing link and clicks Cancel in the edit form, Then the link reverts to the values it had before the edit form was opened
- [ ] Given the user is editing a link and clicks the edit button on a different link without clicking Done, Then the first link's changes are discarded and the second link's edit form opens
- [ ] Given the Sortable Links Modal has no links, When the modal is open, Then an empty state message is displayed and the Add Link button is still available
- [ ] Given the user clicks Add Link, When a new link is added, Then the new link appears in the list and its edit form opens automatically

- [ ] Given the Sortable Links Modal has unsaved changes, When the user views the footer, Then an "Unsaved changes" indicator with an amber dot is visible next to the Cancel/Save buttons
- [ ] Given the user removes a link from the list, When the link is removed, Then an undo bar appears showing the removed link's name with an Undo button
- [ ] Given the undo bar is visible, When the user clicks Undo, Then the removed link is restored to its original position in the list and the undo bar disappears
- [ ] Given the user removes a second link while the undo bar is visible, When the second link is removed, Then the undo bar updates to show the newly removed link

### Non-behavioral

- [ ] After removing a link, keyboard focus moves to the Undo button
- [ ] After clicking Undo, keyboard focus moves to the restored link's edit button
- [ ] The Cancel/Save button pattern applies to both the Navigation List and Featured Links blocks
- [ ] The Add Link button is positioned above the scrollable list, not in the footer
- [ ] The Add Link button has visibly bolder font weight than surrounding text
- [ ] The Add Link button and Cancel/Save footer remain visible and do not scroll with the list
- [ ] Cancel and Save buttons are reachable via keyboard (Tab) and activatable via Enter/Space
- [ ] Pressing Escape closes the modal and discards changes
- [ ] The edit-form Cancel button is labeled "Cancel" (not "Remove")
- [ ] The drag handle is positioned to the left of the link label, with edit and delete actions on the right
- [ ] After closing an edit form (Done or Cancel), keyboard focus returns to the button that opened it

### Summary
Three UX polish items for the SortableLinksModal: (A) empty state copy updated to point users toward Add Link, (D) dirty indicator in the footer showing "Unsaved changes" when the working copy diverges + overlay click always blocked, (E) inline undo bar after trash with one-level undo stack and focus management. All changes reuse existing `@wordpress/components` primitives. One new subcomponent (`UndoBar`) in `link-modal-parts.tsx`.

### Decisions worth knowing
- Always block overlay click (`shouldCloseOnClickOutside={false}`) — removes the dynamic toggle. Cancel, X, and Escape are the only exit paths. Matches Notion/Linear form-commit modal convention.
- Dirty indicator uses amber dot + muted text — signals attention without competing with Save's primary blue.
- Undo is one level only — trashing a second link replaces the undo for the first. No undo stack needed at this list scale.
- Focus after trash moves to Undo button. Focus after undo moves to restored link's edit button via existing `pendingFocusLinkId` pattern.
- `lastRemoved` clears on: Undo, next trash, Save, Cancel. Does NOT clear on Add Link, edit, Done, or reorder — undo persists through normal editing actions.
