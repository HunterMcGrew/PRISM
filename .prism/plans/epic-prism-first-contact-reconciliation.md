# Plan: epic-prism-first-contact-reconciliation

> Closed: 2026-06-16

> **REPLANNED (2026-06-16) against the `prism:update` primitives.** The original design reused Epic B's three-way content-hash diff engine and `--dry-run` three-bucket preview. Epic B was superseded and deleted (ADR-0057); the `prism:update` epic shipped `.sync-manifest.json` + `classifyPath` + overwrite/`.bak` in its place. This plan is now re-derived against those primitives and is build-ready — no longer deferred, no longer blocked. The `NEEDS REPLAN` and `DEFERRED` banners are retired.

## Ticket

No Linear ticket. Solves the hardest distribution case: adopting PRISM into a repo that **already has its own setup** (Thrive today; SPC in the future).

## Goal

Let an established team (Thrive, SPC) adopt PRISM **without losing the work they already have** — by giving Atlas a first-contact mode that seeds PRISM's content, runs an initial sync to write the steady-state baseline manifest, and discovers-and-adopts the team's pre-existing skills, architect docs, ADRs, rules, and docs.

## The core problem, re-derived against the new primitives

The original plan feared that a first sync into an established repo with **no baseline `.prism-state.json`** would be a catastrophe: every file would look brand-new or hand-edited, with nothing to diff against. That fear was scoped to Epic B's three-way merge, which genuinely needed a stored base per file.

**The `prism:update` model dissolves most of that fear.** Read `applyIncomingFile` (`scripts/ai-skills/update.ts:130–162`) and ADR-0057 § Decision: the equality check (`consumerHash === incomingHash` → no-op) runs **before** the divergence `.bak` branch. ADR-0057 names this exact case "the no-manifest path" — a pre-manifest install with no recorded base hash still no-ops byte-identical files instead of `.bak`-ing every file. After one pass, `rewriteConsumerManifest` (`update.ts:268`) writes the baseline manifest. **Steady-state needs no hand-authored bootstrap manifest — the first sync synthesizes it as a byproduct.**

So "bootstrap the baseline" is not a diff-engine problem. It collapses into two genuinely-unsolved pieces:

1. **Seed `.prism/` and run the first sync.** `prism:update` (`update.ts:main`) assumes the consumer already has a `.prism/` content root, a `paths.json`, and a `config.json`. An established repo that has never had PRISM has none of these. There is **no consumer-side install/adopt entry point today** — `templates/install/.prism/` is a seed *surface*, but nothing scripts laying it down. First-contact must seed `.prism/` from the install surface, then let the first sync + `rewriteConsumerManifest` establish the baseline manifest. After that, `prism:update` is steady-state forever.

2. **Discover and adopt the team's pre-existing assets.** The team's own `AGENTS.md`, `.claude/` skills, their `docs/`, their own architect docs / ADRs / rules live **outside** `classifyPath`'s `.prism/`-relative namespace (`ownership.ts`). The update flow never touches them — which means it never *adopts* them either. This is the real first-contact gap and exactly what the discovery-flow forward-idea (Hunter, 2026-06-15) addresses: discover them, classify them, and either adopt into canonical source (via `prism:migrate-skill` for skills, standards-routing for rules/docs) or leave-and-record.

**This is an onboarding mode (Atlas), not a sync mode** — a walked, resumable conversation, resumable via `.ai-skills/registry/onboarding-state.json` (`onboarding-state.ts`). No second diff/merge engine is built. Where first-contact needs something the `prism:update` primitives don't provide, the task **extends the existing primitive**, never forks a parallel one.

## Decisions

- **First-contact rides the `prism:update` merge model — no second diff/merge engine.** The overwrite/`.bak` model (ADR-0057) plus the no-op-before-`.bak` ordering means a first sync into an established repo is safe with no baseline manifest: byte-identical files no-op, diverged files are preserved as `.bak`, and `rewriteConsumerManifest` writes the baseline as a byproduct. **Why:** the original Epic-B-based fear (every file looks brand-new with nothing to diff) was scoped to the three-way merge, which is gone; the shipped model degrades gracefully without a manifest by design. Alternatives considered: (a) hand-author a baseline manifest before the first sync — rejected, redundant with `rewriteConsumerManifest`; (b) build a first-contact-specific preview UI mirroring Epic B's three buckets — rejected, the update flow's `reportSummary` already names every `written`/`backed-up` outcome, and a parallel preview is a second engine. → promoted to ADR-0059 (task 11).
- **First-contact needs a new install entry (`prism:adopt`) because `prism:update` assumes `.prism/` already exists.** `update.ts:main` reads the consumer's `paths.json` and `config.json` and operates on an existing `.prism/` content root — none of which an established never-onboarded repo has. `prism:adopt` seeds `.prism/` from `templates/install/.prism/`, then delegates to `runUpdate` for the merge + baseline. **Why:** this extends the existing primitive (`runUpdate` is reused, not forked) rather than building a parallel engine, honoring the "no second diff/merge implementation" constraint. Alternatives considered: fold seeding into `prism:update` behind a `--first-contact` flag — rejected, it muddies the steady-state refusal logic and the source==consumer guard; a separate entry keeps each flow's preconditions clean. → promoted to ADR-0059 (task 11).
- **Discovery sweep generalizes `prism:migrate-skill`, it does not re-implement it.** For skills, Atlas invokes the existing `pnpm prism:migrate-skill <path>` CLI, which already handles all four generated shapes and normalizes the ID to the org namespace (`normalizeSkillId`). For rules/docs/ADRs, the sweep routes via the existing rule-placement test (`.prism/SPEC.md`) and `classifyPath` carve-outs. **Why:** the migrate machinery exists and is tested; the discovery flow's job is orchestration and the adopt/leave decision, not decomposition. Alternatives considered: a bespoke bulk-discovery script — rejected, it would duplicate `migrate-skill.ts` and drift from it. → promoted to ADR-0059 (Decision (b) states the sweep generalizes `prism:migrate-skill` and the rule-placement test, never re-implementing decomposition). install-layout.md § first-contact carries the seed-and-sync half of the flow; the discovery-sweep design lives in the ADR alongside the repo-agnostic principle, since it is a design decision rather than an install-layout mechanic.
- **The established team's own architect docs / ADRs / docs stay consumer-owned by default — the sweep confirms-and-records, and migrates only on explicit user opt-in.** Flat `.prism/architect/*.md` and `.prism/spec/adrs/*.md` are already `consumer` per `CONSUMER_OWNED_GLOBS`; `prism:update` never touches them. The docs layout is recorded into the `documentation` config block via the existing `detectDocLayout`. **Design call (added in the 2026-06-16 amendment):** the survey now also surfaces architect-docs/ADRs found at non-standard user-supplied paths. The default for all of these stays *confirm consumer-owned, no file move* — but if the user explicitly asks for a doc/ADR to be PRISM-managed, the sweep offers to migrate it into the PRISM-owned `_toolkit/` tree. Default is leave; migration is opt-in only. **Why:** ADR-0058 retired the paired-dev-doc mandate — PRISM no longer dictates doc shape, so adopting a team's docs means *recording* their layout in config, not rewriting their files; the opt-in migration path exists because a generic, repo-agnostic mode must let a team that *wants* PRISM-managed docs say so, rather than hard-coding "consumer-owned" as the only outcome. Alternatives considered: hard-lock all consumer architect-docs/ADRs as never-migratable — rejected, it removes a legitimate user choice and contradicts the fully-generic design. → no promotion needed (codified by `classifyPath` + ADR-0058; the sweep is a confirm-or-opt-in step, not a new rule).
- **No `--dry-run` preview for `prism:adopt` — recover-after-`.bak` is the safety model, not see-before-write.** Briar's self-review surfaced that plan task 3's verification promised `pnpm prism:adopt --prism-source <path> --dry-run` would preview writes, but neither `adopt.ts` nor `update.ts` implements the flag — it was a vestige of the deleted Epic B three-bucket-preview design.
  - **Root cause:** the dry-run line survived the 2026-06-16 replan as a stray reference to the old preview UI; the replanned design already chose ADR-0057's recover-after-write model and Decision #1 above explicitly rejected a first-contact preview as "a second engine." The verification line contradicted the plan's own design.
  - **Alternatives considered:** (A) implement a real `--dry-run` — thread a `dryRun` flag through `runUpdate → applyIncomingFile / applyDeletedFile / writeIncoming / backupConsumerFile / rewriteConsumerManifest` and a parallel one through `seedConsumerContentRoot`, collecting would-write outcomes and printing without touching disk. (B) retract the claim and verify against a throwaway fixture.
  - **Chosen approach:** (B) retract. Beats (A) because the safety a preview would buy is already structural and load-bearing: seed **never overwrites** an existing consumer file (`consumerHash === null → written`, skip otherwise — `adopt.ts:seedConsumerContentRoot`), and the sync pass only touches PRISM-owned paths, no-ops byte-identical files, and `.bak`-snapshots every divergence with a non-clobbering name the run summary prints (`update.ts:applyIncomingFile`, ADR-0057). The first-contact catastrophe ("first run eats my existing setup") cannot occur, so a preview duplicates `reportSummary`'s after-the-fact account of a fully-reversible run. (A) also adds net-new conditional behavior to `update.ts` — the steady-state merge core ADR-0057 froze — doubling every write site into a would-write/write branch on PRISM's most safety-critical path, contradicting Decision #1's "no second engine" rejection.
  - **Implementation guidance:** task 3 verification is now a throwaway-fixture run asserting the `AdoptSummary` and the `.sync-manifest.json` baseline. No code change to add `--dry-run`. Clove's Lane-1 fix pass only updates the plan-promised verification behavior (already amended) — `adopt.ts`/`update.ts` need no dry-run handling.
  - **→ promoted to ADR-0059 (task 11)** — fold into ADR-0059's Decision section alongside Decision #1; the "recover-after-`.bak`, no preview engine" posture is the same design principle and belongs in the same durable record.
- **First-contact mode is permanently, fully repo-agnostic — SPC is treated exactly like Thrive or any third-party repo, with zero team-specific code.** First-contact detects and adopts whatever an established repo already has (its skills, architect docs, ADRs, rules, docs — in standard *or* non-standard locations) through one generic path: survey the user about each asset class and its paths → run the discovery sweep (auto-detection cross-checked against the user's stated paths) → construct everything correctly into canonical PRISM structure (adopt) or record as consumer-owned (leave). The entire flow is driven by detection + `config.json` only; there is never a branch that keys on "is this SPC." SPC's existing setup, doc format, and asset locations ride the same channel as Thrive's and any other repo's. **Why:** Hunter ruled "we will treat SPC just like Thrive or some other random repo" (History 2026-06-16) — first-contact is generic by design, not generic-pending-a-future-special-case. A repo-specific branch is the failure mode this decision permanently forecloses. Alternatives considered: keep the question open until SPC onboards and special-case its shape then — rejected, it invites a team-specific code path the agnostic design is built to prevent; the survey + discovery + seed-and-sync flow already handles arbitrary established repos, so there is nothing left to resolve. **This resolves the prior OPEN decision** (was: "OPEN — TBD, needs Hunter + SPC input," see History 2026-06-16). → promoted to ADR-0059's Decision section (the agnostic principle is design-defining; folded in task 11).
