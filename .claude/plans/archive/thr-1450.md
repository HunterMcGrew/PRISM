# Plan: THR-1450

## Ticket

Mega Menu Migration Tool — migrates legacy `gravity-platform-core/mega-menu` block data into the new Gutenberg-based `mega-menu-editor` block structure.

## Goal

Extract the mega menu migration tool from the 592-commit `hmcgrew/thr-1450-mega-menu-migration-tool` branch onto a clean branch off `main`, producing a single squash-mergeable PR with only migration-related changes.

---

## Decisions

- The old branch (`hmcgrew/thr-1450-mega-menu-migration-tool`) is the **source of truth for migration code**. All migration-related file content comes from there.
- **New branch name:** `hmcgrew/thr-1450-mega-menu-migration-clean` — fresh branch off current `main`.
- **Extraction method:** `git checkout` for new files, `git diff ... | git apply` for modified files. No cherry-picking commits.
- Non-migration collateral changes are **dropped** (SlidingPanel, url-utilities, MegaMenuEditor.php cosmetic changes, admin/index.tsx blank line).
- The `gravity/v1` → `thrive/v1` namespace rename is **in scope** because the migration API routes use `thrive/v1` and all routes in MegaMenuBlocksApi.php must be consistent. The 2 frontend consumers (`resolver.props.ts`, `page.tsx`) must match.
- `esc_url`/`esc_attr` Brain\Monkey stubs belong in individual migration test files that need them, NOT in global `Pest.php` — per PR review feedback from jmotes.
- `MegaMenuEditorTest.php` gets 2 comment wording updates (cosmetic but part of the branch diff — keeps tests consistent with the API constant visibility change).
- **Sanitize callback test changes** in `MegaMenuBlocksApiTest.php` are included because they test the `wp_kses_post` comment/behavior added to `MegaMenuBlocksApi.php`.
- `WP_REST_Response::header()` stub is added to the test file because `getMegaMenuBlocks()` now calls `$response->header('Referrer-Policy', 'no-referrer')`.
- **Option key prefix is `platform_settings`**, not `gravity_platform` — aligns with the established convention in `thrive-sss`. Safe to rename without data migration (mega menu editor is staging-only as of 2026-04-10).
- **Legacy block name** (`gravity-platform-core/mega-menu`) must be a `public const` on `MegaMenuMigrationService` — single source of truth for the string used in finder logic and test helpers.
- **`PUBLISHED_OPTION_KEY` and `DRAFT_OPTION_KEY` stay `public`** — tests reference these constants to avoid string drift. Other production classes should use the public static methods (`getDraftSerializedBlocks()`, etc.) rather than reading options directly. Architect context documents this convention.
