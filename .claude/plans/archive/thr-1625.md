# Plan: THR-1625

## Ticket

https://linear.app/tractru/issue/THR-1625/hide-dont-show-again-checkbox-on-manually-launched-legacy-import-modal

## Goal

Hide the "Don't show again" checkbox when the legacy-import modal is launched manually via the "Import from Legacy Menu" header button — keep it visible only on the auto-open path where it's meaningful.

---

## Decisions

- Replace `isMigrateModalOpen: boolean` with `migrateModalTrigger: "auto" | "manual" | null`. The boolean conflated two entry points with different UX contracts — the new shape encodes intent in the type system and prevents a sibling-flag sprawl if a third entry point is ever added.
- The `<CheckboxControl>` renders only when `trigger === "auto"`. On manual, the checkbox is absent from the DOM (not disabled or hidden). Reason: the user didn't invoke the auto-open flow, so the escape hatch for that flow has no context.
- ~~`MigrateModal` keeps its `open: boolean` prop. The `null` case is collapsed to `open={false}` at the parent boundary in Header.~~ **Retired 2026-04-18 (Eric review round 2):** In practice, the separate `open` prop required a `trigger={... ?? "auto"}` fallback that was never read (modal returned early on `!open`). Two synced props + a dead fallback is not simpler than one nullable prop. `MigrateModal.trigger` is now `MigrateModalTrigger | null` — null means closed, non-null means open. The modal early-returns on `!trigger`.
- Naming: use `trigger` consistently — not `mode`, `source`, or `openedBy`.

---

## Implementation Tasks

### Clove (implementation)

1. **Replace open-state with trigger union in `editor.tsx`**
   - Swap `useState<boolean>(false)` for `useState<"auto" | "manual" | null>(null)`
   - Rename `isMigrateModalOpen` → `migrateModalTrigger`, `setIsMigrateModalOpen` → `setMigrateModalTrigger`
   - Auto-open `useEffect` (~line 288): `setMigrateModalTrigger("auto")` when condition met
   - `handleMigrate` success close (~line 330): `setMigrateModalTrigger(null)`
   - Any other close paths: set to `null`
   - Pass the new state and setter down to `<Header>` in place of the old pair

2. **Update `Header.tsx` prop surface**
   - Replace `isMigrateModalOpen: boolean` + `setIsMigrateModalOpen: Dispatch<SetStateAction<boolean>>` props with `migrateModalTrigger: "auto" | "manual" | null` + `setMigrateModalTrigger: Dispatch<SetStateAction<"auto" | "manual" | null>>`
   - `handleOpenMigrateModal` → `setMigrateModalTrigger("manual")`
   - `handleCloseMigrateModal` → `setMigrateModalTrigger(null)`
   - At the `<MigrateModal>` render site, pass `open={migrateModalTrigger !== null}` and a new `trigger={migrateModalTrigger ?? "auto"}` prop (the fallback is unreachable because the modal returns `null` when `open` is false — but TS needs a concrete value)

3. **Update `MigrateModal.tsx`**
   - Add to `MigrateModalProps`: `trigger: "auto" | "manual"`
   - Destructure `trigger` in the component
   - Wrap the entire `<div className="thrive-migrate-modal__checkbox">...</div>` block (lines 92–100) in `{trigger === "auto" && ( … )}`
   - No other changes

4. **Type-check and smoke-test**
   - `cd backend/plugins/gravity-platform-core && pnpm run check-types`
   - Manual smoke:
     - Load editor with legacy menu, no dismissal → modal auto-opens, checkbox visible
     - Check "Don't show again" + close → reload editor → modal does not auto-open
     - Click "Import from Legacy Menu" button → modal opens, checkbox NOT rendered
     - Cancel / X / successful import all close cleanly in both entry points

### Eli (documentation)

None. No user-facing behavior change to the auto-open flow; the manual-launch checkbox was an unintended surface. No existing docs reference the checkbox behavior on the manual path.

## Acceptance Criteria

### Behavioral
- [ ] Given an admin opens the mega menu editor with a legacy menu present and no prior dismissal, When the editor loads, Then the modal opens automatically and the "Don't show again" checkbox is visible and functional
- [ ] Given the modal is open via auto-open, When the admin checks "Don't show again" and closes the modal, Then the preference persists and the modal does not auto-open on the next editor load
- [ ] Given the admin clicks the "Import from Legacy Menu" button in the editor header, When the modal opens, Then the "Don't show again" checkbox is not rendered anywhere in the modal
- [ ] Given the admin previously dismissed the modal via checkbox, When they manually open the modal and close it (Cancel, X, or after Import), Then their dismissal preference is unchanged
- [ ] Given the modal is open via manual launch, When the admin tabs through the modal, Then tab order flows body → Cancel → Import (no checkbox stop)

### Non-behavioral
- [ ] The modal's visual layout (body text, warning notice, spacing, X alignment) is identical between auto and manual entry points, except for the checkbox
- [ ] Error state ("Try Again" / "Close") is unaffected on either entry point
- [ ] Successful migration success toast still fires regardless of entry point
