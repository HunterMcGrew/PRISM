# Plan: THR-952

## Ticket

https://linear.app/tractru/issue/THR-952/panel-hero-youtube-and-vimeo-videos-showing-info

## Goal

Fix YouTube and Vimeo video embeds in the Panel Hero block so they fill the container at all viewports (no black bars) and provide native mute/unmute controls on hover.

---

## Decisions

- Scope limited to `panel-hero` block only — deprecated `hero-three-panels` is not touched
- Secondary panels are image-only — no video changes needed in `MediaItem` or `SecondaryInnerPanelBlock`
- Video sizing fix uses a `fillContainer` prop on `ThirdPartyVideoPlayer` — but `object-fit: cover` does NOT work on iframes; needs oversized-iframe + crop approach. Standalone `thrive-video` block remains unaffected
- Flash fix: use CSS container query units (`cqw`/`cqh`) instead of ResizeObserver for iframe cover sizing — eliminates client JS, flash on initial render, and hydration timing bugs. `FillContainerIframe` becomes an RSC. Parent wrapper gets `container-type: size`, iframe uses `width: max(100cqw, calc(100cqh * 16 / 9))` / `height: max(100cqh, calc(100cqw * 9 / 16))`
- `ThirdPartyVideoPlayer` must use the `toAllow` prop instead of hardcoding the `allow` attribute — existing bug where the prop is passed but ignored
- No `background=1` for Vimeo — it disables all controls including unmute, which conflicts with the requirement
- Vimeo uses explicit params: `autoplay=1&muted=1&loop=1&title=0&byline=0&portrait=0&controls=1` — clean look with native controls on hover
- YouTube uses explicit params: `autoplay=1&mute=1&loop=1&controls=1&playlist={id}` — native controls on hover
- No mute/unmute controls needed — overlay intentionally blocks interaction (visual "light up" effect only), so `controls={false}` for cleaner look
- Work within the existing three-component structure (`LocalVideo`, `YoutubeVideo`, `VimeoVideo`) — each platform has different embed requirements

## Debugged Issues

### object-fit: cover has no effect on iframes (YouTube/Vimeo)
- **Status:** `fixed`
- **Fixed in:** `FillContainerIframe.tsx` — RSC-first "use client" leaf with ResizeObserver that oversizes the iframe to cover the container at 16:9, centered with translate(-50%, -50%), cropped by parent overflow-hidden
- **File:** `frontend/components/thrive-video/ThirdPartyVideoPlayer.tsx:36`
- **Root cause:** CSS `object-fit` is ignored on `<iframe>` elements — it only applies to replaced elements like `<img>` and `<video>`. YouTube/Vimeo enforce 16:9 inside their player, so the iframe stretches to fill the container but the video content letterboxes with black bars when the container is not 16:9.
- **Recommended fix:** Replace the `object-cover` approach with the "oversized iframe" technique. When `fillContainer` is true, `ThirdPartyVideoPlayer` should become a `"use client"` wrapper that: (1) uses a ResizeObserver on the container to read its dimensions, (2) calculates the minimum iframe size to cover the container at 16:9 — `width = max(containerW, containerH * 16/9)`, `height = max(containerH, containerW * 9/16)`, (3) centers the iframe with `top: 50%; left: 50%; transform: translate(-50%, -50%)`, and (4) relies on the parent's existing `overflow-hidden` to crop. Alternatively, a CSS-only approach using `min-width: max(100%, calc(100% * 16/9 / var(--aspect)))` could work if container queries are acceptable.
- **Suggested tests:** Visual regression test or Storybook story showing the panel hero with a YouTube/Vimeo video at various container aspect ratios (portrait, landscape, square). Unit test confirming the ResizeObserver sets correct dimensions.

### Hover overlay blocks pointer events to iframe video controls
- **Status:** `fixed` → `deferred` (out of scope — overlay is intentional visual "light up" effect; mute/unmute via native controls not required for this ticket)
- **Fixed in:** N/A — reverted pointer-events-none change

### No iframe "cover" sizing — 16:9 enforced by YouTube/Vimeo causes black bars

- **File:** `frontend/components/thrive-video/ThirdPartyVideoPlayer.tsx:34-37`

### FillContainerIframe has no scale buffer — black bars at near-16:9 container ratios
- **Status:** `open`
- **Severity:** Medium
- **Environment:** Frontend — staging, Panel Hero with YouTube video at `/josue-testing`
- **File:** `frontend/components/thrive-video/FillContainerIframe.tsx:36-37`
- **Root cause:** The oversized-iframe math (`max(100cqw, calc(100cqh * 16 / 9))` / `max(100cqh, calc(100cqw * 9 / 16))`) computes the exact minimum size to cover the container at 16:9. When the container is close to 16:9 (e.g. 1389.75×800px = ~1.737:1), the iframe is barely larger in one dimension and exactly equal in the other (1422.22×800px — zero vertical overflow). YouTube's player adds a few pixels of internal chrome/padding, so the actual video content renders smaller than the iframe, exposing thin black bars that can't be cropped because there's no overflow in that dimension.
- **Steps to Reproduce:**
  1. Navigate to a Panel Hero with a YouTube video where the container is close to 16:9 (e.g. ~1390×800px at typical desktop width with 3 panels)
  2. Observe thin black bars on top/bottom of the video
- **Expected behavior:** Video fills the panel completely with no visible bars
- **Actual behavior:** Thin black bars visible at top because the iframe height equals the container height exactly — no vertical overflow to crop YouTube's internal padding
- **Recommended fix:** Add a small scale buffer to the FillContainerIframe sizing. Multiply by ~1.02–1.05 so the iframe always exceeds the container by enough to crop platform chrome: `width: max(100cqw, calc(100cqh * 16 / 9 * 1.02))` / `height: max(100cqh, calc(100cqw * 9 / 16 * 1.02))`. This ensures a few percent overflow in both dimensions regardless of how close the container is to 16:9.
- **Suggested tests:** Update existing FillContainerIframe tests to verify the scale buffer in the CSS values. Add a Storybook story with a near-16:9 container to catch this edge case visually.
- **Linear:** `not synced`

#### Why this was missed

The oversized-iframe technique was designed to handle containers that are *not* 16:9 — wide containers (black bars top/bottom) or tall containers (black bars left/right). The math correctly oversizes the iframe in one dimension to maintain 16:9 coverage. What wasn't accounted for:

1. **YouTube/Vimeo players are not pixel-perfect 16:9 renderers.** The platforms add internal UI chrome — title bars, control overlays, safe-area padding — that consumes a few pixels of the iframe's content area. The actual video content renders slightly smaller than the iframe dimensions. This is invisible when the iframe has significant overflow (the crop hides it), but becomes visible when the iframe matches the container exactly in one dimension.

2. **The Panel Hero container is often near-16:9 by coincidence.** `PanelHeroBlock.tsx` uses fixed heights (`h-[800px]` for ≤3 panels, `h-[1200px]` for 4). At common desktop widths (1366–1440px), the main panel (`xl:w-[75%]`) ends up at ~1024–1080px wide, giving a ratio of ~1.28:1–1.35:1 (far from 16:9 — oversized iframe has plenty of overflow, no issue). But at wider viewports or when the browser is fullscreen on a 1920px monitor, the main panel reaches ~1440px wide, giving 1440×800 = exactly 1.8:1 — within rounding distance of 16:9 (1.778:1). At this precise ratio, the math produces zero or near-zero overflow in one dimension.

3. **Testing didn't cover the near-16:9 boundary.** The Storybook stories test at 375/768/1440 viewport widths. At 1440px viewport, the Panel Hero container (75% of available width minus gap) lands close to 16:9 — but the test only verifies that the iframe dimensions are ≥ container dimensions (`assertVideoCoverSizing`), not that there's *enough* overflow to absorb platform chrome. The assertion `iframe >= container` passes even with zero buffer. A visual Storybook test might have caught this, but automated pixel-comparison wasn't in scope.

4. **Local dev testing didn't hit this ratio.** Hunter confirmed YouTube worked locally — likely at a slightly different viewport width where the container ratio was far enough from 16:9 to produce visible overflow. The staging page `/josue-testing` happened to render at the exact sweet spot.

#### How to fix

Add a scale buffer of 2–5% to the `FillContainerIframe` CSS sizing. This guarantees the iframe always exceeds the container by enough to crop any platform chrome, regardless of how close the container ratio is to 16:9:

```css
/* Before — exact cover, zero buffer */
width: max(100cqw, calc(100cqh * 16 / 9));
height: max(100cqh, calc(100cqw * 9 / 16));

/* After — 2% buffer ensures overflow in both dimensions */
width: max(100cqw, calc(100cqh * 16 / 9 * 1.02));
height: max(100cqh, calc(100cqw * 9 / 16 * 1.02));
```

The buffer is small enough to be visually imperceptible (cropping an extra ~16px at 800px height) but large enough to absorb YouTube/Vimeo's internal padding at any container ratio. The `overflow-hidden` on the parent already handles the crop — no other changes needed.

#### How to prevent this class of issue

- When using oversized-element techniques for iframe cover behavior, always add a scale buffer to account for platform-internal rendering that reduces the usable content area
- Test at viewport widths that produce near-target-aspect-ratio containers, not just standard breakpoints
- Visual regression tests (Chromatic/Percy) would catch this automatically since the black bars are visible in screenshots
