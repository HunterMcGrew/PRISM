## BMAD Research (Reference for Eli)

External evidence supporting the BMAD comparison. All sourced from BMAD's own GitHub issues and community discussions — not opinion or personal experience.

### Source 1: Context Window Consumption
**Source:** [GitHub Issue #1343 — Reduce BMAD Agent Context Window Usage](https://github.com/bmad-code-org/BMAD-METHOD/issues/1343)

- BMAD agents consume **67%+ of Claude's 200K context window** during activation, leaving only ~30% for actual work
- The TEA (Technical Engineering Agent — BMAD's architect equivalent) knowledge base alone consumes **86% of context** (~172,750 tokens out of 200K)
- "Complex tasks fail due to context overflow, forcing users to start new sessions mid-task and lose conversation history"
- Proposed fix: 4-phase optimization to reduce from 67% to 25% — still in progress

**Our approach:** Manifest-based context loading means agents only load architect docs relevant to the files being touched. A debugger investigating one file doesn't load the entire architecture. Skills have focused, scoped prompts — not one monolithic agent.

### Source 2: Agent Coordination Failures on Real Projects
**Source:** [GitHub Issue #446 — BMad needs feature-driven approach (real usage on brownfield project)](https://github.com/bmad-code-org/BMAD-METHOD/issues/446)

- "Agents operate in complete isolation...No shared context or memory." Developer had to manually relay information between agents.
- Dev agent "arbitrarily picked story 1.4 without checking if there were unimplemented stories before it (1.1, 1.2, 1.3)"
- Story 1.1 "specifically referenced the technical analysis document" but the agent "completely ignored this critical link and proceeded without reading it"
- Data fabrication: agent "substituted unrelated data instead of acknowledging failure" — reported 160GB disk usage when actual was 54GB
- Architecture-first assumption doesn't work for bug investigation: "you cannot design architecture before reproducing the bug"

**Our approach:** Plan files are the shared context. Every skill reads the plan before starting — decisions, history, debugged issues, review findings all flow between skills without manual relay. Sasha (debugger) exists specifically for investigation-first workflows.

### Source 3: Stale Documentation
**Source:** [GitHub Issue #1218 — Auto-update project documentation after code changes](https://github.com/bmad-code-org/BMAD-METHOD/issues/1218)

- "document-project generates static documentation snapshots that become outdated as code evolves"
- Impact: "AI agent hallucinations," "wasted debugging time when agents work from outdated context," "integration issues from misaligned understanding"
- Developers must "manually remember to re-run documentation workflows" — discipline-dependent, prone to failure
- Proposed fix: git hooks + scheduled checks — still proposed, not implemented

**Our approach:** Architect context files are loaded via manifest matching whenever the relevant files are touched. The agent reads both the doc and the code in the same session — drift is detectable because the two are compared live. Briar's self-review includes a docs staleness check.

### Source 4: Context Rot — Industry-Wide Problem
**Source:** [Solving Context Rot: GSD vs BMAD vs Taskmaster](https://themenonlab.blog/blog/context-rot-ai-coding-agents)

- "The first few prompts are magical—clear responses, accurate code. Thirty minutes into a complex feature, something shifts. The agent starts forgetting decisions made earlier."
- Quality degrades measurably after 30+ exchanges in any framework
- BMAD's approach: specialized personas that hand off via structured documents — but the article notes the core context window limitation remains

**Our approach:** Plan files persist ACROSS sessions, not just within them. Skills have focused prompts that fit within context. The plan's decision log means the agent doesn't need to remember — it reads.

### Source 5: Forced Code Review Issues
**Source:** [GitHub Issue #1332 — Code review forces minimum 3 issues](https://github.com/bmad-code-org/BMAD-METHOD/issues/1332)

- "BMAD code review workflow enforces a minimum of 3 issues per review, which forces reviewers to find nitpicks even when code is genuinely good"
- Causes "endless review cycles and developer fatigue"

**Our approach:** Briar and Eric review what's actually there. If the code is clean, they say "Ship it." No artificial minimums.

### Source 6: Cascading Spec Failures
**Source:** [GitHub Issue #1904 — API contract specification gap causes cascading mismatches](https://github.com/bmad-code-org/BMAD-METHOD/issues/1904)

- After completing the full BMAD planning pipeline, implementation produced "8 cascading bugs on a single flow"
- Root cause: "~65% responsibility on specs and ~35% on the coding agent"
- The planning phase produced specs that didn't hold up during implementation

**Our approach:** Winston's architecture step produces decisions and acceptance criteria that are tested during implementation. Briar validates AC against the actual diff. The feedback loop catches spec-implementation gaps within the same ticket lifecycle.

- Specialized skills — 10 focused agents, each expert in one job. An architect that never writes code. A reviewer that never approves PRs. A debugger that finds root causes without jumping to fixes.

**The before/after comparison:**

### Task 4: Write Our Approach — Why We Built Our Own (BMAD Comparison)

**File:** `our-approach.md`

Content:

**Framework evaluation:**
- The team evaluated existing agent frameworks (BMAD and others) before building a custom system.
- BMAD was evaluated through hands-on testing on a real project over a two-week period before building the custom system. It was not used in production on the Thrive project.

**Where generic frameworks fall short (frame as architectural analysis):**

1. *Context persistence architecture* — BMAD's "living context" approach doesn't reliably prevent regressions across sessions. Documented fixes can be undone because the framework's context model loses coherence when sessions restart. Our approach: persistent plan files with explicit decision logs that agents must read and respect before modifying code. Decisions are append-only — you can't accidentally lose them.

2. *Documentation freshness* — framework-generated architecture docs degrade quickly because there's no enforcement mechanism tying them to code. When the code changes, the docs don't know. Our approach: manifest-based context loading means architect docs are loaded when the files they describe are touched. Drift is detectable because the agent reads both the doc and the code in the same session.

3. *Performance and context efficiency* — generic frameworks load broad context regardless of task scope. Every session pays the cost of loading the entire framework's context, even for a one-line fix. Our approach: manifest matching loads only the architect docs relevant to the files being touched. Specialized skill prompts are focused and scoped — a debugger doesn't load architecture patterns, and a reviewer doesn't load implementation scaffolding.

**Why custom was the right call:**
- Tight integration with the team's actual tools: Linear for ticket management, GitHub for PRs/reviews, VS Code as the primary editor
- Custom rules encode the team's specific standards (WCAG 2.1 AA, TypeScript/PHP conventions, block architecture patterns) — not generic best practices
- Custom skills match the team's actual handoff points (Nora pulling ticket context from Linear, Briar running self-review before PR, Eric posting directly to GitHub)
- The system evolves with the team instead of fighting framework assumptions

**The tradeoff (acknowledge honestly):**
- More upfront investment in rules and skills authoring
- The investment compounds — every rule written applies to every future PR, every architect context file reduces future context dumps
