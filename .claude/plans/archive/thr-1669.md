## Execution Model

**Handoffs are automatic across the whole loop.** Every persona invoked here (Nora, Eli, Eric) treats their normal "ready for the next persona?" confirmation as pre-answered yes. The loop runs unattended — pausing for confirmation between steps would stall it indefinitely.

### Worktree isolation

The entire loop runs inside a dedicated git worktree so the user's main checkout stays untouched. Worktree path: `/Users/hunter/Documents/TracTru/thrive-docs-1669`.

- **Iteration 1 setup** — if the worktree directory doesn't exist, run `git worktree add /Users/hunter/Documents/TracTru/thrive-docs-1669 origin/main` from the main repo. This creates an isolated checkout that the loop owns.
- **Every iteration** — `cd /Users/hunter/Documents/TracTru/thrive-docs-1669` as the first action, before invoking Nora. All branching, committing, pushing, and PR-opening happens inside this worktree. Skills running in the loop see the worktree as their working directory.
- **Loop end** — when the queue is empty, run `git worktree remove /Users/hunter/Documents/TracTru/thrive-docs-1669` from the main repo to clean up. Note this in the final `## History` entry.

This means the user's `codex-skill-sync-scaffold` branch and any working-tree state in the main checkout stay frozen for the entire run.

### Stop conditions

- Queue empty → loop terminates naturally.
- Three consecutive iterations fail (Nora can't start, Eli can't write, push fails, etc.) → append a `## Loop Pause` entry with details and end the loop without firing the next iteration.
- User interrupts → loop honors the interrupt.

### Loop invariants

A few invariants hold the loop together — each one earns its place from a specific failure mode:

- **Nora runs every iteration.** She sets Linear status and branches from clean main. Skipping her drifts Linear out of sync with git and risks branching from a stale tree.
- **One block per iteration.** Single-block iterations let the harness fire compaction between them; chaining multiple grows context until the loop loses track of the queue.
- **Sequential only.** Every block doc may touch the same `_meta.js` files in `docs/content/{user,dev}/blocks/`. Parallel branches conflict on it; sequential pulls latest `main` per iteration and avoids the conflict entirely.
- **Two Eric passes max per PR.** Initial review plus one re-review after Eli's fix. Anything beyond that risks stalling the queue on a single tricky block — leftover issues become async human follow-up.

## Decisions

- Sub-tickets already exist (THR-1670 → THR-1736). Nora starts each one — no ticket creation needed in the loop. Reason: tickets were created upfront so the loop body stays lean.
- One branch + PR per sub-ticket, branch name from Linear's `gitBranchName` field. Reason: matches project branch convention and keeps Linear ↔ git wired correctly via Nora.
- Sequential, not parallel. Reason: every block doc may touch the same `_meta.js` file in the docs sidebar — parallel branches would conflict on it. Sequential pulls latest `main` per iteration via Nora and avoids conflicts entirely.
- Eric reviews twice per PR (initial + post-fix). Reason: catches whether Eli's fixes actually addressed the issues, while keeping the queue moving — extending past two passes would let one tricky block stall the whole loop.
- Skip Briar on docs-only PRs. Reason: her checklist (types, a11y, build, tests) mostly no-ops on markdown changes. Eric on the PR is the substantive gate for docs.
- `colophon` (THR-1679) is `[dev-only]` — user doc already exists at `docs/content/user/blocks/colophon.md`. The loop skips writing a user doc for that ticket only.
- The loop runs inside a dedicated worktree at `/Users/hunter/Documents/TracTru/thrive-docs-1669`. Reason: keeps the user's main checkout (and current branch) frozen for the entire unattended run — branch-hopping in the main repo every iteration would leave them on a random doc branch when they returned.
- **Pair-PR rule** — when a block in the queue is a nested-pair parent or child (per `block.json` `parent` declarations), document both halves in one iteration's PR. Reason: cross-links between parent and child docs (`./parent.md`, `../child.md`) only resolve when both files exist on the same branch. Sequential pair iterations leave a gap where the first PR's links 404 until the second merges. Pair PRs eliminate that gap and reduce reviewer load — one PR with both halves is faster to scan than two PRs with overlapping context. The PR closes both Linear tickets via `Closes THR-PARENT` + `Closes THR-CHILD` lines (Linear's GitHub integration supports multiple closes per PR). Branch + PR title use the parent ticket: `THR-PARENT: Add docs for <parent> + <child>`.
