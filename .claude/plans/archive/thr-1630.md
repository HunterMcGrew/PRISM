- **Root-cause adjacency scenarios:** based on the ticket's root cause description, write scenarios that verify the class of bug isn't present elsewhere. Example: if the root cause is "null check missing on X", write a scenario that verifies other similar surfaces handle the null case.

## Decisions

- Research phase first: real-world QA test plan specialist role definition drives the expanded skill scope, not invention.
- Tag-range mode stays intact as one of several input shapes — not replaced.
- Release-PR workflow ownership unchanged: team lead still owns release PRs; Reese produces the checklist the lead attaches.
- No new personas — this expands Reese, does not spin up a new specialist.
- Mode dispatch over skill split — shared writing rules, regression heuristics, and tester-audience framing make a single dispatcher cheaper than sibling skills. Splitting would force 70%+ duplication and drift over time.
- **Four modes in Phase 1** (Release, Sprint/Group, Feature/PR, Bug-fix Verification). Three deferred modes in Future Phases: pre-implementation AC-derived testing, exploratory charter / SBTM, scheduled regression / smoke. Ship nothing speculative.
- Bug-fix Verification shipped in Phase 1 because the output shape is genuinely different from Feature/PR — structured around the Linear bug report (repro steps become verification scenarios, root cause drives adjacency testing), not around the feature diff. Detection is automatic for 90% of cases.
- **Natural-language mode detection, not `mode:` syntax.** Reese reads prompt words ("release", "sprint", "PR", "hotfix", "verify fix", "retest") alongside input shape (tag pair, PR number, branch name, commit range) **and Linear ticket labels** (single PR whose ticket has the `bug` label auto-dispatches Bug-fix Verification). Core rule: **infer by default from data, override from words.** If signals agree → dispatch silently. If data says bug but prompt is generic, Reese dispatches Bug-fix and announces it in greeting. If ambiguous or conflicting → Reese asks in his own voice. No rigid override syntax — keeps the skill file human per the 4.7 audit strategy.
- **Inputs are flexible upstream of mode.** Feature/PR accepts PR number, PR URL, or branch name (all resolve to the same thing via `gh` — branch without a PR falls back to `origin/main..<branch>`). Sprint/Group accepts multiple PRs or a commit range. Release accepts a tag pair or compare URL.
- **Feature/PR mode inlines Linear AC** when the PR title contains a THR-NNNN. Strengthens traceability — testers verify acceptance directly from the checklist without cross-referencing Linear.
- Existing Release mode output is unchanged — Phase 1 is additive. Byte-identical regression against a historical tag pair is the contract.
- **Domain knowledge splits to `.claude/architect/qa-test-planning.md` in Phase 1** — not deferred. Test design techniques, risk heat map, regression signals, equipment dealership context, QA writing rules live there. SKILL.md holds behavior: mode dispatch, per-mode workflows, authors-ship closing. Split now because (a) we're expanding the file enough to cross the threshold anyway, (b) Phase 2 needs a place to land, and (c) other skills (Briar reviewing QA output, Clove writing test plans) can reference the reference material without re-reading SKILL.md.
- Reese's authors-ship closing stays intact — only the PR subject-line template branches per mode. THR-1631's extension of the authors-ship pattern to other personas is orthogonal and must remain compatible.
- `AGENTS.md § 0` Reese routing row is a pre-existing gap this ticket surfaces, not one it introduces. Fixed here.
- Docs updates are Eli's job — four AI-skills docs reference Reese (`overview.md`, `how-it-works.md`, `workflows.md`, `personas.md`) and need the expanded scope reflected.
