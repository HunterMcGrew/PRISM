# Plan: THR-1456

## Ticket

THR-1456 — Remove Apollo from product route browser bundles

## Goal

Eliminate `@apollo/client` from visitor-facing browser chunks on equipment and showroom pages by replacing client-side `ServiceFactory` usage with server-mediated patterns.

---

## Decisions

- `useEquipment` / `useCompareItem` fetch via `/api/equipment/[id]` route handler, not `ServiceFactory` — user-initiated reads can't be pre-fetched server-side; route handler keeps Apollo server-only while supporting client-triggered fetches (same pattern as `/api/form`).
- `ShowroomBlock` converted to RSC (dropped `"use client"`) — follows Promotions block pattern: accepts `isEditor` + `children`, editor passes all interactive UI as children. Visitor-facing component renders only `CategoriesNav` + `ProductCard` (no DnD, no selectors, no hooks).
- `ShowroomEditorLayout` created as `"use client"` component in frontend — encapsulates DnD context, sensors, and sortable items. Lives in frontend so `@dnd-kit` resolves from `frontend/node_modules`, but only ever imported by backend `edit.tsx`.
- `ShowroomListItem` gets `"use client"` directive — uses `useSortable` hook, only imported by backend editor now.
- `ProductCategorySelector` gets `"use client"` directive — missing directive is a bug; component uses `useState`/`useEffect`.
- Editor can continue calling Apollo via `ServiceFactory` / `resolver.props` — Apollo in the editor (server-side or backend plugin build) is fine; the goal is keeping it out of visitor-facing client chunks only.
