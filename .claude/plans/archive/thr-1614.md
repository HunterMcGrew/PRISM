## Goal

Fix the Mega Menu Locations block so the "Add Location" button renders correctly on initial load and after clearing all locations, without requiring a hide/reopen workaround.

---

## Decisions

- **Scope is the `useSelect` pattern only — not the render gate.** The existing `locationPosts?.length > 0` gate is correct once data loads through the right registry: it properly hides the button in "loading", "all selected", and "no locations exist" cases. Sasha's proposed gate change (`locationPostsFromWp !== null && locationPostsFromWp.length > 0`) would also show the button when all locations are already selected, opening an empty dropdown — a UX regression. Tight surgical fix is safer.
- **Fix pattern:** `import { store as coreStore } from "@wordpress/core-data"` plus `useSelect((select) => select(coreStore).getEntityRecords(...), [])`. Matches [apps/mega-menu-editor/components/header/Header.tsx:4](backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/components/header/Header.tsx#L4). Reason: the Mega Menu editor runs inside a standalone `@wordpress/data` registry. Module-level `select("core")` is bound to the **default** registry; `useSelect`'s subscription is wired to the **active** registry. Crossing wires prevents the re-render when the fetch lands. The parameterized `select` inside `useSelect` resolves store references against the active registry, so subscription and read target align.
- **Scenario C (InnerBlocks TypeError) is out of scope.** User observed the editor functions normally despite the console error. Likely a transient load-order artifact. File separately if it proves user-impacting.
- **Promotions block carries the same anti-pattern but is out of scope.** `backend/plugins/gravity-platform-core/src/blocks/promotions/edit.tsx` uses module-level `select("core")`. No `parent` constraint on its `block.json`, and not in the mega menu editor's exclude list — so it could be inserted into a mega menu pane and hit the same bug. Separate ticket (see follow-ups below). Not bundled here because scope discipline matters more than opportunistic cleanup.
- **Architect context gets fixed in this PR.** [.claude/architect/backend-editor-blocks.md](.claude/architect/backend-editor-blocks.md) was teaching the broken pattern as canonical. Updated in the same branch so the doc and the code land together and no future block gets written with the same bug.

## Debugged Issues

### Add Location button missing on first load and after clearing all locations
- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-locations/edit.tsx` — replaced `import { select, useSelect } from "@wordpress/data"` with `import { useSelect } from "@wordpress/data"` + `import { store as coreStore } from "@wordpress/core-data"`. Rewrote the `locationPostsFromWp` `useSelect` to use the parameterized `select` with the imported `coreStore` reference. Listbox render gate on line 290 (`locationPosts?.length > 0 &&`) left intact per Winston's scope decision.
- **Severity:** High
- **Environment:** Mega Menu editor page (wp-admin, `toplevel_page_thrive-mega-menu`)
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-locations/edit.tsx:16, 52-58, 60-63, 290`
- **Root cause:** The Locations block uses module-level `select("core")` from `@wordpress/data` inside its `useSelect` callback. The Mega Menu editor is a **standalone block editor with its own `@wordpress/data` registry** (`<BlockEditorProvider>` wrapped by the Mega Menu app), not the global/default registry. `select("core")` resolves against the default registry — not the mega menu's — so `getEntityRecords("postType", "locations", ...)` never triggers the core-data resolver for the active registry. The store subscription is effectively disconnected: `locationPostsFromWp` stays `null`, `locationPosts` collapses via `|| []` to `[]`, and the Listbox gate `locationPosts?.length > 0` hides the "Add Location" button. Hide+reopen forces a remount at a moment when the default registry's core-data store has been populated by another component (likely via the admin shell's own `coreStore`-keyed selectors in `apps/mega-menu-editor/`), so the remounted useSelect reads cached data synchronously.
- **Steps to Reproduce:**
  1. Navigate to the Mega Menu editor page (`wp-admin → Mega Menu`)
  2. Insert a fresh Mega Menu Locations block (or load a page with an existing one populated)
  3. Observe the block contents — no "Add Location" button
  4. Hide and reopen the block's container (collapse/expand the pane) — button appears
  5. Alternative: with a populated block, remove every location one at a time until none remain — button does not reappear
- **Expected behavior:** "Add Location" button renders on initial mount as soon as the locations fetch completes, and remains visible after all selections are cleared, without any hide/reopen workaround.
- **Actual behavior:** Button is absent on first load and absent after clearing all locations. Only appears after hiding and reopening the block's container.
- **Recommended fix:** Import `store as coreStore` from `@wordpress/core-data`; rewrite useSelect to `useSelect((select) => select(coreStore).getEntityRecords(...), [])`; change the render gate on line 290 from `locationPosts?.length > 0` to `locationPostsFromWp !== null && locationPostsFromWp.length > 0` so that the button visibility is decoupled from the filtered-subset length. The `coreStore` import + parameterized `select` pattern matches what `apps/mega-menu-editor/components/header/Header.tsx` already uses.
- **Suggested tests:** Manual QA in the Mega Menu editor as listed in the implementation task. No new unit tests — editor `edit.tsx` is not Jest-tested per project convention.
- **Linear:** `synced`
