# Install Layout

PRISM is a multi-platform AI toolkit, and the install layout is how its content lands across Claude Code, Codex, and Cursor inside a consumer's repo.

## The need: every platform is first-class

PRISM ships skills, rules, ADRs, architect docs, templates, and references into a consumer's codebase. Three AI coding platforms are first-class targets: Claude Code, Codex, and Cursor. Each one has its own root directory the platform auto-loads at session start — `.claude/` for Claude Code, `.codex/` for Codex, `.cursor/` for Cursor. The auto-load is what makes "rules show up in the agent's context without anyone Reading them" possible. It's what consumers expect; it's what makes the toolkit usable instead of theoretical.

Two kinds of content sit in the install. **Platform-specific outputs** — skills, native config files — live where the platform expects them and look different on each one. **Platform-agnostic content** — rules, ADRs, architect docs, templates, references, plans, lessons — looks identical regardless of which platform reads it. The same writing-voice rule applies whether the agent runs on Claude or Codex.

## What the requirements force

Three things fall out of "every platform is first-class" plus "most content is platform-agnostic":

- **The platform-agnostic content reaches every platform's auto-load surface.** If only Claude Code auto-loads it, Codex and Cursor consumers operate without rules — second-class by structural accident, not by design choice.
- **A single canonical source.** Editing the writing-voice rule in one place propagates to all three platforms. Three hand-edited copies drift the moment anyone touches one of them.
- **Agent-written content has one source, not three.** A plan or lesson that the agent updates mid-session can't sit in three platform copies — concurrent edits race against the next build.

The first requirement points away from "everything in `.claude/`." The second points away from "three hand-maintained copies." The third points away from "copy plans into platform dirs alongside the rules."

## The natural fit: bifurcation

Two surfaces, not one. Platform-agnostic canonical content lives at `.prism/<area>/` — neutral, doesn't privilege any platform. Platform dirs hold build-time copies of the read-only canonical content, regenerated by `pnpm prism:build`, plus their own platform-specific outputs. Each platform's auto-load surfaces its copy; the canonical is the source of truth.

Two alternatives the reader would naturally consider — and why neither works:

- **Symlinks from each platform dir to `.prism/`.** Cross-platform symlink support is the wall. Windows requires elevated privileges or developer mode; symlinks in git repos break in unpredictable ways across Mac, Linux, and WSL setups consumers actually use. The portability cost outweighs the duplication cost.
- **`.prism/` only, no platform copies.** Forces every session to Read 5–10 files from `.prism/` at start to surface the rules and architect docs. Claude Code's free auto-load gets thrown away; Codex and Cursor sessions get slower. The build-copy step is what keeps platform auto-load working.

Agent-written content (`.prism/plans/`, `.prism/lessons.md`) sits at canonical only. No mirror, no copy. A plan edit lands in one place; the next build doesn't overwrite it because the build doesn't touch it.

## Platform limits force a build layer

Auto-load mechanisms across the three platforms don't share a path. Claude Code reads `.claude/`; Codex reads `.codex/`; Cursor reads `.cursor/`. None of them auto-loads `.prism/`, and "configure the platform to auto-load `.prism/` instead" isn't a thing any of them support. The bifurcation can't rely on the platforms — the build is what bridges canonical to per-platform.

`scripts/ai-skills/build.ts` runs the bridge:

- **Content copy.** For each platform dir, copy the read-only canonical areas from `.prism/`. Every copied area carries an `.ai-skill-generated` marker at its root so the build knows which files it owns.
- **Drift detection.** `pnpm prism:check` runs the same logic in dry-run mode. If a platform copy has been hand-edited or canonical was edited without a rebuild, the check fails and reports the file. CI runs this.
- **Cleanup on rename.** If a canonical file is renamed or removed, the next build deletes the orphaned platform copies (recognized via the marker) so the platform doesn't accumulate stale content.
- **Path guard.** A standalone module at `scripts/ai-skills/path-guard.ts` runs early in the build. It scans canonical content and fails when it finds `.claude/<area>/`, `.codex/<area>/`, or `.cursor/<area>/` references in copied areas. This is the rule that makes "cite `.prism/<area>/<file>` everywhere in canonical" enforceable instead of aspirational.

## The cross-reference convention

Canonical content cites `.prism/<area>/<file>` — never a platform-dir path. Every platform's copy of the citing file resolves correctly through its own auto-load: Claude reads the rule from `.claude/rules/<rule>.md`, sees the citation, and resolves it against the canonical location. Codex and Cursor do the same.

Citing `.claude/rules/<rule>.md` inside content that ships to all three platforms is the failure mode the path guard catches. A Codex consumer doesn't have a `.claude/` dir — the citation breaks the moment the rule reaches them.

The convention has two scoped exceptions:

- **Skill bodies under `.claude/skills/<id>/SKILL.md`** are platform-specific outputs. They're allowed to reference platform-specific paths because they only run on that platform.
- **Agent-written content (plans, lessons)** doesn't get copied, so a plan that mentions `.codex/rules/foo.md` in its acceptance criteria doesn't propagate the reference anywhere. The path guard explicitly skips non-copied areas.

## Layout map

What lives where, after the bifurcation:

| Path                          | Source of truth         | Notes                                                                                                  |
| ----------------------------- | ----------------------- | ------------------------------------------------------------------------------------------------------ |
| `.prism/rules/`               | Canonical               | Cross-cutting rules; copied to every platform dir                                                      |
| `.prism/architect/`           | Canonical               | Path-scoped context, including `manifest.json`; copied                                                 |
| `.prism/spec/adrs/`           | Canonical               | Architectural Decision Records; copied                                                                 |
| `.prism/templates/`           | Canonical               | Shared templates (PR description, AC, bug report, etc.); copied                                        |
| `.prism/references/`          | Canonical               | Operational flows reused across skills; copied                                                         |
| `.prism/SPEC.md`              | Canonical               | Tier 0 meta-doc; copied                                                                                |
| `.prism/plans/`               | Canonical (only)        | Agent-written; not copied                                                                              |
| `.prism/lessons.md`           | Canonical (only)        | Agent-written; not copied                                                                              |
| `.claude/skills/<id>/`        | Platform-specific build | Generated from `.ai-skills/skills/<id>/`; not under `.prism/`                                          |
| `.claude/rules/`, etc.        | Build copy              | Mirror of `.prism/rules/`; managed marker at the root                                                  |
| `.codex/...`, `.cursor/...`   | Build copy              | Same shape as `.claude/` for the read-only areas; platform-specific outputs alongside                  |
| `templates/install/`          | Distribution surface    | What ships into a consumer repo. Sub-layout mirrors the bifurcation: `.prism/`, `.claude/`, `.codex/`, `.cursor/`, plus `AGENTS.md.tmpl` at root |

## When canonical changes

Editing rules or architect docs follows a fixed loop:

1. Edit the file under `.prism/<area>/`.
2. Run `pnpm prism:build`. Platform copies regenerate; the path guard runs; drift is reset.
3. Commit both the canonical edit and the regenerated platform copies. CI runs `pnpm prism:check` to confirm nothing was missed.

`pnpm prism:check` is the safety net — if a canonical edit ships without a rebuild, CI fails before it merges. The same check catches hand-edits to platform copies (someone "fixing" `.claude/rules/code-comments.md` directly instead of editing the canonical).

## Trade-offs the bifurcation accepts

- **Two-step edit.** Every canonical change costs one rebuild. The check guards against forgetting; the cost is one command.
- **More surface in `paths.json` and `build.ts`.** The bifurcation adds `canonical.contentRoot`, `generated.platformContentCopies`, and roughly 150 lines to `build.ts`. Worth it for symmetry across platforms.
- **Reserved namespaces.** `.prism/` joins `.claude/`, `.codex/`, `.cursor/`, and `.agents/` as reserved at consumer repo roots. Consumers whose project happens to use `.prism/` for unrelated reasons would conflict — the same kind of conflict the existing platform-dir reservations already imply.
- **Non-Claude platforms require a local build before first use.** In this repo, `.claude/` is committed (dogfood install) but `.codex/`, `.cursor/`, `.agents/`, and `.generated/` are gitignored as build outputs — the Codex and Cursor copies only exist after `pnpm prism:build`. A fresh clone needs the build before opening Codex or Cursor against the repo. Consumer teams choose per-platform whether to commit their copies or treat them as generated; the build copy is a function of canonical either way. `prism:check` treats a platform without managed content as opt-out (signal: the area-level `.ai-skill-generated` marker is absent) rather than as drift, so a fresh clone doesn't false-fail on the gitignored platforms — drift detection kicks in once the contributor runs `prism:build` and the markers land.
