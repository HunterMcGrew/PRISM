## Decisions

- **Scope is bounded to `main` as of `b7929faa3`.** The original ticket scoped six topics including Docker layer caching, the `CACHE_BUST` plugin manifest pattern, the base image consumption contract, the `Dockerfile.admin-base` two-Dockerfile architecture, and the `publish-admin-base.yml` workflow — none of which exist on `main` yet. They live in the in-flight CI/Build Optimization epic (PR #1917 / THR-1769). Documenting them now would describe a state that isn't real yet and would be wrong for any agent or teammate reading `main` today. We document what's on `main`, mark the gaps as known follow-ups, and update post-merge.

- **Skip `docs/content/dev/operations/ci-operations.md` entirely on this PR.** The ticket's "Base image rebuild" operations section describes the manual-trigger workflow for `publish-admin-base.yml`, which doesn't exist on `main`. There are no other operator procedures unique to the build pipeline today (PR validation is auto-triggered by `validate-pr.yml`; release builds are auto-dispatched by `create-release.yml`). When the epic merges and the rebuild workflow becomes real, the ops section becomes a real procedure to document — that's a follow-up.

- **Document `Dockerfile.admin` as it actually is, not as it's about to become.** The current file is monolithic — apt, curl, Playwright deps, Xdebug, WP CLI, Corepack, composer, plugin download, theme copy, all in one. The epic splits the stable layers out into `Dockerfile.admin-base` and pins admin to it. We describe the monolith truthfully. The architecture doc's Beat 4 (platform limits + custom layer) names the dual-Dockerfile choice as "admin uses the WP base image for prod parity, frontend uses node-alpine for size" — not "admin uses a published base image." That's the post-epic story.

- **Three jobs, not four.** `build.yml` on `main` has `build`, `plugin-tests`, and `docker-compose-build`. The epic splits `docker-compose-build` into separate `docker-build-admin` and `docker-build-frontend` jobs to enable per-Dockerfile cache scopes. We document three jobs.

- **Pair-naming pattern.** Use the same base filename across both halves of the pairing — `.claude/architect/ci-build.md` and `docs/content/dev/architecture/ci-build.md`. This matches the precedent set by `plugin-management.md`. Keyword "ci-build" complements the existing "ci-pipeline" keyword without colliding.

- **Architect-doc voice rules apply to both files.** `architecture-doc-shape.md` (four-beat arc, anchor sentence, shopping-list anti-pattern, concrete platform limits, operational-bleed anti-pattern, problem-shape ↔ solution-shape, natural-fit framing) governs the docs-side file. Spec-content rules from `writing-voice.md` (onboarding voice, explain-the-why, keep it short) govern the architect-side file. Both follow `documentation.md` conventions for placement, frontmatter, and cross-references.

- **Eli owns all four tasks.** Both new files are documentation work — agent-facing and teammate-facing — and the manifest + documentation.md updates are paperwork that follows from creating new docs. No source code touched.

## Review Issues

### Missing article in dev doc Beat 4 paragraph

### "apt installs:" misclassifies curl-based installers

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md:108` and `docs/content/dev/architecture/ci-build.md:107`
- **Problem:** The "apt installs:" bullet lumps Azure CLI and Composer 2.7.6 with the genuinely apt-installed packages. In `Dockerfile.admin` lines 62-63, both are installed via curl (`https://aka.ms/InstallAzureCLIDeb` and `https://getcomposer.org/installer`).
- **Suggested fix:** Split into two bullets — "apt installs: gh, git, nodejs, rsync, sudo, zip" and "curl-based installers: Azure CLI, Composer 2.7.6". Apply in both files.

### Composer install asymmetry — incomplete rationale

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md:80` (Job: `plugin-tests` paragraph)
- **Problem:** The doc says plugin-tests "Owns its own composer install — does not share state with the `build` job. The plugin-tests job is PR-only; folding its composer install into the always-running `build` job would add it to the release path for no purpose." The "no purpose" framing is the secondary reason. The substantive reason: the two jobs run composer in different modes — `build`'s release path passes `--no-dev` to plugin installs (`build.yml:142`) so release artifacts ship without dev deps; `plugin-tests` deliberately omits `--no-dev` (`build.yml:263`) so the test runs have access to dev deps. Sharing composer state would conflict. An agent reading the current text might try to merge the steps as an "obvious optimization" and quietly break either the release artifact size or the tests.
- **Suggested fix:** Replace "for no purpose" with the actual reason: "The two jobs run composer in different modes — `build`'s release path uses `--no-dev` for plugin installs so release artifacts ship without dev deps; `plugin-tests` keeps dev deps so the tests can run. Sharing composer state would conflict." Mirror in the dev doc if the same framing is reused there.

### Missing `composer test:install` step in plugin-tests

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md:74-82` (Job: plugin-tests section)
- **Problem:** Before running tests, `plugin-tests` runs `composer test:install` from `gravity-platform-core` (`build.yml:268-271`), which installs the WordPress test environment against the MariaDB service. The architect doc lists the test runs but skips this prerequisite. A reader trying to mirror the job locally would miss why the tests can't run cold.
- **Suggested fix:** Add one sentence after the MariaDB description: "Before running tests, `composer test:install` (in `gravity-platform-core`) installs WordPress against the MariaDB service so the integration tests have a real DB to talk to."

### `THRIVE_TOKEN` dual purpose not surfaced

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md` (no current mention) and `docs/content/dev/architecture/ci-build.md` (no current mention)
- **Problem:** `build.yml` declares `THRIVE_TOKEN` as required (`build.yml:24-25`) and uses it in two places: composer GitHub auth for private packages (`build.yml:130`) and as the custom release-action token (`build.yml:223`). The release-action use is non-obvious — there's an inline comment `# Uses a custom token so that deployment workflows can be triggered when a release is created` because GITHUB_TOKEN-triggered releases don't fire downstream `workflow_run` events. An agent updating the workflow would likely swap to GITHUB_TOKEN and silently break the deploy chain.
- **Suggested fix:** Add a short bullet to the Release-tag gating section (or a new "Secrets" subsection): "`THRIVE_TOKEN` is the custom GitHub token used by the release-action — required because GITHUB_TOKEN-triggered releases don't fire downstream `workflow_run` events, breaking the deploy handoff. Same token is reused for composer GitHub auth on private packages."

### `release/v*` branch convention not documented

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md:13-16` (The reusable workflow shape) and `docs/content/dev/architecture/ci-build.md:33-37`
- **Problem:** `validate-pr.yml` skips PRs whose head branch starts with `release/v` (`validate-pr.yml:10`): `if: ${{ !startsWith(github.head_ref, 'release/v') }}`. The convention exists so release PRs aren't built twice (once on PR push, once on tag push). The architect doc lists `validate-pr.yml` as the PR caller but doesn't mention the skip. A reader who renames the release branch convention would silently break the deduplication.
- **Suggested fix:** Add a one-line note under the `validate-pr.yml` bullet: "Skips PRs whose head branch starts with `release/v` — those PRs are validated on the tag-push side via `create-release.yml` to avoid building twice."

### HOST_UID/HOST_GID host-environment phrasing reads muddled

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/architect/ci-build.md:111`
- **Problem:** Current line: "Defaults to 33 — correct for OrbStack/Mac and Windows-filesystem WSL2 workspaces; needs override for native Linux or WSL2-filesystem workspaces." The "Windows-filesystem WSL2 workspaces" / "WSL2-filesystem workspaces" pair reads as garbled — both phrases contain WSL2 and "filesystem" but mean opposite things. `Dockerfile.admin:24-28, 95-100` uses clearer phrasing.
- **Suggested fix:** Match the Dockerfile's phrasing: "Defaults to 33 — correct for OrbStack/Mac and for Windows workspaces stored on the Windows filesystem. Override (`--build-arg HOST_UID=$(id -u)`) is needed for native Linux or for Windows workspaces stored on the WSL2 filesystem."
