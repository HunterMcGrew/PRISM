# Plan: THR-1656

## Ticket

https://linear.app/tractru/issue/THR-1656/mega-menu-mobile-pane-renders-subtitles-that-should-be-desktop-only

## Goal

Stop the Thrive Mega Menu's mobile pane from rendering `description` subtitles under `links-with-icon` entries, without touching the legacy mega menu or the non-mega mobile menu.

---

## Decisions

- The bug lives in the **Thrive** mega menu mobile path (`frontend/blocks/mega-menu-renderer/` + `frontend/components/mega-menu-renderer/`), not the legacy mega menu path (`frontend/blocks/mega-menu/` + `frontend/blocks/site-header/MobileMegaMenu.tsx`). The ticket's initial file hints and the `NavigationPaneLink` mention are for the legacy system and are a red herring here.
- Fix site is [`frontend/blocks/mega-menu-renderer/resolver.props.ts:446`](../../frontend/blocks/mega-menu-renderer/resolver.props.ts#L446) inside `parseLinkWithIcon`. Removing `description: toNonEmptyString(attrs?.description)` from the returned object stops subtitles from entering the mobile nav model at the source.
- `MegaMenuMobileLink.description` has no other consumer in the mobile path. After the resolver change, [`MegaMenuMobileMenu.tsx:210-212`](../../frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L210-L212) (the conditional passing description) and [`MegaMenuMobileMenu.tsx:242-246`](../../frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L242-L246) (the rendering block) become dead. Also remove the `description?: string` field on `MegaMenuMobileLink` in [`mobile-nav-model.ts:9`](../../frontend/components/mega-menu-renderer/mobile-nav-model.ts#L9).
- Legacy mega menu files are NOT touched â€” `frontend/blocks/mega-menu/resolver.ts`, `MobileMegaMenu.tsx` (site-header), `NavigationPaneLink.tsx`, and `frontend/components/mega-menu/panes/*` stay as they are. The legacy mega menu has its own subtitle behavior (via `NavigationPaneLink`), but that's a separate codepath and outside scope for this ticket.
- `MobileMenu.tsx` (the non-mega mobile menu) is NOT touched.

## Debugged Issues

### Thrive mega menu mobile pane renders `description` subtitles under links-with-icon entries

- **Status:** `fixed`
- **Fixed in:** `frontend/blocks/mega-menu-renderer/resolver.props.ts` (dropped `description` from `parseLinkWithIcon` output); `frontend/components/mega-menu-renderer/mobile-nav-model.ts` (removed `description?: string` from `MegaMenuMobileLink`); `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx` (removed dead description prop + render block). Regression tests added in `resolver.props.test.ts` and `MegaMenuMobileMenu.test.tsx`.
- **Severity:** Medium (S3) â€” cosmetic, links still work. Priority is Urgent (launch blocker today).
- **Environment:** Frontend, mobile viewport (< `lg` breakpoint), any dealer site where the Thrive Mega Menu is enabled and a `mega-menu-links-with-icon` block is authored with `description` values.
- **File:** [`frontend/blocks/mega-menu-renderer/resolver.props.ts:446`](../../frontend/blocks/mega-menu-renderer/resolver.props.ts#L446) â€” `parseLinkWithIcon` copies `description: toNonEmptyString(attrs?.description)` onto every `MegaMenuMobileLink`. Rendered downstream at [`MegaMenuMobileMenu.tsx:210-212`](../../frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L210-L212) and [`MegaMenuMobileMenu.tsx:242-246`](../../frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L242-L246).
- **Root cause:** The mobile nav resolver intentionally carries `description` from authored `mega-menu-link-with-icon` block attrs into the mobile model, and the mobile view intentionally renders it for `links-with-icon` groups. The product decision â€” mobile mega menu is title-only â€” wasn't applied when this mobile pathway was built. Legacy mega menu rendering (`NavigationPaneLink`) is not involved in this codepath.
- **Steps to Reproduce:**
  1. In the Thrive Mega Menu editor, create a navigation item with a pane containing a `mega-menu-links-with-icon` block.
  2. Author at least one link with a non-empty `description` (title: "Parts Request", description: "Parts Request Form").
  3. Publish the menu and load a dealer frontend on a mobile viewport (< `lg` breakpoint).
  4. Open the mobile mega menu and tap into the nav item to reach the second-level pane.
  5. Observe each link renders title with the description stacked underneath in lighter text.
- **Expected behavior:** Each link shows its title only on mobile. Desktop continues to show title + description.
- **Actual behavior:** Each link shows title + description on mobile.
- **Recommended fix:**
  1. Remove `description: toNonEmptyString(attrs?.description),` from the object returned by `parseLinkWithIcon` in [`resolver.props.ts:446`](../../frontend/blocks/mega-menu-renderer/resolver.props.ts#L446).
  2. Remove the now-unused `description?: string` field from `MegaMenuMobileLink` in [`mobile-nav-model.ts:9`](../../frontend/components/mega-menu-renderer/mobile-nav-model.ts#L9).
  3. Remove the dead conditional at [`MegaMenuMobileMenu.tsx:210-212`](../../frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L210-L212) (the `description={...}` prop on the `SecondLevelGroup` map) and the dead `description` prop + rendering block in `MegaMenuMobileLinkAnchor` at lines 223 and 242-246. The component's `description` prop ceases to exist.
- **Suggested tests:**
  - Update [`frontend/blocks/mega-menu-renderer/__tests__/resolver.props.test.ts`](../../frontend/blocks/mega-menu-renderer/__tests__/resolver.props.test.ts) â€” assert that `parseLinkWithIcon` output does NOT include `description`, even when the input block attrs define one. Covers the resolver regression.
  - Update [`frontend/components/mega-menu-renderer/__tests__/MegaMenuMobileMenu.test.tsx`](../../frontend/components/mega-menu-renderer/__tests__/MegaMenuMobileMenu.test.tsx) â€” assert that a `links-with-icon` group rendered on mobile does NOT render the `description` text in the DOM, even if the mock data carries it. Guards against someone re-adding description pass-through in the view.
  - Update the `MegaMenuMobileMenu.stories.tsx` fixture so the story exercises a `links-with-icon` group with authored descriptions â€” Chromatic catches visual regressions.

## Acceptance Criteria

### Behavioral

- [ ] Given a mega menu column with icon links that have subtitles, When the menu is viewed on a mobile viewport and the user taps into the parent item, Then each link in the pane shows its title only â€” no subtitle text underneath.
- [ ] Given the same mega menu on a desktop viewport, When the user opens the mega menu, Then subtitles continue to render under icon links as they did before this fix (no regression to desktop behavior).
