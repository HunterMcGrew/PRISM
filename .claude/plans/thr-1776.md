### Eli (documentation) — Winston audit 2026-04-28 (three open issues)

These tasks come from Winston's post-merge audit. Three jmotes threads on PR #1928 remain **unresolved** (confirmed via GitHub GraphQL `isResolved` field). All three touch `docs/content/user/configuration/admin-toolbar.md`.

**A. Fix "Purge All Gravity Forms" — When to use it (line 66 area)**

Current text:
> **When to use it:** After editing a form — adding, removing, or reordering fields, changing labels, updating notification settings — and you want the new version to appear on the live site immediately.

This is inaccurate. `backend/plugins/gravity-platform-core/includes/Setup/FormsSetup.php:34` hooks `gform_after_save_form` and calls `revalidateService->revalidateByTag("gravity_form-{$form['id']}")` on every save. `frontend/lib/forms/fetch-form.ts:27` tags cache entries with both `CONTENT_TYPES.FORMS` (`gravity_forms`) and the per-form singular tag (`gravity_form-{formId}`). The singular tag matches the auto-revalidation tag, so an individual form edit already auto-clears that form's cache. Manual purge is not required after a normal form save.

Replace with:
> **When to use it:** If a form change isn't appearing on the live site after saving it. Form saves normally trigger an automatic cache refresh — use this option if the new version still isn't showing after a minute or two.

**B. Clarify access for the Dealer role (line 27 area)**

Current NOTE:
> Only users with the **Administrator** role see this toolbar. Editors, authors, contributors, and subscribers don't have access to it.

`backend/plugins/gravity-platform-core/includes/Roles/RolesBuilder.php` defines a custom `dealer` WordPress role built on `editor` + GF/WooCommerce capabilities. It is NOT the `administrator` role. `CacheClearToolbar.php:26` gates visibility with `current_user_can('administrator')`, which checks role name. **Dealer-role users cannot see or use this toolbar.**

The doc's primary audience is dealers — they should know this toolbar isn't available under their usual Dealer account. Expand the NOTE to add:

> Dealer accounts don't have Administrator access by default — if you're a dealer who needs to purge the cache and don't see this toolbar, contact your site administrator or the Thrive team.

**C. Add Vercel / hosting scope note (line 11 area, Overview)**

jmotes flagged that "we have a handful of dealers on WPE's Atlas platform still where WPGraphQL Cache purging is required instead." The doc currently has no mention of hosting scope.

`RevalidateService.php:99-101` always calls the Next.js `/api/revalidate` endpoint on the configured `frontendDomain`. WPE Atlas sites don't run the same headless Next.js layer — the revalidation either silently no-ops (if `frontendDomain` is unconfigured, `shouldRevalidate()` returns false) or calls the wrong endpoint. There is an additional UX pitfall: `handleClearCache` redirects to `?thrive_cache_cleared=1` regardless of whether `revalidateByTag` succeeded, so WPE Atlas users see a green success notice even though nothing was cleared.

Add a `[!NOTE]` or inline sentence near the top of the Overview scoping this toolbar to **Vercel-hosted Thrive sites**. Suggested text:

> **Hosting note:** This toolbar purges Thrive's Next.js frontend cache and applies to dealer sites hosted on Vercel. If your site is hosted on WP Engine's Atlas platform, cache is managed through WP Engine's own tools — the Purge Frontend Cache menu won't affect your site's caching layer.

Place this after the opening paragraph (line 15-area), before the `### When you'd reach for this toolbar` subsection.

**D. Reply to the three GitHub threads**

Once A, B, and C are fixed: reply to each of the three unresolved jmotes threads on PR #1928 (lines 27, 66, 11) with a brief confirmation of what changed and resolve them.

## Decisions

- **File path: `docs/content/user/configuration/admin-toolbar.md`.** Matches the existing `configuration/` pattern alongside `recaptcha-setup.md`. The placement test in `.claude/architect/documentation.md` doesn't apply here (it's a Block Documentation Rule); for non-block configuration docs, `configuration/` is the established home.
- **Doc title: "WP-Admin Toolbar," not "Purge Frontend Cache."** The toolbar may grow beyond cache-purging in the future (it's plausible — sites this size often add other dev tools to the admin bar). Naming the doc after the surface, not the only current feature, lets it grow without a rename.
- **Document every sub-item, not just the parent.** The AC says "all available options"; the parent alone doesn't refresh anything — clicking it just expands the dropdown. The sub-items are the value-add and each does something different.
- **Use the actual toolbar labels ("Purge Frontend Cache," "Purge X").** The existing inline references in `colophon.md` and `regional-offers-carousel.md` use "Clear Cache," which doesn't match the code. The new doc uses the correct wording, and the existing references get updated to match.
- **Note administrator-only visibility explicitly.** Editors, authors, and contributors don't see the toolbar — the doc must state this so support staff don't waste time looking for it on the wrong account.
- **Take fresh screenshots showing the dropdown expanded.** The existing `colophon/clear-cache.png` shows only the collapsed parent. A reader of the new doc benefits from seeing what the expanded dropdown actually looks like.
- **Centralize the template-part workflow in `admin-toolbar.md`, link from block docs.** PR #1928 review surfaced that the colophon doc was inlining a workflow that will recur in every template-part-affected block doc (header, sidebar, future template parts). Promoting it to a named subsection in admin-toolbar with a stable anchor lets future block docs link to it instead of duplicating it. Matches the plan's North Star — admin-toolbar is the canonical reference.
- **Frame the homepage-save as a workaround, not a homepage-specific requirement.** ~~The real reason is that `wp_template_part` saves don't fire any revalidation hook, so re-saving any page is needed to trigger a path-based revalidation.~~ **Reversed 2026-04-28** — Hunter retested and confirmed the homepage-save step was never required. On local, the 5-second `ON_DEMAND_CACHE_TIME` fallback catches template-part edits without any manual action; on Vercel staging/prod, **Purge Site Settings alone is sufficient** because template parts are part of the `GET_WORDPRESS_GLOBALS_QUERY` response (verified in `frontend/lib/wp/functions/get-site-globals.ts:194-196` — `globalTemplateParts.headerJSON` and `footerJSON` come from the same `allSettings` query that everything else in site globals comes from). `revalidateTag('site_globals')` invalidates the `fetchSiteGlobals` cache entry and propagates to page renders that consumed it. The homepage-save step was folklore from a misobservation.
- **THR-1782 reframed.** Originally filed as "PostRevalidateHandler should hook `wp_template_part` so template-part edits auto-revalidate." The premise — that template-part edits don't get pushed live without manual intervention — was wrong. Purge Site Settings is the lever, and it works. Auto-revalidation on `save_post` for `wp_template_part` is still a possible UX improvement (one click less for the admin), but its priority drops sharply: it's a convenience, not a fix for broken behavior. Recommend closing THR-1782 or downgrading it.
- **Land THR-1776 before THR-1723 (Option A).** PR #1903 was still open when THR-1776 was scoped, so `regional-offers-carousel.md` lived only on that branch. Rather than block on a merge, this PR ships the new admin-toolbar doc first and PR #1903 absorbs the link swap as a review fix on its own branch. Lower coordination cost — both updates land in their natural ticket.
