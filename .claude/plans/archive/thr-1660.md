## Decisions

- `startAtEnvironmentIndex` is a **post-filter index** — the number counts into the list of environments that will actually be dispatched, matching the order operators see in the run log. Slice runs after filter.
  - **Why:** The method's own doc-comment says the parameter is "useful for skipping environments that have already been processed" — that means dispatched, which means post-filter. The prior implementation contradicted its own comment and produced off-by-K restart math where K = the number of skipped envs before the restart point. The THR-1660 prefix additions widen that K, making the footgun more likely to surface.
- **Dispatch log includes `[displayIndex/total]` in absolute, post-filter coordinates.** `displayIndex` = `startAtEnvironmentIndex + iterationIndex`, `total` = `startAtEnvironmentIndex + environments.length`.
  - **Why:** Operators watching a cancelled run need to copy the last successfully-dispatched index directly into `START_AT_ENVIRONMENT_INDEX`. Displaying the absolute (pre-resume-offset) index removes mental math — what you see in the log is what you paste into the env var.
- **`EnvVarUpdateService` follows the same filter-then-slice ordering and logs the same index format** — not strictly required by this ticket's original scope, but folded in because the two services share the operator-facing contract and divergence is worse than the extra lines of diff.
  - **Why:** An operator using one tool shouldn't discover that the other tool counts differently. Keeping the convention shared across the bulk-service family is the only scalable answer, and the duplication isn't worth a separate ticket.
- **Abstract `dispatchWorkflow` signature grows by two parameters (`displayIndex`, `total`)** rather than logging in the base class.
  - **Why:** Subclasses render dispatch-specific verbs ("deployment" vs "WP core update") and should keep owning that phrasing. Passing iteration coordinates through the signature keeps responsibility where it belongs — the base class owns iteration, subclasses own the dispatched action. Two small params beats a generic base-class log that loses context.
- **`START_AT_ENVIRONMENT_INDEX` stays an index, not a name.** Renaming to `START_AT_ENVIRONMENT_NAME` ("resume after ogem") is a better UX but a larger, separately-scoped change — filed as a future follow-up rather than bundled in here.

### Followup: architecture-doc-shape rule

- **Architecture docs in `docs/content/dev/architecture/` open with a four-beat arc: need → technical flows → natural fit → platform limits + custom layer.**
  - **Why:** Mirrors how architects actually reason through a problem (understand the need → think through technical flows → research existing solutions → build custom only where nothing fits). Makes design choices visible rather than collapsing them into procurement narrative. Prevents regression to "lead with the mapping" structure.
- **Architecture and operations docs live in paired files.** `docs/content/dev/architecture/<topic>.md` explains shape and reasoning; `docs/content/dev/operations/<topic>.md` owns procedures and keystrokes. Cross-link both ways.
  - **Why:** Two distinct readers (first-time architect vs. 3am operator) with different needs — a single doc that serves both compromises both. The Two-Reader Model was first surfaced abstractly on 2026-04-22; this followup applies it concretely.
- **The four-beat arc and supporting principles are codified at `.claude/architect/architecture-doc-shape.md`, referenced from `documentation.md`.** Scoped writing standards (like this one, which applies only to architecture docs) live in `.claude/architect/` with a manifest key so they load only when relevant files are in scope; cross-cutting writing standards live in `.claude/rules/`. ### Followup: CI pipeline architecture (context for the doc rewrite)

These aren't general rules — they're the load-bearing architectural facts about the CI pipeline itself that the `ci-pipeline.md` rewrite must surface. Captured here so they survive from conversation into Eli's prose intact.

- **Every dealer is two deployment targets: Cloudways Autonomous (WordPress backend) + Vercel (Next.js frontend), deployed backend-first.**
  - **Why:** The frontend queries GraphQL schema from the backend. If the frontend ships before the backend schema, dealer sites return schema-missing errors until the backend catches up. Backend-first with success verification before starting frontend prevents the gap. Every dealer therefore needs scoped secrets and deploy history for both halves independently, and operators need to dispatch backend-only or frontend-only for schema-safe hotfixes.
- **GitHub Actions is the orchestration layer that enforces backend-before-frontend sequencing — not just a dispatch layer.**
  - **Why:** Two independent workflows that happen to deploy the right things aren't sufficient — we need a coordinator that starts the frontend deploy _only after_ the backend deploy returned success. GitHub Actions provides that via workflow dependencies (`needs:`), status gates, and workflow-chaining triggers. This orchestration capability is a first-class feature we're leveraging, not a side-effect of using Actions for dispatch. It's also the single biggest reason git-integration auto-deploy is insufficient: git integration has no primitive for "wait for platform A success before triggering platform B."
- **Git-integration auto-deploy (offered by both CWA and Vercel) was considered and rejected.**
  - **Why:** Git integration has three structural gaps against our requirements — no orchestration (can't sequence backend success → frontend start), no selective dispatch (can't do backend-only or frontend-only hotfixes), and no fleet-wide fan-out primitive. Keeping dispatch in GitHub Actions keeps ordering, selectivity, and coordination under our code. This is load-bearing enough that the doc must preempt the reader's "why not just use git integration?" question explicitly.
- **GitHub Environments is the single source of truth for the fleet list.**
  - **Why:** Any fan-out primitive that requires a parallel list of dealers (matrix strategies, static arrays, external JSON) drifts against reality as dealers are added or removed. `workflow_dispatch` reading the environments list at runtime means the primitive IS the manifest — one source of truth, no drift. This is why matrix was rejected, not the 256-job cap.
- **Three platform rate limits stack to force resumability: GitHub's 6-hour per-job timeout + CWA's deploy rate limit + Vercel's deploy rate limit.**
  - **Why:** Combined with 400+ dealers and two-stage per-dealer deploys, a full release routinely exceeds 6 hours wall-clock. The GitHub 6-hour cap alone wouldn't force resumability — it's the intersection with the two downstream rate limits that makes "no single workflow job can finish a release" the actual constraint. Resumability isn't cleverness; it's the only shape that fits.
- **TypeScript + Octokit for the `ci-scripts/` layer.**
  - **Why:** Single language across the team's tooling (no bash/Python divergence). TS catches API-shape bugs that thin test coverage misses. Octokit's typed surface gives contractual confidence on the REST calls we depend on — no hand-built queries that drift against the API.
- **Backend deploys use rsync (via the WPE Deploy GitHub Action, which is rsync under the hood). Frontend deploys use Vercel's build-trigger.**
  - **Why:** Captures what "Deploy Site dispatches to a dealer" actually means per-half. Useful context for anyone diagnosing a deploy failure on one target vs. the other.
- **Fleet homogeneity is the support model. A half-deployed fleet breaks the single-codebase promise.**
  - **Why:** The moment dealers run different versions, support becomes "figure out which version this dealer is on before diagnosing." That cost is borne by everyone downstream (support, QA, on-call engineering) and undermines the whole reason we keep a single codebase for 400+ sites. Resumability keeps the fleet in a single-version state even when a release spans multiple workflow jobs.
