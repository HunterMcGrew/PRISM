# Plan: THR-1616

## Ticket

https://linear.app/tractru/issue/THR-1616/mega-menu-mobile-locations-links-open-in-new-tab

## Goal

Converge the Thrive mega menu (desktop + mobile) on editor-controlled link targeting — remove the URL-shape `urlIsExternal → "_blank"` fallback so links default to same-tab unless the editor explicitly set `target="_blank"` via block settings.

---

## User Stories

- As a mobile visitor on a dealer site, I want tapping a Locations link in the mega menu to open in the same tab, so that I stay in my browsing context and don't see a black flash during transition.
- As a mobile visitor, I want all mobile mega menu panels (featured links, navigation list, links with icons) to open links in the same tab unless the editor configured otherwise, so that navigation feels consistent.
- As a desktop visitor, I want a top-level mega menu link to open in the same tab unless the editor configured it as a new-tab link, so that the same link behaves the same way on desktop and mobile.
- As an editor, I want my explicit "open in new tab" choice in the block settings to be the only thing that triggers new-tab behavior, so that my link configuration is predictable.

## Decisions

- Mega menu link `target` is editor-controlled only — the renderer does not default to `_blank` based on URL shape (`urlIsExternal`). URL-shape inference substitutes a heuristic for editor intent, and blocks without a target UI (Locations, `type="link"` top-level nav) have no way to override it. Only `isLinkTarget(link.target)`-validated editor input drives target.
- Scope includes both mobile (`resolver.props.ts`) and desktop (`MegaMenuNavigationItem.tsx`) Thrive mega menu rendering — fixing only one side creates cross-breakpoint inconsistency on the same link.
- Legacy mega menu renderers (`blocks/site-header/MobileMegaMenu.tsx`, `components/navigation-menu-item/NavigationMenuItem.tsx`) are explicitly out of scope — they serve the pre-Thrive menu, which may have different dealer expectations about auto-external behavior.
- `urlIsExternal` utility is not modified — it has valid consumers outside the mega menu (e.g. `LocalVideo`) where URL-shape inference for offsite navigation is still appropriate. The bug was using URL shape to decide `target`, not the utility itself.
- Editor-set `rel` is emitted regardless of `target`. This preserves the editor UI contract — `LinkControl.tsx` exposes `Link rel` as an independent input, and `setLinkRel` is a first-class utility. When `target="_blank"` and no editor-set `rel`, `rel="noopener noreferrer"` is applied as a security default. This decision was revised during Briar's self-review — the original "rel only when target" rule silently dropped legitimate editor data (e.g. `nofollow` on a same-tab sponsor link).

## Debugged Issues

### Mobile mega menu auto-opens absolute URLs in new tab
- **Status:** `fixed`
- **Fixed in:** Removed `urlIsExternal → "_blank"` fallback from `parseLocationsGroup`, `normalizeLink` string branch, and `normalizeLink` object branch in `frontend/blocks/mega-menu-renderer/resolver.props.ts`. Also removed the parallel branch in desktop `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx`. Target is now editor-controlled only; `rel` attaches only when a target is set.
- **Severity:** High
- **Environment:** Mobile (iOS/Android), production/staging. Surfaced under the Locations panel; latent in all other panel types.
- **File:** `frontend/blocks/mega-menu-renderer/resolver.props.ts:208-211` (Locations), `:304-307` and `:326-330` (`normalizeLink`, used by featured-links, navigation-list, links-with-icons, top-level nav)
- **Root cause:** The resolver uses `urlIsExternal(href)` (shape check: `href.startsWith("http")`) as a fallback to default `target="_blank"` and `rel="noopener noreferrer"`. Locations has no editor UI to override this, so any location URI that arrives scheme-prefixed is auto-tagged external and opens in a new tab. Other panel types honor an editor-set target first, but fall back to the same heuristic when no target is set — so absolute internal URLs in those panels are also auto-`_blank` on mobile.
- **Steps to Reproduce:**
  1. On mobile, open a dealer site using the Thrive mega menu
  2. Tap the hamburger menu, navigate to the Locations panel
  3. Tap a location link
  4. Observe: link opens in a new tab (source page flashes black during transition)
- **Expected behavior:** Location links open in the same tab. Links in all mobile mega menu panels open in the same tab unless the editor has explicitly configured `target="_blank"` on the block.
- **Actual behavior:** Location links always open in a new tab when the URI starts with `http`. Featured-links / navigation-list / links-with-icons links open in a new tab when the editor hasn't set a target and the URL is absolute.
- **Recommended fix:** Remove the `urlIsExternal → "_blank"` fallback in three places. Default `target` to `undefined` unless the editor explicitly set a valid `LinkTarget`.
  1. `parseLocationsGroup` (`:208-211`) — drop the `target`/`rel` derivation; always `undefined` (Locations has no per-location editor target).
  2. `normalizeLink` string branch (`:301-315`) — always `undefined` for raw string links.
  3. `normalizeLink` object branch (`:326-333`) — keep `isLinkTarget(link.target) ? link.target : undefined`; drop the `urlIsExternal` else-branch. Preserve editor-set `rel` only when a target is set.
- **Suggested tests:**
  - `resolver.props.test.ts` — add cases: Locations with absolute-URL URIs default to same-tab; `normalizeLink` with no editor target defaults to same-tab regardless of URL shape; editor-set `_blank` is honored.
  - `MegaMenuMobileMenu.test.tsx` — assert rendered `<a>` has no `target` attribute for locations and for other panels without editor-configured targets.
- **Linear:** `not synced`

### `rel` silently dropped when editor sets rel without target
- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** `normalizeLink` object branch in `resolver.props.ts` — `rel` is now derived as `editorRel ?? (target === "_blank" ? "noopener noreferrer" : undefined)`. Editor-set `rel` is preserved regardless of `target`. Test case `"does not attach rel when there is no target, even if the editor set rel"` was flipped to `"preserves an editor-set rel on a same-tab link (no target)"` to assert the new contract. Plan Decision #5 updated; architect context updated.
- **File:** `frontend/blocks/mega-menu-renderer/resolver.props.ts:313-315` (new `normalizeLink` object branch)
- **Problem:** The new logic `rel = target ? (toNonEmptyString(link.rel) ?? "noopener noreferrer") : undefined` discards an editor-set `rel` when the editor did not also set `target`. This contradicts the editor UI: `backend/plugins/gravity-platform-core/src/components/editor-sidebar/LinkControl.tsx:94` exposes `Link rel` as an independent input from the target checkbox, and the codebase has `setLinkRel` as a first-class utility (per `.claude/architect/mega-menu.md:144`: "setLinkTarget(link, undefined) deletes both target and rel — callers use setLinkRel to set rel independently"). A dealer setting `rel="nofollow"` on a same-tab sponsor link in the editor will now see that value silently dropped on the frontend. Plan Decision #5 calls for this exact behavior, but was made without inspecting the editor UI.
- **Suggested fix:** Decouple — emit editor-set `rel` regardless of `target`. Keep the `noopener noreferrer` fallback only when `target="_blank"` and no editor-set `rel`. Proposed:
  ```ts
  const target = isLinkTarget(link.target) ? link.target : undefined;
  const editorRel = toNonEmptyString(link.rel);
  const rel = editorRel ?? (target === "_blank" ? "noopener noreferrer" : undefined);
  ```
  Then update Plan Decision #5 and the "does not attach rel when there is no target" test case to reflect the new contract.

- **Suggested fix:** Either (a) leave as-is and note in the plan that the attribute is now unconditional, or (b) drop the `data-testid` entirely since the jest mock supplies it for tests regardless.
