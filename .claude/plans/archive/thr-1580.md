# Plan: THR-1580

## Ticket

THR-1580: Add PR labeling system to Eric's review skill

## Goal

Give Eric (the PR review skill) a two-label system that lets the lead dev scan the PR list and immediately know two things: how long the review will take (effort) and how much to trust Eric's verdict (confidence).

---

## Decisions

- **Two labels per PR, always.** Every labeled PR gets exactly one effort label and one confidence-slot label. Never one, never three. The two dimensions are independent signals — effort tells "how long", confidence tells "how much trust."
- **Zero labels = not ready.** When critical or major issues exist, Eric applies no labels. The absence of labels is the signal — no dedicated "blocked" label needed.
- **`review:has-minors` takes the confidence slot.** It is not a third label — it replaces `confidence:high` or `confidence:needs-judgment` when unresolved minors exist. This keeps the label count at exactly two.
- **Flag labels retired.** `flag:security`, `flag:shared-code`, `flag:new-pattern`, and `flag:a11y` do not exist as GitHub labels. Those observations are called out in the summary comment body instead. Why: flags added noise as labels (3+ labels per PR), and the summary comment is where reviewers read details anyway.
- **Three-state decision gate replaces priority cascade.** The old single-label priority cascade (first match wins from a ranked list) was replaced because it collapsed two independent signals into one. The new gate has exactly three outcomes: (1) critical/major → 0 labels, (2) unaddressed minors → effort + has-minors, (3) all clear → effort + confidence.
- **Design iterated through 3 versions.** v1: multi-label (PR #1766). v2: single-label priority cascade (original state of PR #1767). v3: two-label decision gate (current state of PR #1767, on branch `-2`). Each iteration was driven by user feedback — v1 was too noisy, v2 lost the confidence signal, v3 balances both.

## Label Inventory

### Effort — how long will the human review take?

| Label | Color | Criteria |
|---|---|---|
| `effort:glance` | `0E8A16` (green) | Only plan files, docs, config, or copy changed. No logic changes. |
| `effort:quick` | `FBCA04` (yellow) | Single concern, 3 or fewer files with logic changes. Tests present for new logic. |
| `effort:deep` | `D93F0B` (red) | More than 3 files with logic changes, multiple concerns, cross-cutting (frontend + backend). Default when criteria are ambiguous. |

### Confidence — how much should the reviewer trust Eric's verdict?

| Label | Color | Criteria |
|---|---|---|
| `confidence:high` | `0E8A16` (green) | Zero issues, or all issues are minor and clearly actionable. No ambiguity, no UX judgment calls, no untestable behavior. |
| `confidence:needs-judgment` | `E4E669` (yellow-green) | Eric couldn't make the call — UX tradeoffs, business logic correctness, ambiguous requirements, or behavior Eric couldn't verify. |
| `review:has-minors` | `FBCA04` (yellow) | Unaddressed minor issues remain. Takes the confidence slot — the reviewer needs to check whether the minors matter. |

## Decision Gate Logic

```
IF critical or major issues exist:
  → apply 0 labels (strip any existing)

ELSE IF unaddressed minors remain:
  → apply effort:<level> + review:has-minors

ELSE (all clear — zero issues or all minors addressed/acknowledged):
  → apply effort:<level> + confidence:<level>
```

"Developer-acknowledged" = the PR author replied as the last comment on a minor thread. No magic words required.

## Acceptance Criteria

### Behavioral
- [ ] Given Eric reviews a PR with no critical or major issues and no unresolved minors, When Eric applies labels, Then exactly two labels are applied: one effort label and one confidence label
- [ ] Given Eric reviews a PR with unresolved minor issues, When Eric applies labels, Then exactly two labels are applied: one effort label and `review:has-minors`
- [ ] Given Eric reviews a PR with critical or major issues, When Eric finishes the review, Then no labels are applied
- [ ] Given Eric re-reviews a PR, When Eric starts the review, Then all previous review labels are stripped before new ones are applied
- [ ] Given a PR has security concerns, shared-code blast radius, new patterns, or a11y observations, When Eric writes the summary, Then those observations appear in the summary comment body, not as labels

### Non-behavioral
- [ ] No `flag:*` labels remain in the strip command or label definitions
- [ ] The decision gate has exactly three outcomes — no fourth path exists
- [ ] Every labeled PR gets exactly two labels — the skill text explicitly says "never one, never three"
