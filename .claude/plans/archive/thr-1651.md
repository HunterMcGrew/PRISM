## Decisions

- **The THR-1610 control was built on a false premise.** `core/columns` does not have a container-level horizontal alignment control. The author of THR-1610 assumed it did and implemented the feature to match. Removing the control restores parity with core's actual behavior and matches the product intent ("mega-menu-columns is a stripped-down core/columns"). Reason: the feature has no product requirement behind it, only a mistaken mental model.
- **Remove rather than fix.** The specificity fix from the earlier THR-1651 approach works, but fixing a feature that shouldn't exist is wasted motion. Removal is the correct resolution given the feature has never shipped.
- **Keep child-column content alignment intact.** `mega-menu-column` (singular, the child block) has its own `horizontalAlignment` for content inside a single column. This matches `core/column`, is unrelated to the container-level control being removed, and stays.
- **Keep `BLOCK_ALIGNMENT_MAPPINGS` in `theme.ts`.** It's a cross-block primitive used by child-column content alignment and `extend-column-block.tsx`. Not THR-1610-specific.
- **Restore `ColumnBlock`'s `w-full` default.** The earlier THR-1651 decision to have the flex container own child sizing depended on `horizontalAlignment` existing on the container. With the container-level control gone, `ColumnBlock` needs its own width default again (pre-THR-1651 behavior).
- **Retitle the ticket.** Original title was "Mega Menu Columns horizontal alignment setting not applied." The resolution is now "Remove horizontalAlignment from mega-menu-columns to match core/columns." Flag in the PR so reviewers read the diff as intended.
- **None of this is in production.** The mega menu and these blocks haven't shipped, so no migration, no data loss, no deprecation cycle needed.

## Debugged Issues

### Mega Menu Columns horizontal alignment — specificity loss, not wiring

- **Status:** `superseded` — ticket pivoted to feature removal; specificity bug moot
- **Fixed in:** N/A — removing the feature eliminates the bug
- **Severity:** Medium
- **Environment:** Mega Menu editor (wp-admin) AND frontend (staging + local)
- **File:**
  - Editor loss: `backend/plugins/gravity-platform-core/src/blocks/mega-menu-columns/editor.scss:14`
  - Frontend loss: `frontend/blocks/columns/alignment-classes.ts:14` + `frontend/blocks/columns/ColumnBlock.tsx:58`
- **Root cause:** The attribute is registered, the toolbar writes it, and the mapped classes render. But the CSS specificity math in the THR-1610 followup fix (c78c00397) was wrong. `[&>*]:flex-none` compiles to `.<class> > * { flex: none }` with specificity **(0,1,0)** — not (0,1,1) as assumed. Editor scss at (0,2,0) wins; frontend ties with `w-full` on `ColumnBlock` at (0,1,0). Moot now — the feature is being removed.
- **Linear:** `synced`

- **Suggested fix:** Swap the two lines so the order matches `main`. Minimizes the diff's surface area and keeps the "just put it back where it was" contract clean. Defer if reviewers disagree — it's bikeshed territory.
