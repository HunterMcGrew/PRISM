# Branch Plan: feature/thr-1132-mega-menu-locations-block-2

## Ticket

https://linear.app/tractru/issue/THR-1132/locations-list-block-for-mega-menu

## Goal

Add a `mega-menu-locations` block that allows editors to select, reorder, and display locations inside a mega menu column, with optional sidebar images.

---

## Decisions

- Component lives in `frontend/components/mega-menu-locations/MegaMenuLocations.tsx` — matches all other mega-menu blocks and `code-standards-ts.md` rule ("Block components live in `frontend/components/`, not `frontend/blocks/`"). Corrected from initial placement in `frontend/blocks/` per jmotes' review.
- `resolver.props.ts` is used because the shared `resolveProps` function is called by `resolver.ts`. Backend `edit.tsx` fetches independently via `ServiceFactory` rather than importing `resolver.props.ts` — both access the same `LocationService.getItemsByIds`.
- `IDataAccessLayer<T>` uses a single generic — `TId` removed. Only 1 of 13 repos needed non-string IDs, which doesn't justify a generic parameter. Repos with non-string native IDs (e.g. events) handle conversion internally. Keeps the contract consistent with `getItemById(id: string)`.
- `MegaMenuContext.tsx` and `MegaMenuEditor.tsx` are stubs pending PRs #1626 and #1627. Stub comment stays until those PRs land; stub removal is a follow-up task.
- Props type must be named `MegaMenuLocationsBlockProps` (not `BlockFrontendProps`) and must not extend `BlockAttributes` — new blocks follow the decoupled pattern from `.claude/rules/component-props-decoupling.md`.
- `noItemsConfig` is required on `MegaMenuLocationsBlockProps` — the resolver always provides it via `DEFAULT_ATTRIBUTES`; `?` operator on it was a symptom of the coupled `extends BlockAttributes` pattern.

## Acceptance Criteria

### Behavioral

- [ ] Given the WordPress block editor is open, when the user opens the block inserter inside a mega-menu-column, then the Locations List block appears as an insertable option
- [ ] Given the Locations List block is inserted, when the editor opens the Locations panel in the sidebar, then a dropdown lists all published locations from WordPress
- [ ] Given one or more locations are selected, when the user drags a location card to a new position, then the order updates in both the editor preview and on the frontend
- [ ] Given a location is displayed in the editor, when the user clicks the remove button on its card, then the location is removed from the list and from the saved attributes
- [ ] Given the Sidebar Image One selector is empty, when the user uploads an image, then it appears as the first sidebar image in the editor preview
- [ ] Given Sidebar Image One is set, when the user removes it, then the image clears and the second image slot collapses if it was also empty
- [ ] Given the no-items config controls, when the editor sets a custom empty-state message, then the frontend renders that message when no locations are selected
- [ ] Given no locations are selected and noItemsConfig type is "none", when the page renders, then the block renders nothing (no container or placeholder)
- [ ] Given locations are selected and saved, when the frontend page renders, then locations appear in the same order as configured in the editor
- [ ] Given one or two sidebar images are configured, when the frontend renders, then the image(s) appear in a sidebar column beside the locations list

### Non-behavioral

- [ ] Block is only insertable inside a `mega-menu-column` block (enforced via `block.json` parent constraint)
- [ ] Alt text from the WordPress media library is stored and rendered on sidebar images
- [ ] The empty-state message is accessible — wrapped in an `aria-live="polite"` region so screen readers announce it
