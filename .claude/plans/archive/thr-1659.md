## Decisions

- Do not modify legacy `frontend/blocks/site-header/MobileMegaMenu.tsx`. **Why:** inherited don't-touch decision from THR-1617 — legacy file is being phased out. **How to apply:** scope all fixes to `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx` only.
- Fix for THR-1659 is a one-line removal of `focus-visible:text-primary` from `ITEM_CLASSES`. **Why:** the CSS swap is the responder to whatever browser heuristic is triggering the phantom highlight — removing it closes the entire bug path regardless of the exact WebKit mechanism. We considered redirecting `<Dialog initialFocus>` instead; Winston's read of Headless UI v2.2.9 source confirms `InitialFocus` is disabled on touch devices via `matchMedia("(pointer: coarse)")`, so that fix would be a no-op on iPhone. **How to apply:** single-line CSS change, no `SlidingPanel` modification, no shared-component surgery.
- Focus indication in the mobile mega menu relies on the browser's default UA focus outline on `<a>` elements (keyboard nav). **Why:** the UA outline uses the browser's internal focus-visible heuristic, which is narrower than WebKit's CSS `:focus-visible` match — it only fires on genuine keyboard interaction, so no phantom flash on touch. No custom focus class needed. **How to apply:** do not add `focus-visible:outline-*` or `focus-visible:ring-*` — they have the same `:focus-visible` trigger as the removed color swap and would reintroduce the iOS bug path.
- Active-link color (`text-primary`) and focus indication are semantically different concerns. **Why:** this is the third pass at the same user-visible bug (PR #1797, PR #1828, now THR-1659), each via a different mechanism all funneling through `text-primary` on a pseudo-class matching focus/hover. The structural cause is conflating focus state with active-page state via a shared color token. **How to apply:** active-link indication (when implemented in a future ticket) must use `aria-current="page"` with multi-channel styling (font weight + border accent + color, not color alone). It must NOT share a CSS selector with focus state. Never use `focus-visible:text-*` color swaps on anchor elements in this component tree.
- The root cause mechanism is not fully verified on device. **Why:** Sasha's initial hypothesis (Headless UI FocusTrap calls `.focus()` on the first anchor) was partially correct for desktop but is disabled on touch per Headless UI v2.2.9 source. What *actually* triggers the phantom on iOS remains uncertain — could be an iOS-specific `:focus-visible` heuristic edge case, sticky focus state from tap, or WebKit's handling of focus transfer during element unmount. **How to apply:** the CSS removal is a category eliminator, not a targeted patch — it closes the path without requiring mechanism certainty. If a similar bug appears in another surface, device-debug before guessing.

## Debugged Issues

### Mobile mega menu: first `<Link>` anchor in newly-mounted panel appears pre-highlighted (post-#1828 residual)
- **Status:** `fixed`
- **Fixed in:** [MegaMenuMobileMenu.tsx:32-33](frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L32) — removed `focus-visible:text-primary` from `ITEM_CLASSES`. Lock-in test added at [MegaMenuMobileMenu.test.tsx](frontend/components/mega-menu-renderer/__tests__/MegaMenuMobileMenu.test.tsx) in the accessibility describe block, asserting no anchor across top/second/third level panels carries the class.
- **Severity:** Medium (S3)
- **Environment:** iOS Safari (confirmed by Laura Gray, 2026-04-22 staging screenshots on Linear ticket THR-1617 comment thread); Android Chrome suspected, not yet confirmed
- **File:** `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx:33` (the `ITEM_CLASSES` constant)
- **Root cause (revised after device QA on 2026-04-22 surfaced the actual mechanism):** The first THR-1659 fix (removing `focus-visible:text-primary`) did NOT resolve the bug — user confirmed on iPhone via ngrok tunnel that the phantom still appears, specifically on the first `<a>` in the FIRST third-level panel opened (not subsequent third-level panels). That asymmetry pointed at a coordinate-based mechanism, not a focus-state mechanism. The actual cause: **`frontend/styles/globals.css` lines 10-18 define a global `a:hover` rule** (`a:hover { color: theme("textColor.link.hover"); }`) that colors every anchor on the site `primary-light` on hover. On iOS Safari, tapping an element applies sticky `:hover` state at that tap coordinate; when React re-renders the mega menu panel, whichever `<a>` lands at that coordinate inherits the sticky state and the global rule colors it. Tap on first navigation-list button (Y≈100) → third-level renders with first link at Y≈80-100 → coordinate overlap → phantom. Tap on second navigation-list button (Y≈200) → third-level renders with first link still at Y≈80-100 → no overlap → no phantom. Buttons are immune because `a:hover` only applies to `<a>`. Tailwind's `hoverOnlyWhenSupported` future flag is not enabled in `frontend/tailwind.config.js`, and the global `a:hover` rule has no `(hover: hover)` media guard — so it fires on touch devices. This is the **same** underlying mechanism THR-1617 / PR #1797 originally identified for sticky hover, just surfacing through a different CSS source (globals.css base rule instead of a local `hover:text-primary` utility on `ITEM_CLASSES`). PR #1797 only closed the local path, not the global one.
- **Steps to Reproduce:**
  1. On iOS Safari, open a dealer site using the new mega menu at the mobile breakpoint
  2. Tap the hamburger icon to open the mega menu
  3. Tap an expandable top-level item (dropdown)
  4. Tap a second-level "navigation-list" item to open its third-level panel
  5. **Observe:** the first link in the third-level panel renders in the primary/active color before the user taps anything
- **Expected behavior:** No link is pre-highlighted in any panel. Active/primary color applies only to the link matching the current page URL (via `aria-current` or equivalent) or on genuine keyboard focus (a ring, not a text-color swap).
- **Actual behavior:** The first `<Link>` anchor in the newly-mounted panel renders `text-primary` because `:focus-visible` matches programmatic focus from Headless UI's FocusTrap on anchor elements under WebKit's heuristic.
- **Final fix (after four passes):** Three-layer `ITEM_CLASSES` in `MegaMenuMobileMenu.tsx:33`:
  1. `!text-[var(--mega-menu-text-color)]` — default color (gray), `!important` to beat any base-layer anchor color rules.
  2. `hover:text-primary` — mouse-hover brand color, restoring the original pre-PR-#1797 behavior that was inadvertently lost. Pointer users now see the dealer's brand primary on hover (not the lighter `primary-light` from the global `a:hover` rule they've been seeing since #1797).
  3. `[@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]` — touch-device override with `!important`. Beats the non-important `hover:text-primary` via cascade on touch devices, neutralizing iOS sticky hover. Skipped on pointer devices, letting layer 2 take over.
  Tailwind utilities sit in `@layer utilities`, which wins over `@layer base` regardless of specificity. The earlier justification "component is `lg:hidden` so desktop hover doesn't matter" was wrong twice over — real-pointer users DO hit this component at narrow viewports (responsive debugging, iPad with mouse, tablet modes), AND the original PR #1797 hover affordance (`hover:text-primary`) was a documented design intent that got silently dropped. Scoping by device capability (`hover: none`) rather than by viewport (`max-lg:`) correctly addresses both the touch-phantom bug and the pointer-hover-feedback concern while restoring the original brand-color hover for mouse users.
- **Suggested tests:**
  - Unit: assert `ITEM_CLASSES` does not contain `focus-visible:text-primary` (locks in the fix; jsdom can't reproduce WebKit's heuristic). If the fix adds a ring-based focus indicator, assert its presence so a future refactor doesn't silently drop keyboard a11y.
  - Manual: real-device iOS Safari pass — open mega menu, drill to third-level panel, verify first link is not pre-highlighted. Repeat on Android Chrome. Verify the fix does not regress desktop behavior at `lg` breakpoint.

## Review Issues

### Default `!important` blocks `hover:text-primary` from ever applying on mouse hover
- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** [MegaMenuMobileMenu.tsx:32-33](frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L32) — removed the `!` prefix from Layer 1's `text-[var(--mega-menu-text-color)]`. Class specificity (0,1,0) already beats the base-layer `a { color: var(--wp--elements--link--text) }` (0,0,1), so the `!important` was never earning its keep and was actively blocking Layer 2's `hover:text-primary` on mouse hover. Final class string: `"py-2 pl-8 pr-4 text-lg font-semibold text-[var(--mega-menu-text-color)] hover:text-primary [@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]"`. Existing tests still pass (they assert class-name presence; the `!` removal doesn't change any asserted pattern). Device QA required on desktop mouse at narrow viewport (should now show primary on hover) and iPhone (should still be phantom-free).
- **File:** `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx:33`
- **Problem:** Current `ITEM_CLASSES` has three layers: `!text-[var(--mega-menu-text-color)]` (Layer 1, `!important`), `hover:text-primary` (Layer 2, no `!important`), `[@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]` (Layer 3, `!important`). On mouse hover, Layer 1 and Layer 2 both match (Layer 1 because `.class` matches any state, Layer 2 because `:hover` matches). Layer 1 has `!important`, Layer 2 does not. Per CSS cascade rules, `!important` beats non-`!important` declarations of the same property **regardless of specificity** — so Layer 1 (0,1,0 !imp) wins over Layer 2 (0,1,1 no-!imp) and the link stays gray on mouse hover. The `hover:text-primary` layer can never take effect. Tests currently pass because they assert class-name presence in the rendered `className` string, not computed styles — they verify the class is attached, not that it wins cascade. Device QA on desktop mouse at narrow viewport will show gray on hover, not the restored primary. This is the exact same symptom as the earlier "hover is gone entirely" Major, just via a different cascade path.
- **Verified via cascade analysis, not on device.** Iteration B of this fix (`!text-[var(...)] hover:!text-[var(...)]`) showed gray on mouse hover — user confirmed that. Iteration D (current, `!text-[var(...)] hover:text-primary [@media(hover:none)]:hover:!text-[var(...)]`) has the same Layer 1 `!important` but swapped Layer 2's target class. The Layer 1 behavior hasn't changed — it still eats Layer 2 on hover. Only Layer 3 (which also has `!important` and higher specificity via `:hover`) can beat Layer 1 — and Layer 3 only matches on touch.
- **Suggested fix — preferred:** Remove the `!` from Layer 1. New class string:
  ```ts
  const ITEM_CLASSES =
      "py-2 pl-8 pr-4 text-lg font-semibold text-[var(--mega-menu-text-color)] hover:text-primary [@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]";
  ```
  Rationale: Layer 1's `!important` was added defensively against Turbopack cache weirdness, but it's not needed for cascade correctness. `.text-[var(...)]` at specificity 0-1-0 already beats the base-layer `a { color: var(--wp--elements--link--text) }` at 0-0-1 without needing `!important`. Dropping it lets Layer 2's `hover:text-primary` win on mouse hover (0,1,1 no-!imp beats 0,1,0 no-!imp). Layer 3 still wins on touch because it's the only `!important` in the mix.
  - **Cascade verification after fix:**
    - Default state, any device: Layer 1 (0,1,0) beats globals `a {}` (0,0,1) → gray ✓
    - Mouse hover: Layer 2 (0,1,1) beats Layer 1 (0,1,0) → primary ✓
    - Touch hover: Layer 3 (!imp) beats everything → gray (phantom neutralized) ✓
- **Suggested fix — alternative:** Add `!` to Layer 2 (`hover:!text-primary`). Works because both Layer 1 and Layer 2 are then `!important`, and Layer 2 has higher specificity (0,1,1 vs 0,1,0). But this adds another `!important` to the stack — cleaner to drop the one on Layer 1 that isn't earning its keep.
- **Test updates needed:** none — existing assertions still pass with either fix (the class names don't change in fix 1; just the `!` goes away).
- **Device verification after fix:** confirm iPhone still phantom-free AND confirm desktop mouse at narrow viewport shows primary color on mega menu link hover. This is the verification step that was missed on the last push — PR description was updated before device QA happened.

- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** [MegaMenuMobileMenu.tsx:32-33](frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L32) — swapped `hover:!text-[var(--mega-menu-text-color)]` for `[@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]` so the override only fires on touch devices. Pointer-device users now fall through to the global `a:hover` rule and get the expected brand-color hover feedback. - **File:** `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx:32`
- **Problem:** `hover:!text-[var(--mega-menu-text-color)]` fires on every device type — touch AND mouse. The plan's justification ("component is `lg:hidden` so desktop users don't see it") only holds for production desktop at full viewport. Real-pointer users who hit the mobile menu (narrow browser window, responsive debugging, iPad with Magic Keyboard/mouse, tablet modes) now see zero hover feedback where they used to see the brand primary-light color change. Flagged by the user during final review — Briar missed this on the correctness sweep by taking the plan's justification at face value instead of pressure-testing the "desktop never sees this" assumption.
- **Applied fix:** Scoped the hover override to touch devices only using a `@media (hover: none)` arbitrary variant. Final class string:
  ```ts

      "py-2 pl-8 pr-4 text-lg font-semibold !text-[var(--mega-menu-text-color)] [@media(hover:none)]:hover:!text-[var(--mega-menu-text-color)]";
  ```
  Behavior: on touch devices (`hover: none`), the override fires and neutralizes sticky hover. On pointer devices (`hover: hover`), the override is skipped and the global `a:hover` rule fires normally, producing the expected primary-light hover feedback. Device verification required after merge: confirm iPhone still clean (phantom dead) AND confirm desktop at narrow viewport (responsive DevTools or narrow window) shows primary-light hover on mega menu links.
