# Plan: THR-1626

## Ticket

https://linear.app/tractru/issue/THR-1626/design-review-legacy-import-modal-auto-show-behavior-vs-savedpublished

## Goal

Decide whether the legacy-import modal's auto-show behavior should change based on saved/published mega menu state, or whether the existing user-controlled "Don't show again" checkbox is the right mechanism. Resolution: redesign dismissal as per-user (localStorage) with implicit dismissal via Skip/Import, remove the checkbox, and guard auto-show with a mega-menu-exists check.

## Design

### Summary

Reverses the THR-1612 dismissal model. Moves from site-wide WP option + explicit checkbox to per-user localStorage + implicit dismissal (button labels carry the intent). Cancel → Skip. Adds reassurance copy pointing at the toolbar button. Adds a mega-menu-exists guard to the auto-show condition. Conditional warning copy and primary button label ("Import and Review" vs "Import and Replace") based on whether a draft or published menu exists.

### Decisions worth knowing

- **Per-user (localStorage), not site-wide (WP option).** The collaborator scenario — Sarah in marketing opens the editor after Hunter dismissed — was the failure mode of the THR-1612 design. Per-user storage fixes it natively; any user gets their own first-time onboarding.
- **No checkbox.** Implicit dismissal: Skip and Import both permanently suppress; × and Escape are session-only (soft close for "I'll come back to this later").
- **Auto-show guards are 3-layered:** legacy menu exists → mega menu does NOT exist on site → user hasn't dismissed in this browser. All three must be true for auto-show.
- **Toolbar button shares the same guard as auto-show (partially).** Button visibility: `hasLegacyMenu && !hasExistingMenu`. Once a mega menu is built (draft or published), the toolbar button disappears too — onboarding affordances live only as long as the onboarding. Recovery for edge case: delete the draft to bring the button back.
- **No warning notice, no "Import and Replace" button.** Because the modal is only reachable when no mega menu exists yet, there is nothing to replace. The re-import flow is removed entirely — simpler, and removes the destructive-by-default risk of a "big blue button" that reflexive clicks could use to wipe real work.
- **Single modal variant.** First-time import only. One copy, one button label ("Import and Review").
- **Reassurance copy is doing the work the checkbox used to do.** "You can also import anytime from the 'Import from Legacy Menu' button in the header" earns the right to remove the checkbox by naming the re-entry path.
- **THR-1612 PHP surfaces are removed, not deprecated.** THR-1612 has not shipped to production, so there is no backwards-compatibility constraint. The follow-up ticket should fully remove the site option `platform_settings_mega_menu_migration_dismissed`, its constant, the `/migrate/dismiss` endpoint and route registration, and the `dismissed` field from the `/migrate/status` response.

## Decisions

- This plan supersedes THR-1612's dismissal model — THR-1612 shipped site-wide option + checkbox, THR-1626 reverses that to per-user localStorage + implicit dismissal.
- The modal's auto-show trigger already checks `hasLegacyMenu` (existing guard from THR-1612). This design adds a second check: `!hasMegaMenuOnSite` — if a mega menu exists in any state, the modal doesn't auto-show.
- All close gestures (× / Escape / Skip / Import) persist dismissal. The reassurance copy ("You can also import anytime from the toolbar button") earns this simpler model — the user never loses access to the import flow, so unified dismissal isn't a trap. Matches onboarding modal convention across SaaS products (Jakob's Law).
- THR-1612 has not shipped to production — no backwards compatibility required for the removed PHP surfaces.
- **localStorage key is unscoped** — `thrive_mega_menu_import_dismissed`, value `"1"` on presence. WP admin sessions are user-scoped server-side; per-user key scoping adds plumbing without solving a real problem for an admin task with low device churn.
- **localStorage access is guarded** — read and write are wrapped in try/catch. On failure, treat dismissal as "not dismissed" so the modal can still appear and close; the user just loses the persistence, not the ability to dismiss in the moment.
- **Toolbar button is always visible when a legacy menu exists** — `hasLegacyMenu` is the only guard. Button is decoupled from draft/published state. Rationale: Import only writes to the draft option, never the published menu. The published menu is untouchable until the user explicitly clicks Publish, so re-import can never affect what site visitors see. Draft overwrite is recoverable via revisions. Nielsen #3 (user control and freedom) + Jakob's Law: a persistent affordance is safer than a conditional one because there is no dead-end state the user has to discover a recovery path for.
- **`force: true` is restored on the frontend migrate call** — since the button is now reachable when a draft or published menu exists, the backend 409-on-conflict check would block legitimate re-imports. Passing `force: true` is the explicit "user clicked Import from a visible button, they mean it" signal. Safety comes from (a) published menu being write-protected at the backend level (import writes only to draft) and (b) revisions providing rollback on the draft.
- **Auto-show and button guards are intentionally different.** Auto-show: `hasLegacyMenu && !hasDraft && !hasPublished && !dismissed` — protects the user from an unsolicited modal interrupting their work. Button: `hasLegacyMenu` — provides a persistent, user-initiated recovery path. The passive affordance is gated for respect; the active affordance is always available for control.

## Acceptance Criteria

### Behavioral
- [ ] Given a site with a legacy menu and no mega menu yet, When an admin opens the Mega Menu editor for the first time in their browser, Then the import modal auto-shows.
- [ ] Given a site where a mega menu draft or published version already exists, When any user opens the Mega Menu editor, Then the import modal does NOT auto-show but the "Import from Legacy Menu" toolbar button remains visible.
- [ ] Given the user is in the modal, When they click Skip, Then the modal closes and does not auto-show again in this browser for this user.
- [ ] Given the user is in the modal, When they click Import and Review, Then the modal closes, the imported menu loads in the editor as the new draft, and the modal does not auto-show again in this browser for this user.
- [ ] Given the user has an existing draft or published mega menu, When they click the "Import from Legacy Menu" button and confirm Import and Review, Then the current draft is overwritten with the legacy import AND the published menu (if any) is not affected until the user explicitly publishes the new draft.
- [ ] Given the user is in the modal, When they click × or press Escape, Then the modal closes and does not auto-show again in this browser for this user.
- [ ] Given a site with a legacy menu, When the user opens the Mega Menu editor, Then the "Import from Legacy Menu" button is visible in the editor toolbar regardless of draft or published state.
- [ ] Given a site with NO legacy menu, When the user opens the Mega Menu editor, Then the "Import from Legacy Menu" button is hidden and the modal never shows.
- [ ] Given the modal was auto-opened on first visit, When a user reads it, Then they see no warning notice, the primary button reads "Import and Review", and the reassurance copy reads "The Import button is in the toolbar if you need it later."
- [ ] Given the user manually opened the modal from the toolbar Import button, When a user reads it, Then the reassurance copy is hidden (no toolbar reminder is shown because the user just clicked the toolbar button).

### Non-behavioral
- [ ] No "Don't show this again" checkbox is rendered anywhere in the modal.
- [ ] No warning notice about replacing existing menus is rendered anywhere in the modal.
- [ ] The modal has a single variant with primary button reading "Import and Review" — there is no "Import and Replace" label anywhere.
- [ ] Dismissal is persisted to localStorage — not to WP site options or user meta.
- [ ] The backend `PUT /migrate/dismiss` endpoint, its route registration, and the `MIGRATION_DISMISSED_OPTION_KEY` constant are removed.
- [ ] The `dismissed` field is removed from the `/migrate/status` response.
- [ ] Import writes only to the draft — the published menu is never modified by the import call.
- [ ] Keyboard and screen reader accessibility: Tab order covers Skip → Import and Review, Escape closes the modal and persists dismissal, focus returns to the triggering element on close.
