7. **Write `docs/content/dev/architecture/caching-and-revalidation.md`** — paired companion to the architect file.

   **Goal:** a teammate landing on this doc cold should be able to (a) understand where caching happens and why each layer exists, (b) choose between primitives without guessing, (c) know what's changing with the data-layer reshape and Next.js 16, (d) follow the touch-it-fix-it migration policy.

   **Voice:** narrative onboarding-style per `writing-voice.md`. Lead with the "why" before the "what." One concrete example per primitive. No mandate voice. Cite the lessons that drove the decisions.

   **Sections — required, in this order:**

   - **Overview** — one paragraph: the layered model, why we don't have a single global cache, the source-as-lifetime-driver framing. Cross-link to ADR-0021 (caching lives at the source axis).

   - **Why caching is layered** — explain the constraint that drives the layering. WordPress is webhook-driven (fast invalidation via `revalidateTag`); Space Station is polling-only (no push, longer TTL acceptable). A unified service-level cache can't honor both source policies — one always wins. The architecture pushes lifetime decisions down to the source. **Why this section exists:** answers the reader's first question — "why isn't there just one cache?" — before they ask it.

   - **The three caching tiers** — for each tier, four bullets: what it caches, where the primitive lives, what tags it uses, how it invalidates.
     - Block resolvers (`resolver.ts`, `resolver.props.ts`) — block-owned data, wrapped in `unstable_cache`, tagged by content type.
     - Supplementary fetchers (`single-templates/*/fetchers.ts`) — route-owned reads, wrapped in `cache(unstable_cache(...))` per the headless-architecture rule. Cite that rule.
     - Route handlers (`frontend/app/api/...`) — client-initiated, per-route caching strategy.

   - **The cache primitives** — explain each, with one short code example for each:
     - `"use cache"` — Next.js directive (15.5+, becomes canonical in 16). Top-level module functions only, serializable args. Public API around it can be any shape — class, function, factory — it's just thin delegation. Cite the 2026-04-28 lesson where Winston tangled this up.
     - `unstable_cache` — the current workhorse. Used in resolvers and supplementary fetchers. Will be retired in the Next.js 16 transition.
     - `cache()` from React — request-scoped de-dup. Used as the outer wrapper around `unstable_cache` in supplementary fetchers (per headless-architecture rule).
     **Why this section exists:** the three primitives look interchangeable until you've been bitten. This makes the choice tree explicit.

   - **Lifetime by source** — explain `ON_DEMAND_CACHE_TIME` (short fallback TTL because webhook handles fast invalidation) vs `POLLING_CACHE_TIME` (longer TTL because no push mechanism exists). Show how each maps to a source: WordPress → on-demand, Space Station → polling. Explicitly walk through why a service composing both (e.g. `OfferService`) can't have a single TTL — one source policy always loses. Cite the 2026-04-28 lesson on caching layer placement. **Why this section exists:** the "service-level caching is wrong here" rule reads as arbitrary without this scenario walked through.

   - **Revalidation tags** — the conventions and the gotcha. Per-source tag namespace. Explicit callout: `site_globals` already propagates to template-part content via the same `GET_WORDPRESS_GLOBALS_QUERY` response (cite `frontend/lib/wp/functions/get-site-globals.ts:194-196`) — this is the THR-1782 misobservation root cause; documenting it here heads off future re-derivation. **Why this section exists:** the "what tag invalidates what" question is the most-asked caching question on the team and currently has no central answer.

   - **Current state vs. target shape vs. Next.js 16** — the four-state migration-aware status block. Echo the structure used by `data-layer.md` exactly:
     - **Current State** — caching is partly at services (legacy), partly at repos (the new shape post-ADR-0021), partly missing (some entities have no caching at all). Name a couple specific examples without listing exhaustively.
     - **Target Shape** — per-source caching at the data-access layer (per ADR-0021). Service composition reads already-cached source results. Cross-cutting tags handle invalidation across compositions.
     - **Next.js 16 Target** — `"use cache"` directive becomes the canonical primitive. `unstable_cache` retired. The file shape stays the same; the wrapper changes.
     - **Touch-It-Fix-It Migration Policy** — new caching code goes in target shape from day one. Existing code migrates on touch (per the data-layer-boundaries rule scope section). No standalone migration tickets unless an entity is touched substantially enough to justify a full reshape.

   - **Review rules** — what to flag at PR review. Mirror the architect file list with two-three sentence narrative for each. Severity guidance.

   - **Cross-links** — to ADR-0021, `data-layer.md` companion in `docs/`, `headless-architecture.md` rule, the source-lifetime lesson.

8. **Write `docs/content/dev/architecture/routing-and-page-resolution.md`** — paired companion to the architect file.

   **Goal:** a teammate landing on this doc cold should be able to (a) trace how a URL becomes a page, (b) know where data fetches live in the page lifecycle, (c) understand what the single-template pattern is for, (d) add a new content type without guessing.

   **Voice:** same as task 8. The pipeline walkthrough is the spine of the doc — everything else is supporting.

   **Sections — required, in this order:**

   - **Overview** — one paragraph: catch-all routing + content-type discovery + single-templates + block tree composition. Cross-link to `frontend-app.md` companion and the caching doc.

   - **The page-resolution pipeline** — walk through it end-to-end with a single concrete example (an Offer URL like `/offers/labor-day-special`). Step-by-step:
     1. Request hits `/offers/labor-day-special` → catches in `app/[...slug]/page.tsx`
     2. Slug query → `node-by-uri` → returns content node with `__typename: "Offer"`
     3. Content type matched against the single-template registry → `single-templates/Offer.tsx` resolves
     4. Template fetches main + supplementary data via its `fetchers.ts` (cross-link caching doc for the wrap-with-`cache(unstable_cache(...))` pattern)
     5. Template generates metadata via `meta.ts` helpers
     6. Template hydrates the block tree from the `blocks` field
     7. Each block resolver fetches its own block-owned data
     **Why this section exists:** every other section is a refinement of this pipeline. New developers need the map before the legend.

   - **The single-template pattern** — one component per content type, not a generic page renderer. Trade-offs: more boilerplate, but type-safe per-content-type composition and per-template flexibility (e.g. an Offer page composes differently from a Career page). When to add a new single-template (new content type) vs. extend an existing one. **Why this section exists:** the alternative ("why not a generic renderer?") is the obvious question; answering it preempts well-meaning refactor proposals.

   - **Where data fetches live** — three places: block resolvers (block-owned), single-template fetchers (route-owned), route handlers (client-initiated). Same three tiers as the caching doc but viewed from the routing angle. Cross-link the caching doc for primitives and TTL details.

   - **Metadata composition** — `meta.ts` helpers, per-template overrides, OG tags, canonical URLs. How metadata fits into RSC rendering and when it gets generated.

   - **Sitemap generation** — `sitemap.ts` and the per-content-type sitemap files; where the URL list comes from; how often it regenerates.

   - **The catch-all and edge cases** — what happens when `node-by-uri` returns null (the 404 path), how redirects are handled (cross-link the `/api/redirects` route handler), how preview mode interacts with the catch-all if applicable.

   - **Adding a new content type** — concrete checklist as a numbered list:
     1. Register the content type in the single-template map
     2. Create `single-templates/SingleX.tsx`
     3. Create `single-templates/single-x/` fetcher folder if supplementary data is needed
     4. Add to sitemap config if the content type should appear in sitemaps
     5. Add to `meta.ts` helpers if the content type has special metadata
     6. Add to the architect manifest if a new directory pattern is introduced
     **Why this section exists:** every "how do I add a new content type" search currently ends in code-spelunking. This is the answer.

   - **Current state vs. target shape vs. Next.js 16** — four-state migration-aware status block. The single-template pattern is settled — Current State and Target Shape are largely the same. Next.js 16 changes affect the metadata generation flow (verify against Next.js 16 release notes when writing). Touch-It-Fix-It Policy: stable code stays; new content types go in the established pattern.

   - **Cross-links** — to `frontend-app.md` companion, caching doc, `data-layer.md` companion, ADR-0021.

## Decisions

- **Caching docs supplement, never override, the merged ADR-0021 and `data-layer.md`.** The new file carries the cross-cutting story (route handlers, supplementary fetchers, revalidation tag conventions, lifetime-by-source) and links into `data-layer.md` for the per-source mechanics. Reason: the data-layer reshape is the source of truth for data-access internals; a competing caching doc would fork it.
- **All three new durable artifacts use the four-state migration-aware status block.** Current State → Target Shape → Next.js 16 Target → Touch-It-Fix-It Policy. Reason: matches the convention from the data-layer reshape that landed today; future readers calibrate on a consistent shape.
- **ADR-0021 collision: rename the constants-leak ADR to 0023, not the data-layer one.** Reason: `0021-data-layer-reshape.md` is older, more-cited, and merged today. Renumbering the smaller ADR is the lower-blast-radius fix.
- **One new ADR: ADR-0024 (rsc-by-default-use-client-is-load-bearing).** ADR-0023 (the renamed constants-leak) becomes a specific anti-pattern derived from ADR-0024's general rule. Cross-link both ways.
- **Skipped ADR-0024 caching-lives-at-the-source.** Reason: ADR-0021's Decision section already captures this rule as one of its four axes. A separate ADR would duplicate the decision and create a "which one is canonical?" question. The new caching architect file cites ADR-0021's caching axis directly.
- **Architect files pair with `docs/content/dev/architecture/` companions.** Reason: precedent from `plugin-management.md`. Architect files stay tight for fast agent loading; the human-readable narrative lives in `docs/`.
- **Forms architecture deferred** — Q2 forms redo will produce its own architecture; documenting current shape would be wasted.
- **RSC general rule earns a standalone ADR rather than a fourth architect file.** Reason: it's decision-shaped (one rule, one rationale) and gets referenced from multiple architect files and rules. ADR fits the shape better than a long architect doc.
