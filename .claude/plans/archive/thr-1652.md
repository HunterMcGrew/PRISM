## Decisions

- Width lives on the **outer** `.block-editor-block-list__block` wrapper via `getEditWrapperProps` in `mega-menu-column/index.tsx`, not on `useBlockProps` style. The Gutenberg editor renders every block as two nested divs; `useBlockProps` styles land on the inner div (parent: `display: block`), where `flex-basis` has no layout effect. See `.claude/architect/backend-editor-blocks.md` § Styling the Outer Block Wrapper.
- `flex-1 w-full` on the inner className is harmless and unconditional. Both classes are effectively inert on the inner div — `flex-1` is ignored because the inner's parent isn't a flex container, and `w-full` is redundant because a block-level child of a block parent already fills its width. They stay because they match the frontend `ColumnBlock` pattern and avoid reasoning about conditionals that don't do anything.
- ~~Do not add `flex-1` or `w-full` to the Mega Menu Column editor className unconditionally.~~ **Retired 2026-04-21** — this Decision was written when `flex-basis` was on the inner div (first-pass fix); it assumed the classes would override an inline `flex-basis` on the same element. With width now on the outer wrapper, those classes can't override anything they aren't on.

## Debugged Issues

### Inline `flex-basis` lands on the wrong element — outer block wrapper is the flex item

- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/index.tsx` — added `getEditWrapperProps(attributes)` that returns `{ style: { flexBasis, flexGrow: 0, flexShrink: 0 } }` when `attributes.width` is set. Also removed redundant `flex: 1` from `mega-menu-column/editor.scss:2` (parent-scoped rule in `mega-menu-columns/editor.scss:15` still handles the default case) and dropped the now-dead inline `flexBasis` from `edit.tsx`. Kept the `!attributes.width && "flex-1 w-full"` conditional on the className per plan `## Decisions`.
- **Severity:** Medium (S3)
- **Environment:** wp-admin Mega Menu editor (backend plugin); frontend unaffected
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/index.tsx` (block registration needs `getEditWrapperProps`)
- **Root cause:** DOM evidence from the editor shows the column renders as two nested divs: an outer `.block-editor-block-list__block.wp-block` wrapper (`display: block`) and the inner `useBlockProps`-spread div that carries our `className` and `style`. The flex item inside `.mega-menu-columns-inner` is the **outer** wrapper, not the inner div. Gutenberg's browser info confirms: "The display: block property on the parent element prevents flex-basis from having an effect." So the inline `flex-basis: <width>` written in `edit.tsx` targets a block-level descendant of the flex item — it has no layout effect. The two prior diagnoses on this ticket assumed `useBlockProps` merged onto a single flex-item element; the DOM dump disproves that assumption.
- **Steps to Reproduce:**
  1. Open the Mega Menu editor in wp-admin
  2. Select a Mega Menu Column block with siblings
  3. Set Width to `10%`, `25%`, `10rem`, or `200px` in the sidebar
  4. Inspect the selected column — outer `<div class="block-editor-block-list__block wp-block">` has no width style; inner `<div ... style="flex-basis: 10%">` carries the intended value but is not a flex item
- **Expected behavior:** Editor preview resizes to the configured width, matching the frontend.
- **Actual behavior:** Inline `flex-basis` lands on a non-flex-item; the outer wrapper (the real flex item) inherits `flex: 1` from `mega-menu-columns/editor.scss:15` and renders equal width regardless of attribute.
- **Recommended fix:**
  - Add `getEditWrapperProps(attributes)` to `registerBlockType` in `mega-menu-column/index.tsx`. Return `{ style: { flexBasis: attributes.width, flexGrow: 0, flexShrink: 0 } }` when `attributes.width` is set, empty object otherwise. This is Gutenberg's idiomatic API for styling the outer block wrapper from a block's own registration — it lands style on the flex item, where `flex-basis` actually does something.
  - Keep `.mega-menu-columns-inner > .block-editor-block-list__block { flex: 1 }` in `mega-menu-columns/editor.scss:15` — it handles the default (unsized) case. When `getEditWrapperProps` writes inline flex props, they override the class rule (inline beats class specificity).
  - Remove `flex: 1` from `mega-menu-column/editor.scss:2` — redundant with the parent-scoped rule and creates two overlapping sources of flex behavior. Keep `min-width: 0` for overflow safety.
  - Remove or leave the inline `flexBasis` in `edit.tsx`'s `useBlockProps` style — it's on the wrong element and does nothing, either way. Preferred: remove it to avoid future confusion. Leave the `!attributes.width && "flex-1 w-full"` conditional on the className since the inner div's own flex behavior still matters for its own children.
  - `apps/mega-menu-editor/styles.scss:260-273` broad rule (`.editor-styles-wrapper .block-editor-block-list__layout { flex: 1 }`) is not the decisive cause here. It targets the inner (layout) element, which isn't the flex item. No change needed unless we discover a separate issue.
  - Verify: 4 columns with mixed widths (`25%`, unset, unset, unset) render with one fixed-width column and three equal-width siblings filling the rest. Confirm frontend unchanged.
- **Suggested tests:**
  - Manual editor verification: set `25%`, `10rem`, `200px`; confirm each resizes the outer wrapper; confirm clear returns to default equal-width behavior.
  - Consider asserting `getEditWrapperProps` behavior in a Pest-like unit test if block registration is testable — low value unless there's a pattern for it.

### Mega Menu Column width attribute never applied to editor DOM

- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/edit.tsx:38-54` — bound `flexBasis: attributes.width` in `useBlockProps` style, and made `flex-1 w-full` conditional on `!attributes.width`. Added an inline comment explaining why so the class pair doesn't silently creep back in.
- **Severity:** Medium (S3)
- **Environment:** wp-admin Mega Menu editor (backend plugin); frontend is unaffected
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/edit.tsx:38-50`
- **Root cause:** `useBlockProps` hard-codes `className: "... flex-1 w-full"` and `style: { minHeight: "2rem" }`. `attributes.width` is saved by `handleWidthChange` and serialized to the block delimiter, but never read into the editor's rendered style or class — so the preview never resizes. `flex-1` (`flex: 1 1 0%`) and `w-full` (`width: 100%`) would also override any `flexBasis` even if it were added, so both must be removed from the className when `attributes.width` is set.
- **Steps to Reproduce:**
  1. Open the Mega Menu editor in wp-admin
  2. Select a Mega Menu Column block
  3. Set Width to `25%`, `10rem`, or `200px` in the sidebar
  4. Observe the editor preview does not resize
- **Expected behavior:** Editor preview resizes to the configured width and unit, matching the frontend.
- **Actual behavior:** Editor preview always renders at default flex-grow width; the attribute is saved but visually ignored.
- **Recommended fix:**
  - In `edit.tsx`, bind `attributes.width` to `useBlockProps` style: `style: { minHeight: "2rem", ...(attributes.width ? { flexBasis: attributes.width } : {}) }`.
  - Drop `flex-1 w-full` from the className when `attributes.width` is set — mirror the frontend `ColumnBlock` rule (`!width && "w-full"`).
  - Add a short code comment on the className explaining why `flex-1` and `w-full` are conditional — these classes override `flexBasis` and have re-appeared in this file more than once.
  - No frontend change required: `ColumnBlock.tsx:44-75` already applies `flexBasis: width` and omits `w-full` when width is set. Existing tests in `ColumnBlock.test.tsx:120-155` cover the frontend path.
- **Suggested tests:**
  - Backend: manual editor verification — set %/rem/px, confirm preview resizes; clear the control, confirm revert.
  - Frontend: no new tests (already covered). Optional: Storybook story for ColumnsBlock with a width-configured child if not already present.
