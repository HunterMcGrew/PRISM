## Decisions

- Unregister stretchy variations in `bootstrap.tsx` after `registerCoreBlocks()` — scoped to the mega menu editor page only, doesn't affect the post/page editor where stretchy text is wanted
- Fix the Browse All popup by dispatching the synthetic Escape on `document.activeElement` instead of `document` — targets the focused element inside the Popover so the event bubbles through the Popover's DOM tree
- `core/heading` stretchy variation is also excluded even though the user only mentioned paragraph — heading is in the same allowed blocks list and the same variation mechanism
- No test code needed — both fixes are WordPress runtime API calls and DOM interactions that can only be verified manually in the editor
- The `Object.defineProperty` keyCode/which shim is the correct approach — alternatives (blur + `closeIfFocusOutside`, calling Dropdown `close()` via ref, CSS hide) all lose on determinism or ownership. The synthetic Escape is a boundary adapter to WordPress's `useDialog` internal contract, not an abstraction we're inventing. Winston confirmed on 2026-04-16.
- Using `{ get: () => 27 }` rather than `{ value: 27 }` for the keyCode/which shim — `keyCode`/`which` are accessor properties on `KeyboardEvent.prototype`, so overriding accessor-with-accessor sidesteps legacy browser quirks around accessor-to-data conversions.
- The variation-unregistration pattern is not mega-menu-specific — it applies to any standalone editor app using `BlockEditorProvider`. Promoted to `backend-apps.md` (general pattern) with a cross-reference from `mega-menu.md` (specific instance).
- Timing dependency on theme script load order is not a new fragility introduced by this PR — `src/blocks/index.tsx:99` uses the same `wp.domReady(() => wp.blocks.unregister*())` shape and has been in place for years. This is the established codebase convention for post-registration cleanup.

## Debugged Issues

### Stretchy Paragraph variation appears in mega menu quick inserter

- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/bootstrap.tsx` — added `wp.blocks?.unregisterBlockVariation?.()` calls for both `core/paragraph`/`stretchy-paragraph` and `core/heading`/`stretchy-heading` after `registerCoreBlocks()`
- **Severity:** Low
- **Environment:** wp-admin mega menu editor
- **File:** `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/bootstrap.tsx:24`
- **Root cause:** THR-1373 (commit 38665abc7) removed global `unregisterBlockVariation` calls from `blocks/index.tsx` to enable stretchy text in the post/page editor. Since `core/paragraph` and `core/heading` are in `MEGA_MENU_ALLOWED_BLOCKS`, the quick inserter shows their stretchy variations. The custom InserterSidebar (Browse All) is not affected because it uses `getBlockTypes()` (no variations) not `getInserterItems()`.
- **Steps to Reproduce:**
  1. Open the mega menu editor in wp-admin
  2. Click the + inserter button inside a mega menu column
  3. Observe "Stretchy Paragraph" appears in the quick inserter list
- **Expected behavior:** Stretchy Paragraph and Stretchy Heading should not appear in the mega menu editor inserter
- **Actual behavior:** Stretchy Paragraph appears in the quick inserter (+) but not in Browse All
- **Recommended fix:** Add `unregisterBlockVariation("core/paragraph", "stretchy-paragraph")` and `unregisterBlockVariation("core/heading", "stretchy-heading")` in `bootstrap.tsx` after `registerCoreBlocks()` — scoped to the mega menu editor page only
- **Suggested tests:** Manual — verify stretchy variations don't appear in the mega menu inserter; verify they still appear in the post/page editor
- **Linear:** `not synced`

### Browse All popup does not dismiss when sidebar opens

- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/editor.tsx:589-596` — assigned `keyCode` and `which` (both `27`) to the synthetic Escape via `Object.defineProperty` after construction so WordPress's `useDialog` Escape handler (`event.keyCode === 27`) actually fires. Kept dispatch from `document.activeElement` so the event bubbles through the Popover's dialog node.
- **Severity:** Medium
- **Environment:** wp-admin mega menu editor
- **File:** `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/editor.tsx:589`
- **Root cause (verified):** WordPress's Dropdown Popover closes on Escape via the `useDialog` hook (`@wordpress/compose/build/hooks/use-dialog/index.cjs:63-68`), which checks `event.keyCode === ESCAPE` (where ESCAPE === 27 from `@wordpress/keycodes`). Our synthetic `new KeyboardEvent("keydown", { key: "Escape", bubbles: true, cancelable: true })` only sets `event.key` — the `KeyboardEvent` constructor's init dictionary does **not** honor `keyCode` (it's a deprecated property). At dispatch time `event.keyCode === 0`, so the condition fails and `onClose` is never called. The first fix (changing dispatch target from `document` to `document.activeElement`) addressed the bubble path correctly, but the event still has the wrong shape — the listener now fires but exits early at the keyCode guard.
- **5 Whys:**
  1. Why doesn't the popup close? → Synthetic Escape doesn't reach `onClose`.
  2. Why? → WordPress checks `event.keyCode === 27`, but our event has `keyCode === 0`.
  3. Why is keyCode 0? → `new KeyboardEvent(..., { keyCode: 27 })` silently ignores `keyCode` in the init dict.
  4. Why was this missed originally? → The original implementer (commit `da5d423e3`) verified visually only; the Sidebar was conditionally rendered at the time (`isSettingsOpen ? <Sidebar /> : null`), so the popup may have appeared to disappear because nothing was overlapping. That conditional was later reverted (current code: `<Sidebar />` unconditional at editor.tsx:780), making the still-open popup visible.
  5. Why no test? → Manual verification was the only suggested test. WordPress runtime API behavior with synthetic events is inherently tricky to unit-test, but even a reproduction in a Storybook/dev fixture would have caught this.
- **Steps to Reproduce:**
  1. Open the mega menu editor in wp-admin
  2. Click the + inserter button inside a mega menu column
  3. Click "Browse All" in the quick inserter popup
  4. Observe the sidebar opens but the popup stays visible
- **Expected behavior:** The inline inserter popup should close when the sidebar inserter opens
- **Actual behavior:** The popup remains open, overlapping with the sidebar
- **Recommended fix:** Add `keyCode` and `which` to the synthetic event via `Object.defineProperty` after construction (the constructor ignores them in the init dict), and keep dispatching from `document.activeElement` so the event bubbles through the Popover's dialog node:
  ```js
  const escapeEvent = new KeyboardEvent("keydown", {
  	key: "Escape",
  	bubbles: true,
  	cancelable: true,
  });
  // WordPress's useDialog escape handler checks event.keyCode === 27,
  // not event.key === "Escape". KeyboardEvent's constructor ignores
  // keyCode in the init dict, so we have to assign it post-construction.
  Object.defineProperty(escapeEvent, "keyCode", { get: () => 27 });
  Object.defineProperty(escapeEvent, "which", { get: () => 27 });
  (document.activeElement ?? document).dispatchEvent(escapeEvent);
  ```
  Alternative cleaner approach: blur the active element (`(document.activeElement as HTMLElement)?.blur()`) — the Dropdown's `closeIfFocusOutside` handler will see focus has left the container and call `close()` after a debounce. Slightly less direct but avoids the keyCode hack. Recommend the keyCode fix because it matches the user pressing Escape (which is what we're conceptually doing).
- **Suggested tests:** Manual — click Browse All in the quick inserter and verify the popup closes immediately. If feasible, add an editor smoke test that mounts the Editor, clicks the inserter, clicks Browse All, and asserts the popup container is removed from the DOM.
- **Linear:** `not synced`
- **Historical note:** No "lost-in-merge" fix was found. The user's recollection may stem from commit `da5d423e3` (which introduced both the synthetic-Escape attempt AND `isSettingsOpen ? <Sidebar /> : null` conditional rendering). The conditional rendering was lost at some point — current `editor.tsx:780` renders `<Sidebar />` unconditionally — but that conditional was for the settings sidebar, not the inserter popover. The popover keyCode bug has likely been latent since `da5d423e3`.
