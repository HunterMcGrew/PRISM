# Plan: THR-1461

## Ticket

https://linear.app/tractru/issue/THR-1461/support-yoast-robotssearch-crawl-settings-in-graphql-and-frontend

## Goal

Restore the fleet-default "block crawling of internal search URLs" rule by defaulting Yoast's `deny_search_crawling` toggle to ON at the storage layer, so the Yoast UI matches what's actually emitted in `robots.txt`.

---

## Decisions

- **The original THR-1461 work surfaced the three Yoast crawl-cleanup toggles (`denySearchCrawling`, `denyWpJsonCrawling`, `denyAdsbotCrawling`) via GraphQL on `ThriveSettings.yoastRobotsSettings` and consumed them in `frontend/app/robots.ts` to generate `robots.txt` dynamically.** Original PR #1764. Replaces the prior unconditional `Disallow: /?q=` and `Disallow: /search` rules with Yoast-driven equivalents. The frontend uses `?q=` instead of WP's `?s=` because that's the search-param convention on the Next.js side.
- **The frontend fallback `yoast?.denySearchCrawling ?? true` only catches null/undefined.** Reason: when the GraphQL field is unavailable (e.g. PHP not yet deployed) the search-block rule still emits. Confirmed correct by the existing test at `frontend/app/__tests__/robots.test.ts`.
- **Default Yoast's `deny_search_crawling` to `true` at storage, not at the resolver or frontend.**
  - **Root cause of the regression:** Yoast's stock default for the toggle is OFF. After THR-1461 made the rule Yoast-driven, every un-touched Yoast install reports `denySearchCrawling: false`, the `?? true` fallback never fires (it's a stored `false`, not null/undefined), and the rule disappears from `robots.txt` for the entire fleet.
  - **Alternatives considered:**
    - Flip the PHP resolver default in `YoastRobotsSettings::resolve_settings()` from `false` to `true`. Rejected — desyncs Yoast UI from reality (toggle reads OFF in wp-admin while rules are still emitted), violating the user's "intuitive: UI matches robots.txt" requirement.
    - Restore the unconditional `SEARCH_DISALLOW` rules in `frontend/app/robots.ts` and ignore the Yoast toggle for this specific key. Rejected — same UI/reality desync, plus it discards the integration work the original ticket established.
    - WP-CLI `wp option update wpseo` directly. Rejected — Yoast settings live inside a serialized option array; a direct write would clobber unrelated keys in the same option group.
  - **Chosen approach:** sentinel-guarded one-time `WPSEO_Options::set('deny_search_crawling', true)`, invoked via a new `wp thrive set-yoast-crawl-defaults` CLI command from a step in `.github/workflows/deploy-site.yml`. `WPSEO_Options::set` is Yoast's supported writer (precedent in `wordpress-seo-premium/premium.php` and `wordpress-seo-premium/classes/upgrade-manager.php`) — preserves the rest of the serialized option array and runs Yoast's validation. The sentinel option `thrive_yoast_crawl_defaults_applied` ensures the migration runs exactly once per site so explicit dealer choices stick after the first run.
  - **Implementation guidance:** new command class at `backend/plugins/gravity-platform-core/includes/Commands/SetYoastCrawlDefaultsCommand.php`, registered alongside the existing `wp thrive *` commands in `wp-cli.php`. Workflow step calls the named command instead of inlining a multi-statement `wp eval`. Step is placed after the "activate all plugins" step so Yoast is guaranteed loaded. No `if:` flag on the step — gated by the sentinel inside the command, not the workflow. Both WPE and Cloudways `post-deploy.sh` runners pick it up identically.

- **Refactored mid-implementation from inline `wp eval` to a `WP_CLI_Command` class.** Reason: the inline approach crammed three statements + two guards into a single-quoted heredoc string, which read poorly and would have made future edits brittle. Extracting to a command class matches the established `Thrive_Core\Commands\*` pattern (`ReindexCommand`, `SetThemeLinkColorCommand`) and shrinks the workflow step to a one-line `thrive set-yoast-crawl-defaults` invocation. No behavior change — the guards, the writer, and the sentinel are identical.

- **No tests added for the new command class.** Reason: existing convention — neither `ReindexCommand` nor `SetThemeLinkColorCommand` has tests, and adding `tests/Unit/Commands/` would establish a new pattern off a single thin command. The branching is small (sentinel guard + Yoast guard + dry-run) and load-bearing only at first run on each site; verification is via the post-deploy staging check in Task 2. If Briar flags coverage, we add then — pattern-establishing work earns its own ticket.

- **Yoast safe defaults initialized at the variable declaration, not assigned in the catch block.**
  - **Root cause:** With Yoast as a required Thrive plugin and the deploy-time migration setting `deny_search_crawling` ON for every fleet site, a null/undefined `yoast` value never means "Yoast says nothing" — it always means "we couldn't read the toggle" (catch block fired, transient network error, our `gravity-platform-core` plugin not loaded, PHP resolver not registered). All null sources are abnormal states; they all deserve the same fail-safe behavior.
  - **Alternatives considered:**
    - `?? true` at the consumption site (`yoast?.denySearchCrawling ?? true`). Rejected — puts the failsafe far from the seam where the failure happens; reader has to mentally connect the consumption site to the catch block 10 lines up.
    - Catch block assigns safe defaults; resolved-null leaves yoast null. Mid-pass take. Rejected — differentiates two null sources that don't have a real semantic distinction. Resolved-null parent (broken plugin) and catch-block (transient failure) both mean "we couldn't read the toggle"; both deserve the same failsafe.
  - **Chosen approach:** Initialize `let yoast: YoastRobotsSettingsDTO = { denySearchCrawling: true, denyWpJsonCrawling: false, denyAdsbotCrawling: false }` at declaration. Assign from `yoastResponse.data.allSettings.thrive?.yoastRobotsSettings ?? yoast` so resolved-null parent keeps the defaults. Catch block becomes one line of `console.warn` — failsafe is implicit in the var's starting value. Consumption site uses `yoast.denySearchCrawling ?? false` (per-field fallback for the inner DTO nullability).
  - **Implementation guidance:** Failsafe shape lives in one place (var declaration). All three null paths (catch fires, resolved-null parent, missing PHP) behave identically — search-block rule emits, opt-in toggles stay off. Type narrows from `YoastRobotsSettingsDTO | null` to `YoastRobotsSettingsDTO` so downstream uses `yoast.x` instead of `yoast?.x`. Tests cover all three paths via the same assertion (search rules emitted, no AdsBot rule). When this entity migrates to the DAL/service layer per [THR-1802](https://linear.app/tractru/issue/THR-1802), the init-time defaults move into the service method's local state.

- **The three URL patterns Yoast's toggle emits are hardcoded in Yoast's PHP, not configurable.** Confirmed in `wordpress-seo/src/integrations/front-end/robots-txt-integration.php:120-123`. Yoast's toggle ON emits exactly: `Disallow: /?s=`, `Disallow: /page/*/?s=`, `Disallow: /search/` under `User-agent: *`. None of these touch `/` or affect site-wide indexing. Cosmetic mismatch worth noting: Yoast UI displays `?s=` (its native search param) while our `robots.txt` shows `?q=` (our frontend's search param) — same intent, different literal patterns. Not blocking.

## Implementation Tasks

1. Add a new `wp thrive set-yoast-crawl-defaults` CLI command:
   - `backend/plugins/gravity-platform-core/includes/Commands/SetYoastCrawlDefaultsCommand.php` — `WP_CLI_Command` subclass with sentinel guard, Yoast `class_exists` guard, and `--dry-run` flag.
   - `backend/plugins/gravity-platform-core/wp-cli.php` — register alongside the existing `thrive reindex` and `thrive set-theme-link-color` commands.
   - `.github/workflows/deploy-site.yml` — append the command to `backend/post-deploy.sh` in a new step between the existing "activate all plugins" and "update platform settings" steps. No `if:` flag — sentinel handles idempotency.

2. Verify on a staging site after the first deploy:
   - `wp option get thrive_yoast_crawl_defaults_applied` → `1`
   - `wp option get wpseo --format=json | jq '.deny_search_crawling'` → `true`
   - `curl -s https://<site>/robots.txt | grep -E '\?q=|/search'` → three Disallow lines under `User-agent: *`
   - Yoast SEO → Settings → Crawl optimization shows the "Prevent crawling of internal site search URLs" toggle ON.
   - Toggling it OFF in the UI and re-fetching `robots.txt` (after the 60s cache window) drops the search-block rules. Toggling back ON restores them.

3. Update the PR body with the storage-layer rationale and the sentinel design — reviewers landing on this followup need to understand why we didn't take the resolver-default shortcut.

## Acceptance Criteria

### Behavioral

- [ ] Given a fleet site that has never had Yoast configured, When the deploy workflow runs after this change ships, Then the "Prevent crawling of internal site search URLs" toggle in Yoast SEO → Settings → Crawl optimization reads ON.
- [ ] Given a fleet site where the toggle was just set ON by the migration, When a visitor or crawler fetches `/robots.txt`, Then the response contains three `Disallow:` lines covering search URL patterns under `User-agent: *`.
- [ ] Given a site where the toggle has been set ON by the migration, When an admin turns the toggle OFF in Yoast UI and waits for the cache window (~60 seconds), Then `/robots.txt` no longer contains the search-block rules.
- [ ] Given a site where an admin has previously turned the toggle OFF intentionally and the migration has not yet run, When the deploy workflow runs, Then the migration sets the toggle ON anyway (one-time forward migration). After that first run, dealer choices are respected on subsequent deploys.
- [ ] Given any site, When the deploy workflow runs a second (or third, or Nth) time, Then the migration step does not re-modify the Yoast option — the sentinel prevents it.

### Non-behavioral

- [ ] The deploy step uses Yoast's `WPSEO_Options::set` writer, not a direct `wp option update wpseo`.
- [ ] The sentinel option key is `thrive_yoast_crawl_defaults_applied`.
- [ ] The deploy step runs after plugin activation so Yoast is loaded.

## Review Issues

### Frontend `?? true` fallback on `robots.ts:157` contradicts the "UI matches reality" decision

- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/app/robots.ts:157`
- **Problem:** `yoast?.denySearchCrawling ?? true` defaulted to `true` (search-block rules emitted) when the Yoast GraphQL field is null/undefined. With the storage-layer migration shipped in this followup, that fallback created a Yoast-UI-vs-robots.txt divergence in any case where the GraphQL field is missing.
- **Fixed in:** `frontend/app/robots.ts` — consumption site uses `?? false` (faithfully follows Yoast for non-null reads). Catch block assigns safe defaults (`{ denySearchCrawling: true, denyWpJsonCrawling: false, denyAdsbotCrawling: false }`) so transient fetch failures fail safe by emitting the search-block rule. One test flipped to match new behavior ("Yoast returns no settings → no search rules" — resolved-null is broken-plugin territory); the "Yoast settings query fails" test retains its original assertion (search rules emitted via catch-block failsafe). All 16 robots tests pass.
