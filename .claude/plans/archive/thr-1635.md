## Decisions

- Detect the site-search page by path (`SEARCH_PAGE_PATH` + `usePathname()`), not by `defaultRefinement === "all"` â€” non-search pages also use `"all"` to apply default filters, so refinement-based detection misclassifies them and fires the scroll-to. **Verified** by Sasha on 2026-04-20.
- `isSiteSearch` has exactly one real consumer (`ScrollTo.tsx`). Four sibling components (`SearchBox`, `ClearRefinements`, `CurrentRefinements`, `SidebarRefinementsFilter`) accept the prop but never use it â€” dead threading to be removed.
- Colocate the `usePathname()` check inside `ScrollTo.tsx` (Option B) rather than computing it in `KronosSearchBlock` and threading it down (Option A). Rationale: Option B makes the only real consumer self-sufficient, removes the dead prop threading the ticket AC already requires us to clean up, and avoids re-introducing the coupling as a later follow-up.
- Preserve the `isTyping` guard in `ScrollTo.tsx:32-36`. It intentionally skips the scroll when a search input has focus, cooperating with PR #1552 (THR-1422). Do not remove it as part of the cleanup.
- Add `"use client"` to `ScrollTo.tsx`. The file uses hooks but currently relies on its parent's directive â€” `code-standards.md` Â§ Server/Client says each hook-using component must own its directive.
- Do not bundle PR #1552 (THR-1422) into this PR. The two fixes cooperate on the same middleware but touch different files and belong to different tickets; one-ticket-per-PR is load-bearing for squash-merge traceability, changelog grouping, and QA tracking. Ship as sibling PRs targeting the same release window. Note the dependency in the PR description.
- Bundle into the mega menu release per Lennon.

## Debugged Issues

### Kronos block auto-scrolls on non-search pages with `defaultRefinement="all"`

- **Status:** `fixed` (fixed by Clove on 2026-04-20)
- **Severity:** High (S2)
- **Environment:** Production â€” `colemantractor.com/spring-blowout-special`. Reproduces on any page whose Kronos block has `defaultRefinement="all"` (the schema default).
- **File:** `frontend/blocks/kronos-search/KronosSearchBlock.tsx:64`
- **Root cause:** `isSiteSearch = defaultRefinement === "all"` is an overloaded classifier â€” both the site-search page AND non-search "show all inventory" pages use `"all"`. `ScrollTo.tsx:13` seeds `hasInteracted` to `true` when `isSiteSearch` is true, so the first InstantSearch state change on mount fires `scrollIntoView` without any user interaction. Compounded by `schema.ts:31` where `DEFAULT_ATTRIBUTES.defaultRefinement` is `"all"` â€” any block saved without overriding gets the buggy classification.
- **Steps to Reproduce:**
  1. Place a Kronos block on a non-search page with content above it (e.g. a carousel).
  2. Ensure `defaultRefinement` is `"all"` (this is the schema default â€” blocks without explicit override qualify).
  3. Load the page.
  4. Observe: page auto-scrolls to the Kronos block on initial mount, skipping content above.
- **Expected behavior:** Page loads at the top; no auto-scroll until the user interacts (click or input).
- **Actual behavior:** Page auto-scrolls to the Kronos block on initial InstantSearch state sync.
- **Recommended fix:** Drive `isSiteSearch` from the URL path using `usePathname()` from `next/navigation` compared against `SEARCH_PAGE_PATH` (`/search`). Two shapes to consider:
  - **A (minimal):** replace the computation at `KronosSearchBlock.tsx:64` with the path check; leave prop threading as-is.
  - **B (cleaner):** move the `usePathname()` check inside `ScrollTo.tsx` (its only real consumer), drop the `isSiteSearch` prop from `ScrollTo`, and remove the dead prop threading in `SearchBox`, `ClearRefinements`, `CurrentRefinements`, `SidebarRefinementsFilter`, `RefinementsBar`, `DesktopFilters`, `EquipmentListBlock`, and the `SearchBox` test.
- **Suggested tests:**
  - Unit test for `ScrollTo` asserting no `scrollIntoView` call on initial mount when the path is not `/search`.
  - Unit test asserting the existing behavior on `/search` is preserved (scroll on state change without prior interaction).
  - Add a storybook story for Kronos with `defaultRefinement="all"` on a non-search path â€” visual regression coverage for the scroll-position check.
- **Linear:** `synced`

## Review Issues

### `ScrollTo.tsx` middleware is registered without cleanup (pre-existing)

- **Severity:** `minor`
- **Status:** `fixed` (Fixed in: `frontend/blocks/kronos-search/ScrollTo.tsx:50` â€” effect now returns `addMiddlewares(middleware)` so the unsubscribe fires on unmount / dep change)
- **File:** `frontend/blocks/kronos-search/ScrollTo.tsx:29-44`
- **Problem:** The effect calls `addMiddlewares(middleware)` every time its dependencies change (on mount, on `hasInteracted` flip, on `isSiteSearch` change). There's no return cleanup, so earlier middlewares accumulate on the InstantSearch instance. Pre-existing behavior â€” not introduced by this PR; surfaced during review.
- **Suggested fix:** Capture the return value of `addMiddlewares` (it's the unsubscribe function) and return it from the effect for cleanup. Out of scope for THR-1635 â€” follow-up ticket.

### `ScrollTo.test.tsx` last test has brittle cleanup

- **Severity:** `minor`
- **Status:** `fixed` (Fixed in: `frontend/blocks/kronos-search/__tests__/ScrollTo.test.tsx:115-131` â€” `appendChild` / `input.remove()` now wrapped in try/finally so the input is torn down even on assertion failure)
- **File:** `frontend/blocks/kronos-search/__tests__/ScrollTo.test.tsx` (last test â€” "skips scrolling when a search input is focused")
- **Problem:** The test creates a DOM input and removes it after the final `expect`. If the expectation fails, the input isn't removed and leaks into subsequent tests. Happens to be the last test in the file today, so leakage is harmless â€” but rule-of-three says move it to `afterEach` or try/finally before someone adds a new test after it.
- **Suggested fix:** Move the `document.body.appendChild` / `removeChild` lifecycle into a `beforeEach`/`afterEach` pair, or wrap in try/finally. Minor â€” defer or fix in a follow-up.

### `useState(isSiteSearch)` seed relies on remount on route change

- **Severity:** `minor`
- **Status:** `fixed` (Fixed in: `frontend/blocks/kronos-search/ScrollTo.tsx:20` â€” `hasInteracted` now seeds to `false` unconditionally; the site-search bypass is expressed in the middleware guard at line 44, not by overloading `hasInteracted`)
- **File:** `frontend/blocks/kronos-search/ScrollTo.tsx:14-17`
- **Problem:** `useState(isSiteSearch)` only uses the pathname-derived value on mount. If `ScrollTo` ever stays mounted across a client-side navigation between `/search` and a non-search path, `hasInteracted` would keep its stale initial seed. Next.js App Router typically unmounts page subtrees on route change, so this is unlikely to manifest in practice â€” but it's an implicit assumption worth flagging.
- **Suggested fix:** Either add an effect that resets `hasInteracted` when `isSiteSearch` changes, or document the assumption. Not needed for the fix to ship correctly â€” the bug reproducer (load-and-observe) doesn't depend on cross-route persistence.

### Generic `Props` type name in `ScrollTo.tsx`

- **Severity:** `minor`
- **Status:** `fixed` (Fixed in: `frontend/blocks/kronos-search/ScrollTo.tsx:10` â€” renamed `interface Props` to `interface ScrollToProps`; destructure annotation updated)
- **File:** `frontend/blocks/kronos-search/ScrollTo.tsx:10`
- **Problem:** `interface Props` uses the generic name banned by `component-props-decoupling.md` and `frontend-components.md`. Type names must be `{ComponentName}Props` for greppability, IDE clarity, and refactoring safety.
- **Suggested fix:** Rename to `ScrollToProps`.
