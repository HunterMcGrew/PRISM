# Plan: thr-1634

## Ticket

[THR-1634 — Upgrade Lilac to post standups directly to Slack via MCP](https://linear.app/tractru/issue/THR-1634)

Originally filed as "Fix Lilac standup link format for Slack WYSIWYG composer" (1-pt format fix). Scope expanded after PR #1812 was opened: team lead reviewed the paste-format fix and asked why Lilac couldn't post directly to Slack instead of producing a block to paste. Linear description updated to reflect the expanded goal.

## Goal

Make Lilac a one-command standup tool — she pulls yesterday's PR activity, asks what you're doing today and whether you're blocked, then posts the composed standup to `#tractru-dev` via Slack MCP, with mandatory user confirmation of the exact rendered message before posting.

2. **Template restructure** (`.claude/templates/standup-summary.md`). Replace the current status-axis section structure (`Working on:` / `Merged:` / `Reviewed:`) with an authorship-axis structure:
   - Top-level sections: `Your PRs:` (PRs the user authored) and `Reviewed:` (PRs authored by others that the user reviewed)
   - Per-entry status label: `[merged]`, `[in review]`, or `[draft]` — appended after the PR number

   - Add a new "Render targets" subsection documenting that link syntax differs by target: `[#NNNN](url)` for paste, `<url|#NNNN>` for API
   - Add the new outer sections the full standup now covers: `Project`, `Yesterday / Friday & Weekend`, `Today`, `Blockers`

3. **Status detection logic** (`.claude/skills/thrive-standup-summary/SKILL.md`). For each PR in the rendered list, derive the status label from GitHub data:
   - `[merged]` — PR is closed and merged in the window
   - `[draft]` — PR is in draft state
   - `[in review]` — PR is open and not draft (covers both "reviewers requested" and "no reviewers yet, but author signaled ready")
   - Document the detection rule in the skill under a new "Status labels" subsection

4. **Continuation detection logic** (same SKILL.md). For each entry in `Your PRs:`, query the PR's commit history and check whether any commits authored by `$USERNAME` predate `$SINCE_DATE`. If yes, prepend `Continued ` to the entry. Applies to both `[merged]` and `[in review]` statuses within `Your PRs:`. Does not apply to `Reviewed:` entries — "continued review" is a different semantic and is out of scope.

5. **Interactive prompting flow** (SKILL.md). After generating the `Yesterday` section, Lilac:
   - Shows the Yesterday block in-chat with the Project line prefilled (`Thrive`) and an auto-generated "when" label (`Yesterday:` on Tuesday–Friday, `Friday & Weekend:` on Monday, or the user's specified range)
   - Prompts: "What are you going to do today?" — accepts multi-line input
   - Prompts: "Any blockers?" — accepts multi-line input or the literal word `None`
   - Assembles the full 4-section standup
   - Renders it in API-format (mrkdwn) and shows the user the exact message that will be posted, along with the target channel
   - Asks: "Post to `#tractru-dev`, or would you rather paste it yourself?"

6. **Slack MCP post path** (SKILL.md + the discovery result from Task 1). On user confirmation:

## Decisions

- **Scope expanded from format fix to full Slack-post flow** — team lead reviewed PR #1812 and pushed for direct posting instead of pasting. The format fix remains valid as one slice of the larger goal; we don't throw it away, we build on it.
- **One ticket, one branch, one PR** — Hunter's call. THR-1634 covers the whole expanded scope. PR #1812 absorbs the v2 work; no followup PR. The PR has no reviews yet so amending is free.
- **Sections organized by authorship, not status** — `Your PRs:` and `Reviewed:` replace the `Merged` / `Working on` / `Reviewed` precedence structure. Reason: Jonathan (team lead) flagged that the old Merged section conflated "you merged" with "you reviewed that merged," which reads ambiguously. Authorship-first structure matches how a lead dev scans the channel.
- **Three status labels — `[merged]`, `[in review]`, `[draft]`** — covers every realistic PR state without ambiguity. Considered a fourth `[open]` for "open but no review requested" and rejected it: opening a non-draft PR signals "review me," the formal request is a GitHub UX quirk.
- **`Continued ` prefix on Your PRs entries when the user has commits before the window start** — gives the lead dev a free signal about slow-moving PRs. Computed from PR commit history, no memory required. Scope-limited to Your PRs; Reviewed section doesn't get it because "continued review" is a different concept.
- **Lilac always emits standard markdown link syntax (`[#NNNN](url)`), regardless of delivery path** — the Slack MCP's posting tool accepts standard markdown and translates to Slack's raw mrkdwn internally; the WYSIWYG composer accepts standard markdown on paste. mrkdwn is an implementation detail the MCP wrapper owns — Lilac never emits it. The v1 decision that shipped in PR #1812 was correct for both surfaces; the v2 plan's original render-target split was overengineered for a problem that didn't exist. Corrected in Briar's review round.
- **Explicit user confirmation before every post — shows the exact rendered message** — no exceptions, no auto-posting on success. Slack posts are permanent enough that "review what you're about to post" is a reasonable default for any write action.
- **Default channel `#tractru-dev`, configurable per-invocation, never inferred** — hard-coding the default keeps the common case fast; per-invocation override supports "post this one to #planning" without editing config; never inferring keeps Lilac from guessing wrong.
- **Graceful fallback to paste-format when Slack MCP is unconnected or the post fails** — never leave the user stranded without a working output. The existing v1 paste behavior IS the fallback; no new code, just reuse.
- **Project line hardcoded to "Thrive"** — multi-project standups aren't a current need. Revisit when they are.
- **Every Lilac invocation starts fresh** — no memory across runs of Today/Blockers answers. Repeating the same day means answering again; that's scope creep we're not doing.
- **Lilac posts as a bot user named "Lilac"; the first line of the message names the human** (`Hunter's standup ~`). Bot-user simplifies Slack app configuration; the attribution line keeps the post clearly from Hunter, on Lilac's behalf.
- **Section labels render as bold, spacer lines hold a zero-width space.** The first real run of the four-subsection template posted through `slack_send_message` failed with `MCP error -32602 … invalid_blocks` — the Slack MCP's validator rejected the `###` top-level headings. A second post using bold labels succeeded but rendered every section flush against its neighbors, because Slack's message renderer collapses empty lines between paragraphs. The accepted resolution: every section label (top-level prompts and Yesterday subsections alike) renders as `**Label:**` on its own line, and every paragraph break between sections is a spacer line containing one U+200B character. Empty lines collapse; U+200B lines don't.
- **Yesterday section restructured into four subsections — `Merged` / `In Review` / `Continued` / `Reviewed`.** First real-run feedback (DM to Jonathan, 2026-04-20) surfaced two issues with the authorship-axis v2 structure: Slack collapsed blank lines between sections into nothing (`ThriveWhat did you do yesterday?` on a single line), and the two-bucket `Your PRs` / `Reviewed` split didn't carry enough signal — Hunter wanted Merged / In Review / Continued / Reviewed as named subsections instead. Assignment is first-match-wins: Reviewed wins if the user didn't author; then Merged if the PR merged in-window (merge wins over continuation — a merged PR always reports under Merged regardless of whether its commits predate the window); then Continued if the PR is open and has pre-window commits; then In Review for any remaining open authored PR. The `Continued ` prefix on entries is retired — continuation is a subsection instead. Top-level prompts render as `###` headings and subsection labels as bold, with blank lines between every heading and its content to force Slack to render paragraph breaks. The when-label (`Yesterday:` / `Friday & Weekend:`) is dropped — redundant with the subsection structure. Attribution line (`<Name>'s standup ~`) is also dropped — the Slack bot-on-behalf-of already names the author.
- **Task 1 discovery — Slack MCP `slack_send_message` schema verified.** Server UUID: `f1202eb1-6612-4f44-8789-14b7704ac6c4`. Required parameters: `channel_id` (a Slack channel ID like `C12345`, NOT a `#name`) and `message` (the body). The MCP documents that messages use standard markdown and handles any mrkdwn translation internally. Sibling tools `slack_send_message_draft` and `slack_schedule_message` are decoys — the former creates drafts instead of posting, the latter schedules for later delivery. Neither is what Lilac wants; the skill rejects any tool whose name ends in `_draft`, `_schedule`, or `_canvas`. Channel-name-to-ID resolution happens at runtime via `slack_search_channels`. Skill is schema-driven: Phase 5.3 loads the chosen tool's schema via `ToolSearch select:<tool-name>` and maps Lilac's concepts to whatever parameter names the schema advertises — no hardcoding. End-to-end post path still pending first-invocation verification, but the assumptions the skill operates under are now verified from the tool's actual description.
- **Slack collapses blank lines between any two bold paragraphs, including `**What did you do yesterday?**`→`**Merged:**`.** Supersedes the earlier "one blank line is enough between a top-level Yesterday label and its first subsection" guidance. A third real-run DM to Jonathan (2026-04-20) returned the standup rendered with those two bold labels flush against each other — same collapse pattern as the earlier section-to-section case. The uniform rule: a U+200B spacer sits between every top-level prompt (`What project(s)…?`, `What did you do yesterday?`, `What are you going to do today?`, `Blockers:`) and its content — whether that content is plain text or another bold label — and between adjacent top-level sections. Subsection labels inside Yesterday (`**Merged:**`, `**In Review:**`, `**Continued:**`, `**Reviewed:**`) keep a plain blank line to their entries; entries are non-bold, so the bold-to-nonbold break renders fine without a spacer. One rule, one exception (subsection → entries), no case-by-case carve-outs.

### The Slack MCP `slack_send_message` tool accepts standard markdown, not mrkdwn

- **Severity:** `critical`
- **Status:** `fixed`

### Tool disambiguation is under-specified — draft and schedule variants can silently hijack the post

- **Severity:** `major`
- **Status:** `fixed`
- **File:** `.claude/skills/thrive-standup-summary/SKILL.md:315-320`
- **Problem:** The Slack MCP exposes three "message" tools: `slack_send_message` (sends now — the right one), `slack_send_message_draft` (creates a draft, doesn't send), and `slack_schedule_message` (schedules for later). ### Behavioral

- [ ] Given Lilac is invoked on a workday, When she finishes pulling PRs, Then she shows the user a preview with the Yesterday section split into up to four subsections in order: `Merged`, `In Review`, `Continued`, `Reviewed`. The Project line is prefilled (`Thrive`). No when-label, no attribution line.
- [ ] Given an authored PR merged within the window, When Lilac assigns it to a subsection, Then it lands under `Merged` — even if it had pre-window commits (merge wins over continuation)
- [ ] Given an authored open PR has commits authored by the user before the window start, When Lilac assigns it to a subsection, Then it lands under `Continued` — covers both `[in review]` and `[draft]` states
- [ ] Given an authored open PR has no pre-window commits, When Lilac assigns it to a subsection, Then it lands under `In Review` — covers both `[in review]` and `[draft]` states
- [ ] Given a subsection has no entries, When Lilac renders the Yesterday section, Then that subsection header is omitted (no empty `**Merged:**` / `**In Review:**` / `**Continued:**` / `**Reviewed:**` block)
- [ ] Given a PR is open and not in draft state, When Lilac renders its status label, Then she shows `[in review]`
- [ ] Given a PR was merged within the window, When Lilac renders its status label, Then she shows `[merged]`
- [ ] Given a PR is in draft state, When Lilac renders its status label, Then she shows `[draft]`
- [ ] Given the preview is shown, When Lilac prompts for today's plan, Then she accepts multi-line input from the user
- [ ] Given today's plan is captured, When Lilac prompts for blockers, Then she accepts multi-line input or the literal word `None`
- [ ] Given both prompts are answered, When Lilac assembles the full standup, Then she renders it as standard markdown with every section label (top-level and subsection) as `**Bold:**` on its own line — no `#` / `##` / `###` heading syntax — with a U+200B spacer line between every top-level prompt and its content (including when the content is another bold label like `**Merged:**`) and between adjacent top-level sections, while subsection labels inside Yesterday use a plain blank line to their non-bold entries, and she names the target channel in the preview
- [ ] Given the user confirms posting and the Slack MCP is connected, When Lilac posts, Then the message lands in `#tractru-dev` and Lilac confirms success with a permalink if the MCP returns one
- [ ] Given the user declines the post, When Lilac handles the decline, Then she re-renders the standup using standard-markdown link syntax so the user can paste it into Slack manually
- [ ] Given the Slack MCP is not connected, When Lilac reaches the post step, Then she skips the post, re-renders in paste format, and tells the user why
- [ ] Given the Slack MCP post call fails, When Lilac handles the failure, Then she reports the reason, re-renders in paste format, and does not auto-retry

### Non-behavioral

- [ ] Default post channel is `#tractru-dev` — configured, not inferred; per-invocation override is supported
- [ ] Lilac never posts to Slack without an explicit user confirmation of the exact rendered message
- [ ] Links render as standard markdown (`[#NNNN](url)`) on both delivery paths — the Slack MCP wrapper accepts it, and the WYSIWYG composer converts it to rich-text on paste
- [ ] The standup renders identically for post and paste — one render format, no duplicate templates
- [ ] Channel name resolves to a Slack channel ID at runtime via `slack_search_channels`; channel lookup failure falls through to the paste path
- [ ] When the Slack MCP is unavailable, Lilac's output remains functional via the paste fallback
