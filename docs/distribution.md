# Distribution

How PRISM gets into a consumer team's codebase, how updates flow, and what stays under team control. The install layout is bifurcated — canonical platform-agnostic content at `.prism/`, build-time copies at each platform dir. See [ADR-0031](https://github.com/HunterMcGrew/PRISM/blob/main/.prism/spec/adrs/_toolkit/0031-bifurcated-install-layout.md) for the decision and reasoning.

## Primary model: npm (public)

PRISM is published publicly to npm as `@huntermcgrew/prism`. The recommended way to adopt it is via `npx` — no clone, no global install, no PATH setup:

```bash
cd your-repo
npx @huntermcgrew/prism adopt
```

Updates work the same way:

```bash
npx @huntermcgrew/prism update
```

See [Adopt PRISM into your repo](./adopt-prism.md) for the full consumer-facing guide, including pinning to a specific version and alternative install paths.

## Alternative models: checkout-based

For air-gapped environments, teams that want to customize PRISM's source directly, or contributors to PRISM itself, checkout-based installs are supported. These were the primary distribution model before the npm publish.

**Vendor PRISM inside your repo** — clone PRISM as a subdirectory and run `pnpm prism:adopt` from inside the clone. The command auto-detects the enclosing consumer repo:

```bash
cd your-repo
git clone https://github.com/HunterMcGrew/PRISM.git PRISM
cd PRISM && pnpm install && pnpm prism:adopt
```

**Sibling repo with `--consumer` flag** — clone PRISM beside the consumer repo and pass `--consumer <path>`:

```bash
cd prism
pnpm prism:adopt --consumer ../ktc-codebase
pnpm prism:update --consumer ../ktc-codebase
```

See [Adopt PRISM into your repo](./adopt-prism.md) for the global `prism` command and the `--consumer` flag details.

## What gets installed where

When `npx @huntermcgrew/prism adopt` (or `pnpm prism:adopt` from a checkout) runs, the consumer's repo receives a bifurcated layout: canonical content at `.prism/<area>/` plus a build-time copy of the read-only areas inside each platform dir.

### Canonical content (one source per file)

Sourced from `templates/install/.prism/<area>/` and written to the consumer's `.prism/<area>/`. Tokens are substituted from the consumer's `.ai-skills/config.json` at sync time.

| Source (in PRISM)                                       | Destination (in consumer)                | Notes                                                                                          |
| ------------------------------------------------------- | ---------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `templates/install/.prism/SPEC.md.tmpl`                 | `consumer/.prism/SPEC.md`                | Tier 0 meta-doc; tokens substituted                                                            |
| `templates/install/.prism/rules/`                       | `consumer/.prism/rules/`                 | Universal rules always copied; opt-in rules copied if listed in `config.rules.optIn`           |
| `templates/install/.prism/spec/adrs/`                   | `consumer/.prism/spec/adrs/`             | All toolkit-level ADRs copied                                                                  |
| `templates/install/.prism/architect/`                   | `consumer/.prism/architect/`             | Toolkit-meta docs. Team-specific architect docs are generated by Winston in Phase 3.           |
| `templates/install/.prism/architect/manifest.stub.json` | `consumer/.prism/architect/manifest.json` | Renamed from `.stub.json`. Team-specific routes get added by Winston.                          |
| `templates/install/.prism/templates/`                   | `consumer/.prism/templates/`             | All shared templates copied                                                                    |
| `templates/install/.prism/references/`                  | `consumer/.prism/references/`            | Operational flows reused across skills                                                         |

### Build copies (regenerated by `pnpm prism:build`)

After the canonical content lands, `prism:build` runs locally on the consumer's machine and writes byte-identical copies of the read-only areas into each platform dir. Each copied area carries an `.ai-skill-generated` marker at its root so the build can manage it across renames.

| Canonical                  | Build copies regenerated at                                                          |
| -------------------------- | ------------------------------------------------------------------------------------ |
| `.prism/rules/`            | `.claude/rules/`, `.codex/rules/`, `.cursor/rules/`                                  |
| `.prism/architect/`        | `.claude/architect/`, `.codex/architect/`, `.cursor/architect/`                      |
| `.prism/spec/`             | `.claude/spec/`, `.codex/spec/`, `.cursor/spec/`                                     |
| `.prism/templates/`        | `.claude/templates/`, `.codex/templates/`, `.cursor/templates/`                      |
| `.prism/references/`       | `.claude/references/`, `.codex/references/`, `.cursor/references/`                   |
| `.prism/SPEC.md`           | `.claude/SPEC.md`, `.codex/SPEC.md`, `.cursor/SPEC.md`                               |

`.prism/plans/` and `.prism/lessons.md` are agent-written and stay canonical-only — no platform copies. Mirroring those would race against in-session edits.

### Platform-specific outputs (generated from `.ai-skills/`)

Skills are not under canonical at all. They're generated from `.ai-skills/skills/<id>/` to platform outputs:

| Source                                | Destination (in consumer)                              | Notes                                            |
| ------------------------------------- | ------------------------------------------------------ | ------------------------------------------------ |
| `.ai-skills/skills/<id>/`             | `consumer/.claude/skills/<id>/SKILL.md`                | Built via `prism:build` with token substitution  |
| `.ai-skills/skills/<id>/`             | `consumer/.agents/skills/<id>/SKILL.md`                | Codex skill output                               |
| `.ai-skills/skills/<id>/`             | `consumer/.codex/agents/<id>.toml`                     | Codex agent adapter                              |
| `.ai-skills/skills/<id>/`             | `consumer/.cursor/skills/<id>/SKILL.md`                | Cursor output                                    |

### Enforcement floor (build-generated, build-time copy)

The build also emits the enforcement runtime into both the live `.claude/` tree and the install seed so new adopters get the floor on first `prism adopt`. The canonical source lives in `.ai-skills/hooks/`; the build writes copies to both destinations.

| Canonical source                  | Build copies written to                                                       | Notes                                       |
| --------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------- |
| `.ai-skills/hooks/`               | `.claude/hooks/`, `templates/install/.claude/hooks/`                          | Runtime hooks: `ownership-guard.mjs`, `run-gates.mjs`, `evidence-ledger.mjs`, `gates.json`, `lib/` |
| `.ai-skills/hooks/settings.json`  | `.claude/settings.json`, `templates/install/.claude/settings.json`            | Hook wiring config; ships with `{"hooks":{}}` until floor is enabled |

See [The Enforcement Floor](./ai-skills/enforcement-floor.md) for what these files do and how to enable or disable enforcement.

### Top-level anchor files

| Source                                       | Destination (in consumer) | Notes                                  |
| -------------------------------------------- | ------------------------- | -------------------------------------- |
| `templates/install/AGENTS.md.tmpl`           | `consumer/AGENTS.md`      | Cross-platform constitution; tokens substituted |
| `templates/install/.claude/CLAUDE.md.tmpl`   | `consumer/CLAUDE.md`      | Claude-specific behavioral guidance; tokens substituted |

`AGENTS.md` stays at the root because it applies uniformly to Claude, Codex, and Cursor. `CLAUDE.md` stays at the root because that's where Claude Code expects to find it.

### Files NOT copied

- The PRISM repo's own `.prism/` and `.claude/` dogfood install
- `.ai-skills/skills/` canonical sources (built into platform outputs at sync time, not copied raw)
- `scripts/ai-skills/` (the generator runs from PRISM's clone, not the consumer's)
- `package.json` and tooling — the consumer's `package.json` stays untouched
- Plans and lessons from PRISM (`.prism/plans/`, `.prism/lessons.md`) — these are PRISM's own working memory, not toolkit-level content

## State file

After the first sync, the consumer repo has `.ai-skills/.prism-state.json` recording:

```json
{
  "prismCommit": "abc1234...",
  "syncedAt": "2026-05-02T19:30:00Z",
  "config": "<hash of config.json at sync time>",
  "files": {
    "AGENTS.md": "<content hash>",
    ".prism/rules/writing-voice.md": "<content hash>",
    ".prism/spec/adrs/_toolkit/0031-bifurcated-install-layout.md": "<content hash>",
    "...": "..."
  }
}
```

This enables three-way merge on subsequent syncs:

- **File unchanged in consumer + changed in PRISM** → update silently.
- **File changed in consumer (hash differs from synced) + changed in PRISM** → conflict. Print the diff, ask the user how to resolve.
- **File changed in consumer + unchanged in PRISM** → leave alone. The user has hand-edited; their version wins.
- **File unchanged in consumer + unchanged in PRISM** → no-op.

State tracks the canonical files at `.prism/<area>/`. Build copies under `.claude/<area>/`, `.codex/<area>/`, and `.cursor/<area>/` aren't tracked individually — they're regenerated from canonical, and `pnpm prism:check` catches any hand-edits.

Hand-authored files (the consumer's own skills, their own rules with new names, their own ADRs above the highest toolkit number) are never touched — they don't appear in the state file, so the sync layer doesn't see them.

## Pulling updates

**From npm (recommended):**

```bash
cd your-repo
npx @huntermcgrew/prism update
```

**From a checkout:**

```bash
cd ~/work/prism
git pull origin main           # or whichever branch your org tracks
pnpm install                   # if package.json changed
pnpm prism:update --consumer ../ktc-codebase
```

`prism update` refreshes canonical content in the consumer repo and regenerates platform copies. CI on the consumer side can run `pnpm prism:check` to confirm canonical and platform copies stay in sync.

The sync prints a summary: files updated, files preserved as user-edited, conflicts requiring resolution. Conflicts open the diff in the consumer's repo and don't auto-resolve.

## What stays under team control

After sync:

- The consumer's `.ai-skills/config.json` — owned by the team. Edit to change ticket prefix, tech stack, opt-in rules, etc. The next sync re-reads it.
- Hand-authored skills under `.claude/skills/` (without a corresponding `.ai-skills/skills/<id>/` in PRISM) — never managed.
- `.prism/plans/` and `.prism/lessons.md` — agent working memory; sync doesn't touch them.
- Anything in the consumer's repo outside `.prism/`, `.claude/`, `.agents/`, `.codex/`, `.cursor/`, and the `AGENTS.md` / `CLAUDE.md` anchor files — sync doesn't touch. (`.generated/` was a pre-Phase-1.5f staging directory; removed in ADR-0044.)

## Branching and updates upstream

PRISM uses phase branches for major work. Phase 1 (foundation), Phase 1.5 (bifurcated layout + tokenization), Phase 2 (Atlas onboarding), and Phase 3 (Winston codebase scan) have all shipped to `main`. Consumer teams can choose:

- **Track `main`** — get changes only when phases merge to main.
- **Track a phase branch** — early access to in-progress work; expect breakage.

Teams should pin to specific versions (`npx @huntermcgrew/prism@1.0.0 adopt`) for production stability.
