# Plan: THR-1612

## Ticket

https://linear.app/tractru/issue/THR-1612/mega-menu-migration-modal-reconsider-one-time-dismissal-ux

## Goal

Make the mega menu migration modal's dismissal behavior explicit — add a "Don't show this again" checkbox so suppression is a user choice, not an implicit side effect of conditions changing. Provide a re-surface path so users who dismissed the modal can still trigger it manually.

---

## Design

**Mock:** `.claude/design/mocks/thr-1612-migration-modal-dismissal.md`

### Summary
Redesigned the migration modal with tighter copy (~20 words vs ~42), a styled warning notice (amber bg, left border) for the "will replace existing menu" case, a `CheckboxControl` for explicit dismissal, and missing states (error + success toast). All components are from `@wordpress/components` — no new patterns. The checkbox persists a site option; the existing "Import from Legacy Menu" button in the Header serves as the re-surface path with no new UI needed.

### Decisions worth knowing
- Warning uses WP admin notice-warning visual treatment (amber bg, 4px left border) — not just an emoji prefix in a `<p>` tag
- Checkbox preference stored as a WP site option (`platform_settings_mega_menu_migration_modal_dismissed`), not user meta — typically one admin manages the mega menu
- The "Import from Legacy Menu" button in the Header is always visible when `hasLegacyMenu` is true, regardless of dismissal preference — this IS the re-surface path (checkbox only suppresses auto-open)
- Dropped the `→` arrow from "Import and Review" button — no page navigation occurs
- Error state replaces body text with error notice + "Try Again" primary action
- Success state is a WP snackbar toast, not an in-modal confirmation

### Open questions
1. Should the dismissed preference be per-user (user meta) or site-wide (site option)? Pixel recommends site option — one admin typically manages the mega menu. Confirm before implementation.

## Decisions

- Current modal auto-open logic (`hasLegacyMenu && !hasDraft && !hasPublished`) is condition-based with no persistence — THR-1612 adds explicit persistence via a "Don't show this again" checkbox
- No DB flag exists yet for modal dismissal — this is net-new, using a WP site option keyed as `platform_settings_mega_menu_migration_dismissed` (not `_modal_dismissed` — the concept is dismissing the migration prompt, not a specific UI element). Follows the established `platform_settings_mega_menu_*` convention from `MegaMenuBlocksApi.php`.
- Option constant lives in `MegaMenuBlocksApi` as `private const MIGRATION_DISMISSED_OPTION_KEY` — alongside the other 4 option key constants. Private because only the API class reads/writes it.
- **Site option, not user meta** — mega menu configuration is a single-admin operation. Site-level gives a simpler mental model: "this site has seen the migration prompt."
- The auto-open condition becomes: `hasLegacyMenu && !dismissed` — the `!hasDraft && !hasPublished` guards are replaced by the explicit preference. If the user unchecks "Don't show this again," the modal resumes auto-opening.
- The `/migrate/status` endpoint already returns `hasLegacyMenu`, `hasDraft`, `hasPublished` — extend it to also return `dismissed: bool`. No new endpoint needed.
- Dismissal preference is persisted via a new `PUT /thrive-mega-menu/{id}/migrate/dismiss` endpoint — sets or clears the option. Added to `MegaMenuBlocksApi` alongside the existing migration routes.
- Re-surface path is the existing "Import from Legacy Menu" button in the Header — no new UI surface needed. The button always appears when `hasLegacyMenu` is true. The checkbox only suppresses auto-open, not manual trigger.
- Warning notice uses styled box (amber bg, left border) instead of emoji-in-paragraph — matches WP admin notice convention
- Body text shortened from ~42 words to ~20 — the title already explains the action, body focuses on "starting point" and "review before publishing"
- Error and success states added: error replaces modal body with notice + "Try Again"; success closes modal and shows a WP snackbar toast
- Error handling stays simple — the existing `handleMigrate` catch block already keeps the modal open on failure. Change: set an `error` state variable instead of dispatching a snackbar, so the modal can render Pixel's error state inline. "Try Again" calls `onConfirm` again, clearing the error state first.
- The migration API already returns structured error responses with `message` fields and proper HTTP status codes (404, 409, 500). `apiFetch` surfaces the `message` on throw — no status code parsing needed on the frontend.

## Acceptance Criteria

### Behavioral
- [ ] Given an admin opens the mega menu editor with a legacy menu present and the modal not dismissed, When the editor loads, Then the "Import from Legacy Menu" modal appears automatically
- [ ] Given the modal is open, When the admin checks "Don't show this again" and closes the modal (Cancel, X, or Import), Then the modal does not auto-open on the next editor load
- [ ] Given the modal has been dismissed via checkbox, When the admin clicks "Import from Legacy Menu" button in the editor header, Then the modal opens (manual trigger always works regardless of dismissal preference)
- [ ] Given the admin previously checked "Don't show this again," When they open the modal manually and uncheck the checkbox and close, Then the modal resumes auto-opening on future editor loads
- [ ] Given the admin has an existing draft or published menu, When the modal is open, Then a styled warning notice (amber background, left border) appears: "This will replace your existing draft or published menu."
- [ ] Given a migration fails, When the error occurs, Then the modal shows an error message ("The import couldn't be completed. Your existing menu was not changed.") with "Close" and "Try Again" buttons
- [ ] Given a migration fails and the admin clicks "Try Again," When the retry succeeds, Then the modal closes and a success toast appears
- [ ] Given a migration succeeds, When it completes, Then the modal closes and a toast notification confirms: "Menu imported — review your changes below."

### Non-behavioral
- [ ] The close button (X) in the modal header is right-aligned, not centered
- [ ] All modal states are keyboard-navigable (Tab through checkbox → Cancel → primary action, Escape to close)
- [ ] Warning and error notices are announced to screen readers (`role="alert"`)
- [ ] The dismissal preference persists across browser sessions (stored server-side, not in local storage)

## Review Issues

### Dismissed preference lost on successful migration
- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** Removed `localDismissed` state from MigrateModal — checkbox now calls `onDismissedChange` directly on toggle, updating parent state and persisting via API immediately. No local state to lose on unmount.
- **File:** `editor.tsx:330` / `MigrateModal.tsx:38-44`
- **Problem:** When migration succeeds, `handleMigrate` calls `setIsMigrateModalOpen(false)` directly, unmounting the modal. `MigrateModal.handleClose` — which persists the dismissed preference via `onDismissedChange` — never fires. If a user checks "Don't show this again" and then clicks "Import and Review," the preference is silently lost.
- **Suggested fix:** In `editor.tsx`, persist the dismissed preference in the `handleMigrate` success path before closing the modal. Read the current `dismissed` state and compare to the modal's checkbox value — or pass the checkbox state back through `onConfirm` so the parent can persist it.
