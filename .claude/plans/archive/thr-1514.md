# Plan: THR-1514

## Ticket

https://linear.app/tractru/issue/THR-1514/fix-qa-failures-from-stretchy-text-implementation-thr-1373-follow-up

## Goal

Fix two QA failures from the THR-1373 stretchy text implementation: FitText not stretching inside column blocks, and HTML anchor (`id`) not passed to the frontend.

---

## Decisions

- `width: 100%` lives in FitText's `mergedStyle`, not in consumers — it's a measurement invariant, not a layout choice
- `prose` is excluded when `fitText` is true — FitText controls font-size via binary search so prose typography adds no value, and its max-width interferes with measurement. This avoids needing `max-w-none` which would defeat `container` max-width
- Anchor fix scoped to HeadingBlock and ParagraphBlock (`id={anchor?.trim() || id}`), not BlocksRenderer — narrower blast radius, these are the blocks where anchors matter
- `anchor` added to both block schemas as `anchor?: string` — WordPress sends it as a block attribute
- Storybook stories for FitText visual regression — Jest can't test DOM measurement; Storybook captures the actual rendering behavior that broke
- Anchor extraction belongs in `BlocksProcessor.php`, not frontend resolvers — follows established innerHTML → attrs pattern (button, image), uses proper DOM parsing with DOMDocument + wp_kses sanitization, eliminates regex in Node. WordPress stores the anchor as `id="..."` in innerHTML; `attrs.anchor` is not reliably present. BlocksProcessor normalizes this before GraphQL so the frontend just reads `attrs.anchor`.
- Frontend heading/paragraph resolvers removed — they existed solely for anchor extraction. With BlocksProcessor handling it, the resolvers add no value. Fewer dynamic imports, less code.
- `extractIdFromHTML` utility removed — no longer needed. PHP DOMDocument is more robust than a regex for HTML parsing.

## Debugged Issues

### FitText doesn't stretch inside column blocks with non-stretch alignment
- **Status:** `fixed`
- **Fixed in:** FitText.tsx — added `width: '100%'` to mergedStyle; ParagraphBlock.tsx — added `max-w-none` when fitText is true
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — any page with a stretchy heading/paragraph inside a Columns block whose column has horizontal alignment set to center (or any non-stretch value)
- **File:** `frontend/components/utility/FitText.tsx:54` and `frontend/blocks/heading/HeadingBlock.tsx:74`
- **Root cause:** Two factors combine: (1) HeadingBlock only applies `w-full` when `!isInnerBlock` (line 74), so headings inside columns have no explicit width. (2) FitText captures `element.clientWidth` at the start of its binary search (line 54), but without `width: 100%`, a heading in a `flex-col items-center` column is shrink-wrapped to its content width. The binary search then finds the largest font that fits within that small content width — which is approximately the default font size. The column gets `items-center` from `BLOCK_ALIGNMENT_MAPPINGS.center` when horizontal alignment is set to "center", or from manual className.
- **Affected layout contexts:**
  - Column with `horizontalAlignment: "center"` → `items-center` (confirmed)
  - Column with `horizontalAlignment: "left"` → `items-start` (same mechanism)
  - Column with `horizontalAlignment: "right"` → `items-end` (same mechanism)
  - Column with default alignment → `items-stretch` (NOT affected — child stretches to full width)
  - Any other flex/grid parent where FitText element isn't explicitly full-width
- **Padding/margin interaction:** Padding classes (`px-9 py-5`, `block-gap`, or custom spacing) and inline styles from `buildBlockStyles` (colors, CSS vars) do NOT affect FitText measurement — `clientWidth` and `scrollWidth` both account for padding consistently. No width/max-width is set by `buildBlockStyles`.
- **Paragraph `prose` class interaction:** ParagraphBlock applies the `prose` class, which sets `max-width: 65ch`. Since `ch` is font-size-relative, this creates a subtle interaction: `clientWidth` is captured once at the initial font size, but `max-width: 65ch` recomputes as FitText mutates font-size during binary search. The element may grow wider than the captured `containerWidth` during iteration. Adding `width: 100%` alone may not fully fix paragraph FitText when `prose` is present — the `max-width: 65ch` constraint may need to be overridden with `max-w-none` when `fitText` is true.
- **Steps to Reproduce:**
  1. In WordPress editor, add a Columns block with one column
  2. Set the column's horizontal alignment to "center" (or add `items-center` class)
  3. Inside the column, add a Stretchy Heading variation
  4. Publish and view the frontend
- **Expected behavior:** The stretchy heading scales to fill the column width
- **Actual behavior:** The heading renders at approximately its default font size, not stretched
- **Recommended fix:** Add `width: '100%'` to FitText's `mergedStyle` object (line 96-101 in FitText.tsx). This is a fundamental requirement of the measurement algorithm — FitText must fill its container to measure against the available space, not the content size. This approach is better than conditionally adding `w-full` in HeadingBlock/ParagraphBlock because it fixes the issue for any parent layout, not just flex columns. For ParagraphBlock, also add `max-w-none` to the className when `fitText` is true to override the `prose` max-width constraint.
- **Suggested tests:** Storybook stories for visual regression testing — add stories for FitText inside a flex-col items-center parent, FitText at top level, and FitText inside a column with each alignment option. Jest tests for `width: 100%` in rendered style.
- **Linear:** `not synced`

### HTML anchor attribute not passed to frontend element
- **Status:** `fixed`
- **Fixed in:** `BlocksProcessor.php` — added `processBlockAnchor` + `extractAnchorFromInnerHTML` methods to extract `id` from heading/paragraph innerHTML using DOMDocument. Frontend resolvers and `extractIdFromHTML` utility removed. Components already have `anchor?.trim() || id` wiring.
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — any page with a heading or paragraph that has an HTML anchor set in the editor
- **File:** `frontend/blocks/BlocksRenderer.tsx:125` and `frontend/blocks/heading/HeadingBlock.tsx:38`
- **Root cause:** WordPress stores the HTML Anchor field as an `anchor` attribute on the block. For blocks without a resolver (like heading), attrs are spread as props — so `anchor` is available. But BlocksRenderer always sets `id: context.id` (line 125), which is an auto-generated value from `BlockRenderSession.nextId()` (e.g. `main-heading-1`). HeadingBlock and ParagraphBlock destructure `id` but never read the `anchor` prop. The user-entered anchor value is silently ignored.
- **Steps to Reproduce:**
  1. In WordPress editor, add a Heading block (stretchy or regular)
  2. In the block sidebar, set the HTML Anchor field to `my-anchor`
  3. Publish and inspect the heading element on the frontend
- **Expected behavior:** The heading element has `id="my-anchor"`
- **Actual behavior:** The heading element has `id="main-heading-1"` (auto-generated)
- **Recommended fix:** In HeadingBlock and ParagraphBlock, prefer `anchor` over auto-generated `id` when present: `id={anchor ?? id}`. Add `anchor` to both block schemas. Same fix applies to both the FitText and non-FitText rendering paths. Consider also fixing this at the BlocksRenderer level for all blocks, but that's a broader change.
- **Suggested tests:** Add test verifying that when `anchor` prop is provided, it's used as the element `id` instead of the auto-generated one. Test both fitText and non-fitText paths.
- **Linear:** `not synced`

### `max-w-none` defeats `container` max-width on root-level fitText paragraphs
- **Status:** `fixed`
- **Fixed in:** Replaced prose + max-w-none with !fitText && prose in ParagraphBlock. Updated tests.
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — any page with a root-level stretchy paragraph (`!isInnerBlock`)
- **File:** `frontend/blocks/paragraph/ParagraphBlock.tsx:62-64`
- **Root cause:** ParagraphBlock applies `prose`, `max-w-none`, and `container` to the same element when `fitText && !isInnerBlock`. Tailwind utilities (`max-w-none`) cascade after components (`.container`), so `max-w-none` overrides both `prose`'s `max-width: 65ch` (intended) and `container`'s `max-width: 90%` (collateral). The element has no max-width constraint at all. Since there's no page-level width wrapper — each block manages its own width — a root-level fitText paragraph renders wider than a root-level fitText heading (which correctly respects `container`'s 90% via HeadingBlock).
- **Steps to Reproduce:**
  1. Add a Stretchy Paragraph at the top level (not inside a column)
  2. On the same page, add a Stretchy Heading at the top level
  3. View the frontend and compare widths
- **Expected behavior:** Both fitText elements respect `container`'s `max-width: 90%` and render at the same width
- **Actual behavior:** The paragraph ignores the 90% constraint and renders wider than the heading
- **Recommended fix:** Remove `prose` from ParagraphBlock's classNames when `fitText` is true (instead of keeping `prose` and overriding with `max-w-none`). FitText controls font-size via binary search, so `prose`'s typography rules don't add value. This eliminates `max-w-none` entirely and lets `container`'s `max-width: 90%` work correctly. Change `fitText && "max-w-none"` to `!fitText && "prose"` (invert the condition).
- **Suggested tests:** Update ParagraphBlock test that checks for `max-w-none` — it should now verify `prose` is absent when fitText is true. Add test verifying `prose` is present when fitText is false.
- **Linear:** `not synced`

### QA report: stretchy paragraph in column appears non-stretchy
- **Status:** `not a bug`
- **Severity:** N/A — expected behavior
- **Environment:** Frontend — stretchy paragraph inside second column of a Columns block
- **File:** `frontend/components/utility/FitText.tsx`
- **Root cause:** Not a code defect. FitText IS rendering correctly (confirmed by `width: 100%`, `white-space: nowrap`, `overflow: hidden` inline styles). The binary search lands on `font-size: 16px` because the paragraph text (~80+ characters) is too long for the half-width column container. FitText finds the largest integer font-size where the nowrap text fits without overflow — for long text in a narrow container, that result can equal or be smaller than the browser default (16px), making the stretchy effect invisible.
- **Steps to Reproduce:**
  1. Add a Columns block with two columns
  2. In the second column, add a stretchy paragraph with a full sentence (~80+ characters)
  3. View the frontend
- **Expected behavior:** FitText renders and computes the largest font-size that fits — which for long text may be the same as or smaller than default
- **Actual behavior:** Exactly as expected — FitText computes 16px, which looks like a normal paragraph
- **Recommended fix:** No code fix needed. This is a content guidance issue — stretchy text works best with short, punchy text (1–5 words). A full sentence in a half-width column doesn't have room to grow beyond default size. QA/content team should be informed that this is by design.
- **Suggested tests:** none needed
- **Linear:** `not synced`
