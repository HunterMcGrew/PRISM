## Debugged Issues

### Wheel/Tab lock on site-search page after autocomplete submit

- **Status:** `fixed`
- **Severity:** High
- **Environment:** Was 100% reproducible in Chrome + Firefox, local dev + production build. Not browser-specific, not bundler-specific.
- **File:** `frontend/components/site-search/Autocomplete.tsx`. The destination page (`/search`) was innocent — `ScrollTo` and `KronosSearchBlock` worked fine when `/search` was reached by direct link, refresh, or bookmark. The bug only reproduced via navigation from Autocomplete's submit paths, and narrowing further, only when `<ViewResultsButton>` sat as a non-`ComboboxOption` child of `<ComboboxOptions static>`.
- **Root cause:** `<ViewResultsButton>` was rendered as a non-`ComboboxOption` `<button>` inside `<ComboboxOptions static>`. Clicks traversed Headless UI's options state machine and left Combobox state in a condition that blocked wheel and Tab events on the destination page until any click anywhere released it. The destination's wheel events reached `window` with `defaultPrevented: false` / `cancelable: false` — the page simply wasn't the scroll target because the Combobox state machine was still holding the focus/event context.
- **Steps to Reproduce (historical):**
  1. Navigate to any dealer page with the site-search in the header
  2. Click the search input in the header (AdvancedSearch / Autocomplete)
  3. Type a broad query like `BX`
  4. Click "View all Results" at the bottom of the suggestions dropdown
  5. Destination page (`/search?q=BX&searchType=all`) loads
  6. Try to scroll with mousewheel or trackpad — no scroll happens until clicking anywhere on the page
  7. Hit Tab at this point — focus exits the page to browser chrome instead of moving to the first tabbable element
- **Fixed in:** [PR #1841](https://github.com/TracTru/thrive/pull/1841), commit `298896aec`. Moved `<ViewResultsButton>` outside `<ComboboxOptions>` so it renders as a sibling inside the visible dropdown panel. Visually identical — still inside the `data-headlessui-state={open}` wrapper — but no longer a child of Headless UI's options state machine. Also reverted `ScrollTo.tsx` to `origin/main` (the `hasInteracted` gate added during diagnosis was defending against a disproved hypothesis and suppressed legitimate scroll-to-top on refinement changes).
- **Suggested tests:** Playwright e2e (still recommended as follow-up) that clicks "View all Results" then fires a `wheel` event and asserts `window.scrollY > 0`. Jest cannot reproduce this because it's a browser-level scroll/focus behavior. Existing 37 Jest cases still assert the button's handler behavior (navigate, reset, callback).
- **Linear:** `synced` — PR #1841 round 2 context appended to the ticket description 2026-04-22.

| 6 | Sasha-era gate: `ScrollTo` only scrolls after `hasInteracted` | `ScrollTo.tsx` | Kept — improved UX (no half-scroll on landing) but didn't fix wheel lock | `scrollIntoView({behavior: smooth})` is NOT the cause; ScrollTo is innocent |

**Disproved hypotheses (important negative evidence — do not re-investigate):**

- *Focus-trapping on the combobox input.* Instrumentation confirmed `document.activeElement === body` throughout the broken state, no `focusin` events firing during the 3-second watch window.
- *CSS scroll-lock.* `getComputedStyle(document.body).overflow === "visible"`, same for `html`. No inline `overflow: hidden`. No `scroll-behavior: smooth` anywhere in the active codebase. No `inert` attributes used in `frontend/`.
- *`scrollIntoView({behavior: smooth})` hijacking the wheel.* Clove's gate at [ScrollTo.tsx:44](frontend/blocks/kronos-search/ScrollTo.tsx#L44) eliminates the programmatic smooth scroll on initial page hydration. Bug persists after that fix. Experiment #6.
- *Missing `blur()` on submit handlers.* `handleViewAllResultsClick` already calls `resetState()` which blurs the input — and the user's focus ends up on `body`, which is what blur achieves. Explicit `document.activeElement.blur()` also had no effect.
- *Wheel `preventDefault` listener somewhere.* Wheel instrumentation captured 58 events; all `defaultPrevented: false`.
- *Next.js App Router scroll-to-top.* Experiment #1.
- *`nextjs-toploader` / progress bar.* Experiment #2.
- *Async chunk loading during hydration.* Experiment #3.
- *React 19 concurrent-mode pending transition.* Implicit in experiments #4 and #5 — the bug is deterministic across browser engines and build modes.
- *`ScrollTo` / `KronosSearchBlock` themselves.* User confirmed: both components work fine on other pages (equipment-list, showroom, etc.). The destination page is innocent.

**What we still know about the bug (confirmed observations):**

- **Specific to Autocomplete's navigation to `/search`.** All Autocomplete submit paths (Enter, magnifying glass, View all Results) reproduce. Clicking a dropdown suggestion (which navigates to a product permalink instead of `/search`) does NOT reproduce. Direct navigation to `/search` (refresh, bookmark, direct link) does NOT reproduce.
- **`cancelable: false` on wheel events** — unusual signal; wheel events are typically `cancelable: true` in Chrome.
- **Tab exits the document to browser chrome** — the page's focus scope is in a state that Tab navigation can't enter.
- **A click anywhere on the page restores both wheel and Tab.**
- **Focus is on `body`** throughout the broken state.
- **User can scroll for a "split second" before the page fully loads** — whatever the trigger is, it activates right around when the destination page commits.

**Next investigation direction:**

The bug is in what Autocomplete does uniquely when navigating to `/search`. Candidates worth investigating:
- Headless UI `Combobox` internal state at the moment of submit. The `<ComboboxOptions static>` panel stays rendered (via `static` prop) and `ViewResultsButton` is a non-ComboboxOption `<button>` inside it — clicks may not trigger Headless UI's normal close/blur flow. Worth instrumenting Combobox's internal state across the navigation.
- `<ComboboxInput>` ref behavior across navigation. The Autocomplete is in the layout header — it persists across navigation. Its Combobox state does NOT unmount. Anything Headless UI sets up globally (document listeners, focus traps, scroll locks) persists too.
- Order-of-operations: `router.push` before `resetState` vs after. Currently `router.push` fires first, then `resetState`. Maybe the navigation commits before the combobox state is cleaned up, leaving orphaned listeners.
- Diff Autocomplete vs a minimal `<a href="/search?q=BX">` link that navigates directly. If the `<a>` link doesn't reproduce, the bug is 100% in what Autocomplete does differently.
