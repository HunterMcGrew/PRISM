# Plan: THR-1458

## Ticket

THR-1458 — Type-safe block registration: enforce resolver↔view prop contract

## Goal

Make `registerBlock` generic so TypeScript catches resolver/view prop mismatches at compile time, then fix all pre-existing mismatches.

---

## Decisions

- Make the existing `BlockRegistration` generic — `BlockRegistration<Props = any>` — instead of introducing a second interface. `BlockService` references `BlockRegistration` without a type param, which resolves to `BlockRegistration<any>` via the default. One type, zero new interfaces.
- `NoInfer<Props>` on `BlockRegistration.resolver` — the view component defines the canonical Props type; the resolver is checked for compatibility, not used for inference. This matches the actual data flow where the renderer (`BlocksRenderer`) provides `BaseBlockProps` (`id`, `blockIndex`, `blockName`, `innerHTML`, `isInnerBlock`, `isLastBlock`) and the resolver provides block-specific data.
- The generic enforcement happens at the `block-registry.ts` call site via `registerBlock<Props>()`, not inside `BlockService.registerBlock()` — this keeps the runtime service simple and avoids threading generics through the resolver cache and resolve pipeline. `BlockService` doesn't change at all.
- Skip `createBlock` factory for now — the generic `registerBlock` already provides compile-time safety. A factory can be added later if a pattern emerges, but it's not needed to close this ticket.
- `currentSlideIndex` on HeroCarouselBlock made optional — it's client-side interactive state, not server-resolved data. Required props on a view component should reflect what the component genuinely can't function without, not what happens to be available.
- HeroCarouselBlock Props decoupled from `BlockAttributes` — component defines its own data requirements (`autoHeight: boolean`, `autoplay: boolean`, etc.) instead of `extends BlockAttributes`. This is the first block migrated to the decoupled pattern documented in `.claude/rules/component-props-decoupling.md`.
- For blocks without resolvers, the `resolver` field is omitted from `BlockRegistration<Props>`. The default pass-through in `BlockService.resolveBlock()` (`{ ...attrs, ...meta, innerHTML }`) is not type-checked by the registry — that's a runtime concern outside this ticket's scope.

## Phases

### Phase 1: Type infrastructure

**Files changed:**
- `frontend/lib/interfaces/block.ts` — make `BlockRegistration` generic with `<Props = any>` default
- `frontend/lib/services/block/block-registry.ts` — make `registerBlock` generic

**Steps:**
1. Parameterize `BlockRegistration` in `interfaces/block.ts`:
   ```ts
   export interface BlockRegistration<Props = any> {
       resolver?: () => Promise<{ default: BlockResolver<Props> }>;
       childrenAsRoot?: boolean;
       view: ComponentType<Props>;
   }
   ```
2. Update `registerBlock` in `block-registry.ts`:
   ```ts
   function registerBlock<Props>(
       blockType: string,
       registration: BlockRegistration<Props>
   ) {
       BlockService.getInstance().registerBlock(blockType, registration);
   }
   ```
3. `BlockService` — no changes. Its `Map<string, BlockRegistration>` resolves to `Map<string, BlockRegistration<any>>` via the default.
4. Run `tsc --noEmit` to surface all mismatches.

### Phase 2: Fix hero-carousel

The only real mismatch after NoInfer. The resolver is typed as `BlockResolver<BlockAttributes>` but the component expects `BlockFrontendProps` (extends `BlockAttributes` with `currentSlideIndex`, etc.). `currentSlideIndex` is client-side interactive state — make it optional on the component, update resolver `satisfies` to use `BlockFrontendProps`.

### Phase 3: Add resolver unit tests

Add tests for resolvers that perform data transformation in `resolveProps`, verifying the output shape matches what the view component expects.

### Phase 4: Verify and clean up

1. Run `tsc --noEmit` — zero errors
2. Run existing tests — no regressions
3. Update this plan
