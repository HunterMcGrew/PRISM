# Plan: THR-1606

## Ticket

https://linear.app/tractru/issue/THR-1606/mega-menu-editor-clicking-linksbuttons-navigates-instead-of-selecting

## Goal

Prevent interactive elements (links, buttons) in mega menu editor blocks from navigating when clicked — they should select the block for editing instead.

---

## Implementation Tasks

### Clove (implementation)

1. Add missing `import "./editor.scss"` to mega-menu-link-with-icon/edit.tsx
2. Add missing `import "./editor.scss"` to mega-menu-links-with-icon/edit.tsx
3. Add `pointer-events: none` for `a` and `button` elements in card/editor.scss
4. Add `button` to existing `pointer-events: none` selector in mega-menu-featured-banner/editor.scss

---

## Decisions

- Link-with-icon blocks had correct CSS rules but missing imports — root cause was dead editor.scss files, not wrong selectors
- Card block scoped to `.wp-block-gravity-platform-core-card` (WP block class) rather than a mega-menu-specific selector — prevents accidental navigation in all editor contexts
- Featured banner keeps `.editor-styles-wrapper` scoping since the mega menu standalone editor does render that class (confirmed in BlockEditor.tsx)

## Acceptance Criteria

### Behavioral

- [ ] Given a mega menu is open in the editor, when clicking on a link-with-icon element, then the block is selected for editing (no navigation occurs)
- [ ] Given a mega menu is open in the editor, when clicking on a card block, then the block is selected for editing (no navigation occurs)
- [ ] Given a mega menu is open in the editor, when clicking on the featured banner button, then the block is selected for editing (no navigation occurs)
- [ ] Given any of the above blocks in the editor, when using keyboard navigation to reach the block, then the block can be selected and edited normally

### Non-behavioral

- [ ] Changes are CSS-only in editor.scss files — no frontend rendering impact
- [ ] Existing block editing functionality (toolbar, sidebar controls) remains unaffected
