# Plan: THR-1526

## Ticket

https://linear.app/tractru/issue/THR-1526/bug-mobile-mega-menu-shows-no-navigation-items

## Goal

Fix the mobile mega menu rendering no navigation items due to a broken `c.type` comparison introduced by the THR-1457 block-views refactor.

---

## Decisions

- Replace `c.type === NavigationMenuItemBlock` with `c.props.blockName === BLOCK_TYPES.NAVIGATION_MENU_ITEM` — the `blockName` prop is set by `BlockChunk` for every block element and is stable regardless of component wrapping strategy
- Remove the direct import of `NavigationMenuItemBlock` from `MegaMenuBlock.tsx` — it's no longer needed for the type comparison and was only used for the broken `c.type` check
- THR-1475 (SlidingPanel useEffect removal) is NOT the cause — confirmed clean and unrelated
- **Refactor: move nav item extraction to `resolveProps`** — eliminates `Children.toArray` + `isValidElement` introspection entirely. Resolver extracts mobile nav data from `block.innerBlocks` on the server. Desktop rendering still uses `{children}` from BlocksRenderer. This avoids all sensitivity to `next/dynamic` wrapper boundaries.
- `resolveProps` (not `resolveChildren`) is the correct tool — `resolveChildren` can't serve both desktop rendering (all children) and mobile data extraction (filtered props)
- Location filtering inlined in mega-menu resolver (not shared via resolver.props.ts) — 3-line filter doesn't justify a shared file with only 2 consumers
- MegaMenuBlock Props decoupled from `BlockAttributes` per THR-1452 migration — `MegaMenuBlockProps` replaces `BlockFrontendProps`
- MobileMegaMenu now imports `MobileNavItem` from MegaMenuBlock instead of `NavigationMenuItemBlockProps`

## Debugged Issues

### Mobile mega menu shows no navigation items
- **Status:** `fixed`
- **Fixed in:** `MegaMenuBlock.tsx` — replaced `c.type === NavigationMenuItemBlock` with `c.props.blockName === BLOCK_TYPES.NAVIGATION_MENU_ITEM`
- **Severity:** High
- **Environment:** Local dev (`pnpm run dev`, localhost:3000). Reportedly not visible on staging — possible `next/dynamic` dev-vs-prod behavior difference (see Additional Context below).
- **File:** `frontend/blocks/mega-menu/MegaMenuBlock.tsx:37`
- **Root cause:** `c.type === NavigationMenuItemBlock` compares the raw component function (imported from `../navigation-menu-item/NavigationMenuItemBlock`) against the element's actual `type`, which is a `next/dynamic()` wrapper (from `block-views.tsx`). These are different JavaScript references, so the filter always returns an empty array and `mobileNavItems` is empty.
- **Steps to Reproduce:**
  1. Run `pnpm run dev` in the frontend directory
  2. Open the site on a mobile viewport (or resize to < `lg` breakpoint)
  3. Tap the hamburger menu to open the mobile mega menu
  4. Observe the SlidingPanel opens but contains no navigation items
- **Expected behavior:** The mobile mega menu shows all top-level navigation items (service, equipment, parts, etc.) as clickable links/expandable rows.
- **Actual behavior:** The SlidingPanel opens but the `<ul>` is empty — no items are rendered because `menuLinks` is derived from the empty `mobileNavItems` array.
- **Recommended fix:** Replace `c.type === NavigationMenuItemBlock` with `c.props.blockName === BLOCK_TYPES.NAVIGATION_MENU_ITEM`. The `blockName` prop is set by `BlockChunk` (BlocksRenderer.tsx:127) for every rendered block element and is a stable string identifier (`"gravity-platform-core/navigation-menu-item"`). Remove the now-unused `NavigationMenuItemBlock` import. Import `BLOCK_TYPES` from `@frontend/lib/constants/block-types`.
- **Suggested tests:** Add a test for `MegaMenuBlock` that verifies `mobileNavItems` extraction works when children have `blockName` props (simulating `BlockChunk` output). Test both expandable and non-expandable items.
- **Linear:** `synced`

### Additional Context

**Why staging may still work:** Three hypotheses (ordered by likelihood):
1. `next/dynamic` resolves imports eagerly during production SSR, making `c.type` resolve to the actual component. In dev mode, the lazy wrapper stays as the element type.
2. Staging hasn't been deployed with THR-1457's changes.
3. Nobody tested the mobile mega menu on staging.

**Introducing commit:** `a80592489` — THR-1457: client-side code splitting for block and template registries (#1646), merged 2026-03-23. Changed `BlockService.loadView()` from `const mod = await reg.view(); return mod.default` (raw component) to `return this.registrations.get(blockName)?.view` (dynamic wrapper).

**Not the cause:** THR-1475 (SlidingPanel useEffect removal) is confirmed clean. The SlidingPanel correctly passes `openPanel` directly to `<Dialog open={openPanel}>` and the close handler works. The panel opens — it's the content that's empty.
