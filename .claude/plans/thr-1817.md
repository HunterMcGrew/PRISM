## Decisions

- **Migrate the block's image attribute to `image: IImage`, native shape, with a WP `deprecated[]` migration for legacy `backgroundImage: string` saved content.**
  - **Root cause:** Hunter flagged that `backgroundImage: string` plus the three flat dimension fields landed during the original bug fix (`backgroundImageId`, `backgroundImageWidth`, `backgroundImageHeight`) is the wrong shape. `IImage` already carries `id`, `sourceUrl`, `altText`, `title`, and `mediaDetails: { width, height }` as a single object — the four flat fields decompose what should be one. The architect docs had no doctrine on image attribute shape (ADR-0019 covered links; nothing covered images), so the original bug fix codified the wrong target.
  - **Alternatives considered:** (a) keep the four flat fields and ship the bug fix as-is, file a follow-up ticket for the IImage migration — rejected because shipping a known antipattern with our own architect doc cementing it costs more in technical debt than absorbing the migration now; (b) **adapter pattern** per ADR-0019 (keep flat fields at schema, build IImage in the handler) — rejected because the legacy shape is just `backgroundImage: string` (a single URL), which migrates cleanly via WP's `deprecated[]` parse-time hook without needing the adapter complexity; (c) **native shape with WP `deprecated[]`** — chosen.
  - **Chosen approach:** schema attribute is `image: IImage`. Drop `backgroundImage`, `backgroundImageId`, `backgroundImageWidth`, `backgroundImageHeight` from `BlockAttributes`. Register a `deprecated: [{ ... }]` entry in the block registration that detects legacy `backgroundImage: string` saved content (`isEligible(attrs) { return typeof attrs.backgroundImage === "string"; }`) and migrates it to `image: IImage` at parse time. Frontend component (`MegaMenuFeaturedBanner.tsx`) reshapes its prop from `backgroundImage?: string` to `image?: IImage` so the frontend consumes the same shape the editor stores. Skew bug stays fixed because `image.mediaDetails.width/height` feeds `ResponsiveWrapper` exactly as `mega-menu-info-banner/edit.tsx` already demonstrates.
  - **Implementation guidance:** mirror `frontend/blocks/mega-menu-info-banner/schema.ts` for the schema shape, `backend/plugins/gravity-platform-core/src/blocks/mega-menu-info-banner/edit.tsx` for the `handleMediaSelect` and `ResponsiveWrapper` wiring, and `backend/plugins/gravity-platform-core/src/blocks/button-group-item/index.tsx` for the `deprecated[]` migration pattern. The `LEGACY_PLACEHOLDER_BACKGROUND_IMAGE` URL is dropped at migration time (legacy URL → `DEFAULT_ATTRIBUTES.image` with empty `sourceUrl`) so the resolver guard for the placeholder URL becomes a defensive case rather than a regularly-triggered one. Component layout behavior is unchanged — `<Image fill>` with `src={image.sourceUrl}` still renders the same.
- **Closed the architect-doc gap by writing ADR-0027 ("Block Image Attributes Use IImage") and adding `backend-editor-blocks.md` § Image Attributes alongside the existing § Link Attributes.**
  - **Root cause:** ADR-0019 codified `ILink` for block link attributes with an encounter protocol and a canonical pattern reference. `IImage` had no equivalent doctrine — domain models used it consistently, several blocks used it, but new block authors had no architect-doc surface telling them this was the canonical shape. The Featured Banner author landed flat fields on this branch precisely because there was no guidepost.
  - **Alternatives considered:** a generalized ADR covering "block attributes use domain types where they exist" — rejected because the encounter protocol is shape-specific (what counts as legacy, what counts as a clean migration, where the canonical example lives); ADR-0019's link-specific framing has worked in practice and ADR-0027 mirrors it deliberately. A `.claude/rules/` entry — rejected because rules don't carry the alternatives-considered + consequences narrative that survives future re-litigation.
  - **Chosen approach:** ADR-0027 modeled exactly on ADR-0019's structure (Context, Decision with `IImage` shape, native-vs-adapter migration tree, encounter protocol, canonical references). Cross-referenced from `backend-editor-blocks.md` § Image Attributes. The original `backend-editor-blocks.md` § MediaUpload + ResponsiveWrapper Preview Pattern (added during the bug fix) is reframed to describe the `IImage.mediaDetails` shape instead of the flat-field shape — same ResponsiveWrapper math, correct schema target.
  - **Implementation guidance:** future block image attributes follow ADR-0027. Reviewers (Eric, Briar) flag `imageUrl: string` or split-field clusters on new attributes, citing the ADR. The encounter protocol governs absorb-vs-ticket for existing blocks (`card`, `feature`, `hero-card`, `hero-cards`) when they're touched. No existing-block migrations are in scope for this ticket.
- **Backwards compatibility:** existing block instances saved with `backgroundImage: string` are migrated at parse time by the `deprecated[]` entry. `ResponsiveWrapper` accepts `undefined` for `naturalWidth` / `naturalHeight` and falls back to intrinsic-dimension rendering, so legacy content (which has no stored `mediaDetails`) renders without skew until the editor next replaces the image. No data migration script needed.
- **`MediaSelector` is part of the legacy split-field pattern, not a canonical control for `IImage` blocks.**
  - **Root cause:** Eric's PR-review caught ADR-0027 §Decision describing `MediaSelector` as "the single editor control" that emits "the consumer's chosen shape" — both claims diverge from source. `MediaSelector.tsx:30, 42-49` hardcodes its emit type to `MediaSelectorImage` (`{ imageUrl, mediaId, mediaWidth, mediaHeight, altText }`), structurally the same legacy split-field cluster the ADR is phasing out. Six blocks consume it and inherit the antipattern by construction. The ADR cited it as canonical while simultaneously phasing out the shape it manufactures — a contradiction the doc can't survive.
  - **Alternatives considered:** (a) reshape `MediaSelector` to emit `IImage` — rejected because the wrapper would become a single-consumer abstraction once its six current consumers migrate, which is premature; the canonical pattern (inline `MediaUpload` with handler-builds-`IImage`) is already established by `mega-menu-info-banner/edit.tsx` and reads cleanly without the indirection. (b) leave the ADR paragraph as-is and treat `MediaSelector` as out-of-scope — rejected because the divergence misleads every future agent that loads ADR-0027 via the manifest, and `architect-doc-verification.md` holds doc-class drift at minimum Major. (c) **flag `MediaSelector` as part of the phase-out, name inline `MediaUpload` as canonical, mark the wrapper as a deletion candidate** — chosen.
  - **Chosen approach:** rewrite the MediaSelector paragraph in ADR-0027 §Decision per task 10 above. The encounter protocol then naturally covers `MediaSelector`'s six consumers (`card`, `hero-card`, `hero-cards`, `mega-menu-locations`, `navigation-menu-item`'s `LocationPaneSettings` and `EquipmentPaneSettings`) without needing a special carve-out — they're legacy blocks with legacy image attributes, same encounter rule.
  - **Implementation guidance:** ADR text-only edit, no code changes for `MediaSelector` itself in this ticket. Future migration of any `MediaSelector` consumer will follow the encounter protocol — when the last consumer migrates off, that's the moment to delete `MediaSelector.tsx`.
- **Apply the deprecation pattern's three legs to `mega-menu-featured-banner` — close the migration-service write-side leak and add a read-side resolver adapter.**
  - **Root cause:** the original PR shipped Eric-approved with leg 1 only (WP `deprecated[]` array in `index.tsx`). Two legs were missed: `MegaMenuMigrationService::buildFeaturedBanner()` (`MegaMenuMigrationService.php:681–714`) still emits `attrs.backgroundImage` for every fresh dealer migration — a write-side leak that produces legacy-shape content indefinitely; and the frontend resolver has no read-side fallback for already-published `platform_settings_mega_menu_blocks` content sitting on dealer sites in legacy shape, which renders as `image: undefined` (no banner image) until a human re-saves through the editor. The architect docs had no doctrine on what a "complete" deprecation looks like, so the missing legs landed silently.
  - **Alternatives considered:** (a) keep the new `image` schema and dual-attribute the block by adding `backgroundImage` back to current attributes — rejected because dual-attribute is the dual-source-of-truth antipattern ADR-0027 explicitly phases out; (b) add a mapping in `BlocksProcessor.php` — rejected as the wrong layer (BlocksProcessor handles `core/button`, `core/image`, `core/heading`, `core/paragraph` for post content GraphQL; mega menu blocks flow through `MegaMenuBlocksApi` and never touch BlocksProcessor); (c) server-side normalization in `MegaMenuBlocksApi::transformBlocks()` — rejected because the resolver is already the centralization point for attribute defaults (placeholder guard lives there) and PHP doesn't enforce the IImage type contract; (d) **fix the migration service emit shape (leg 2) and add a typed legacy-shape adapter to the resolver (leg 3)** — chosen.
  - **Chosen approach:** `MegaMenuMigrationService::buildFeaturedBanner()` rewritten to emit `image: IImage` directly (mirrors `migrate()`'s shape — same `mediaDetails: undefined` fallback when dimensions are missing). Three failing tests in `NormalizerAndStructuralTest.php` updated to assert on the new shape. Frontend `schema.ts` adds `LegacyBackgroundImage` and `BlockAttributesWithLegacy` types per ADR-0028's naming convention. `resolver.props.ts` consumes `Partial<BlockAttributesWithLegacy>` and falls back to building IImage from legacy fields when `attrs.image` is absent. Same `LEGACY_PLACEHOLDER_BACKGROUND_IMAGE` guard applies on the legacy path.
  - **Implementation guidance:** the legacy adapter logic mirrors `index.tsx`'s `migrate()` — same shape transformation, just at read time instead of editor-parse time. Both produce `mediaDetails: undefined` when legacy dimensions weren't stored; `ResponsiveWrapper` and `next/image` both handle undefined dimensions via intrinsic-size fallback. Keep the adapter function under ~30 lines per the architect doc's caution against turning it into a data-hydration layer (if it grows past that, the right answer is a one-shot WP migration script, not more adapter logic).

- **Codify the deprecation pattern as a doctrine in ADR-0028 + paired sections in `backend-editor-blocks.md` and `frontend-blocks.md`.**
  - **Root cause:** ADR-0019 and ADR-0027 each implicitly required the same three-leg pattern (WP `deprecated[]`, PHP migration service emit, frontend resolver adapter) but neither captured the framework. THR-1817 made the cost concrete — the migration-service leg got forgotten silently because no doctrine made it visible. Without codification, every future deprecation rediscovers the three legs from existing examples and the same leg keeps getting missed.
  - **Alternatives considered:** (a) document only THR-1817's specific changes, leave the framework implicit — rejected because the same three legs apply to every future block deprecation, and this is the moment to capture the doctrine while it's fresh; (b) extend ADR-0027 with a "complete deprecation" section — rejected because ADR-0027 is image-specific; the framework generalizes to links, future buttons/CTAs, and any block attribute migration (one ADR per concrete shape, one ADR for the framework keeps the responsibilities clean); (c) **new ADR-0028 generalizing the framework, with paired operational sections in the two architect files (backend carries legs 1+2, frontend carries leg 3)** — chosen.
  - **Chosen approach:** ADR-0028 published with the three-leg framework, the `Legacy<DeprecatedAttributeName>` naming convention, the same-name escape hatch (`Legacy<AttributeName>_<ShapeQualifier>`), and a retirement criterion. `backend-editor-blocks.md` § Deprecating Block Attributes covers the WP `deprecated[]` entry shape and the PHP migration service grep test. `frontend-blocks.md` § Legacy Attribute Adapter Pattern covers the type composition (`BlockAttributes` clean + `Legacy*` sibling + `BlockAttributesWithLegacy` intersection) and the resolver shape. Both architect files cross-reference each other and the ADR. THR-1817 becomes the canonical reference implementation.
  - **Implementation guidance:** future block deprecations follow ADR-0028's three-leg framework. Authors run the leg-2 grep test (`grep -rn "'blockName' => '<block-name>'"`) before opening the PR. Reviewers (Eric, Briar) flag any backend `deprecated[]` addition without a paired frontend leg as Major. Legacy types retire in their own deletion ticket once production content has been verified clean, never as part of a feature ticket.

- **`BlockAttributes` stays canonical; `BlockAttributesWithLegacy` is the resolver-facing intersection.**
  - **Root cause:** widening `BlockAttributes` with optional legacy fields would let `edit.tsx` write `setAttributes({ backgroundImage: "..." })` — a regression vector the type system should rule out. The legacy fields are only valid as read inputs to the resolver; they should never appear in any write surface.
  - **Alternatives considered:** (a) widen `BlockAttributes` with optional legacy fields — rejected for contaminating the editor's write surface and silently allowing legacy emission; (b) keep `BlockAttributes` clean and use ad-hoc `as` casts in the resolver — rejected because the legacy shape becomes invisible to TypeScript and reviewers; (c) **separate `Legacy*` shape intersected only at the resolver boundary** — chosen.
  - **Chosen approach:** `BlockAttributes` is the canonical write contract. `LegacyBackgroundImage` is read-only. `BlockAttributesWithLegacy = BlockAttributes & LegacyBackgroundImage` is consumed only by `resolver.props.ts` via `ResolveProps<Props, BlockAttributesWithLegacy>`. The editor, the component, and tests all keep importing `BlockAttributes` (no change from the current code).
  - **Implementation guidance:** only the resolver imports `BlockAttributesWithLegacy`. Each new deprecation round adds another sibling `Legacy*` type to `schema.ts` and intersects it into `BlockAttributesWithLegacy`. The legacy shape never leaks past the resolver boundary — the editor and component types stay clean.

- **Naming legacy shapes after the deprecated attribute, not by version.**
  - **Root cause:** v1/v2/v3 naming carries no information about the shape. A future reader investigating saved legacy content has to dig through git history to find out what `LegacyAttributesV2` represented. Shape-descriptive names are self-documenting.
  - **Alternatives considered:** (a) version numbers (`LegacyAttributesV1`, `V2`) — rejected as opaque; (b) shape qualifiers as the default (`LegacyImage_FlatFields`) — rejected as more verbose than needed when the deprecated attribute name is unique; (c) **default to `Legacy<DeprecatedAttributeName>` with a shape-qualifier escape hatch only when the same attribute name has been deprecated multiple times** — chosen.
  - **Chosen approach:** default `Legacy<DeprecatedAttributeName>` (e.g. `LegacyBackgroundImage`); escape hatch `Legacy<AttributeName>_<ShapeQualifier>` when WP's `isEligible` checks structure rather than presence (rare). The grep test makes the convention work — `LegacyBackgroundImage` is what a debugger would search for when investigating legacy content.
  - **Implementation guidance:** the type body is exhaustive — the name picks the primary deprecated attribute, the body lists every legacy field that was part of that deprecation round (dimension siblings, paired URL/target fields, etc.). ADR-0028 codifies the rule and the rare exception case.

- **Keep `handleMediaSelect`'s `mediaDetails: { width: media.width ?? 0, height: media.height ?? 0 }` shape; defer harmonization with `migrate()` to a separate ticket.**
  - **Root cause re-evaluation:** Eric flagged the asymmetry between `handleMediaSelect` (`?? 0`) and `migrate()` (`undefined`) as Minor, suggesting we match the latter. On second look, the asymmetry is intentional context-specific defense, not a contract violation. `handleMediaSelect` runs against a fresh WP `MediaUpload` payload — WP reliably returns `width`/`height` for image media, so the `?? 0` fallback is theoretical. `migrate()` runs against legacy saved content where dimensions may genuinely be missing, so `undefined` correctly signals "dimensions unknown, fall back to intrinsic" through `ResponsiveWrapper`. The architect doc's `mediaDetails: undefined` callout in § Backwards Compatibility is scoped to the migration path, not the onSelect handler.
  - **Alternatives considered:** (a) match `migrate()` here (`undefined` fallback) — rejected because `mega-menu-info-banner/edit.tsx:65-68` (the canonical reference) uses the same `?? 0` shape; changing this block alone diverges from the canonical sibling. (b) harmonize both `mega-menu-info-banner` and `mega-menu-featured-banner` `handleMediaSelect` handlers in this PR — rejected as out-of-scope drive-by refactor. (c) **defer to a separate ticket if codebase-wide harmonization is desired** — chosen.
  - **Chosen approach:** keep `?? 0` in `handleMediaSelect` to stay consistent with the canonical pattern. Document the deferral in `## Review Issues` with Eric's flag and the deferral rationale, and promote the asymmetry rationale to `backend-editor-blocks.md` § MediaUpload + ResponsiveWrapper Preview Pattern so the next reviewer doesn't have to re-litigate it.
  - **Implementation guidance:** no source code change. Architect-doc addition: a "Dimension fallback asymmetry — onSelect vs migrate" paragraph in `backend-editor-blocks.md` § MediaUpload + ResponsiveWrapper Preview Pattern, slotted between the "Build the full IImage object" sentence and the Canonical example callout. Names both shapes, gives the reason (fresh upload vs legacy data, ResponsiveWrapper fallback math), and tells future readers not to harmonize by reflex.

- **Root cause:** `<ResponsiveWrapper naturalWidth={854} naturalHeight={400}>` hardcodes a fixed aspect ratio (~2.135:1) and forces the child `<img>` to fill the box at `width: 100%; height: 100%`, stretching any image whose intrinsic aspect differs from 854:400.
- **Steps to Reproduce:**
  1. Open a page with a Mega Menu Featured Banner block (or insert one).
  2. Select the block to open its InspectorControls.
  3. Expand the "Background Image" panel.
  4. Upload or select a background image whose intrinsic aspect ratio is not 854:400 (e.g. 16:9, 4:3, 5:4, or square).
  5. Observe the preview in the sidebar.
- **Expected behavior:** preview displays the image without distortion regardless of source aspect ratio.
- **Actual behavior:** image is stretched/skewed to fit the hardcoded 854×400 box.
- **Recommended fix:** match the `feature` / `mega-menu-info-banner` pattern — widen `BlockAttributes` with `backgroundImageId?: number`, `backgroundImageWidth?: number`, `backgroundImageHeight?: number`; capture all four (id, url, width, height) in `handleImageChange`; pass `attributes.backgroundImageWidth` / `attributes.backgroundImageHeight` to `ResponsiveWrapper`; set `MediaUpload` `value={attributes.backgroundImageId ?? 0}`; clear all four on remove. See `## Implementation Tasks` and `## Decisions` for the full step list and rationale.
- **Suggested tests:** none — editor `edit.tsx` files aren't unit-tested in this codebase. Manual visual verification in wp-admin with images of varying aspect ratios.
- **Linear:** `synced`

## Review Issues

### Decorative `<Image>` now renders WP-supplied altText, conflicting with `role="presentation"`

- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx`
- **File:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx:62`
- **Problem:** The frontend `<Image>` keeps `role="presentation"` (decorative) but now reads `alt={image.altText ?? ""}`. When a dealer uploads through the WP media library, alt text is captured by default — so the rendered `<img>` ends up with `role="presentation"` AND a non-empty `alt`, which is an ARIA conflict. The image is decorative (the heading/description carry the meaning), so alt must remain empty per WCAG. Previous code hardcoded `alt=""` for this exact reason. The plan said to wire `image?.altText ?? ""`, but that wasn't load-bearing — preserving the decorative contract is.
- **Suggested fix:** revert to `alt=""` and drop the `image.altText ?? ""` read on line 62. Keep `role="presentation"`.

### handleMediaSelect mediaDetails fallback diverges from migrate()

- **Severity:** `minor`
- **Status:** `deferred`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-featured-banner/edit.tsx:85-88`
- **Reviewer:** Eric (PR #1962)
- **Problem:** `handleMediaSelect` writes `mediaDetails: { width: media.width ?? 0, height: media.height ?? 0 }`; `migrate()` in `index.tsx:110-117` uses `mediaDetails: undefined` when either dim is missing. Eric flagged the asymmetry could produce `NaN%` padding-bottom in `ResponsiveWrapper` if `0/0` ever fires.
- **Reason for deferral:** the asymmetry is intentional context-specific defense, not a contract violation. `handleMediaSelect` runs against a fresh WP `MediaUpload` payload — WP reliably returns `width`/`height` for image media, so the `?? 0` fallback is theoretical. `migrate()` runs against legacy saved content where dimensions may genuinely be missing, so `undefined` correctly signals "dimensions unknown, fall back to intrinsic" through `ResponsiveWrapper`. The canonical pattern (`mega-menu-info-banner/edit.tsx:65-68`) uses the same `?? 0` shape; changing this block alone would diverge from the canonical reference. The architect doc's "graceful intrinsic-size fallback" guarantee in § Backwards Compatibility is scoped to the migration path, not the onSelect handler.
- **Follow-up if pursued:** harmonize both `mega-menu-info-banner/edit.tsx` and `mega-menu-featured-banner/edit.tsx` together in a separate ticket; isolated fix here creates new divergence.

### IImage.id type drift between DEFAULT_IMAGE and migrate()

- **Severity:** `minor`
- **Status:** `deferred`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-featured-banner/index.tsx:106` vs `frontend/blocks/mega-menu-featured-banner/schema.ts:6`
- **Reviewer:** Eric (PR #1962)
- **Problem:** `DEFAULT_IMAGE.id` is `"0"` (string), while `migrate()` builds `id: backgroundImageId ?? 0` (number) and `handleMediaSelect` uses `id: media.id` (number from `WPMedia`). `IImage.id: string | number` allows both, but freshly-created and migrated blocks ship the same shape with different runtime types.
- **Reason for deferral:** the pattern is codebase-wide. `mega-menu-info-banner/schema.ts:5` uses `id: "020"` (string) for its `DEFAULT_IMAGE`, and its `handleMediaSelect` stores numbers from `WPMedia.id`. `IImage.id: string | number` is permissive precisely because the codebase doesn't enforce one shape — the renderer doesn't compare IDs. Fixing only this block would diverge from the canonical sibling without harmonizing the broader convention. Worth a separate ticket if the team wants codebase-wide normalization (likely landing as part of the encounter protocol's ongoing `IImage` migrations).

### Missing resolver test: legacy URL without dimensions

- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `frontend/blocks/mega-menu-featured-banner/__tests__/resolver.props.test.ts`
- **File:** `frontend/blocks/mega-menu-featured-banner/__tests__/resolver.props.test.ts`

## Acceptance Criteria

### Behavioral

- [ ] Given a Mega Menu Featured Banner block on a page, When I select the block and expand the Background Image panel in the right sidebar and choose a 16:9 image, Then the preview displays the image with its true 16:9 proportions and no distortion.

- [ ] Given an existing Featured Banner block migrated from legacy saved content, When I select the block in the editor and then click "Replace" to choose a new image, Then the preview displays the new image at its true proportions.
- [ ] Given a dealer site with a Mega Menu Featured Banner block published before this change (carrying `backgroundImage: string` in `wp_options`), When the page is rendered on the live frontend, Then the background image displays correctly with no missing-image regression — the resolver's legacy adapter normalizes the legacy fields into the canonical `image` shape on read.
