3. **Write `backend/seeds/README.md`.** Human-facing rulebook. Sections in order:
   - **Purpose** — one paragraph, "local development seed data, never staging/prod."
   - **The commit-it rule** — "If you seed something that doesn't already have a seeder here, write one and commit it. Future you, future your teammate, and future agents are all the audience."
   - **Naming convention** — `seed-<entity>.php`. Supports a positional `cleanup` arg.
   - **Idempotency convention** — every seeded post tagged with `_thrive_seeded = 1` meta. Cleanup mode wipes only those.
   - **How to run** — show the canonical `wp eval-file` command with `--skip-plugins=wpae-acf-add-on` workaround included in every example.
   - **Gotchas** — distilled from the THR-1418 events seeder thread:
     - Cleanup must query `$wpdb->postmeta` directly. `WP_Query` and `get_posts` get filtered by entity-specific ORM gates (TEC's date scoping is the canonical example) and may not return seeded rows even when they're in the DB.
     - Timezone: WP's `wp_timezone_string()` returns a numeric offset (`-05:00`) when no named timezone is set, and TEC silently fails to compute UTC fields from offset-format inputs. Default to `'UTC'` when `get_option('timezone_string')` is empty.
     - TEC `tribe_create_event` parses `EventStartDate` as `Y-m-d` only and pulls time from a separate `EventStartTime` field. Pass them split, not as `Y-m-d H:i:s`.
     - Plugin conflict: local has a `pmae_admin_notices` redeclaration between `wpae-acf-add-on` and `wp-all-export-csv-excel-xml-for-acf`. `--skip-plugins=wpae-acf-add-on` is the universal workaround for `wp eval-file`.
   - **Local only — never run on staging or prod.** Repeat at top and in every example command. Warnings get ignored when abstract; repetition is the cost of safety.
   - **Future runner** — note that a `seed-all.php` runner will be added when seeder #2 lands; for now, run individual seeders by name.

## Decisions

- **Folder lives at `backend/seeds/`, not top-level `seeds/` or `backend/scripts/seeds/`.**
  - **Root cause:** these scripts only run inside the WordPress admin container — top-level placement misfits the boundary, nested-under-scripts buries the seam. Flat under `backend/` is the most discoverable for a new contributor scanning the repo.
  - **Alternatives considered:** (a) top-level `seeds/` — wrong scope, suggests broader applicability than there is; (b) `backend/scripts/seeds/` — buries the convention under a less specific concern, weaker discoverability for new devs; (c) keep in `backend/scripts/` with a naming pattern — relies on convention without a physical home, which is stickier-broken than place-based conventions.
  - **Chosen approach:** `backend/seeds/` flat. Beats the alternatives because the place IS the convention; `ls backend/seeds/` answers "what can be seeded?" without anyone having to remember a naming pattern.
  - **Implementation guidance:** the architect doc routes on this exact path. Future seeders default here. If an agent writes a seeder in `backend/scripts/` by mistake, code review catches it (the architect doc is a written norm).

- **Establish the foundation now, even with one concrete seeder.**
  - **Root cause:** the original "wait for case #2" framing assumed no historical demand. In reality, manual test-data setup has been an unmet need across multiple tickets, and a new contributor with a bare environment is the immediate trigger that makes this load-bearing now.
  - **Alternatives considered:** (a) defer the folder until seeder #2 lands — leaves the lessons unwritten and the new dev rediscovering them; (b) put the seeder in `backend/scripts/` and rely on naming convention — weaker discoverability, doesn't communicate "this is the place to commit seeders."
  - **Chosen approach:** build the folder + README + architect doc in this PR. Beats deferral because the folder creates the place where the convention lives; conventions stick when they have physical homes.
  - **Implementation guidance:** keep the architect doc tight (~80 lines) so per-route load cost stays low until the folder earns more content. The README absorbs the longer narrative for humans.

- **Defer `seed-all.php` runner until seeder #2 lands.**
  - **Root cause:** with one seeder, a runner is unnecessary scaffolding — `wp eval-file` against the single file is the runner. Building a runner now front-loads complexity for hypothetical future seeders.
  - **Alternatives considered:** (a) build the runner now — over-fits the abstraction; (b) skip mention entirely — leaves the next person guessing.
  - **Chosen approach:** README notes the runner is planned for the second-seeder PR. Beats both alternatives because it preserves intent without spending complexity budget.
  - **Implementation guidance:** when seeder #2 lands, `seed-all.php` is one-day of work — discover all `seed-*.php` files in the directory, run each, aggregate output.

- **Architect-doc routing starts narrow at `backend/seeds/**`, not `backend/scripts/**`.**
  - **Root cause:** routing on `backend/scripts/**` would catch agents writing seeders in the wrong directory but would also load the doc for unrelated scripts work, paying load cost on every routed edit.
  - **Alternatives considered:** (a) wide route immediately — over-loads the doc; (b) no route at all — relies on agents discovering the README, which doesn't happen in routed edits.
  - **Chosen approach:** narrow route on `backend/seeds/**` and `backend/scripts/seed-*.php` (catches misplaced seeders specifically, not all scripts). Beats both alternatives because it's targeted to where the doc actually applies.
  - **Implementation guidance:** monitor for misplaced seeders during PR review for ~3-6 months. If it happens, widen the route. If it doesn't, the narrow form was right.
