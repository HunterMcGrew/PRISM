# Plan: THR-1221

## Ticket

THR-1221 ‚Äî Refactor CallToActions to be an RSC component (follow-up from THR-1238, THR-1455, THR-1457)

## Goal

Refactor `CallToActions` and its component tree to React Server Components, eliminating client JS for link-type CTAs and removing the client-side form fetch waterfall.

---

## Decisions

- `useCTAProps` hook replaced by pure function `resolveCTA()` ‚Äî no hooks, no side effects, runs on server. Returns a discriminated union by CTA kind (`link | form | call | print`).
- `CTAFactory` removed ‚Äî `CallToActions` calls `resolveCTA()` directly and renders the appropriate component per kind.
- `useGetGravityForm` removed from CTA path ‚Äî forms pre-fetched server-side via `fetchFormsForCTAs()` helper, passed as `formsById` prop.
- Link-type CTAs (email, link, phone, video, credit_application) render as RSC ‚Äî zero client JS. `CTA` component already has no `"use client"` directive.
- Interactive CTAs get thin dedicated client components: `FormCTAButton`, `CallCTAButton`, `PrintCTAButton`.
- Form and call modals delegate to the existing `SharedModals` via `AppContext.openModal()` ‚Äî no new modal components. `FormCTAButton` calls `populateDynamicFormFields` on the pre-fetched form then passes it via `openModal({ type: "form", form: populatedForm })`. `CallCTAButton` calls `openModal({ type: "call" })`. `SharedModals` already handles rendering, closing, and loading state.
- `populateDynamicFormFields` runs inside `FormCTAButton` before calling `openModal` ‚Äî the form arrives at `SharedModals` already populated with item-specific dynamic fields + UTM params.
- `ProductOverview` keeps `CallToActions` as a direct import (client path) for this ticket ‚Äî `buildSummary` depends on client-side `ProductTemplateContext` state that updates live as the user configures composite product options; it can't be hoisted to the RSC parent. Defer children-prop restructure to a composite product RSC ticket. `formsById` passed as a new prop from `Product.tsx`.
- Other 4 consumers (PromotionOverview, RentalCTAs, OfferOverview, EquipmentOverview) are already RSC ‚Äî just add `formsById` from resolver/page.
- `formsById` is a required prop (`Record<string, GfForm>`) ‚Äî consumers without form CTAs pass `{}`. Makes the data contract explicit and prevents silent failures.
- Interactive secondary CTAs (form, call, print) use `<button>` elements, not `<a onClick>` ‚Äî fixes existing keyboard accessibility issue where `<a>` without `href` is not focusable.

## Implementation Steps

### Step 1: Pure CTA resolution ‚Äî `resolve-cta.ts`

Extract the `useCTAProps` switch into a plain function. Returns a discriminated union:

```ts
type ResolvedCTA =
  | { kind: "link"; href: string; label: string; isPrimary: boolean; isProminent?: boolean; openInNewTab?: boolean; icon: IconType }
  | { kind: "form"; formId: string; label: string; isPrimary: boolean }
  | { kind: "call"; label: string; isPrimary: boolean }
  | { kind: "print"; label: string; isPrimary: boolean }
  | null;
```

Handles: `credit_application` ‚Üí link, `email` ‚Üí link, `link` ‚Üí link, `phone` ‚Üí link, `video` ‚Üí link, `form` ‚Üí form, `call` ‚Üí call, `print` ‚Üí print.

No hooks, no context, no side effects. Write tests ‚Äî this has the most branching logic.

### Step 2: Server form fetching ‚Äî `fetchFormsForCTAs()`

Add to `lib/forms/fetch-form.ts`:

```ts
export async function fetchFormsForCTAs(
  ctas: ICallToAction[],
  wpUrl: string
): Promise<Record<string, GfForm>>
```

- Extracts unique `formId`s from CTA arrays
- Calls `fetchForm()` for each
- Returns a `Record<formId, GfForm>`
- Called by resolvers/pages, result passed as `formsById` prop

### Step 3: Thin client components

**`FormCTAButton`** (`"use client"`):
- Props: `form: GfForm`, `label`, `isPrimary`, `formFieldsItem?`, `buildSummary?`
- Calls `populateDynamicFormFields(form, formFieldsItem, buildSummary, utmParams)` on click
- Then calls `openModal({ type: "form", form: populatedForm })` via `AppContext`
- Renders `Button` (primary) or `<button>` with icon (secondary)

**`CallCTAButton`** (`"use client"`):
- Props: `label`, `isPrimary`
- Uses `useContext(AppContext)` ‚Üí `openModal({ type: "call" })`
- Renders `Button` (primary) or `<button>` with icon (secondary)

**`PrintCTAButton`** (`"use client"`):
- Props: `label`, `isPrimary`
- `onClick: () => window.print()`
- Renders `Button` (primary) or `<button>` with icon (secondary)

All three use `<button>` (not `<a onClick>`) for secondary style ‚Äî keyboard accessible by default.

### Step 4: Refactor `CallToActions` to RSC

- Remove `"use client"`
- Add required `formsById: Record<string, GfForm>` prop
- Call `resolveCTA()` per CTA
- Render `kind: "link"` as RSC (`CTA` component ‚Äî strip `onClick` from its props, link-only now)
- Render `kind: "form"` as `<FormCTAButton form={formsById[formId]} formFieldsItem={...} buildSummary={...} />`
- Render `kind: "call"` as `<CallCTAButton />`
- Render `kind: "print"` as `<PrintCTAButton />`
- No modal, no provider wrapping ‚Äî `SharedModals` handles everything

### Step 5: Update consumers

**RSC consumers** (straightforward ‚Äî fetch forms, pass `formsById`):
- `RentalCTAs` ‚Äî already has `ServiceFactory` access, easiest first
- `PromotionOverview` ‚Äî fetch in parent resolver/page
- `OfferOverview` ‚Äî fetch in parent resolver/page
- `EquipmentOverview` ‚Äî fetch in parent resolver/page

**`ProductOverview`** (`"use client"` ‚Äî minimal change):
- Add `formsById: Record<string, GfForm>` prop
- `Product.tsx` (RSC parent) calls `fetchFormsForCTAs()` and passes result
- `CallToActions` stays imported inside `ProductOverview` ‚Äî runs as client component there
- Still gets the server-fetch win (no `useGetGravityForm` waterfall) and simplified internals
- Defer children-prop restructure to separate ticket

### Step 6: Cleanup

- Delete `CTAFactory.tsx`
- Delete `useCTAProps.ts`
- Remove `useGetGravityForm` from CTA import graph
- Remove `onClick` prop from `CTA` component (link-only now)
- Update `CallToActions.stories.tsx`

## What Stays Client

- `FormCTAButton` ‚Äî `populateDynamicFormFields` (needs `window.location.href`, `document.cookie`), `useUTMParams`, `openModal`
- `CallCTAButton` ‚Äî `openModal({ type: "call" })` via AppContext
- `PrintCTAButton` ‚Äî `window.print()`
- `SharedModals` (existing, unchanged) ‚Äî renders form/call modals

## What Moves to Server

- CTA type resolution / prop building
- Form data fetching (eliminates client waterfall via `/api/form`)
- Link CTA rendering (email, link, phone, video, credit_application) ‚Äî zero JS
- Grid layout rendering
- Icon rendering (`react-icons/fa` for link CTAs)

## What's Eliminated

- `CTAFormModal` / `CTAFormModalContext` ‚Äî not needed, `SharedModals` already handles this
- `useGetGravityForm` in CTA path ‚Äî replaced by server-side `fetchFormsForCTAs()`
- `useCTAProps` hook ‚Äî replaced by pure `resolveCTA()` function
- `CTAFactory` component ‚Äî no longer needed
- Local modal state in `CallToActions` (`currentForm`, `isFormOpen`) ‚Äî delegated to `SharedModals`

---

## Risks

- **Form pre-fetch adds server work** ‚Äî but cached via `ON_DEMAND_CACHE_TIME` and eliminates a client waterfall, so net positive.
- **Storybook** ‚Äî Not a concern. `experimentalRSC: true` is enabled and `storybook-addon-module-mock` is already used for server-side mocking (see PromotionsBlock, EmployeesBlock, NavigationBlock stories). Mock `fetchFormsForCTAs` the same way.
- **`react-icons/fa` in RSC** ‚Äî icon components render `<svg>`, should work as RSC. Verify with build that tree-shaking still works.
- **`populateDynamicFormFields` mutates the form object** ‚Äî `FormCTAButton` should spread/clone the form before mutating to avoid corrupting the prop.
