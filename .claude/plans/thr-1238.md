## Goal

Cut unnecessary client JS on every page load—**without** relying on `next/dynamic` around header search / shared modals (tried; **no meaningful bundle win**).

**Direction:**

1. **Server-only** GraphQL for Gravity Form **reads** where practical; **Server Action** for **`submitGfForm`** so form UI is not on the client Apollo path (THR-1454 scope).
2. Further steps: **RSC-first** form props, **`/api/form`** cleanup, and **non-form** client Apollo (product **`ServiceFactory`** paths)—**separate tickets** after THR-1454.

## Decisions

- **Reverted (2026-03-19):** `next/dynamic` for `AdvancedSearch`, `MobileSearch`, and `SharedModals` / `GravityForm`; `create-product-service.ts` + `ServiceFactory` delegation. Focus shifted to **RSC + Server Action** instead.
- **Submit server action:** `submitFormAction` in `lib/forms/submit-form-action.ts` uses **`WORDPRESS_URL`** only (no client-supplied base URL). Client sends **`FormData`** with `formId`, JSON `fieldValues`, and `file_{fieldId}` entries for uploads; server merges files and runs **`getClient(..., "upload").mutate`** (Apollo stays server-only for this path).
- **Client helper:** `lib/forms/serialize-form.ts` (`serializeForm`) strips `File` from field values for JSON and lists files for `FormData` (client-safe; not `"use server"`). Tests: `lib/forms/__tests__/serialize-form.test.ts`.
- **`lib/forms` consolidation:** Server-only **`fetch-form.ts`** (`fetchForm`, `fetchAllForms`), **`populate-form-fields.ts`**, **`submit-form-action.ts`**, **`serialize-form.ts`**; tests + **`__fixtures__/equipment.fixture.ts`**, **`index-record.fixture.ts`**. Gravity fetch lives in `lib/forms`, not mixed with client-only helpers.

### RSC-first retrieval (future — not THR-1454)

Definitions are resolved on the **server**; client components receive **`GfForm`** (or a map) via **props / context** populated from RSC or block resolvers.

| Area | Direction |
|------|-----------|
| **Gravity Form block** | Already resolver + `fetchForm` → `GravityFormBlock`. |
| **Career / Location** | Already server `fetchForm`. |
| **Kronos `Results` / default search page** | RSC or block props: `fetchForm(wpUrl, getQuoteFormId)` → pass **`quoteForm`** into client block (`Hits` already accepts it). |
| **`EquipmentResults`** | Resolver or RSC parent: `fetchForm` for **`ctaFormId`** → pass **`ctaForm`**; drop **`useGetGravityForm`**. |
| **`CallToActions` / `useCTAProps`** | RSC parents collect form IDs from CTAs, **`fetchForm`** server-side, pass **`formsById`** (or per-CTA forms); hook does not fetch. |
| **`SharedModals`** | **`mainContactForm`** from layout/context (server-fetched from **`mainContactFormId`**). For **`openModal({ type: "form", formId })`** without `form`, prefer resolver-supplied **`form`** on blocks; narrow fallback only if needed. |
| **Cleanup** | Remove or isolate **`useGetGravityForm`**; remove **`/api/form`** when unused, or document preview-only. |

InstantSearch blocks stay **client** for Algolia/Typesense; **form payloads** are still loaded **outside** that boundary and passed down.

- 2026-03-21: Added `fieldErrors` prop to `HtmlField` and `SectionField` for `dynamic()` type compatibility.
- 2026-03-21: Removed `searchClient` from `AppContext` and `AppContextFields` entirely. Moved `createSearchClient` into each consumer (`KronosSearchBlock`, `EquipmentListBlock`, `AdvancedSearch`) via local `useRef` init.
- 2026-03-21: **Finding — move to consumers had no bundle impact.** Webpack was already splitting typesense into its own chunk (chunk `133`, 485KB) before our changes. Moving the import from AppContext to consumers just changed which module owns the reference — the chunk and its page-manifest inclusion were unchanged. Root cause: the block-registry is statically imported by the layout, and webpack statically analyzes every `() => import(...)` in it, so ALL block chunks (including typesense via AdvancedSearch → SiteHeaderBlock) appear in the RSC client manifest and load as `<script async>` on every page.
- 2026-03-21: **Finding — `useEffect` + `import()` in AdvancedSearch also has no effect.** Webpack statically analyzes ALL `import()` calls regardless of where they appear in code (top-level, inside useEffect, inside callbacks). Chunk 133 remains in `index.html` as a `<script async>`. The only viable path to removing heavy block-level chunks from the page manifest is option 2: restructure the block-registry so block modules are loaded by runtime string lookup rather than static `() => import(...)` arrow functions — which are fully visible to webpack's static analysis.

### Layout bundle — eager dependency removal (next PR)

Bundle analysis (2026-03-21) found the root layout ships ~2.3MB raw / ~600KB gzipped to **every page**. Remaining eagerly-loaded dependency trees:

#### ~~1. Lazy-load `createSearchClient()` in AppContext (~60KB gzip)~~ — DONE

**Chain:** `layout.tsx` → `AppContextProvider` → `AppContext.tsx` ("use client") → `createSearchClient()` → `TypesenseInstantSearchAdapter` → typesense + algoliasearch-helper
**Fix:** Replaced static `import createSearchClient` with a dynamic `import()` inside a `useEffect` in `AppProvider`. `searchClient` state starts `null` (already handled by all consumers) and is set after the async import resolves. Changed `import { SearchClient }` to `import type` in both `AppContext.tsx` and `app-context.ts`.

#### 2. ~~Dynamic import `MultiSelectField`~~ (~40KB gzip) — DONE in THR-1456

All form field components dynamically imported in `GravityFormsField.tsx`. Layout dropped ~97KB raw.

#### 3. Lazy-load `AdvancedSearch` in header (~27KB gzip)

**Effort:** Low
**Chain:** `layout.tsx` → `BlocksRenderer` → `SiteHeaderBlock` → `DesktopHeader` → `AdvancedSearch.tsx` ("use client") → react-instantsearch
**Problem:** Search UI imports react-instantsearch on every page even though it's only visible when user opens search.
**Fix:** Wrap in `next/dynamic` with `ssr: false` in the header component.
