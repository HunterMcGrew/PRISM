# Plan: THR-1591

## Ticket

THR-1591 — Mega menu migration: service pane links-with-icons stretching and icon fallback

## Goal

Fix two defects in migrated service-pane links-with-icons blocks: (1) icon tiles stretch to ~50% of the pane, (2) links with no legacy icon render a phone icon instead of no icon.

## Design

**Status:** Ready for implementation
**Author:** Pixel
**Date:** 2026-04-16

### Summary
Bug 1 (layout stretching) is a single Tailwind class fix. The V2 `MegaMenuLinksWithIcon` container applies `lg:grid lg:grid-cols-2` with no max-width. When the migration emits a 1-column Gutenberg wrapper (no cards), the component fills the full 1280px pane and each tile stretches to ~640px. With cards present, the parent column constrains it to 640px and tiles are ~320px — the correct size.

### Decisions worth knowing
- Add `max-w-screen-sm` (640px) to `MegaMenuLinksWithIcon.tsx:26` — matches the width the component gets when cards are present (50% of 1280px container-xl). Tiles render at the same ~320px width regardless of whether cards exist next to them.
- Left-aligned, not centered. F-pattern scanning, consistency with all other mega menu panes, and stable spatial anchor (link positions don't shift when content count changes).
- Fixed pixel cap (`max-w`), not a percentage. `max-w` is an upper bound — when a parent column constrains the component to less than 640px (e.g. cards present), it's a no-op. It only kicks in when the component has more space than it needs. Can never cause overflow or compression.

## Decisions

- Both bugs originate in `MegaMenuMigrationService.php` (THR-1450, PR #1733) — same migration PR as THR-1589 / THR-1590.
- Bug 1 fix: `max-w-screen-sm` (640px) on the `MegaMenuLinksWithIcon` container — matches the 50% column width the component gets when cards are present. Left-aligned, not centered. Pixel confirmed.
- Bug 2 fix: root-cause across all three layers (migration, parent resolver, child resolver). `null` means "no icon" for migrated data. The `|| "call"` fallback in the resolvers is replaced with explicit null handling so empty/null icons pass through as `null` instead of coercing to `"call"`.
- Schema default and component default parameter stay as `"call"` — new blocks inserted in the editor should start with a phone icon. The fix only affects how null/missing icons from migration are handled.
- Migration omits the `icon` key from block attrs when the legacy value is null/non-string — WordPress block rendering omits attrs matching defaults, so this is the idiomatic pattern.
- `MEGA_MENU_LINK_ICONS` must live in `frontend/lib/constants/icons.ts`, not in the `"use client"` component — exporting it from a client module causes server-side resolvers to receive a client reference proxy instead of the real object, breaking `in` checks. This is the root cause of ALL valid icons becoming "call".

## Debugged Issues

### Service-pane links-with-icons tiles stretch to full pane width
- **Status:** `fixed`
- **Fixed in:** `MegaMenuLinksWithIcon.tsx` — added `max-w-screen-sm`
- **Severity:** High
- **Environment:** Frontend — migrated mega menu service panes where a content row contains links-with-icons but no cards.
- **File:** `frontend/components/mega-menu/MegaMenuLinksWithIcon.tsx:26` (rendering) + `backend/plugins/gravity-platform-core/src/services/MegaMenuMigrationService.php:319-332` (migration output)
- **Root cause:** Two-part interaction. `MegaMenuLinksWithIcon` applies `lg:grid lg:grid-cols-2` with no max-width — each column becomes `1fr` of whatever container it lands in. The component was designed assuming it sits inside a half-pane column (legacy layout). The migration service emits a 1-column content row when links exist but cards don't, so links-with-icons sits in a full-width column and the 2-col grid stretches each tile to ~50% of the pane.
- **Steps to Reproduce:**
  1. Open a migrated mega menu service pane that has links-with-icons but no cards.
  2. View at lg breakpoint or wider.
  3. Observe icon tiles stretched to ~50% of the pane width.
- **Expected behavior:** Icon tiles render at a constrained width matching legacy (half-pane column sizing).
- **Actual behavior:** Tiles stretch to fill the full-width content row.
- **Recommended fix:** Pixel to choose between (a) constraining the component with a `max-w-*` utility (recommended — preserves legacy visual and works in any container), or (b) forcing the migration to always emit a 2-column content row (riskier — affects all migrated panes).
- **Suggested tests:** Storybook story for `MegaMenuLinksWithIcon` in a full-width container; visual regression check on a migrated pane.
- **Linear:** `N/A`

### Valid icon keys becoming "call" — RSC client boundary breaks `in` check
- **Status:** `fixed`
- **Fixed in:** Moved `MEGA_MENU_LINK_ICONS` from `MegaMenuLinkWithIcon.tsx` (`"use client"`) to `frontend/lib/constants/icons.ts`. Updated imports in component, resolver, and stories.
- **Severity:** High
- **Environment:** Frontend — any Next.js render path (preview and published). Icons display correctly in the Gutenberg editor (client-side, no RSC boundary) but become "call" when rendered through BlocksRenderer (server-side RSC pipeline).
- **File:** `frontend/components/mega-menu-link-with-icon/MegaMenuLinkWithIcon.tsx:18-21` (defines `MEGA_MENU_LINK_ICONS` in a `"use client"` file) + `frontend/blocks/mega-menu-link-with-icon/resolver.props.ts:18-21` (imports it server-side and uses `in` operator)
- **Root cause:** `MEGA_MENU_LINK_ICONS` is exported from `MegaMenuLinkWithIcon.tsx`, a `"use client"` module. When the server-side resolver imports it during RSC rendering, Next.js replaces the export with a **client module reference proxy** — the actual object keys (`cart`, `link`, `plus`, etc.) don't exist on the proxy. The `in` operator returns `false` for every key, so the fallback `"call"` fires for all icons.
- **5 Whys:** All icons → "call" ← `in` check always false ← `MEGA_MENU_LINK_ICONS` is a client reference proxy on server ← imported from `"use client"` file ← constant was co-located with the component for convenience ← no rule prevented exporting server-consumed constants from client modules.
- **Evidence:** Runtime instrumentation confirmed: `rawIcon: "cart"`, `typeof: "string"`, `inCheck: false`. The icon value arrives correctly — the validation object is the problem.
- **Steps to Reproduce:**
  1. Set a link-with-icon to any valid icon (e.g. cart, link, plus) in the mega menu editor
  2. Open the mega menu editor preview (or view the published frontend)
  3. Observe all icons render as phone icons despite correct values in the editor
- **Expected behavior:** Icons render matching the editor selection
- **Actual behavior:** All icons render as phone (call) — the `in` check against the client proxy always returns false
- **Recommended fix:** Move `MEGA_MENU_LINK_ICONS` to `frontend/lib/constants/icons.ts` (where `CTA_ICONS` and `SOCIAL_ICONS` already live — not a `"use client"` file). Update imports in the component and resolver.
- **Suggested tests:** Resolver unit test: given `attrs.icon = "cart"`, assert resolved icon is `"cart"` (not `"call"`). This would have caught the proxy issue because Jest doesn't enforce RSC boundaries — the test would pass even with the bug, so also add a Storybook story or integration test that exercises the full RSC render path.
- **Linear:** `N/A`

### All migrated links with null legacy icons render a phone icon
- **Status:** `fixed`
- **Fixed in:** Migration emits null (omits icon key) for non-string icons. Parent resolver uses `??` instead of `||`. Child resolver returns `null` for invalid/missing icons via `resolveIcon()` helper.
- **Severity:** High
- **Environment:** Frontend — migrated mega menu link-with-icon blocks where the legacy `icon` field was null/missing.
- **File:**
  - `backend/plugins/gravity-platform-core/src/services/MegaMenuMigrationService.php:587` (migration emits empty string)
  - `frontend/components/mega-menu/link-with-icon/resolver.ts:52` (parent resolver falls back to `"call"`)
  - `frontend/components/mega-menu/link-with-icon/resolver.props.ts:18-21` (child resolver falls back to `"call"`)
- **Root cause:** Three-layer fallback collapses `null` / `""` / invalid to `"call"`.
  1. Migration: `$icon = is_string($link['icon'] ?? null) ? $link['icon'] : '';` — null legacy icon becomes `''`.
  2. Parent resolver: `icon: raw.icon || LINK_WITH_ICON_DEFAULTS.icon` — empty string is falsy → `"call"`.
  3. Child resolver: validates against `MEGA_MENU_LINK_ICONS`, falls back to `"call"` for anything invalid.
  Legacy `DynamicIcon` returns `null` for null/invalid icons, so legacy rendered an empty tile. V2 has no way to distinguish "no icon" from "call" because every non-match collapses to `"call"`.
- **Steps to Reproduce:**
  1. Migrate a legacy mega menu service pane whose link-with-icon entries have `icon: null`.
  2. View the migrated pane on the frontend.
  3. Observe every such link renders a phone icon.
- **Expected behavior:** Links with no legacy icon render no icon (matching legacy).
- **Actual behavior:** Links render a phone (`"call"`) icon.
- **Recommended fix:** Root-cause fix — remove the `|| "call"` fallback in both resolvers and return `null` for empty/invalid. Update the migration to emit `null` (or omit the `icon` key) when the legacy value isn't a valid icon string. Component prop type and `DynamicIcon` already support `null`.
- **Suggested tests:** Jest unit tests for both resolvers covering `null`, `""`, invalid string, and valid key cases. Migration unit test covering null legacy icon → null output.
- **Linear:** `N/A`
