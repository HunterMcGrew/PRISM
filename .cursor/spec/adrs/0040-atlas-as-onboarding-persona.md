# ADR-0040: Atlas as a dedicated onboarding persona

Date: 2026-05-22

## Context

PRISM's existing personas are ticket-flow personas — invoked in response to a specific ticket, PR, or in-flight piece of work. Onboarding is a different shape: it runs once per team install (or on a stack change), not per ticket. The work it does — detecting the stack, generating per-team rules, writing `.ai-skills/config.json`, populating stub anchors — is a long-running, resumable, interactive flow that produces durable artifacts but isn't tied to any one ticket.

Two integration options surface naturally: (1) absorb onboarding into Winston as a sub-mode, or (2) ship onboarding as a standalone script. Neither fits the shape.

Winston is reactive — invoked when an engineer needs an evaluation or a plan. Onboarding is proactive — the engineer launches it and walks through phases. The cadence is different, the invocation surface is different, and the state is different (Winston works against a ticket plan; onboarding works against the entire repo).

A standalone script gives up the conversational surface. Onboarding asks the team for the project name, the ticket prefix, the GitHub repo, the Linear workspace, the product domain, and existing engineering standards. Some of those need clarifying back-and-forth, especially on re-onboarding when the prior config exists. A script can't ask "I see you previously set the ticket prefix to `ACME`, but `pyproject.toml` was just added — does the Python stack change anything about how your team tracks tickets?" A conversational skill can.

## Decision

Atlas is its own skill — a dedicated onboarding persona invoked via "Atlas onboard", "onboard this repo", "set up PRISM here", "configure PRISM for my team", or any first-install / re-onboarding phrase. Atlas lives at `.ai-skills/skills/prism-atlas/` and runs once per team install or on stack change.

Atlas is **cadence-driven**, like Zoe — proactive, not ticket-scoped. The two cadence-driven personas are distinct from each other in cadence (Atlas runs once per install; Zoe runs weekly) and surface (Atlas writes durable config; Zoe writes audit reports), but share the proactive shape that ADR-0037 codified.

Atlas writes:
- `.ai-skills/config.json` — schema-validated team config that the substitution layer (ADR-0030) reads at every build.
- `.ai-skills/registry/onboarding-state.json` — step-completion tracker so an interrupted session can resume from the last completed step.
- Per-team rules generated from the detected stack (Phase 2.3) — including a stack-appropriate `security.md` per ADR-0036.
- Stub-anchor populations in canonical persona sources (Phase 2.4) — filling the `<!-- atlas:* -->` markers that ADR-0032 reserves.

## Consequences

- Atlas joins Zoe as the second cadence-driven persona. ADR-0037's framing (cadence axis distinct from ticket-flow axis) governs both.
- The bifurcated install layout (ADR-0031) means Atlas writes only to canonical (`.ai-skills/` and `.prism/`) — platform mirrors regenerate from the build.
- Resumability is structural: the state file lets a partial run continue across sessions without losing progress. Each step writes its own completion marker.
- Atlas's interactive flow is the conversational surface for what would otherwise be a one-shot script. Re-onboarding (stack change, new branch from a different team config) re-enters the flow with the prior config loaded as defaults.
- Future cadence-driven personas — anything that runs on a non-ticket trigger — should compare their shape against Atlas and Zoe before adopting a different model.
