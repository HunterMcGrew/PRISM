## Decisions

- **#285 closed won't-fix (owner call, 2026-07-06) — no change to `epic-prism-conductor.md`.**
  - **Root cause:** PR #265 (L2) edited one phrase (line 45) in the closed conductor plan while correctly updating all live surfaces, leaving the frozen record internally inconsistent with its line-79 tiering table.
  - **Alternatives considered:** revert line 45 to `Sonnet fleet` (Winston's recommendation — restores frozen-at-close truth); update the line-79 table to Eric→Opus (rejected — rewrites more frozen history).
  - **Chosen approach:** leave as-is. The inconsistency is cosmetic: the plan is closed, the real tiering lives on every live surface, and the fix wasn't worth carrying. Owner made the call directly.
  - **Implementation guidance:** do not "helpfully" fix this file in passing — the won't-fix is intentional. This entry is the record a future reader needs when they spot the mismatch.
- **Category-1 fix shape: de-link to plain skill-ID text, not repoint.**
  - **Root cause:** `../skills/<id>/SKILL.md` from rules/references resolves in only two of four consumer surfaces (`.claude/`, `.cursor/` — where skills are sibling dirs) and dangles in `.prism/` and `.codex/` (Codex skills live at `.agents/skills/`). No single relative path is portable, and skills are platform outputs, not canonical content, so the repo-root-absolute cross-reference convention (install-layout.md § Cross-reference convention) can't cover them either.
  - **Alternatives considered:** moving all 19 to the permanent `INSTALL_RELATIVE_LINK_ALLOWLIST` as "correct by design in the installed tree" — rejected: they are only correct in half the installed surfaces, and the epic's triage already classified them as genuinely dangling.
  - **Chosen approach:** plain-text persona + backticked skill ID. Same information, no false path claim anywhere.
  - **Implementation guidance:** pattern in task 3; skill-routing already refers to skills by ID in prose, so this matches house style.
- **Edits land in canonical `.prism/` files by default; curated seed copies are the exception.** `writeSeedMirror()` auto-mirrors non-curated files at build time — hand-editing their seed copies is drift. The seven curated files in scope (`seed-curation.json` § curated) are hand-maintained in the seed and edited there directly.
- **Corrected links that resolve in the consumer tree but not the raw seed tree (due to `.tmpl` renames of AGENTS.md/SPEC.md) graduate to `INSTALL_RELATIVE_LINK_ALLOWLIST`** — that allowlist exists precisely for correct-by-design-in-installed-tree entries, and each entry carries a comment.
- **`octocat` chosen as the neutral handle for #286** — GitHub's canonical placeholder identity, matching the issue's suggestion.
- **AC recorded in plan only** — tracker is GitHub issues; the three issue bodies already carry per-issue done conditions, and cross-issue AC pasted into each would drift. No tracker AC sync for this plan.
- **No entries graduated to `INSTALL_RELATIVE_LINK_ALLOWLIST`.**
  - **Root cause:** the plan anticipated that corrected links to `AGENTS.md`/`SPEC.md` (files that ship as `.tmpl` in the raw seed tree) might dangle raw despite resolving in the installed consumer tree, since the `.tmpl` extension is stripped only at install time.
  - **Alternatives considered:** pre-emptively allowlisting the `AGENTS.md`/`SPEC.md`-adjacent corrected links without checking the gate first — rejected per the plan's explicit instruction to confirm at the gate rather than guess.
  - **Chosen approach:** ran the gate after emptying `INSTALL_RELATIVE_LINK_TRACKED_VIOLATIONS`; `refResolves()` (`scripts/ai-skills/crossref-lint.ts`) already checks for a `.tmpl` twin of a resolved `.md` path (and vice versa), so all corrected links resolved clean with zero graduations needed.
  - **Implementation guidance:** `INSTALL_RELATIVE_LINK_ALLOWLIST` stays empty; if a future `.tmpl`-adjacent link genuinely dangles, verify against `refResolves`'s twin-check logic before assuming it needs the permanent allowlist.
- **Two additional `../skills/prism-design/SKILL.md#states` links de-linked in `implementation-task-detail.md`, beyond the plan's original 4-link count for that file.**
  - **Root cause:** the pre-fix tracked-violations key for this file (`...::../skills/prism-design/SKILL.md`, no anchor) allowlisted both the bare link (`## Who runs this rule`) and two `#states`-anchored citations elsewhere in the file, because `isInstallRelativeLinkAllowlisted` strips the anchor fragment before the lookup. Emptying the tracked set exposed the two anchored links as newly-visible dangling references.
  - **Alternatives considered:** none — same root cause and same de-link fix as Category 1, discovered via the gate rather than the original file audit.
  - **Chosen approach:** applied the identical de-link convention (drop the link, keep `` `prism-design` § States `` as plain text) to both citations, in canonical and the curated seed copy.
  - **Implementation guidance:** anchor-stripping in the allowlist lookup means one tracked/allowlisted bare-path entry silently covers every anchored variant of that path in the same file — worth checking for when auditing a file for a single reported link.
- **`crossref-lint.test.ts`'s two `isInstallRelativeLinkAllowlisted` tests reseed a synthetic permanent-allowlist entry instead of using `TRACKED_VIOLATIONS[0]`.**
  - **Root cause:** both tests destructured `[firstEntry] = INSTALL_RELATIVE_LINK_TRACKED_VIOLATIONS` to get a real fixture; emptying the set broke that fixture source.
  - **Alternatives considered:** the `allowlistOverrideFn` seam suggested by the plan — inspected but doesn't apply, since it's a parameter of `scanFileForRelativeLinks`, not of `isInstallRelativeLinkAllowlisted` (which reads the module-level Sets directly with no override hook).
  - **Chosen approach:** import `INSTALL_RELATIVE_LINK_ALLOWLIST`, cast it to mutable `Set<string>`, seed one synthetic `(relPath, rawRef)` pair in a `try` block, assert, then delete it in `finally`. Exercises the real lookup and anchor-stripping mechanism without depending on production data.
  - **Implementation guidance:** if `INSTALL_RELATIVE_LINK_ALLOWLIST` ever gains a real entry, these two tests can optionally switch back to it, but the synthetic-seed approach remains valid indefinitely.
