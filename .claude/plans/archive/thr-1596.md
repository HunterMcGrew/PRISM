## Design

**Status:** Ready for implementation
**Author:** Pixel
**Date:** 2026-04-16

### Summary
Responsive padding reduction on mega menu nav items. No text shrinking, no word wrapping, no `flex-wrap`. The padding is where all the excess space lives.

### Approach
Change `px-8` to `px-3 xl:px-8` on `NAV_ITEM_BASE_CLASSES` and `LINK_CLASSNAMES` in MegaMenuNavigationItem.tsx. This gives 12px/side at `lg` (1024px+) and 32px/side at `xl` (1280px+).

### Decisions worth knowing
- **No text shrinking** — dropping to `text-sm` at `lg` buys ~80px but costs legibility at arm's-length screen distances (laptops, propped tablets). The real room is in the padding, not the font.
- **No word wrap** — wrapping breaks the horizontal scan line (Gestalt continuity) and makes items different heights, reading as inconsistency rather than responsiveness. Keep `whitespace-nowrap`.
- **No `flex-wrap`** — a wrapping nav bar is worse than a tight one. If items ever wrap, the design has failed and needs an overflow-to-hamburger mechanism, not CSS papering over it.
- **`px-3` over `px-4`** — 4px/side difference is visually negligible at intermediate widths, but 48px total headroom across 6 items is significant for dealers with longer labels ("Parts & Service", "Heavy Equipment").
- **Touch targets remain adequate** — `py-3` + text height gives ~44px effective height. Horizontal click targets are label-driven, not padding-driven.

### Limitations
If a dealer has 8+ items with long labels, even `px-3` at 1024px could get tight. The real answer for that scenario is an overflow detection mechanism like `useIsMenuOverflowing` — but that's a feature, not this bug fix.

## Decisions

- Header background color issue is a separate ticket under THR-1550 — not in scope for this ticket.
- Fix is flat `px-3` padding (no responsive step-up) — the `xl:px-8` jump caused a visible "padding pop" at 1280px and reintroduced overflow risk on sites with many items.
- MegaMenuEditor outer div needs `overflow-x-clip` (not `overflow-hidden`) — without it, `whitespace-nowrap` nav items that exceed viewport width push the document body wider, causing the header to stop at viewport width while the mega menu extends past it. `overflow-hidden` clips the focus ring's box-shadow vertically; `overflow-x-clip` only clips horizontally and doesn't create a scroll container, so vertical ring rendering is unaffected.
- Popover wrapper needs `focus-within:z-[1]` — adjacent nav items each have an opaque background (`bg-[--mega-menu-background-color]`), which paints over the previous item's focus ring box-shadow on its right side. Raising the focused Popover one z-level above siblings ensures the ring is fully visible on all sides during keyboard navigation.

## Debugged Issues

### Mega menu parent items overflow viewport at intermediate desktop widths
- **Status:** `fixed`
- **Fixed in:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx` — changed `px-8` to flat `px-3` on `NAV_ITEM_BASE_CLASSES`
- **Severity:** Medium
- **Environment:** Frontend, all dealer sites using Thrive Mega Menu, viewport widths ~1024-1200px
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:26` and `frontend/blocks/mega-menu-navigation-item/schema.ts:3-4`
- **Root cause:** The `px-8` padding (64px per item) on `NAV_ITEM_BASE_CLASSES` combined with `whitespace-nowrap` and no `flex-wrap` on the nav container (`NAVIGATION_MENU_CLASSES`) causes items to overflow the viewport at intermediate desktop widths. The legacy `HeaderMenu` has responsive text sizing and `useIsMenuOverflowing` detection — the Thrive Mega Menu has neither.
- **Steps to Reproduce:**
  1. Open a dealer site using the Thrive Mega Menu (`menuType === "mega-menu"`)
  2. Resize the browser to ~1024-1200px wide (just above the `lg` breakpoint)
  3. Observe that right-most parent nav items are clipped by the viewport edge
- **Expected behavior:** All parent mega menu items are fully visible at any viewport width above the `lg` breakpoint, with graceful scaling or overflow handling
- **Actual behavior:** Items overflow the viewport and are clipped at intermediate desktop widths
- **Recommended fix:** Change `px-8` to `px-3 xl:px-8` on `NAV_ITEM_BASE_CLASSES` in MegaMenuNavigationItem.tsx. Apply the same responsive padding to `LINK_CLASSNAMES` (which inherits from `NAV_ITEM_BASE_CLASSES`). No text shrinking, no word wrapping, no `flex-wrap`.
- **Suggested tests:** Storybook stories at multiple viewport widths. Manual check with varying numbers of nav items.
- **Linear:** `not synced`

### Responsive padding jump causes visible "padding pop" at 1280px
- **Status:** `fixed`
- **Fixed in:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx` — removed `xl:px-8`, keeping flat `px-3`
- **Severity:** Medium
- **Environment:** Frontend, all dealer sites using Thrive Mega Menu, viewport crossing 1280px (xl breakpoint)
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:26`
- **Root cause:** `xl:px-8` in `NAV_ITEM_BASE_CLASSES` causes item padding to jump from 12px to 32px per side at 1280px. This creates a visible layout shift and reintroduces overflow risk for sites with many or long nav labels.
- **Steps to Reproduce:**
  1. Open a dealer site with the Thrive Mega Menu containing 6+ items
  2. Slowly widen the browser past 1280px
  3. Observe the nav items visibly jump wider as padding changes from `px-3` to `px-8`
- **Expected behavior:** Nav item padding is consistent across all desktop widths above the `lg` breakpoint
- **Actual behavior:** Padding jumps abruptly at 1280px, shifting the layout and potentially re-causing overflow
- **Recommended fix:** Remove `xl:px-8` from `NAV_ITEM_BASE_CLASSES`, keeping just `px-3`
- **Suggested tests:** Storybook stories comparing behavior at 1279px and 1281px
- **Linear:** `not synced`

### Mega menu container missing overflow clipping pushes page wider than viewport
- **Status:** `fixed`
- **Fixed in:** `frontend/components/mega-menu-editor/MegaMenuEditor.tsx` — added `overflow-x-clip` to the outer div (not `overflow-hidden`, which clipped focus ring box-shadow vertically). Also added `focus-within:z-[1]` to Popover in `MegaMenuNavigationItem.tsx` so the focused item's ring isn't covered by the next sibling's opaque background.
- **Severity:** High
- **Environment:** Frontend, all dealer sites using Thrive Mega Menu, viewport widths where nav items exceed available width
- **File:** `frontend/components/mega-menu-editor/MegaMenuEditor.tsx:22` and `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:63`
- **Root cause:** The mega menu's outer div (`mega-menu relative w-full`) had no overflow clipping. Nav items use `whitespace-nowrap`, so when combined item widths exceed the viewport, they push the document body wider. The header — sized to `w-full` (100% of the viewport, not the document) — doesn't expand with it, creating a visible gap where the header stops short of the page edge. Secondary: adjacent nav items' opaque backgrounds cover each other's focus ring box-shadow because all Popovers sit at the same z-level.
- **Steps to Reproduce:**
  1. Open a dealer site with 8+ mega menu items (or items with long labels)
  2. Resize to a width where combined item widths exceed the viewport (~1024-1200px)
  3. Observe horizontal scrollbar appears and the header background doesn't extend to the right edge
  4. Tab through nav items — focus ring is missing on right side of non-last items
- **Expected behavior:** The mega menu clips overflowing items horizontally, the header spans the full visible width, and keyboard focus rings are fully visible on all sides
- **Actual behavior:** The mega menu pushes the page wider, creating a gap in the header. Focus rings are partially hidden by adjacent items.
- **Recommended fix:** Add `overflow-x-clip` to the outer div (clips horizontal overflow without affecting vertical box-shadow). Add `focus-within:z-[1]` to each Popover wrapper so the focused item stacks above its siblings.
- **Suggested tests:** Visual check with many items at intermediate widths; verify dropdown panels still render correctly (they portal to body); verify focus ring is fully visible on keyboard tab through all items
- **Linear:** `not synced`
