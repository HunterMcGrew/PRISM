# Plan: THR-1570

## Ticket

https://linear.app/tractru/issue/THR-1570/devcontainer-add-github-cli-and-secure-auth-setup-for-claude-code-auto

## Goal

Secure, practical authentication setup for Claude Code and GitHub CLI in the devcontainer.

---

## Decisions

- **No filesystem-persisted tokens** — named Docker volumes rejected. Any token stored on the container filesystem (including volumes mounted into the container) is readable by all VS Code extensions running in the remote. Security over convenience.
- **SSH agent forwarding is automatic — no devcontainer.json config needed.** VS Code's Dev Containers extension forwards the host SSH agent automatically. `forwardAgent` is not a valid devcontainer.json property (never was — checked the spec, schema, and feature request history). The `SSH_AUTH_SOCK` remoteEnv override and `~/.ssh` bind mount were removed because: (1) the override passes a host-only path that can overwrite the auto-forwarded socket, (2) the bind mount exposes private keys to all extensions and agents in the container, contradicting the no-filesystem-tokens security decision. Prerequisite: host must have an SSH agent running with keys loaded (`ssh-add -l`).
- **Git credentials forwarded by VS Code** — built-in credential forwarding works automatically for git operations. No config needed.
- **gh CLI and Claude Code re-auth on rebuild** — accepted tradeoff. Both require `gh auth login` and Claude Code re-auth after a container rebuild. Rebuilds are infrequent enough that this is tolerable.
- **devuser as remoteUser** — required for Claude Code auto mode. Non-root user with workspace ownership via `postCreateCommand`.
- **Corepack prepare must run as devuser with login shell** — `runuser -l devuser -c 'corepack prepare ...'` so HOME is set to `/home/devuser`. Without `-l`, runuser inherits root's HOME and caches pnpm to `/root/.cache/` where devuser can't find it.
- **devuser has zero sudo access** — no NOPASSWD, no password-based sudo, no sudo group membership. No runtime operation requires root. A compromised extension running as devuser can read/write source code but cannot escalate, install packages, or modify system config.
- **All privileged operations live in setup.sh** — the entrypoint runs as root, does everything that needs root (user creation, apt packages, corepack cache), then drops to www-data for Apache. New privileged needs go in setup.sh, not into sudo grants.
- **Ad-hoc root access via host-side docker exec** — developers who need root for debugging run `docker exec -it -u root thrive-admin-1 bash` from their host terminal. Intentionally outside the devcontainer context — unreachable by extensions or AI agents.
- **devuser is devcontainer-only** — created by setup.sh (devcontainer entrypoint), not by the Dockerfile. In production, setup.sh never runs, so devuser never exists. www-data runs Apache in all contexts. Prod is unaffected.
- **Docker socket not mounted** — the devcontainer has no access to the host's Docker daemon. Prevents container escape and keeps the blast radius contained.
- **GitHub CLI added via devcontainer feature** — `ghcr.io/devcontainers/features/github-cli:1` installs gh. Credentials are not forwarded by the feature.
- **Docs live in a standalone reference page** — `docs/content/dev/getting-started/devcontainer-reference.md`. Covers what persists across rebuilds, what doesn't, and re-auth steps. Both `local-setup-mac.md` and `local-setup-windows.md` link to it (same devcontainer, same gh flow regardless of host OS). Avoids duplicating instructions in two setup guides.

## Debugged Issues

### setup.sh crashes on stale image — useradd exit code 9
- **Status:** `fixed`
- **Fixed in:** `.devcontainer/setup.sh:16` — guard with `id devuser` check
- **Severity:** High
- **Environment:** local devcontainer (VS Code)
- **File:** `.devcontainer/setup.sh:16`
- **Root cause:** `useradd -m -s /bin/bash devuser` fails with exit code 9 ("username already exists") when the Docker image was built from a commit that still created devuser in the Dockerfile. `set -euo pipefail` propagates the exit code, killing the container.
- **Steps to Reproduce:**
  1. Build the Docker image from commit `27d9fd7fc` or earlier (devuser baked into image)
  2. Check out commit `f6a754b01` (devuser creation moved to setup.sh)
  3. Open devcontainer in VS Code (do not rebuild image)
  4. Container exits with code 9 immediately
- **Expected behavior:** Devcontainer starts successfully regardless of whether the image has devuser pre-baked
- **Actual behavior:** Container exits with code 9; VS Code reports "container thrive-admin-1 exited (9)"
- **Recommended fix:** Make useradd idempotent: `id devuser &>/dev/null || useradd -m -s /bin/bash devuser`
- **Suggested tests:** none needed — entrypoint idempotency is verified by rebuilding without image cache
- **Linear:** `not synced`
