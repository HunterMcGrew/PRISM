## Decisions

- **Framework knowledge lives in SKILL.md, not in rules files** — each persona's reasoning frameworks are tightly coupled to their personality and operational behavior. Separating them into rules files would lose the integration between "how they think" and "how they talk."
- **ux-patterns.md is a shared architect file, not a rule** — it captures the building code that Clove implements against and Briar/Eric review against. Pixel owns the full design framework; ux-patterns.md is the implementation-facing subset. Wired via manifest.json glob patterns.
- **Manifest.json uses glob patterns for ux-patterns.md** — `frontend/blocks/**/*.tsx` and `frontend/components/**/*.tsx` don't conflict with existing prefix patterns (`frontend/blocks/`, `frontend/components/`), allowing both docs to load in parallel.
- **"How [persona] thinks" sections are operational behavior, not personality flavor** — the reasoning principles are instructions for how the persona reasons through decisions, not decorative personality text. This distinction is explicit in each skill file.
- **Mandatory behaviors are defined as failure states** — "Opus reinforced" means anti-patterns are explicitly named as things the persona must NOT do, with concrete descriptions of what the failure looks like.
- **Domain context (equipment dealership) included in each persona** — each persona gets a tailored dealership context section because the same domain facts have different implications for design (Pixel), triage (Nora), and implementation (Clove).
