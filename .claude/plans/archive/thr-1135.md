# Branch Plan: feature/hmcgrew/thr-1135-info-banner-block-for-mega-menu-2

## Goal

Add a `mega-menu-info-banner` Gutenberg block and its frontend `MegaMenuInfoBanner` component, replacing the old `InfoBanner`.

---

## Decisions

- Replaced `InfoBanner` with `MegaMenuInfoBanner` — the new component uses `WithSkeletonImage` instead of raw `next/image`, integrates with `MegaMenuContext` for close behavior, and uses the shared `LinkTarget` type guard.
- `MegaMenuInfoBanner` is `"use client"` because it uses `useMegaMenuContext` hook.
- `config.ts` exports `BlockAttributes`, `DEFAULT_ATTRIBUTES`, and `MegaMenuInfoBannerBlockConfig` — follows standard block pattern.
- `link.ts` type guard file is shared with PR #1613 (THR-1443) — merged that branch to avoid duplication.
- MegaMenuContext stub created locally — real implementation comes from PR #1543. Remove stub once that PR merges.
- `ServiceNavigationPane` updated to use `MegaMenuInfoBanner` instead of deleted `InfoBanner` — prop names mapped (`buttonTarget` → `buttonUrlTarget`), outer border wrapper removed (now inside component).
- Block wrapper uses `.view.tsx` extension (Pattern 3) — component lives in `frontend/components/info-banner/`, block folder has only CMS wiring. Per jmotes review feedback.
- MegaMenuProvider callback props renamed from `closeMenu`/`closeDropdown` to `onCloseMenu`/`onCloseDropdown` per jmotes review. Context value methods (`closeMenu`/`closeDropdown`) keep verb names — they're API methods, not component callbacks.
- `heading` and `buttonLabel` are required on frontend component and block props — resolver guarantees non-empty values via `DEFAULT_ATTRIBUTES`. If editors don't want both, they should use standalone heading/paragraph blocks instead.
- Editor `help` text marks both fields as "Required" — no WordPress validation needed, just guidance.
- Component removes default parameter values (`= "Learn More"`, etc.) — defaults live in the resolver via `DEFAULT_ATTRIBUTES`, not in the component.
