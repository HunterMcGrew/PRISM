## Design

**Status:** Ready for implementation
**Author:** Pixel
**Date:** 2026-04-29

### Summary

Restructure the Mega Menu Featured Banner editor sidebar so the image controls stop reading as Text Color actions. Wrap the `MediaUploadCheck` block in its own `<PanelBody title="Background Image">`, put the preview first, then the action row underneath. Replace/Remove stay in the sidebar (not the toolbar) — they're infrequent, deliberate actions that don't earn toolbar real estate.

### Decisions worth knowing

- **New `PanelBody` for the image group, not a flat label inside the existing panel.** Considered an inline section header (`<p>BACKGROUND IMAGE</p>`) like the current Text Color pattern. Rejected — Gestalt common region needs a real shared boundary, and the existing Text Color section is already a Nielsen #4 inconsistency we shouldn't propagate.
- **Preview first, buttons under.** The image is the content; buttons act on it. Image-caption convention (Finder, Photos, every CMS).
- **Two-state button row.**
  - No image set: single full-width primary `+ Select Image` (was "Background Image"). One obvious next action.
  - Image set: two equal-weight buttons. Replace neutral (drop `is-primary`), Remove `is-destructive` (red outline/text, not red-filled — destructive action is reversible, don't overweight it).
- **Toolbar stays untouched.** Replace/Remove are property edits, not block-level operations. Toolbar is reserved for move/duplicate/options.
- **Empty-state copy goes sentence case** — "No image selected" instead of "No Background Image Selected." Passive empty state, not a warning.

### Reused components

- `PanelBody` from `@wordpress/components` (already used for "Featured Banner Settings")
- `Button` from `@wordpress/components` (already used)
- `MediaUpload` / `MediaUploadCheck` / `ResponsiveWrapper` from `@wordpress/block-editor` (already used)

No new components. No new patterns. Pure restitch of existing primitives.

### Layout

```
▾ Featured Banner Settings
  HEADLINE / DESCRIPTION / CTA TEXT / CTA LINK / LABEL / LINK REL / TEXT COLOR
  (image controls removed from this panel)

▾ Background Image                            ← new PanelBody
  ┌─────────────────────────┐
  │  [preview / empty slot] │                 ← preview first
  └─────────────────────────┘
  [ Replace ]  [ Remove ]                     ← image-set state
                  -- or --
  [ + Select Image          ]                 ← no-image state, full-width primary
```

### Accessibility notes

- `PanelBody` provides the labeled landmark — screen readers will announce "Background Image" as the section heading.
- Buttons keep their existing labels; Remove's `is-destructive` styling provides a non-color signal (text weight + outline) in addition to color, so colorblind users still distinguish it from Replace.
- No new keyboard interactions — `PanelBody` collapse/expand is already keyboard-accessible.

None. Clove can run with this directly.

## Decisions

- Followup PR title format: `THR-1130 followup: <summary>` per `.claude/rules/git-conventions.md`.
- No global placeholder substitution. The codebase has `getPlaceholderImage(contentType, ...)` for entity-backed content (equipment, employee, location, etc.), but the featured banner is decorative chrome inside a mega menu column with no backing entity. AC requires no image to render when the editor has cleared it.
- **Legacy placehold.co URL normalized at the resolver, not migrated in the database.**
  - **Root cause:** WordPress persists block attribute `default:` values into post content at insert time. Blocks inserted before this fix have the placehold.co URL stored on `block.attrs.backgroundImage` as a real value — the schema change alone only helps new inserts.
  - **Alternatives considered:** (a) WP-CLI migration to scrub the URL from post_content; (b) editor-side `useEffect` that clears the attribute on load; (c) resolver-side normalization to `undefined` whenever the value matches the legacy URL.
  - **Chosen approach:** resolver-side normalization. Beats (a) — no migration to coordinate, no risk of missing a site, no cleanup commit to schedule. Beats (b) — no `useEffect` that mutates editor state on load (anti-pattern per `use-effect-guidelines.md`).
  - **Implementation guidance:** export a single `LEGACY_PLACEHOLDER_BACKGROUND_IMAGE` constant from `schema.ts` so the resolver and any future migration share the canonical value. The check is exact-string equality on the known placeholder URL — do not generalize to "any placehold.co URL," because a dealer could legitimately paste a placehold.co image.

## Debugged Issues

### Featured banner shows broken image placeholder when no background image is set

- **Status:** `fixed`
- **Fixed in:** `frontend/blocks/mega-menu-featured-banner/schema.ts` — removed `backgroundImage` placeholder URL from `DEFAULT_ATTRIBUTES`.
- **Severity:** Medium
- **Environment:** Frontend (Next.js) — also affects editor preview in wp-admin
- **File:** `frontend/blocks/mega-menu-featured-banner/schema.ts:17` (root), `frontend/blocks/mega-menu-featured-banner/resolver.props.ts:15-16` (proximate)
- **Root cause:** `DEFAULT_ATTRIBUTES.backgroundImage` is hard-coded to `"https://placehold.co/854x480?text=background+image"`. The resolver applies this as a fallback whenever `attrs.backgroundImage` is undefined (`attrs?.backgroundImage ?? DEFAULT_ATTRIBUTES.backgroundImage`), so a "no image" state cannot reach the component. The component's `{backgroundImage && (<Image .../>)}` guard at `MegaMenuFeaturedBanner.tsx:60-71` therefore never trips, and Next.js Image attempts to load an unconfigured remote host (`placehold.co` is not in `next.config.js` `images.remotePatterns`), producing the broken-image icon.
- **Steps to Reproduce:**
  1. In WordPress admin, open a page with a mega menu and add (or open) a Mega Menu Featured Banner block inside a column.
  2. Upload a background image, save, then click "Remove" on the image.
  3. Save and view the page on the frontend with the mega menu open.
  4. Observe a broken image icon in the banner area instead of a clean text-only banner.
- **Expected behavior:** Banner renders text content only — no `<img>` element and no broken image placeholder.
- **Actual behavior:** Banner renders an `<Image>` pointing at a placehold.co URL that fails to load, showing a broken image icon.
- **Recommended fix:** Remove the `backgroundImage` entry from `DEFAULT_ATTRIBUTES` in `frontend/blocks/mega-menu-featured-banner/schema.ts`. The field is already optional (`backgroundImage?: string`) and the component handles the falsy case correctly. Both the frontend resolver and the editor's `previewProps` consume this default, so a single change fixes both surfaces.
- **Suggested tests:**
  - Storybook story for the no-image variant.
  - Unit test on `resolveProps` asserting that `attrs.backgroundImage = undefined` produces `result.backgroundImage = undefined`.
- **Linear:** `synced`
