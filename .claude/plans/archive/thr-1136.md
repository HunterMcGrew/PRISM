# Branch Plan: feature/hmcgrew/thr-1136-navigation-list-block-for-mega-menu-2

## Goal

Add a mega-menu-navigation-list block that renders a heading with a configurable multi-column list of links inside the mega menu.

---

## Decisions

- Component lives at `frontend/components/mega-menu-navigation-list/MegaMenuNavigationList.tsx` — follows existing mega-menu component pattern and component-props-decoupling rules. Component is reusable outside block system.
- Block registration via `.view.tsx` pattern: `frontend/blocks/mega-menu-navigation-list/MegaMenuNavigationListBlock.view.tsx` imports the component and serves as the registration point. Allows block system (schema, resolver) to stay thin while keeping component pure and reusable.
- `.view.tsx` wrapper is a passthrough initially but provides a layer for future block-level logic (handlers, providers) without polluting the component.
- `"use client"` is required on the component — uses `useId()` and `useMegaMenuContext()` hooks. Not a problem because block-views.tsx is `"use client"` and uses `next/dynamic()` for proper code splitting.
- Backend editor block at `backend/plugins/gravity-platform-core/src/blocks/mega-menu-navigation-list/` — registers with WordPress, uses SortableLinksModal for link management
- `block.json` declares parent as `mega-menu-column` — navigation list can only be inserted inside mega-menu columns
- `getProps` in resolver.ts filters out links with empty/whitespace labels AND empty/whitespace hrefs — intentional data sanitization
- Component depends on `MegaMenuContext` and `mega-menu-editor/constants` from the mega-menu-editor PR (#1627) — stubs added until that branch merges
- Heading uses `<h3>` with `id={blockId}-heading` and `<ul>` uses `aria-labelledby` — satisfies WCAG 2.1 AA heading/list association
- Random block IDs are acceptable for a11y — `aria-labelledby` only needs a valid reference, not a human-readable one
