## Debugged Issues

### NavigationMenuItem focus ring clipped by overflow-hidden
- **Status:** `fixed`
- **Fixed in:** Removed `overflow-hidden` from `<Popover>` className at line 76
- **Severity:** Medium
- **Environment:** Frontend — all browsers, desktop only (component is `hidden lg:block`)
- **File:** `frontend/components/navigation-menu-item/NavigationMenuItem.tsx:76`
- **Root cause:** The `<Popover>` wrapper has `overflow-hidden` (added in commit `1d4670953` — "Bugfix/mega menu colors header", Dec 2025). The button/link inside uses `focus:ring-2 ring-offset-2`, which renders a `box-shadow` 2px outside the element boundary. `overflow-hidden` clips the ring, making it invisible. Secondary issue: no explicit ring color — Tailwind defaults to semi-transparent blue (`rgb(59 130 246 / 0.5)`), which is low-contrast against the typically dark mega menu background.
- **Steps to Reproduce:**
  1. Navigate to any page with the mega menu navigation bar
  2. Press Tab to move focus through navigation menu items
  3. Observe that no focus ring or highlight is visible on any menu item (buttons or links)
  4. For link-type items, the browser status bar shows the URL — confirming focus moves, but no visual indicator exists
- **Expected behavior:** Each navigation menu item shows a visible focus ring when focused via keyboard
- **Actual behavior:** No visible focus indicator on any menu item — fails WCAG 2.1 Level AA (2.4.7 Focus Visible)
- **Recommended fix:** Replace `overflow-hidden` on the Popover with `overflow-visible` (or remove it), and add an explicit high-contrast ring color (e.g. `focus:ring-white` or a CSS variable). If `overflow-hidden` was needed to clip `rounded-md` corners on the mega menu panel, scope the overflow clipping to the panel only — not the trigger container.
- **Suggested tests:** Add a test asserting the PopoverButton/Link receives focus and has visible focus-related classes. Visual regression test for focus ring rendering.
- **Linear:** `not synced`
