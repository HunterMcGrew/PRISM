# Plan: thr-242

## Ticket

THR-242 — Gravity Forms required field indication

## Goal

Make required fields in Gravity Forms visibly and accessibly identifiable on the frontend by rendering a shared required-indicator atom next to every required field's label or legend.

---

## User Stories

- As a form-filler, I want to see which fields are required before I try to submit, so that I don't waste time filling in optional fields and hitting an error.
- As a screen-reader user, I want required fields announced as "required" during label read-through, so that I get the same scan-ahead affordance that sighted users get from the red asterisk.

---

## Design

No mock. Asterisk-after-label is one of the most universal form conventions on the web; Pixel not needed. Styling and a11y treatment are documented in Decisions below.

## Decisions

- `RequiredIndicator` is an atom, not a wrapper that owns label rendering — a `FieldLabel` wrapper would fight the structural split between `<label>` (single-input fields) and `<legend>` (composite fields). The atom keeps each field's structure legible and the shared concern (the required mark) in one place.
- Asterisk glyph is `*` in `text-red-700`, placed after the label text with `&nbsp;` as the separator — matches the existing color convention from RadioField; `&nbsp;` prevents orphaned-asterisk wrapping on narrow breakpoints.
- Accessibility treatment is `aria-hidden="true"` on the glyph plus a visually-hidden `sr-only` "required" span — color+glyph alone fails WCAG 1.4.1; explicit sr-only text gives SR users scan-ahead parity with sighted users. HTML5 `required` (already present) covers on-focus validation announcement, so `aria-required` is not added manually.
- RadioField's existing `(Required)` text treatment is replaced with `<RequiredIndicator />` — consistency across field types is the point of this ticket; a snowflake RadioField would contradict the goal.
- `labelPlacement !== "HIDDEN"` inconsistency across field components is out of scope — it's a real a11y bug (hidden labels should become `sr-only`, not removed from the DOM), but separate from required-indication. Follow-up ticket to be filed.
- Full Storybook coverage for every Gravity Forms field component is out of scope — significant lift that belongs in its own initiative. This ticket adds a story for `RequiredIndicator` + one representative field story for Chromatic coverage.

## Acceptance Criteria

### Behavioral

- [ ] Given a Gravity Form with a required single-input field (text, email, select, textarea, phone, number, date, time, website, file upload, multi-select), When a user views the form, Then a red asterisk appears immediately after the field's label text.
- [ ] Given a Gravity Form with a required composite field (radio, checkbox, address, name), When a user views the form, Then a red asterisk appears immediately after the field group's title.
- [ ] Given a Gravity Form with a non-required field, When a user views the form, Then no asterisk appears next to the field label.
- [ ] Given a screen reader is active, When a required field's label is read, Then the reader announces the label followed by the word "required".
- [ ] Given a screen reader is active, When a non-required field's label is read, Then the reader announces only the label text, with no "required" announcement.
- [ ] Given a form with radio fields, When a user views it after this change, Then the radio field shows the same red asterisk as other required fields, and the previous "(Required)" text is no longer displayed.

### Non-behavioral

- [ ] Red asterisk color meets WCAG AA contrast ratio (4.5:1) against the form's background.
- [ ] Required indicator is visible on mobile, tablet, and desktop breakpoints without wrapping away from the label.
- [ ] HTML5 form validation behavior is unchanged — required fields still block form submission.
