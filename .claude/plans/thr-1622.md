# Plan: thr-1622

## Ticket

https://linear.app/tractru/issue/THR-1622/mega-menu-editor-preview-clicking-a-nav-link-navigates-to-a-header

## Goal

Prevent navigation when a link-type nav item is clicked in the Mega Menu Editor Preview route, while keeping dropdown triggers interactive.

---

## Decisions

- The preview page runs the **frontend** render pipeline, not the wp-admin editor. The existing `isEditor` prop on `MegaMenuNavigationItem` is not automatically set in this context.
- `MegaMenuContext` is the right home for a preview flag — it's already the canonical cross-component context for the mega menu system, imported by 13 consumers. Adding a dedicated second context would duplicate plumbing.
- Option A (disable links via context flag) preferred over Option B (CSS / click-capture guard). Reason: Option B with `pointer-events: none` on anchors is fragile (leaks through children), and a capture-phase click handler has to distinguish dropdown-button clicks from link clicks — brittle. Option A reuses the existing `isEditor`-style `<span>` branch and is unit-testable.
- `MegaMenuProvider` inherits `isPreview` from parent context when not explicitly set. Reason: `MegaMenuEditor` (frontend) wraps its nav bar in its own `MegaMenuProvider`, which would otherwise shadow a preview-page-level provider. The inner provider reads the outer context and preserves `isPreview` unless explicitly overridden. One-line merge — `isPreview ?? parent?.isPreview ?? false`.
- `isPreview` gates the link branch only (`type === "link"` renders `<span>`). The dropdown panel gate (`!isEditor && buttonBottom > 0`) is unchanged — dropdowns must continue to work in preview.
- Preview-page wrapping is done at the page level, not in the resolver or `BlocksRenderer`. Reason: `BlocksRenderer` is generic and should not learn about preview context; the preview page is the only place that knows it's rendering for preview.
- **Known follow-up (Winston, 2026-04-18):** rendering the preview link branch as an inert `<span>` removes a legitimate editor verification flow. Prior behavior — same-tab navigation to the destination rendering with the *published* mega menu — let editors confirm link targets resolved. The "header-less page" bug only surfaced when draft ≠ published. Follow-up ticket should change the preview link branch to `<a href target="_blank" rel="noopener noreferrer">` so clicks open destinations in a new tab (preview tab stays put, editor gets verification back). Open as a separate ticket to preserve 1803's clean approvals — do not scope-expand this PR.

## Debugged Issues

### Nav item links in editor preview follow their href
- **Status:** `fixed`
- **Fixed in:** `MegaMenuContext` gained `isPreview` (with parent-context inheritance); `MegaMenuNavigationItem` link branch now renders `<span>` when `isEditor || context?.isPreview`; `/mega-menu-editor-preview/page.tsx` wraps `BlocksRenderer` output in `<MegaMenuProvider isPreview>`. 37 unit tests (12 new), 97.67% stmt / 96.15% branch / 100% line coverage on changed files.
- **Severity:** Medium
- **Environment:** Marshall (QA), `/mega-menu-editor-preview` route
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:75-84`; `frontend/app/mega-menu-editor-preview/page.tsx:121-126`
- **Root cause:** `MegaMenuNavigationItem` renders a real Next.js `<Link href={link.href}>` when `isEditor=false`. The `/mega-menu-editor-preview` page calls `BlocksRenderer` without any mechanism to signal "non-interactive render" — `MegaMenuContext` exists but is only provided inside `NavigationMenuPanel`, which wraps the dropdown children, not the nav item trigger itself. Result: clicking a link-type nav item performs a client-side navigation out of the preview.
- **Steps to Reproduce:**
  1. Open the Mega Menu Editor (Marshall or equivalent)
  2. Click into the Preview
  3. Click any link-type nav item
  4. Browser navigates to the link's href; destination renders without the preview's placeholder header
- **Expected behavior:** Clicking a link-type nav item in preview should not navigate; dropdown triggers should still open their mega menu panes.
- **Actual behavior:** Link-type nav items navigate via Next.js `<Link>`.
- **Recommended fix:** Add an `isPreview` flag to `MegaMenuContext` + `MegaMenuProvider`. Wrap the preview page's block render output in `<MegaMenuProvider isPreview>`. In `MegaMenuNavigationItem`, treat `isEditor || context?.isPreview` as the "render as `<span>`" condition for link-type items. Dropdown branch (PopoverButton) is unchanged — no navigation there.
- **Suggested tests:**
  - Unit: `MegaMenuNavigationItem` renders `<span>` when `useMegaMenuContext()` returns `{ isPreview: true }` and `type === "link"`
  - Unit: Dropdown nav items still render `PopoverButton` regardless of `isPreview`
  - Unit: Frontend render (no provider, default context) still renders `<Link>` — no regression in live-site navigation
- **Linear:** `synced`

## Acceptance Criteria

### Behavioral
- [ ] Given the Mega Menu Editor Preview is open, When a simple link-type nav item is clicked, Then the browser does not navigate away from the preview
- [ ] Given the Mega Menu Editor Preview is open, When a dropdown nav item is clicked, Then its mega menu pane opens as expected
- [ ] Given the Mega Menu Editor Preview is open, When a dropdown is open and a link inside a pane is clicked, Then the navigation follows the link (only top-level nav item links are suppressed in preview — links inside dropdown content follow their href as usual)
- [ ] Given a published mega menu on the live site, When a link-type nav item is clicked, Then the browser navigates to the link target (no regression)
- [ ] Given the WordPress block editor canvas, When a link-type nav item is clicked, Then nothing navigates (existing editor behavior preserved)
