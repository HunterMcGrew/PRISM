# Plan: thr-1523

## Ticket

https://linear.app/tractru/issue/THR-1523/color-contrast-utility-programmatic-wcag-contrast-enforcement-for

## Goal

Build a color contrast utility that enforces WCAG contrast ratios programmatically across the theme token system, so contrast failures are caught at definition time rather than discovered in QA or audits.

---

## Background

This ticket emerged from the accessibility audit (see `.claude/plans/thr-1522.md`). During Q2 theming work, color palettes and surface/text token values will become configurable. Without a contrast check at the token level, a designer or developer can introduce a pairing that looks acceptable but fails WCAG minimums — and the failure only surfaces in manual testing.

**Reference:** [MDN Color Contrast Guide](https://developer.mozilla.org/en-US/docs/Web/Accessibility/Guides/Understanding_WCAG/Perceivable/Color_contrast)

**Do not schedule until:** the theming token system architecture is defined. The utility's integration point depends on knowing whether tokens are defined in CSS custom properties, a JS config, a design token JSON file, or a combination.

---

## Decisions

- This is separate from the general accessibility audit ticket — the audit covers existing components; this covers systematic enforcement during theming
- Utility should cover all WCAG contrast tiers: 4.5:1 (normal text), 3:1 (large text and UI components), not just one
- Must check all interactive states (default, hover, focus, active) — not just the default pairing
- Integration point TBD pending theming architecture decision

---

## WCAG Requirements This Utility Enforces

| Criterion | Ratio | Applies To |
|-----------|-------|-----------|
| WCAG 1.4.3 AA — normal text | **4.5:1** | Body copy, labels, captions |
| WCAG 1.4.3 AA — large text | **3:1** | 18pt+ or 14pt bold |
| WCAG 1.4.11 AA — non-text contrast | **3:1** | Buttons, form borders, icons, focus rings |

Exceptions (do not flag): decorative text, disabled/inactive controls, logotypes.

## Candidate Implementation Forms

Assess at design time — choose based on where the theme token system lands:

### Option A: Build-step token validator
- Runs against the token definition file (JSON, JS config) at build time
- Fails the build if a color pairing violates minimum ratios
- Best if tokens are declared in a static config
- No runtime overhead

### Option B: Jest utility for component tests
- `expectContrastRatio(fg, bg, minRatio)` assertion
- Applied in component tests where themed color combinations are rendered
- Best for catching regressions during development
- Does not catch token-level violations before they reach components

### Option C: `useContrastRatio(fg, bg)` hook
- Runtime hook for dynamic theming scenarios where colors are user-configurable
- Returns the ratio and a pass/fail for each WCAG tier
- Best if theming is genuinely runtime-dynamic (e.g. user-selectable themes)
- Not needed if themes are build-time only

A combination of A + B is the most thorough approach for a build-time token system.

## Algorithm Reference

WCAG contrast ratio = (L1 + 0.05) / (L2 + 0.05), where L1 is the lighter relative luminance and L2 is the darker.

Relative luminance formula per WCAG 2.x:
- For each RGB channel: if value ≤ 0.04045, divide by 12.92; otherwise ((value + 0.055) / 1.055) ^ 2.4
- L = 0.2126 × R + 0.7152 × G + 0.0722 × B

**Library options** (if not implementing from scratch):
- [`color2k`](https://color2k.com/) — lightweight, tree-shakeable, supports hex/rgb/hsl
- [`polished`](https://polished.js.org/) — broader color utilities, includes `getLuminance`
- Custom implementation is ~20 lines and removes a dependency

## Open Questions

1. Will theme tokens be defined in a static config file, CSS custom properties, or both? (Determines integration point for Option A)
2. Are themes build-time only, or will users be able to switch themes at runtime? (Determines whether Option C is needed)
3. Should the validator cover only theme tokens, or also ad-hoc inline color values in components?
