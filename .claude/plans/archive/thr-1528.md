# Plan: THR-1528

## Ticket

https://linear.app/tractru/issue/THR-1528/restructure-code-standards-into-universal-overview-ts-php-files

## Goal

Split `.claude/rules/code-standards.md` into three terse Claude rules files (universal + TS + PHP) and rewrite the docs narrative to cover both languages with updated examples.

---

## User Stories

- As a developer using AI agents, I want TS and PHP standards in separate, focused files so the agent loads only the relevant standards for the language I'm working in.
- As a new developer, I want the docs page to explain the full type taxonomy (DTOs, domain models, Props, BlockAttributes) so I understand how data flows through the system.
- As a PHP developer, I want documented naming conventions (snake_case) and security patterns so I don't have to guess at team expectations.

---

## Decisions

- Claude rules files are terse (~80-120 lines) — rule + brief rationale, examples only where the rule is ambiguous without one. The docs page is the rich narrative with full examples and teaching context.
- PHP naming follows PSR conventions (PascalCase classes, UPPER_SNAKE_CASE constants, PSR-4 autoloading) **except** function and method names, which use `snake_case` to match WordPress core. Reverted from camelCase due to WP plugin extension issues. Existing camelCase is grandfathered; new/touched code uses snake_case.
- TS type taxonomy documented as four layers: Interfaces (domain models, `I` prefix) -> DTOs (API shapes, `{Protocol}{Entity}DTO`) -> Props (`{ComponentName}Props`) -> BlockAttributes (`schema.ts`). Connected by `dtoTo*()` mappers in services.
- Data flow pattern (ServiceFactory -> Service -> Repository -> DTO -> domain model) codified in TS standards.
- Docs page stays as ONE page with TS and PHP sections — developers want a single reference, not three tabs.
- `.cursor/rules/` support deferred — noted here for future work. Would be a stub file referencing `.claude/rules/` files.
- `code-standards.md` (overview) has no frontmatter path scope — loads for all conversations. The TS and PHP files get `paths:` frontmatter to scope loading.

## Review Issues

### Universal test rules dropped from overview
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards.md`
- **Problem:** Four universal test rules from the original file were dropped entirely — "Do not break existing tests", "Do not delete tests to make changes pass", "Add tests when changes introduce meaningful risk", "Explain verification steps when behavior changes". These apply to both TS and PHP and don't appear in any of the three new files.
- **Suggested fix:** Add a `## Tests` section to the overview with these four universal rules. The TS and PHP files already cover tooling/framework specifics — the overview should cover the universal behavioral rules.
- **Fixed in:** Added `## Tests` section to `code-standards.md` with all four rules plus Storybook mandate.

### "Do not access DOM APIs unless explicitly requested" dropped
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards-ts.md`
- **Problem:** This rule was in the original Server/Client section and was dropped. It's a useful guardrail — agents shouldn't reach for `document.querySelector` unless asked.
- **Suggested fix:** Add to the Server/Client section in `code-standards-ts.md`.
- **Fixed in:** Added to Server/Client section in `code-standards-ts.md`.

### "we use index for block attributes" dropped
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards-ts.md`
- **Problem:** Informal note from the original Blocks section was dropped. Minor but was an intentional convention.
- **Suggested fix:** Add to the Blocks section in `code-standards-ts.md` — something like "Block attributes are defined in the block's `index` (not `schema.ts` attributes array)." Or clarify what this meant and document it properly.
- **Fixed in:** Clarified in Blocks section — `block.json` for WP-native features, custom attributes defined in block's index via `schema.ts`.

### Resolver splitting size guidance truncated
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards-ts.md`
- **Problem:** Original said "Consider splitting into multiple files only when you have both a props and children resolver and each exceeds ~200 lines." The TS file only kept the first part about when to break out `resolver.props.ts`.
- **Suggested fix:** Add the 200-line guidance back to the Blocks section.
- **Fixed in:** Full size guidance restored to Blocks section in `code-standards-ts.md`.

### TS paths scope matches PHP plugin source directories
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards-ts.md:3-5`
- **Problem:** `backend/**/src/**` in TS paths matches PHP files in `backend/plugins/thrive-sss/src/` (10+ PHP files) and `backend/plugins/thrive-byo/src/` (8 PHP files). PHP rules paths don't include those directories, so PHP files there get TS rules loaded but not PHP rules.
- **Suggested fix:** Narrow TS scope to `backend/plugins/gravity-platform-core/src/**`. Add other plugin `src/` paths to PHP scope, or broaden PHP to `backend/**/src/**`.
- **Fixed in:** Narrowed TS paths to `backend/plugins/gravity-platform-core/src/**`. Broadened PHP paths to include `backend/**/src/**` — covers thrive-sss, thrive-byo, and any future PHP plugins.

### Duplicate block.json bullets in TS standards
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `.claude/rules/code-standards-ts.md:66-67`
- **Problem:** Two consecutive bullets both state `block.json` is for WP-native features only. Second adds `schema.ts` context but repeats the opening clause.
- **Suggested fix:** Consolidate into one bullet.
- **Fixed in:** Merged into single bullet on line 66.
