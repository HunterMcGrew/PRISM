## Goal

Create comprehensive Mega Menu V2 documentation — dev architecture, REST API reference, migration operations guide, and user guide — covering all 11 ticket sections across two audiences.

## User Stories

_Not applicable — documentation ticket. Scope defined by ticket description._

## Decisions

- **Four doc files, not one mega-doc** — one topic per page per documentation.md convention. Separating by audience (dev/user) and category (architecture/references/operations) keeps each file focused and places it correctly in the Nextra sidebar.
- **Dev architecture doc covers sections 1, 2, 7 (arch), 10, 11** — architecture overview, design decisions, mobile menu architecture, feature flag lifecycle, and security model are all "how the system works" content aimed at the same developer audience. One cohesive page.
- **Dev API reference is a separate doc** — REST endpoints are reference material, not narrative architecture. Developers will look this up by endpoint, not read it front-to-back. Includes preview token auth (section 4 internals) and revision storage model (section 9 internals).
- **Dev migration doc is under operations** — migration is a one-time operational task, not architecture. Lives alongside go-live-procedure and dealer-site-spinup in the operations category.
- **User guide is a single page** — covers setup, editor walkthrough, preview, go-live, revisions, mobile behavior, and importing from legacy. One page because the user journey is linear: enable → build → preview → publish → go live. Splitting would fragment a natural workflow.
- **Legacy mega menu is out of scope** — only document V2 and the migration path from legacy to V2. Do not document the legacy system itself.
- **ISR revalidation: document intended behavior** — ticket says 60s ISR but the intent (per Hunter's comment) is longer TTL with on-demand revalidation on save. Document the intended design; note current state only if it differs.
- **Epic plan is the primary source, not the ticket description** — the ticket description is a scope outline. The epic plan (`epic-mega-menu.md`) and the actual codebase are the sources of truth for architecture, decisions, and behavior.
- **Create `.claude/architect/mega-menu.md` as part of this work** — the mega menu is the largest feature system in the codebase but has no architect context file. Promote key architecture and decisions from the epic plan so agent context survives plan closure.
- **Never say "Mega Menu V2" in prose** — use **"Thrive Mega Menu"** for the new system and **"Legacy Mega Menu"** for the old one. Code identifiers that contain `v2` (block name `mega-menu-v2`, component `NavigationMenuItemV2`) stay as-is because they are actual code names. The architecture doc overview should include a terminology note defining these terms.
