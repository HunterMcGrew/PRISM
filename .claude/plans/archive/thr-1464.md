## Debugged Issues

### Offer.stories.tsx â€” `Cannot read properties of undefined (reading 'wpUrl')`
- **Status:** `fixed`
- **Fixed in:** Offer.stories.tsx â€” added `site: getMockSiteGlobals()` to args, removed dead `fetchSiteGlobals` mock
- **File:** `frontend/single-templates/Offer.tsx:28`
- **Root cause:** THR-1221 (commit 7236169b7) added `fetchFormsForCTAs(allCTAs, site.wpUrl)` to `Offer.tsx`. The `Offer` component receives `site` as a prop, but the story at `Offer.stories.tsx:244-248` only passes `item` and `urlPath` â€” not `site`. The story mocks `fetchSiteGlobals`, but that's irrelevant because `Offer` doesn't call it; it receives `site` from its parent. So `site` is `undefined` and `site.wpUrl` crashes.
- **Recommended fix:** Add `site` to the story args: `site: getMockSiteGlobals()`. Also consider whether the `fetchSiteGlobals` mock can be removed since it's unused.
- **Suggested tests:** The existing story test covers this â€” once `site` is passed, it should pass.

## Review Issues

### Duplicated `--testTimeout 30000` flag in package.json
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `frontend/package.json:16` â€” replaced bare `test-storybook --testTimeout 30000` with `pnpm run test-storybook` so timeout is defined once
- **File:** `frontend/package.json:15-16`
- **Problem:** The `test-storybook:ci` script calls the `test-storybook` binary directly with its own `--testTimeout 30000` instead of delegating to `pnpm run test-storybook`, duplicating the timeout value in two places.
- **Suggested fix:** Replace `test-storybook --testTimeout 30000` with `pnpm run test-storybook` in the CI script so the timeout is defined once.

### `concurrently` flags in `test-storybook:ci` are non-obvious
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `.github/workflows/build.yml` â€” added inline comment above the CI step explaining `-k` and `-s first` flags
- **File:** `frontend/package.json:16`
- **Problem:** The `concurrently -k -s first` flags are correct but non-obvious to future maintainers. `-k` kills sibling processes when one exits; `-s first` reports success based on the first process to exit. Without context, someone reading the script may not understand the intent.
- **Suggested fix:** Add a brief note in the PR description explaining what `-k -s first` does (e.g., "kill http-server when tests finish, exit with the test runner's status").
