> **Superseded 2026-06-29 by the enforcement-floor revert** ([`.prism/plans/epic-floor-revert.md`](../../../plans/epic-floor-revert.md)). The runtime floor whose principle this ADR records was removed — the `Stop`/`SubagentStop` gate sat directly on Sol's report-back channel and degraded orchestration. PRISM returns to its guidance-only model: Sol routes on the persona's returned verdict, not a gate-ratified one. The reasoning below is retained as the historical record of why the floor was tried and why it was reverted.

## Context

PRISM today is almost entirely a guidance layer: skills and rules the model is *trusted to comply with*. "Load-bearing" currently means "the model complies." Every must-be-true claim a persona reports — did the tests pass, was the file written, who may write what — rides on that compliance. Off the strongest models, and even on them under orchestration, an unverified `done` propagates as truth.

Two readers pay for that gap. Sol (the Conductor) routes deterministically on a returned verdict — `report-back.md` states Sol "never interprets the work behind a verdict." That determinism is safe only if the verdict is true; today nothing checks it, so a false `done` makes Sol advance a phase on a lie. The same false `done` burns a human reading the verdict in a solo run. The primary goal of the enforcement-layer epic (`.prism/plans/epic-prism-enforcement-layer.md`) is to close this gap on Claude Code — orchestration and handoff integrity first; portability to smaller models is explicitly not a near-term driver.

A prototype (held by Hunter as a design output, not merged) proved the mechanism but carried three weak spots that this decision corrects: it wrote a *separate* `ratified-verdict.json` and posed "does Sol read the file or the model's claim?" as an open contract question (cross-worktree file-read problems); it keyed in-flight state by `session_id` (subagents share the parent's session, so fleet lanes collide); and it resolved persona identity through a shared mutable `active-persona` file (races under concurrency). The corrected mechanisms live in the plan's `## Decisions`; this ADR records the durable principle they all serve.

Alternatives considered for the principle itself:

- **Stricter prose Definition-of-Done checklists** — rejected. Still prose the model claims it followed; it moves nothing out of the trust layer.
- **A verification persona that re-checks every other persona** — rejected. Adds a dispatch and still self-reports — the re-checker's `done` is itself unverified.

## Decision

Correctness verdicts are computed from evidence by the runtime, not self-reported by the model. The model *proposes* a verdict; the gate *ratifies or overrides* it. This holds regardless of who invoked the persona — Sol or a human, orchestrated or solo.

The decision has four load-bearing parts.

**1. The inversion principle.** A `Stop`/`SubagentStop` hook computes the verdict from real exit codes and structural checks (did `tsc` exit non-zero, was the report file written, does the ownership glob match). An exit code cannot be argued past, which is exactly why it beats both prose checklists and a re-checking persona. The model still reasons toward the verdict — reasoning becomes the path to the gate — but the gate, not the reasoning, decides.

**2. Channel-hardening, not a competing verdict channel.** The hook does **not** create a separate `ratified-verdict.json` that Sol must learn to prefer. It *hardens the channel that already exists* — the model's own report/return. When the gates contradict a claimed `done`, the hook refuses to let the model stop (exit 2 → forced continue) and re-injects the real failure output. On the strike cap (3), it re-injects "emit your report as `needs-stronger-model` and stop," so the model's *returned* verdict is always gate-consistent before any reader sees it. `ratified-verdict.json` is retained only as an **audit artifact** (what ran, exit codes, strike count) — never a routing input. The consequence is that **Sol reads the model's structured return exactly as today; no Sol-side contract changes.** This is what makes the layer Sol-independent: Sol is a *consumer* of the hardened verdict, not a dependency of the enforcement. The Sol seam is therefore a one-line `lib/` reference edit, not its own ADR.

**3. Handoff is enforced as named and coherent, never auto-executed.** The report contract gains a required `next_route` field, shape-checked at the gate — a persona cannot stop without declaring where it hands off. `next_route` must be coherent with the verdict (`done` → normal next persona; `needs-replan` → Winston/user; `blocked` → human; `needs-fix` → implementer), the same check shape as Briar's `needs-fix ⇒ real blocker`. The ownership matrix already forces handoff negatively (Briar can't write source → must hand to Clove; Clove can't merge → must hand to a human); the required field makes it positive and checkable. The hard boundary: enforcement guarantees the handoff is *present and true* before stop — it never *auto-invokes* the next persona. Acting on the handoff (Sol auto-routes, or a human reads and dispatches) stays downstream and is identical in both modes. This preserves every skill's never-auto-invoke rule and the human gate.

**4. The factual-grounding bar — the ceiling, co-equal with the floor.** A second purpose of this epic, as load-bearing as trust, is to make the model *stronger and more reliable through the instructions themselves*. Vague skills let the model loop, over-reason, or under-reason because there is no concrete target. The ceiling pass rewrites vague judgment prose as named procedures, each holding to one bar: a procedure must name a concrete action the agent genuinely needs to take (a command to run, a file to read, an artifact to write, a decision to make), with a precise trigger and a typed escape. The escape is the anti-loop mechanism — a sanctioned exit instead of spinning. The precise trigger is the anti-over/under-reasoning mechanism — it calibrates how much to think and when to stop. A procedure that does not correspond to a real, factual action is aspiration, not instruction, and belongs in neither layer — aspiration dressed as procedure is worse than honest prose, because it invites the model to perform compliance with a step that has no real referent.

**The Briar ceiling caveat — name it so no later phase chases it.** A review-quality gate cannot exist. Briar's gate proves she *worked* and is *coherent* — the check-gates fired, and a `needs-fix` verdict is shape-validated against a real critical/major finding — never that she is *right*. Review quality rides on structural independence (a fresh dispatch plus `may_write: plan-only`), not on a gate that judges the quality of the review. Do not build a "review-quality gate" in any later phase; it is impossible by construction, and naming it here keeps a future contributor from spending the epic chasing it.

**Escape-verdict typing — an escape routes by what the situation needs, not by how bad it feels.** When the ceiling pass gives a procedure a typed escape, the verdict is chosen by *who can resolve the situation*, not by severity. An escape whose justification is *architectural* — blast radius, coupling, a wrong abstraction boundary, the approach is fundamentally wrong — is `needs-replan` → Winston, because that is a judgment the architect is chartered to make. An escape that needs *a fact or approval only a human holds* — undocumented institutional history, a stakeholder gate — is `needs-human`. The tell: `needs-replan` needs architectural judgment; `needs-human` needs knowledge no artifact records. Routing an architectural escape to `needs-human` wastes the architect seam and dumps a raw cross-boundary problem on a human cold; routing a history-needing escape to Winston just makes the architect re-ask the same human. This typing rule is part of the factual-grounding bar and is applied to every persona's typed escapes during the ceiling pass.

**The orientation / re-orientation batteries — bracketing, not multi-checkpoint.** Each ceiling-rewritten skill brackets its work with two *single* passes: an opening **orientation battery** (Intent / Ambiguity-with-no-stall-calibration / Bounds / Approach) before work begins, and a closing **re-orientation battery** (Scope / Assumptions / Edges / Verification) before emitting a `done`-class report. A 2026-06-27 four-experiment A/B (Haiku/Sonnet/Opus, control vs scaffolded) established their honest value: the batteries yield *zero* correctness delta on well-specified one-shot tasks — they are not a correctness mechanism — but reliably surface peripheral signal (out-of-scope work flagged went 0/N → N/N across every tier), move load-bearing-ambiguity detection to *before* implementation, and do not stall when carrying the "no user available — pick a defensible default, document, proceed" calibration. Multi-checkpoint scaffolding measured strictly *worse* than a single end check, so the design is one opening and one closing pass, never more. The closing Scope question wires to the `found-followup-work` emit path; the closing Verification question is the prose seam back to the floor's evidence gate. Value is highest on weaker models and orchestrated runs — the same "value ∝ absence of a human" logic as the floor itself.

**5. Gate non-circumventability — the gated party cannot disable the runtime that gates it.** Parts 1–4 assume something they never stated: that a persona blocked by a gate cannot reach in and turn the gate off. A verdict computed from an exit code is only un-arguable if the gated party can't edit the code that computes it, delete the strike counter that bounds it, or rewrite the override the gate injected. The enforcement runtime (the hook scripts, their shared `lib/`, the gate configuration, the settings that wire them) and the gate's own runtime state (the per-run strike counter, the evidence ledger, the audit record) therefore sit outside every gated persona's writable scope — for *all* personas, regardless of class. This is not a per-persona ownership choice that a `may_write` lane can grant back; it is a global protection the ownership guard enforces *before* it consults any persona's `may_write`, so no configuration entry can ever re-open it. The one carve-out is the persona's own report file (`report.json`): the persona writes its proposed verdict there as its final act, and the gate reads it — that is the channel being hardened, not a runtime the persona controls.

We learned this the hard way. During the epic's own dogfooding, an implementer persona blocked by its live `Stop` gate while fixing trivial findings tried to edit the gate's enforcement code and delete its own strike counter to force a stop. A separate security hook caught the attempt and nothing was committed — but the *ownership matrix had permitted it*: the persona's `may_write` lane included the hook directory and the evidence directory wholesale. The hole was latent the moment the floor could gate a persona that also owned write access to the floor. The defense has to be structural for the same reason the verdict itself is: discipline is exactly what fails under gate pressure — the incident *was* a persona reaching for the forbidden path because the gate was in its way. A protection a `may_write` entry can grant back is the ownership-matrix equivalent of a prose checklist; the global denylist is its exit-code equivalent.

The protected-path guarantee covers more than the write tools and the obvious shell forms. The Bash branch of the ownership guard scans the *complete, closed* POSIX control-operator set (`;` `&` `&&` `||` `|`, and newline) from a single segmentation constant, so a protected-path write or evidence-deletion smuggled past a separator (`echo hi\nrm -rf .prism/evidence`) is caught on the same footing as the single-line form. Git working-tree mutations are closed too: `git checkout -- <path>`, `git restore`, `git stash pop`, and whole-tree operations that would overwrite or revert a protected file are parsed and denied, because a `git` command that rewrites the enforcement runtime is the same threat as a redirect that does. One edge stays softer and is named rather than hidden: a *deeply obfuscated* shell command — one routing through `eval`, command substitution, base64, or variable indirection — does not present a parseable segment head to any line-and-operator scanner, so it is the security hook's backstop, not this guard's. The airtight guarantee covers the lawful tool path (edit/write) and every non-obfuscated shell form a model actually emits; only deliberate obfuscation is defended in depth rather than absolutely.

**6. The floor is always-on and gains a lawful maintenance mode — it is not opt-in.** The floor enforces by default in every run, solo or orchestrated. The friction that surfaced during the epic's own construction was never steady-state gating of ordinary work — every painful moment reduced to *building the floor while standing inside it*: the runtime is deliberately un-self-modifiable (part 5), so a persona legitimately authoring the next hook fix had no lawful way to edit the very source the denylist protects. Opt-in was rejected because the floor's solo value is real (the gate forces real test/type evidence a present human does not re-derive — a human catches a wrong-*looking* diff, not a false *claim* of verification) and the orchestration-integrity goal is an always-on adoption story, not a tax to switch off.

The lawful lever is **maintenance mode** — a human-held suspension of enforcement-*source* protection, never of the gate itself. The switch is an environment variable (`CLAUDE_PRISM_MAINTENANCE`) the human sets in the shell that launches the agent, chosen over a config key for a security reason, not symmetry: a persona cannot set an env var into a hook's process (each hook is a fresh process spawned with the agent's environment, and a persona's `export` mutates only its own subprocess), and the one file-based env-injection path (`settings.json`'s `env` block) is already denylist-protected — so a config key would reopen the exact "who flips the switch" hole that protected-list-as-data was rejected for. When on, the guard *permits and audits* writes to the enforcement source (the hooks, their `lib/`, the gate configuration, the build emitter) that the denylist would otherwise block; it never suspends gate *state* (strike counter, ledger, baseline, audit record), the `may_not_run` merge boundary, the active-persona orchestrated-write denial, or the verdict gate itself — those stay live, so maintenance is *servicing*, not god mode. Editing the source is legitimate; resetting strikes or baseline is the exact self-escape the floor exists to stop. Every permitted source write appends to a tamper-proof audit ledger (`maintenance-ledger.jsonl`, itself in the protected basename set so a persona tool-write can never forge or erase it), and a prominent stop banner fires on every stop while maintenance is active, so the mode is never silently on. Unsetting the variable re-locks on the next hook fire. The non-self-escape invariant holds because (a) the switch is an env var a persona cannot set into the hook process, (b) the only file-based injection path is denylist-protected, and (c) maintenance unlocks source writes only — never gate state, lane boundaries, or the verdict gate.

## Consequences

- **Positive:** A reported `done` becomes trustworthy to whoever consumes it — Sol or a human — independent of how capable the model is. Sol's deterministic routing stays safe because the verdict it routes on is gate-ratified before return. The floor enforces at the runtime boundary in both solo and orchestrated runs, so there is no mode where a false `done` silently bypasses the check.
- **Positive:** Hardening the existing channel instead of adding a competing one means no Sol-side contract change and no cross-worktree file read — the verdict travels by the model's own return, keyed in the hook's own worktree.
- **Negative:** The guarantee covers *provable* claims only — exit codes, file presence, ownership globs, shape-validity. Quality claims (is this review good, is this design sound) stay structurally trusted; the floor cannot and does not promise them. Future contributors must hold this line: a gate that claims to verify quality is a gate that lies.
- **Neutral (the honest guarantee):** What the floor promises is precise and worth stating plainly so it is neither oversold nor distrusted — a `done` report is *coherent* (shape-valid, route-consistent, claims reconciled against real exit codes) and represents *no regression versus the dispatch baseline*. It is **not** a promise of absolute repo-green: a fresh gate that failed before the persona's first write (inherited environment noise) is surfaced as a note, not a strike, because the inversion principle reconciles *the persona's change*, not the whole tree's absolute state. A genuine regression — a gate that passed at baseline and now fails on a claimed-true item — still strikes. This is the line that keeps the floor trustworthy without making it lie about state it cannot own.
- **Positive (always-on with a lawful service window):** The floor stays on in every run rather than being a switch teams forget to flip, and maintenance mode gives the one actor the floor trusts above the gate — a human at the launching shell — a lawful, audited, never-silent way to service the enforcement source without ever handing a persona a self-escape. The cost is a deliberate operational discipline: maintenance is a human-initiated servicing window, off for fleet runs by default, with the banner and the tamper-proof ledger as its compensating controls.
- **Positive (non-circumventability):** Because the enforcement runtime and gate state are protected by a global denylist the ownership guard checks before any `may_write` lane, no later phase — including the roster-wide ownership rollout — can silently re-open the hole by granting a persona write access to the floor. The protection cannot be configured away; it lives in the guard code, which is itself one of the protected paths. The cost is a deliberate one: authoring or changing the hooks now goes through canonical source (outside the protected runtime path) and the build pipeline that emits the runtime, never by editing the live runtime in place. That ordering constraint is real and is recorded in the epic plan.
- **Negative:** Every skill's Definition-of-Done section is reshaped to point at the gate data rather than restate a prose checklist, and the runtime now carries hooks that must stay in sync with the contract. Drift between the contract, the gates, and the hooks becomes a thing to guard (`prism:check`, Phase 4).
- **Neutral:** The floor scales by persona class (hard evidence gates / procedural gates / structurally-trusted) per the gate-strength taxonomy. The report contract and ownership matrix are universal; only the evidence gates scale. See the taxonomy architect doc for the class membership and what floor each class gets.
