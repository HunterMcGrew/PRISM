# Plan: THR-1590

## Ticket

https://linear.app/tractru/issue/THR-1590/mega-menu-migration-empty-image-column-persists-when-no-images-exist

## Goal

Eliminate the empty image column that renders in mega menu panes when no valid image can be displayed on the frontend, so remaining content spans the full pane width.

---

## Decisions

- The root cause is a two-part failure: `core/image` blocks without a numeric `id` can't be enriched by the REST API, AND `ColumnBlock` renders empty divs when children produce no output.
- The fix must address both the data enrichment gap (so `core/image` renders when possible) and the rendering gap (so empty columns collapse).
- Add a resolver for `core/image` rather than modifying `ImageBlock` directly — resolvers are the established pattern for bridging CMS data to component props, and this keeps the component clean.
- The resolver prefers CMS-enriched `meta.image` when available; falls back to constructing `IImage` from `attrs.url` / `attrs.alt` — this preserves existing behavior on regular pages while fixing the mega menu path.
- `ColumnBlock` null guard matches `ColumnsBlock` — defense-in-depth for zero-inner-blocks edge case, not the primary fix (React children are element references, not rendered output, so null guard doesn't catch inner blocks that render to null).

## Debugged Issues

### core/image blocks render as empty columns in mega menu panes
- **Status:** `fixed`
- **Fixed in:** `core/image` resolver (`frontend/blocks/image/resolver.ts`)
- **Severity:** Medium
- **Environment:** Frontend — mega menu pane on the live site after migration and publishing via Thrive Mega Menu editor
- **File:** `frontend/blocks/columns/ColumnBlock.tsx:32` (proximate), `backend/plugins/gravity-platform-core/includes/RestApi/MegaMenuBlocksApi.php:514` (root)
- **Root cause:** Two-part failure. (1) When legacy about-pane media data lacks an integer `mediaId`/`id`, `normalizeImageId()` returns 0 and `buildCoreImage()` omits `attrs.id`. The REST API enrichment at `MegaMenuBlocksApi.php:514` requires a numeric `attrs.id` to call `BlocksProcessor::getImage()` and populate `meta.image`. Without enrichment, the frontend `ImageBlock` receives `image = undefined` and returns `null`. (2) Similarly, if `attrs.id` IS present but the WordPress media attachment has been deleted, `getImage()` returns `null`, producing the same result. In both cases, the parent `ColumnBlock` has no null guard — it always renders its wrapper div — creating a visible empty column that takes up flex space.
- **Steps to Reproduce:**
  1. Have a legacy mega menu with an about pane whose media has a `sourceUrl` but no integer `mediaId` (or a `mediaId` pointing to a deleted attachment)
  2. Run the migration tool
  3. Publish the migrated content via the Thrive Mega Menu editor
  4. Open the mega menu on the frontend
  5. Observe the about pane — an empty column is present where the image should be
- **Expected behavior:** When `core/image` cannot render (no usable media ID or deleted attachment), the image column should not take up space.
- **Actual behavior:** An empty `<div class="column-block">` with `flex-basis: 100%` renders in the flex row, consuming proportional width and creating a visible gap.
- **Recommended fix:** Two-part fix: (A) Add a resolver for `core/image` that constructs an `IImage` from `attrs.url` and `attrs.alt` as a fallback when `meta.image` is not populated — this allows images with valid URLs but no media ID to render. (B) Add a null guard to `ColumnBlock` (matching the existing guard in `ColumnsBlock`) so it returns `null` when children produce no visible output — this prevents empty columns from any cause.
- **Suggested tests:** (1) Unit test: `core/image` resolver returns `IImage` from attrs when meta is absent. (2) Unit test: `ColumnBlock` returns null when no children. (3) Migration test: about pane with media lacking `mediaId` produces correct rendering. (4) Storybook: `ColumnBlock` with no children renders nothing.
- **Linear:** `synced`

## Implementation Tasks

1. **Create `core/image` resolver** — `frontend/blocks/image/resolver.ts`
   - Import `"server-only"`, `BlockResolver`, `IImage`, `BlockFrontendProps`, `BlockAttributes`, `IBlockMetadata`
   - `resolveProps`: if `meta.image` has a valid `sourceUrl`, use it (CMS-enriched path). Otherwise, if `attrs.url` is non-empty, construct `IImage` from `{ id: attrs.id?.toString() ?? "", sourceUrl: attrs.url, altText: attrs.alt ?? "", title: "" }`. If neither source has a URL, set `image: null`.
   - Spread remaining attrs (`align`, `href`, `linkClass`, `linkTarget`, `rel`, `scale`, `sizeSlug`, `style`, `width`, `height`, `aspectRatio`) plus `id` and `isInnerBlock` from context.
   - Export as `default { resolveProps } satisfies BlockResolver<BlockFrontendProps, BlockAttributes>`

2. **Register the resolver in block registry** — `frontend/lib/services/block/block-registry.ts`
   - Change the IMAGE registration from `{ view: ImageBlock }` to `{ view: ImageBlock, resolver: () => import("@frontend/blocks/image/resolver") }`

3. **Add null guard to `ColumnBlock`** — `frontend/blocks/columns/ColumnBlock.tsx`
   - Add `if (!children) { return null; }` before the return statement, matching the existing pattern in `ColumnsBlock.tsx:24`

4. **Add resolver unit tests** — `frontend/blocks/image/__tests__/resolver.test.ts`
   - Test: meta.image with valid sourceUrl → returns meta.image (CMS-enriched path preserved)
   - Test: meta absent or empty array, attrs.url present → constructs IImage from attrs
   - Test: meta.image is null (deleted attachment), attrs.url present → constructs IImage from attrs
   - Test: no meta.image, no attrs.url → returns image: null
   - Test: passes through all BlockAttributes fields (align, href, scale, etc.)

5. **Add ColumnBlock null-guard test** — `frontend/blocks/columns/__tests__/ColumnBlock.test.tsx`
   - Test: renders null when children is undefined/null (no inner blocks)
   - Test: still renders when children are provided

6. **Update Storybook stories** — `frontend/blocks/columns/ColumnBlock.stories.tsx`
   - Add story: `Empty` — ColumnBlock with no children renders nothing

## Review Issues

### ColumnBlock null guard breaks intentionally empty columns used as spacers
- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/blocks/columns/ColumnBlock.tsx:32`
- **Problem:** The `if (!children) return null` guard prevents empty columns from rendering — but content editors may intentionally leave a column empty for spacing or layout structure. This is a regression for any page using empty columns as spacers. Additionally, the plan documents that this guard can't catch the actual bug scenario (inner blocks that render to null are still truthy element references), making it defense-in-depth with a real downside.
- **Suggested fix:** Remove the null guard from ColumnBlock. The resolver is the primary fix — it ensures `core/image` renders when `attrs.url` exists. Remove the guard, its 2 tests from ColumnBlock.test.tsx, and the Empty story from ColumnBlock.stories.tsx.

### Test uses `as any` for site context
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** Replaced `{} as any` with `getMockSiteGlobals()` import
- **File:** `frontend/blocks/image/__tests__/resolver.test.ts:5`
- **Problem:** `baseContext` uses `site: {} as any` which violates the no-`any` rule. The resolver doesn't use `site`, but the type system requires it.
- **Suggested fix:** Replace with `site: {} as unknown as ISiteGlobals` or import `getMockSiteGlobals()` from `frontend/__fixtures__/site-globals.fixture.ts`.

### Missing Storybook story for ImageBlock fallback rendering
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** Created `frontend/blocks/image/ImageBlock.stories.tsx` with 5 stories
- **File:** `frontend/blocks/image/` (no `.stories.tsx` exists)
- **Problem:** No Storybook stories exist for ImageBlock. This PR adds a new rendering path (URL fallback via resolver), but there's no Chromatic visual regression coverage for it. Pre-existing gap, but the new behavior should have a story.
- **Suggested fix:** Create `frontend/blocks/image/ImageBlock.stories.tsx` with at least two stories: one with `meta.image` (CMS-enriched path) and one with `attrs.url` fallback (new path). Can be a follow-up PR.
