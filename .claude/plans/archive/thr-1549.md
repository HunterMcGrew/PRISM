# Plan: THR-1549

## Ticket

https://linear.app/tractru/issue/THR-1549/reorganize-backend-src-move-mega-menu-editor-to-apps-rename-component

## Goal

Reorganize `backend/plugins/gravity-platform-core/src/` — move the mega-menu-editor from `admin/` to a new `apps/` directory, rename misleading `index.tsx` files to descriptive PascalCase names, and document the organizing principles for both `apps/` and `admin/`.

---

## Decisions

- `mega-menu-editor/` moves from `src/admin/` to `src/apps/` — it's a standalone application, not an admin utility script. It was originally designed as a separate plugin.
- `src/apps/` is the new category for standalone editor applications that have their own component trees and state management. The upcoming theming utility (Q2 2026) will be the second app.
- `src/admin/` retains only small behavioral scripts that augment existing wp-admin pages (WooCommerce category selector, product image gallery limit, number field scroll behavior).
- Mega-menu-editor components stay in `apps/mega-menu-editor/components/` — they are feature-specific, not shared across blocks. Do NOT move them to `src/components/`.
- `index.tsx` in component directories is renamed to PascalCase component names. The `index.tsx` convention is reserved for block registration entrypoints and true barrel files.
- `src/components/` restructuring deferred — current size (~19 files) doesn't justify the import churn. Documented organizing principle is the priority.
- The directory move preserves the same depth under `src/` (`admin/mega-menu-editor/` → `apps/mega-menu-editor/`), so all external relative imports (to `../../utils/`, `../../../../types/`, etc.) remain unchanged.

### Task 5: Create architect context docs

This is a separate commit from the structural changes.

**5a. Create `.claude/architect/backend-apps.md`**

Document:
- What `src/apps/` is for (standalone editor applications with their own component trees)
- How it differs from `src/admin/` (utility scripts) and `src/blocks/` (block registrations)
- Expected internal structure for an app (bootstrap entrypoint, editor component, components/ subdirectory)
- Naming conventions (PascalCase component files, kebab-case directories)
- Current apps: mega-menu-editor. Planned: theming utility (Q2 2026).
- Import boundaries: apps can import from `src/utils/`, `src/types/`, `@frontend/lib/`, `@wordpress/*`. Apps should NOT import from `src/blocks/` or `src/components/` (shared editor components are for blocks, not apps).

**5b. Create `.claude/architect/backend-admin.md`**

Document:
- What `src/admin/` is for (small behavioral scripts that augment existing wp-admin pages)
- Current contents: WooCommerce category selector, product image gallery limit, number field scroll behavior
- The `index.tsx` entrypoint imports all scripts + the mega-menu-editor app bootstrap
- Standalone applications with their own component trees belong in `src/apps/`, not here

**5c. Update `.claude/architect/manifest.json`**

Add two entries:
```json
{
  "backend/plugins/*/src/apps/": "backend-apps.md",
  "backend/plugins/*/src/admin/": "backend-admin.md"
}
```

**5d. Update `.claude/architect/backend-editor-components.md`**

Add to the "When Adding New Components" section — explicit grouping convention:

- **Flat at root**: shared primitive controls used by 3+ blocks (CardRemoveButton, ItemSelector, ToolbarLink)
- **Feature subdirectory**: components belonging to one feature domain, used by 2+ blocks in that domain (editor-sidebar/, facets-filters/)
- **Not here**: components used by a single block stay in that block's directory. Components for standalone apps stay in that app's `components/` directory (e.g. `apps/mega-menu-editor/components/`).
