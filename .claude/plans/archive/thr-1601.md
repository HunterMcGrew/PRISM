## Decisions

- Use `hasPublished` (not `revisions.length > 0`) for context-aware copy — `hasPublished` means "published at least once," which determines recoverability. A single-publish menu has zero snapshots but the published version survives reset.
- Thread `hasPublished` as a prop through `editor.tsx → Header → ResetModal` — simple prop threading is the established pattern in this component tree. No context or store needed for a single boolean.
- Use `aria-disabled` on Reset button when checkbox is unchecked, `disabled` when resetting — per Pixel spec. Click handler already guards against unconfirmed state.
- Gray disabled state on Reset button (not opacity-on-red) — opacity approach fails contrast; gray → red on unlock feels like a real state change.
- Equal-weight Cancel and Reset buttons — filled secondary + filled destructive. Color alone differentiates intent.

## Acceptance Criteria

### Behavioral
- [ ] Given a user opens the reset modal in the mega menu editor, When the modal appears, Then a checkbox labeled "I understand this will reset my draft" is shown instead of a text input
- [ ] Given the checkbox is unchecked, When the user views the "Reset Mega Menu" button, Then it appears disabled (greyed out) and does not perform the reset when clicked
- [ ] Given the checkbox is checked, When the user views the "Reset Mega Menu" button, Then it becomes enabled and appears red (destructive style)
- [ ] Given the mega menu has been published at least once, When the reset modal opens, Then the body text reads "This will replace your current draft with the default template" and an info line reads "Your published menu is not affected. You can restore from Revisions."
- [ ] Given the mega menu has never been published, When the reset modal opens, Then the body text reads "This will replace your current draft with the default template" and a warning line reads "Since you haven't published yet, this cannot be undone."
- [ ] Given the user clicks Cancel, When the modal closes, Then no reset occurs and the editor state is unchanged
- [ ] Given the checkbox is checked and the user clicks "Reset Mega Menu", When the reset is in progress, Then the button shows "Resetting…" with a spinner and both the checkbox and buttons are disabled
- [ ] Given the reset fails, When the error occurs, Then an error message "An unexpected error occurred. Please try again." appears in the modal
- [ ] Given the user presses Tab in the modal, When navigating with the keyboard, Then focus moves in order: checkbox → Cancel → Reset Mega Menu

### Non-behavioral
- [ ] The Cancel button appears as a filled secondary button (not outline/ghost) with similar visual weight to the destructive button
- [ ] The Reset button is discoverable by screen readers when disabled (uses `aria-disabled`, not `disabled` attribute, when checkbox is unchecked)

## Design

**Status:** Ready for implementation
**Author:** Pixel

### Summary

Four visual refinements to the reset confirmation modal based on screenshot review. All are CSS/markup-level changes — no logic or prop changes needed.

### Clove — implementation notes

**1. Cancel button — filled secondary, not outline**

The Cancel button should render as a filled secondary button with similar visual weight to the Reset button. The `variant="secondary"` is already set — verify that `.components-button.is-secondary` produces a filled style. If WP is rendering it as outline/ghost, add a class override to fill it. The color difference (neutral vs. red) already communicates hierarchy — both buttons need equal *findability*.

**2. Disabled Reset button — replace opacity with gray**

Remove the `opacity: 0.6` approach on `[aria-disabled="true"]`. White text on faded red is a contrast failure. Replace with a neutral gray disabled state:

```scss
.components-button[aria-disabled="true"] {
    cursor: default;
    background-color: #ddd;
    color: #a0a0a0;
    box-shadow: none;
}
```

Gray → red on checkbox check should feel like an *unlock*. The red should mean something when it arrives, not be a faded preview of itself.

**3. X close button — no changes**

Leave it. Top-right is muscle memory (Jakob's Law). No action needed.

**4. Increase vertical spacing between content blocks**

Current spacing is too tight between body and info line (8px). Update to:

```scss
.thrive-mega-menu-reset-modal__info-line {
    margin: 1rem 0 0;    // was 0.5rem — bump to 16px
}

.thrive-mega-menu-reset-modal__checkbox {
    margin-top: 1.5rem;  // was 1rem — bump to 24px
}
```

The checkbox is the action threshold — reading ends, deciding begins. The 24px gap above it creates a visual pause before the commitment step. The 32px before the buttons (`margin-top: 2rem`) stays as-is.

Target rhythm:
```
Body text                        ← "what happens"
        16px
ⓘ Info callout                   ← "what doesn't happen"
        24px                     ← pause before commitment
☐ I understand this will...      ← decision point
        32px                     ← keep as-is
    [Cancel]  [Reset Mega Menu]
```

### Decisions worth knowing
- Gray disabled state over opacity-on-red — opacity approach fails contrast and makes the enabled transition feel weak
- Equal-weight buttons (filled secondary + filled destructive) — color alone differentiates intent; weight differentiates findability
- Wider spacing at the checkbox boundary — the action threshold deserves the largest gap in the modal

## Review Issues

### Prop ordering in ResetModalProps
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `ResetModal.tsx:14-20`
- **Problem:** `hasPublished` (core data/config) is placed before `open` and `isResetting` (control/state) in both the type definition and JSX usage — per `prop-ordering.md`, control/state props should come first.
- **Suggested fix:** Reorder to `open`, `isResetting`, `hasPublished`, then callbacks. Also update the destructuring and JSX call site in Header.tsx.
- **Fixed in:** Reordered type, destructuring, and JSX call site.

### Dead error handling path in ResetModal
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `editor.tsx:527-537`
- **Problem:** `handleReset` in `editor.tsx` catches all errors internally and never re-throws. The modal's `catch` block is unreachable dead code — on failure, the modal closes and a snackbar appears instead of the in-modal error `Notice`. AC item "Given the reset fails, Then an error message appears in the modal" cannot pass.
- **Suggested fix:** Have `handleReset` re-throw after its `createInfoNotice` call.
- **Fixed in:** Added `throw error` after `createInfoNotice` in `handleReset`'s catch block. Error now propagates to the modal, which displays the in-modal `Notice` and stays open.

### Close button selector uses locale-dependent aria-label
- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `styles.scss:165-168`
- **Problem:** `.components-button[aria-label="Close"]` targets the WP Modal close button by its English label. WP's Modal uses `__('Close')` — in a non-English locale, the label would be translated and this selector would stop matching.
- **Suggested fix:** Use `.components-button.has-icon` instead — the close button is the only icon button in the modal header.
- **Fixed in:** Replaced `[aria-label="Close"]` with `.has-icon` class selector.
