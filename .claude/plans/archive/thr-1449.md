## Debugged Issues

### TS2322: `null` vs `undefined` in BlockAttributes.icon
- **Status:** `fixed`
- **File:** `frontend/blocks/mega-menu-link-with-icon/schema.ts:27`
- **Root cause:** `icon` is typed as `keyof typeof MEGA_MENU_LINK_ICONS | null`, but all consumers (IconSelectionGrid, registerBlockType) expect `undefined` for "no value", not `null`. This causes errors at `edit.tsx:117`, `edit.tsx:120`, and `index.tsx:8`.
- **Recommended fix:** Change the type from `| null` to optional: `icon?: keyof typeof MEGA_MENU_LINK_ICONS`. This aligns with how WordPress block attributes and IconSelectionGrid model absent values (`undefined`). Verify that no code path relies on `icon === null` — if any do, update them to check `=== undefined` or `!icon`.
- **Suggested tests:** None needed — this is a type-only fix. The existing `check-types` CI step validates it.
- **Fixed in:** Changed `icon: ... | null` → `icon?: ...` in schema.ts, MegaMenuLinkWithIconProps, DynamicIconProps. Updated story from `null` to `undefined`. Added `?? "call"` fallback in index.tsx registerBlockType.

### TS2322: optional fields leak `undefined` through `||` fallback in edit.tsx
- **Status:** `fixed`
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-link-with-icon/edit.tsx:132,140`
- **Root cause:** `description` and `link` are optional in `BlockAttributes`, so `DEFAULT_ATTRIBUTES.description` and `DEFAULT_ATTRIBUTES.link` are typed as `string | undefined` and `ILink | undefined` respectively, even though the actual default object has concrete values. The `||` fallback doesn't narrow the type because TypeScript sees both operands as potentially `undefined`. At line 132, `TextControl` expects `string | number` but gets `string | undefined`. At line 140, `LinkControl` expects `LinkValue` (`string | ILink`) but gets `ILink | undefined`.
- **Recommended fix:** Use nullish coalescing with a concrete literal fallback: `value={attributes.description ?? ""}` (line 132) and `link={attributes.link ?? DEFAULT_LINK}` (line 140), where `DEFAULT_LINK` is imported from schema.
- **Suggested tests:** None needed — type-only fix validated by `check-types`.
- **Fixed in:** Exported `DEFAULT_LINK` from schema.ts. Changed edit.tsx to use `?? ""` for description and `?? DEFAULT_LINK` for link.

- [ ] Given I am in the WordPress block editor, When I search for and insert the "Thrive Mega Menu Renderer" block, Then the block appears without errors and a preview placeholder is shown in the editor
- [ ] Given a valid preview token has been generated by the mega menu editor, When I visit `/mega-menu-editor-preview?token=<token>`, Then the draft mega menu renders in the preview page without errors
- [ ] Given I am logged in as a WordPress administrator (no token), When I visit `/mega-menu-editor-preview`, Then the draft mega menu renders in the preview page without errors

- [ ] Given the mega menu has published blocks, When I view a page on the frontend that includes the mega menu, Then the desktop mega menu renders all configured panes and navigation items correctly
- [ ] Given I am viewing the site on a mobile device, When I tap the hamburger button, Then the mobile menu opens; When I tap a top-level item, Then the secondary group slides in; When I tap a second-level item with sub-links, Then the third-level links slide in
