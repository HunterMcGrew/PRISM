## Decisions

- Bug lives in `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx` (new mega menu renderer from THR-1449). **Do not modify** the legacy `frontend/blocks/site-header/MobileMegaMenu.tsx` — it is being phased out but still in use.
- Legacy `MobileMegaMenu.tsx` avoids this bug by using a subtle `hover:bg-gray-400/10` background tint on the `<li>` instead of `hover:text-primary` on the link — use as inspiration, not as a shared dependency.
- Tailwind 3.4.19 is in use with `hoverOnlyWhenSupported` not enabled — plain `hover:` variants apply on touch devices.
- Fix approach (pass 1, PR #1797): **drop `hover:text-primary` entirely** on `ITEM_CLASSES` and `iconColorClassName`; keep `focus-visible:text-primary` for keyboard affordance.
- Fix approach (pass 2): **delete the panel-transition `useEffect` entirely** — it was overriding Headless UI Dialog's built-in focus management, causing `focus-visible:text-primary` to fire on programmatic focus on mobile browsers. Headless UI already places focus on the first focusable element in the `DialogPanel` (panel chrome), which is the correct a11y pattern for dialogs.
- The extra Tab press from panel chrome to first menu item is intentional — it matches the WAI-ARIA Dialog pattern (focus on dialog → Tab to content) and gives keyboard users awareness of Back/Close controls before reaching the link list.
- Enabling Tailwind's `hoverOnlyWhenSupported` globally is the structurally-right answer for the whole codebase but out of scope — file a separate ticket if desired.

## Debugged Issues

### Mobile mega menu: link at index N appears pre-highlighted in next sliding panel
- **Status:** `fixed`
- **Fixed in:** [MegaMenuMobileMenu.tsx:32](frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L32) and [MegaMenuMobileMenu.tsx:111](frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx#L111) — removed `hover:text-primary` from `ITEM_CLASSES` and `iconColorClassName`.
- **Severity:** Medium (S3)
- **Environment:** Mobile (iOS Safari primarily), production/staging, new mega menu renderer
- **File:** `frontend/components/mega-menu-renderer/MegaMenuMobileMenu.tsx:33`
- **Root cause:** `ITEM_CLASSES` applies `hover:text-primary` without an `@media (hover: hover)` guard. On iOS Safari, tapping a button at index N leaves "sticky hover" bound to that tap coordinate. After `setActiveTopLevelIndex` triggers re-render and the tapped button unmounts, the new `<li>` element now occupying that same Y-coordinate (index N in the second-level list) inherits the `:hover` pseudo-state and renders in primary (green). The highlight clears on the next tap anywhere else. Tailwind 3.4 does not enable `hoverOnlyWhenSupported` by default, so plain `hover:` variants apply on touch devices.
- **Steps to Reproduce:**
  1. On an iOS mobile browser, open a site using the new mega menu renderer
  2. Open the hamburger menu
  3. Tap an expandable top-level item at index N (e.g. the 2nd or 3rd item)
  4. Observe the second-level panel that slides in
  5. The link at index N in the new panel renders in the active/primary color
- **Expected behavior:** No link pre-highlighted when a new sliding panel opens — primary color reserved for real `hover` (pointer devices) or a future URL-match `aria-current` style.
- **Actual behavior:** Element at the tap coordinate in the new panel renders with `hover:text-primary` styling due to sticky hover on touch devices.
- **Recommended fix:** Surgical scope — modify only `MegaMenuMobileMenu.tsx`. Do not touch the legacy file. Options, in order of preference:
  1. Gate the hover variant with an `@media (hover: hover)` guard — replace `hover:text-primary` with `[@media(hover:hover)]:hover:text-primary` on `ITEM_CLASSES` (line 33) and `iconColorClassName` (line 111). Keeps hover feedback for pointer devices, eliminates sticky-hover bleed on touch. Preferred.
  2. Drop `hover:text-primary` and `focus-visible:text-primary` entirely, substitute a subtle background-tint hover like the legacy pattern (`hover:bg-gray-400/10`). Simpler but loses the primary-color hover feedback entirely — check with design before going this route.
  3. Enable Tailwind's `hoverOnlyWhenSupported` future flag globally in `frontend/tailwind.config.*`. Clean fix for the whole repo but requires a visual-regression sweep across every `hover:*` usage — out of scope for this ticket.
- **Suggested tests:**
  - Unit: assert `ITEM_CLASSES` does not contain an unguarded `hover:` variant (or whichever approach is chosen).
  - Storybook interaction: not possible — sticky hover is a browser behavior, not reproducible in jsdom.
  - Manual: ticket AC already covers real-device verification on iOS.
