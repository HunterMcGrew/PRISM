# Plan: thr-1658

## Ticket

[THR-1658](https://linear.app/tractru/issue/THR-1658/thr-1585-followup-equipment-nav-list-migration-does-not-preserve) — Equipment nav list migration does not preserve legacy column count

## Goal

Migrate each Equipment-pane nav list with the same column count it had in the legacy frontend — preserving the legacy layout rather than imposing a hardcoded default.

---

## Implementation Tasks

### Clove (implementation)

1. Change `transformEquipmentPane()` in [backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php](backend/plugins/gravity-platform-core/includes/Services/MegaMenuMigrationService.php) to pass `columns: 2` for the `firstSection` call and `columns: 1` for the `secondSection` call — matching the legacy `EquipmentNavigationPane.tsx` rendering.
2. Update the existing `secondSection` assertions in [backend/plugins/gravity-platform-core/tests/Unit/Services/MegaMenuMigration/EquipmentPaneTest.php](backend/plugins/gravity-platform-core/tests/Unit/Services/MegaMenuMigration/EquipmentPaneTest.php) — they currently assert `columns: 2` for secondSection which was the bug.
3. Add a focused regression test asserting the firstSection → 2 / secondSection → 1 mapping explicitly, so future changes can't silently revert either side.

---

## Decisions

- **Legacy column count is per-section, not per-instance.** The legacy pane data shape (`EquipmentPaneSettings.tsx`) has no `columns` field — it never exposed column count as editor state. The legacy `EquipmentNavigationPane.tsx` hardcodes `columnCount: 2` on firstSection's `<ul>` and renders secondSection as a single `flex-col` column. The migration mirrors that per-section mapping rather than reading a non-existent data field.
- **The `columns` param on `buildNavList()` is kept, not re-added.** The ticket framed the fix as "re-add the `columns` parameter" but the param already exists (default 1) — it was the two hardcoded `2`s in `transformEquipmentPane()` that were wrong. No signature change needed.
- **The AC phrase "read from legacy source data" was based on the ticket's initial hypothesis.** The actual preservation happens by encoding each section's legacy CSS column count at the call site. Implementation still satisfies every behavioral AC (firstSection → 2, secondSection → 1, no brand-list regression, idempotent).

## Acceptance Criteria

### Behavioral

- [ ] Given a legacy mega menu where an Equipment-pane nav list rendered in N columns, When the migration runs, Then the migrated nav list is emitted with `columns: N` (preserving the legacy layout)
- [ ] Given a legacy mega menu with an Equipment By Location list (secondSection) rendered in 1 column, When the migration runs, Then the migrated secondSection nav list is emitted with `columns: 1`
- [ ] Given a legacy mega menu with an Equipment brand list (firstSection) rendered in 2 columns, When the migration runs, Then the migrated firstSection nav list is emitted with `columns: 2` (no regression from THR-1585)
- [ ] Given a dealer site where the migration has already run once, When the migration runs again on the same legacy menu, Then the output is idempotent — same column counts as the first run
