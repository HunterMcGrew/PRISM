## Debugged Issues

### MegaMenuFeaturedBanner outer container missing `h-40`, breaking `h-full` cascade
- **Status:** `fixed`
- **Fixed in:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx:59` — outer changed to `relative flex h-full min-h-40 w-full flex-col rounded-lg`
- **Severity:** Medium
- **Environment:** Frontend — Parts pane on dealer sites using the new mega menu (observed on Marshall QA)
- **File:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx:59`
- **Root cause:** The outer `<div>` was `flex min-h-40 w-full flex-col rounded-lg` — no explicit `height`, and crucially no `h-full` to inherit the pane's grid-cell height. The inner `<div className="relative flex h-full w-full ...">` uses `h-full`, which per CSS spec requires the parent's `height` to resolve — `min-height` alone does not. `h-full` computed to `auto`, so the inner collapsed to text-content height. `next/image` with `fill` is absolutely positioned against the inner `relative` container, so the image rendered short. The fix is `h-full` on the outer (not a fixed `h-40` — the Parts pane grid cell is taller than 160px in prod, and the banner must stretch to fill it). `min-h-40` remains as a floor for contexts where the parent has no definite height (e.g. standalone block rendering in the editor). `relative` on the outer is defense-in-depth so `fill` positions against it. THR-1130 (PR #1625) introduced the regression when the component was rewritten from scratch.
- **Steps to Reproduce:**
  1. Open Marshall (or any dealer site using the new mega menu) on desktop
  2. Open the mega menu and navigate to the Parts pane
  3. Observe the featured banner image on the right
  4. Compare to the legacy mega menu Parts pane on a site that still runs the legacy block
- **Expected behavior:** The banner image fills the full banner area (at least 160px tall, matching legacy `h-40 min-h-40`).
- **Actual behavior:** The image was constrained to the text-content height of the inner container (~120–130px), leaving a visible gap below the image inside the banner's `min-h-40` outer box.
- **Recommended fix:** Change outer from `flex min-h-40 w-full flex-col rounded-lg` → `relative flex h-full min-h-40 w-full flex-col rounded-lg`. `h-full` lets the banner stretch to fill its grid-cell wrapper (the Parts pane is taller than 160px, and the banner must stretch with it); `min-h-40` remains as a floor for standalone contexts; `relative` on the outer is defense-in-depth so `fill` positions against the outer box.
- **Suggested tests:** Implemented — Jest test `applies relative positioning, h-full, and min-h-40 on the outer so the image fills the allocated pane` replaces the prior anti-assertion; Storybook `Banner` play function now asserts the rendered `<img>`'s bounding rect matches the banner's (width + height), providing Chromatic coverage.
- **Linear:** `synced`

## Review Issues

### Editor preview may still show collapsed image in non-grid contexts
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx` — flattened outer+inner into a single `relative flex min-h-40 w-full items-center rounded-lg p-4` container. Image's `fill` now positions against the outer directly; no `h-full` percentage-height cascade anywhere. Flex-stretch handles grid contexts (Parts pane), `min-h-40` is the floor for standalone contexts (editor, isolated usage).
- **File:** `frontend/components/mega-menu-featured-banner/MegaMenuFeaturedBanner.tsx:59`
- **Problem:** `h-full` on the outer only resolves when the parent has a definite used height (grid-stretched flex/grid item). In the block editor via `backend/plugins/gravity-platform-core/src/blocks/mega-menu-featured-banner/edit.tsx`, the `useBlockProps()` wrapper has no definite height — so `h-full` computes to `auto`, `min-h-40` gives the outer 160px used height but computed `height` is still `auto`, and inner's `h-full` can't resolve. Image collapses to text-content height (~120–130px) inside a 160px outer, same 40px gap as the bug Clove just fixed in the Parts pane. Same root mechanism, just less visible because the outer caps at the min-height floor.
- **Suggested fix:** Out of scope for THR-1623 (user-reported surface is the Parts pane, which is fixed). Follow-up options: (1) inside `edit.tsx`, wrap the component in a container with an explicit height or `h-80`/similar so the cascade resolves in the editor; (2) simplify the component to put `fill`'s image against the outer directly (move `relative` to outer, drop inner's `h-full` dependency) — bigger refactor but eliminates the cascade dependency entirely. Worth a follow-up ticket if editor preview parity matters to content editors.

### Jest test asserts class names (implementation) rather than behavior
- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `frontend/components/mega-menu-featured-banner/__tests__/MegaMenuFeaturedBanner.test.tsx` — test updated to assert the refactor's load-bearing classes (`relative`, `min-h-40`, `items-center`, `p-4`); renamed to "applies the load-bearing layout classes on the outer so the image fills the allocated pane". Kept per Briar's guidance as a refactor tripwire.
- **File:** `frontend/components/mega-menu-featured-banner/__tests__/MegaMenuFeaturedBanner.test.tsx:158-167`
- **Problem:** `applies relative positioning, h-full, and min-h-40 on the outer so the image fills the allocated pane` asserts class-name presence — implementation, not behavior. A refactor that achieves the same visual result via different classes (e.g. inline styles, CSS-in-JS, a different Tailwind utility) would break the test without breaking the feature.
- **Suggested fix:** Defensible as-is — it functions as a tripwire to catch future refactors that silently remove the load-bearing classes (this ticket is exactly the history where that happened). The Storybook `Banner` play function covers behavior (image bounding rect equals banner bounding rect). Leave the Jest test as a class-presence tripwire; flag only if a future refactor needs to change the approach.
