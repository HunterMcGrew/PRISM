# Plan: THR-1619

## Ticket

https://linear.app/tractru/issue/THR-1619 — Mega menu navigation item — migrate link attribute from string URL to ILink via LinkControl

## Goal

Migrate the `mega-menu-navigation-item` block from `attributes.url: string` + `ControlledUrlInput` to `attributes.link: ILink` + `LinkControl`, enabling editor-controlled `target="_blank"` / `rel` on top-level Thrive mega menu nav items. Incidentally harden `LinkControl` to use site-aware `isExternalUrl(href, frontendDomain)` rather than shape-only `urlIsExternal(href)`, and surface the advanced-settings panel when a target is set so auto-behavior is visible to editors.

## Decisions

- **No JS block deprecation.** Pre-prod. Block `save()` returns only `<InnerBlocks.Content />` → no attribute-derived HTML → zero validation risk on schema swap. Worst case: staging testers re-enter a few URLs.
- **`LinkControl` fix is in scope.** Current `handleUrlChange` uses `urlIsExternal(href)` (shape-only). `ControlledUrlInput` already normalizes URLs site-aware before this runs, so the mismatch is mostly edge-case (missing `frontendDomain`, pre-normalized initial state). Swapping for `isExternalUrl(href, frontendDomain)` makes the whole stack use the same site-aware check — defense in depth.
- **`LinkControl` reads `frontendDomain` internally** via `useEntityProp("root", "site", "platform_settings_frontend_domain")`. Matches the pattern in [ControlledUrlInput.tsx:125-129](backend/plugins/gravity-platform-core/src/components/editor-sidebar/ControlledUrlInput.tsx#L125-L129). Avoids plumbing the domain through every consumer.
- **`LinkControl` opens advanced-settings panel by default when a target is set.** Initialize `settingsOpen` to `link.target !== undefined` so an auto-set or pre-set `_blank` is visible at a glance.
- **New attribute name is `link: ILink`** — matches info-banner, featured-banner, link-with-icon.
- **`link` is optional on `BlockAttributes`.** No default `link` object in `DEFAULT_ATTRIBUTES` — dropdown-type nav items store no link at all.
- **Migration rule (PHP):** `{ url: "foo" }` → `{ link: { href: "foo" } }` with `target`/`rel` unset. Editor-controlled going forward.
- **`MegaMenuNavigationItemProps` keeps its name but stops aliasing `BlockFrontendProps`.** Fields declared explicitly on the component, per `.claude/rules/component-props-decoupling.md`. Drops the `BlockFrontendProps` export from schema.
- **No LinkControl unit tests.** Per `.claude/rules/code-standards.md`: backend editor components have no Jest infrastructure. LinkControl behavior verified via Storybook + consumer-block tests + manual AC walkthrough.
- **`isExternalUrl(url, domain)` treats empty domain as internal, not a crash.** Guard added at the top of the function (`if (!domain) return false;`). Closes a latent bug where `new URL("https://")` throws outside the existing try/catch. Additive defense — no behavior change for valid inputs. Retires the pattern in `ControlledUrlInput.tsx:138-142` (external guard around `isExternalUrl`); it stays as harmless defense but is no longer required for new callers.
- **`LinkControl` advanced-settings visibility is derived, not solely state-driven.** `showAdvancedSettings = canEditAdvancedSettings && (expandedByUser || hasTarget)`. Panel shows when the user opened it manually OR when the link has a target set. Chevron is hidden when target is set (no ornamental button). Preserves the "advanced settings hidden by default" affordance while eliminating the invisible-auto-target footgun.

### Downstream impact of the LinkControl fix

Six blocks consume `LinkControl`:

- `mega-menu-info-banner`
- `mega-menu-featured-banner`
- `mega-menu-link-with-icon`
- `panel-hero-main-panel`
- `panel-hero-inner-panel`
- `hero-carousel-slide-cta`

Observable change: when `frontendDomain` IS available (normal editor conditions), behavior is effectively identical because `ControlledUrlInput` already pre-normalizes same-site URLs to relative. When `frontendDomain` is briefly unavailable (first render) or when loading saved content, the site-aware check prevents false-positive auto-`_blank` on same-site absolute URLs. Also: editors who had `_blank` pre-set now see the toggle surfaced without clicking the chevron.

**1b. Inside the component body** — read frontendDomain and widen the fallback:


Then change `handleUrlChange`:

```ts
// Before
const handleUrlChange = (href: string) => {
  const nextLink = setLinkHref(link, href);
  onChange(
    setLinkTarget(nextLink, urlIsExternal(href) ? "_blank" : undefined)

};

// After
const handleUrlChange = (href: string) => {
  const nextLink = setLinkHref(link, href);
  const isExternal = isExternalUrl(href, frontendDomain ?? "");

};
```

`isExternalUrl("", "")` returns false — safe empty-string fallback when the entity prop hasn't resolved yet.

```ts
// After

const [settingsOpen, setSettingsOpen] = useState<boolean>(
  linkObject.target !== undefined
);
```

Note: `linkObject` is already computed on the next line — move the computation above `useState` so the initial state can reference it. Drop the now-duplicate `linkObject` declaration below.

### Clove — Phase 6: PHP migration service

```php
// Before
```

## Review Issues

### LinkControl `isExternalUrl(href, "")` can throw on empty domain

- **Severity:** `critical`
- **Status:** `fixed`
- **Fixed in:** root cause addressed in `frontend/lib/utilities/url-utilities.ts` — added `if (!domain) return false;` guard at the top of `isExternalUrl`. LinkControl reverted the `?? ""` fallback to pass `frontendDomain` through unchanged (now safe). New test case in `frontend/lib/utilities/__tests__/url-utilities/is-external-url.test.ts`.
- **File:** `backend/plugins/gravity-platform-core/src/components/editor-sidebar/LinkControl.tsx:53`
- **Problem:** `isExternalUrl(href, frontendDomain ?? "")` passes an empty domain to `isExternalUrl` when `useEntityProp` hasn't resolved yet (first render, before WP `core` store fetches the site record). Inside the utility, line 61 runs `new URL(base).hostname` where `base = "https://"` — that `new URL("https://")` throws `TypeError: Invalid URL`, and it's OUTSIDE the try/catch (lines 63-80). Result: typing a URL in LinkControl during the brief window before the entity prop resolves crashes `handleUrlChange`, `onChange` never fires, and the editor's edit silently fails to persist. Confirmed `new URL("https://")` throws in Node. Plan's risk analysis cited "line 58: `if (!trimmed) return false`" but that guard is for the url arg, not the domain arg.
- **Suggested fix:** Match the defensive pattern already in `ControlledUrlInput.tsx:138-142` — guard on truthy `frontendDomain` before calling `isExternalUrl`:
  ```tsx
  const isExternal = frontendDomain
    ? isExternalUrl(href, frontendDomain)
    : false;
  ```
  Ripples to zero other files. Aligns with the existing defensive pattern in the same folder.

### LinkControl `settingsOpen` doesn't react to target changes after mount

- **Severity:** `major`
- **Status:** `fixed`
- **Fixed in:** `backend/plugins/gravity-platform-core/src/components/editor-sidebar/LinkControl.tsx`. Replaced `settingsOpen` initial-target state with `expandedByUser` state + `showAdvancedSettings = canEditAdvancedSettings && (expandedByUser || hasTarget)` derivation. Chevron now also gated on `!hasTarget` so it doesn't appear as an ornamental button when the panel is already open. Winston's refinement — prevents a dead-button UX regression that would have replaced the invisibility bug.
- **File:** `backend/plugins/gravity-platform-core/src/components/editor-sidebar/LinkControl.tsx:41-43`
- **Problem:** `useState<boolean>(linkObject.target !== undefined)` initializes once on mount. When a user pastes an external URL, `handleUrlChange` auto-sets `target="_blank"` via `onChange`, the parent re-renders with the new link value, and LinkControl receives the new `link` prop — but `settingsOpen` stays `false` because React's `useState` initializer only runs once. Net effect: the AC "the toggle auto-sets to on AND the advanced-settings panel is expanded so I can see and override" is only satisfied on the next editor session after a reload, not on the user's paste action. The "invisible auto-behavior" this fix was meant to eliminate still happens on the type-triggered path.
- **Suggested fix:** Derive panel visibility instead of tracking it solely in state:
  ```tsx
  const [settingsOpen, setSettingsOpen] = useState<boolean>(false);
  const showAdvancedSettings = canEditAdvancedSettings && (settingsOpen || linkObject.target !== undefined);
  ```
  Then render the panel when `showAdvancedSettings` is true. Chevron click still toggles `settingsOpen`; if target is set, panel stays visible regardless. Satisfies the AC on both mount-with-target and type-triggers-target paths.

### Prop field ordering in `MegaMenuNavigationItemProps`

- **Severity:** `minor`
- **Status:** `fixed`
- **Fixed in:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx`. Swapped `link` and `isEditor` positions in both the interface and the destructure. Order is now `title, type, isEditor, link, children`.
- **File:** `frontend/components/navigation-menu-item/MegaMenuNavigationItem.tsx:19-25, 40-45`
- **Problem:** Per `.claude/rules/prop-ordering.md`: identity → variant → control/state → data → optional → callbacks. Current interface and destructure order is `title, type, link, isEditor, children`. `isEditor` is control/state and should precede `link` (data). Correct order: `title, type, isEditor, link, children`.
- **Suggested fix:** Move `isEditor?: boolean` above `link?: ILink` in both the interface and the destructure.
