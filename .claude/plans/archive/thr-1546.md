# Plan: THR-1546

## Ticket

https://linear.app/tractru/issue/THR-1546/centralize-feature-flag-reads-into-featureflags-service-class

## Goal

Centralize scattered feature flag reads into a single `FeatureFlags` service class with consistent truthiness logic, eliminating a latent bug where `blocks.php` and `functions.php` treat empty strings as "enabled."

---

## Decisions

- All-static class — `FeatureFlags` has no state (reads from constant + `get_option`), so instance creation is unnecessary. Diverges from the handler pattern (which needs DI) but matches the utility nature of the class.
- Canonical truthiness: `!empty($value) && $value !== '0'` — aligns with the existing GraphQL resolver behavior that has been shipping to frontends. Covers ACF true/false fields (`'1'`/`'0'`) and the `get_option` default (`false`).
- MegaMenuEditor migration deferred — `MegaMenuEditor.php` only exists on the THR-1448 branch. This PR ships the service + 3 migrations on `main`. THR-1448 adopts `FeatureFlags::is_enabled()` after this merges.
- Developer docs live in `docs/content/dev/architecture/feature-flags.md` — it's an architectural pattern (ship strategy), not a reference lookup. Agent-facing spec stays in `.claude/architect/php-classes.md`; the two cross-reference but don't duplicate.
- `get_gated_blocks()` and `get_gated_menus()` are per-flag primitives, not aggregate methods — callers iterate flags themselves. Keeps the API minimal per the ticket's "no framework" constraint.
- `register()` moves GraphQL field registration from procedural `settings.php` into the class — wired via `App::registerCoreComponents()`.
- Flag registry moves from global `define()` in `gravity-platform-core.php` to `FeatureFlags::FLAGS` class constant — follows `PrivateSettings::SETTINGS` precedent, co-locates data with the only class that reads it. Extract to a separate file if it crosses ~10 flags.
- Descriptions use plain strings (no `__()`) — class constants must be compile-time evaluable, and GraphQL descriptions are developer-facing, not user-facing.
- No aggregate convenience methods (e.g. `get_disabled_blocks()`) — those are consumer-specific queries, not flag system concerns. Callers iterate `FeatureFlags::FLAGS` themselves.

## Implementation Tasks

1. **Create `includes/Settings/FeatureFlags.php`** — `Thrive_Core\Settings\FeatureFlags` with four static methods:
   - `is_enabled(string $flag_key): bool`
   - `get_gated_blocks(string $flag_key): array`
   - `get_gated_menus(string $flag_key): array`
   - `register(): void` (wires GraphQL fields)

2. **Wire into `App.php`** — add `use Thrive_Core\Settings\FeatureFlags;` and `FeatureFlags::register();` in `registerCoreComponents()`.

3. **Migrate `settings.php`** — remove `gravity_platform_register_feature_flag_fields()` function (lines 137-155) and its `add_action` hook (lines 129-131). Leave all `THRIVE_PLATFORM_PUBLIC_SETTINGS` logic untouched.

4. **Migrate `blocks.php`** — replace the feature flag loop (lines 186-199) with calls to `FeatureFlags::is_enabled()` and `FeatureFlags::get_gated_blocks()`. Add `use` statement at top.

5. **Migrate theme `functions.php`** — replace the feature flag loop (lines 37-52) with calls to `FeatureFlags::is_enabled()` and `FeatureFlags::get_gated_menus()`. Add `use` statement at top.

6. **Write unit tests** — `tests/Unit/Settings/FeatureFlagsTest.php`:
   - `is_enabled()` with `'1'`, `'0'`, `false`, `''`, `null` — verify canonical truthiness
   - `is_enabled()` with nonexistent flag key
   - `get_gated_blocks()` / `get_gated_menus()` with valid key, missing key, key with no gated items
   - Mock `get_option` via Brain\Monkey, define `THRIVE_FEATURE_FLAGS` in test setup

7. **Migrate `MegaMenuEditor.php`** — replace `checkIfMegaMenuIsEnabled()` body with `FeatureFlags::is_enabled('enable_mega_menu')`. Add `use` statement. Update test constant to include both flags.

8. **Update mega menu PR merge order** — when the PR is created, target this PR at the THR-1448 branch.

9. **Create developer docs for feature flags** — `docs/content/dev/architecture/feature-flags.md`:
   - Purpose: what feature flags are in this codebase (ACF on/off toggles, not gradual rollout)
   - When to use: shipping partial features, per-dealer opt-in, kill switches
   - End-to-end flow: ACF field → `wp_options` → `FeatureFlags::is_enabled()` → GraphQL Boolean → frontend `siteGlobals`
   - Block/menu gating: how `blocks.php` and `functions.php` iterate the registry
   - Adding a new flag: step-by-step checklist (ACF → constant → frontend interface → automatic GraphQL)
   - Shipping workflow: create flag first (defaults off), ship code across PRs, enable on staging for QA, enable per-dealer
   - Cross-references to `.claude/architect/php-classes.md` and `FeatureFlags.php`

10. **Update `docs/content/dev/architecture/_meta.js`** — add `"feature-flags": "Feature Flags"` entry.

12. **Move flag registry to class constant** — add `const FLAGS = [...]` to `FeatureFlags.php` with the same data currently in the `define()`. Drop `__()` wrappers on descriptions (plain strings). Remove the `define('THRIVE_FEATURE_FLAGS', ...)` from `gravity-platform-core.php`.

13. **Update `FeatureFlags` internals** — replace all `THRIVE_FEATURE_FLAGS` references with `self::FLAGS`. Remove all `defined('THRIVE_FEATURE_FLAGS')` guards — the constant always exists on the class.

14. **Update `blocks.php`** — replace `THRIVE_FEATURE_FLAGS` with `FeatureFlags::FLAGS`. Remove the `defined()` guard. Add `use Thrive_Core\Settings\FeatureFlags;` if not already present.

15. **Update theme `functions.php`** — same as task 14: `FeatureFlags::FLAGS` instead of `THRIVE_FEATURE_FLAGS`, drop `defined()` guard, add `use` statement.

16. **Update tests** — remove the `define('THRIVE_FEATURE_FLAGS', ...)` blocks from `FeatureFlagsTest.php` and `MegaMenuEditorTest.php`. Tests now read from `FeatureFlags::FLAGS` directly. Verify all existing tests still pass with no behavior change.

## Acceptance Criteria

### Behavioral

- [ ] Given a feature flag is toggled ON in WordPress admin (Platform Settings), When the GraphQL `thrive` settings are queried, Then the flag field returns `true`
- [ ] Given a feature flag is toggled OFF in WordPress admin, When the GraphQL `thrive` settings are queried, Then the flag field returns `false`
- [ ] Given the "Enable Mega Menu" flag is OFF, When the block editor loads, Then the mega menu blocks (mega-menu-column, mega-menu-columns, mega-menu-pane) are not available in the block inserter
- [ ] Given the "Enable Mega Menu" flag is ON, When the block editor loads, Then the mega menu blocks are available in the block inserter
- [ ] Given the "Enable Header" flag is OFF, When viewing the Menus settings in WordPress admin, Then the Mobile Menu and Brow Menu locations are not registered
- [ ] Given the "Enable Header" flag is ON, When viewing the Menus settings in WordPress admin, Then the Mobile Menu and Brow Menu locations are available
- [ ] Given a feature flag has never been set (option does not exist in the database), When the system checks the flag, Then it is treated as disabled

### Non-behavioral

- [ ] All four truthiness checks (blocks.php, settings.php GraphQL, functions.php, and the new FeatureFlags class) produce identical results for the same option value
- [ ] Unit tests cover edge cases: `'1'`, `'0'`, `false`, empty string, `null`, nonexistent flag key
