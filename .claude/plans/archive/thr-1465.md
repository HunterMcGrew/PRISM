# Plan: THR-1465

## Ticket

THR-1465 — Events Calendar block breaks in editor

## Goal

Fix the "invariant expected app router to be mounted" runtime error that crashes the Events Calendar block preview in the WordPress block editor.

---

## Decisions

- Lifted `useRouter` out of the component into an optional `onEventClick` callback prop — keeps the shared component free of Next.js runtime APIs, which don't exist in the wp-admin editor context.
- Moved and renamed `EventsCalendarBlock` → `EventsCalendar` in `frontend/components/events-calendar/` — follows the component-props-decoupling rule that components live in `components/`, not `blocks/`.
- Renamed `BlockFrontendProps` → `EventsCalendarProps` — follows new code standard that Props types are named after the component, not the block system.
- Created `EventsCalendarBlock.view.tsx` as a thin `"use client"` wrapper that provides the router callback for the Next.js frontend — follows the `.view.tsx` convention.
- Editor `edit.tsx` imports the component directly (not the view wrapper) and documents why in a code comment — the view wrapper uses `useRouter` which requires Next.js app router context.
- Replaced deprecated `withSiteGlobals` + manual `useState`/`useEffect` with `usePreviewProps` in editor `edit.tsx` — consistent with other blocks (events-list, showroom, equipment-list, etc.).
- Added code standard: always define a named Props type (`Props` or `{Name}Props`), never inline prop types in function signatures.
