##### Task 16 — Append new mappings to `frontend/lib/constants/theme.ts`

Edit the existing `theme.ts` — add two exports directly AFTER the existing `BLOCK_ALIGNMENT_MAPPINGS` and `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` declarations (keep them grouped as "alignment mappings"). The `BlockToolbarAlignment` type is already declared earlier in the file — reuse it via the same satisfies pattern.

Exact content to append (tab indentation, matching `theme.ts` convention):

```ts
/**
 * Horizontal alignment classes for a Columns container that is always flex-row.
 *
 * Used by the mega menu columns editor preview (parent always flex-row at every
 * breakpoint) and by ColumnsBlock when isStackedOnMobile=false. Distributive
 * values (left/center/right) include an arbitrary-variant [&>*]:flex-none
 * [&>*]:w-auto override — Columns-container children carry flex-1 w-full
 * defaults that fight main-axis distribution; overriding those on the parent
 * is the mechanism that gives justify-* slack to act on. Stretch values
 * (wide/full) omit the override because their children are meant to fill.
 */
export const HORIZONTAL_ALIGNMENT_FLEX_ROW_CLASSES = {
	left: "justify-start [&>*]:flex-none [&>*]:w-auto",
	center: "justify-center [&>*]:flex-none [&>*]:w-auto",
	right: "justify-end [&>*]:flex-none [&>*]:w-auto",
	wide: "justify-stretch",
	full: "justify-stretch",
} as const satisfies Record<BlockToolbarAlignment, string>;

/**
 * Horizontal alignment classes for a Columns container that flips flex-col → flex-row at md.
 *
 * Used by ColumnsBlock when isStackedOnMobile=true. Below md the container is
 * flex-col, so horizontalAlignment targets the cross-axis (items-*); at md+ it
 * flips to flex-row where horizontalAlignment is the main-axis (justify-*) and
 * the distributive child-sizing override engages via the md: prefix only.
 */
export const HORIZONTAL_ALIGNMENT_STACKED_CLASSES = {
	left: "max-md:items-start md:justify-start md:[&>*]:flex-none md:[&>*]:w-auto",
	center:
		"max-md:items-center md:justify-center md:[&>*]:flex-none md:[&>*]:w-auto",
	right: "max-md:items-end md:justify-end md:[&>*]:flex-none md:[&>*]:w-auto",
	wide: "max-md:items-stretch md:justify-stretch",
	full: "max-md:items-stretch md:justify-stretch",
} as const satisfies Record<BlockToolbarAlignment, string>;
```

Placement: directly after the existing `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` block (and before `MAIN_AXIS_ALIGN_MAPPINGS` if it follows, otherwise at the natural next position in the alignment-mappings cluster). Clove: re-read the current `theme.ts` to confirm the exact insertion line — line numbers here reference the concept, not a specific line.

The JSDoc explicitly calls out the `[&>*]:` selector shape because the rest of `theme.ts` uses pure class mappings; without the comment, a future reader scanning the file wonders why two of the mappings look different. No other file edit is required for Task 16 — the barrel (`frontend/lib/constants/index.ts`) uses `export *`, so the new exports are automatically picked up.

- **Mirror the child block pattern exactly.** Add `horizontalAlignment` attribute + `BlockAlignmentToolbar` + map via `BLOCK_ALIGNMENT_MAPPINGS` (items-*), matching the child's implementation from THR-1410. Reason: the ticket's explicit ask is parity with the child; any divergence widens scope and risks regressing existing layouts.
- **Do not touch the existing `verticalAlignment` mapping on the parent.** See Structural Concern #1 below — the existing parent `verticalAlignment → justify-*` produces horizontal-distribution behavior on desktop (flex-row). This is pre-existing from THR-1410 and likely has real content depending on it. Fixing the axis names is out of scope for a 2-point polish ticket.
- **Attribute lives in `index.tsx`, not `block.json`.** Per the THR-1410 decision, all mega-menu block attributes are declared in `index.tsx` so they can reference TypeScript types.
- **Type key uses `keyof typeof BLOCK_ALIGNMENT_MAPPINGS`.** Same typing as the child block's `horizontalAlignment` so both blocks share the same key set (`left | center | right | wide | full`).
- **Apply both alignment classes to `useBlockProps` in both parent and child `edit.tsx`.** Reason: AC #2 requires the editor preview to reflect alignment changes, and the previous version only highlighted the toolbar selection without updating the visible column layout. Both blocks now apply `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS[verticalAlignment]` and `BLOCK_ALIGNMENT_MAPPINGS[horizontalAlignment]` inline via `classNames()` in the `useBlockProps` call. Scope expansion accepted — child block was also updated for consistency even though it's not named in the ticket.
- **Do NOT change the mapping constants themselves.** The parent `ColumnsBlock` view uses `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS → justify-*` which is semantically inverted on flex-row (main-axis = horizontal). That pre-existing axis-naming issue (Structural Concern #1) is out of scope for this ticket — fixing it has a large blast radius across every live site using mega menu verticalAlignment settings and deserves its own ticket with proper QA.
- **Reversed the above decision — added flex-row mappings in this ticket.** User confirmed on 2026-04-16 that axis inversion must be fixed in this ticket. Added `BLOCK_ALIGNMENT_ROW_MAPPINGS` / `BLOCK_VERTICAL_ALIGNMENT_ROW_MAPPINGS` instead of mutating the existing constants (blast-radius preservation). Shared `ColumnsBlock.tsx` handles the flex-col → flex-row breakpoint flip via `max-md:` + `md:` class pairs so behavior is correct in both viewports. The existing flex-col mappings are unchanged; the child `mega-menu-column` block still uses them correctly.
- **Resolved — replaced `*_ROW_MAPPINGS` with `MAIN_AXIS_ALIGN_MAPPINGS` / `CROSS_AXIS_ALIGN_MAPPINGS`.** Hunter's 2026-04-17 lead-dev review flagged the two new constants as redundant with existing primitives. The axis-agnostic utils are the correct shape — consumer picks which axis its horizontal/vertical attribute targets based on container direction. Reverses the prior "added flex-row mappings" decision. Child block + `extend-column-block` migration off `BLOCK_ALIGNMENT_MAPPINGS` stays out of scope per the plan's fix scope guardrail.
- **Skipped plan step 4 (tightening `MAIN_AXIS_ALIGN_MAPPINGS` / `CROSS_AXIS_ALIGN_MAPPINGS` to `as const satisfies Record<...>`).** Reason: `frontend/blocks/group/GroupBlock.tsx:192-201` (`getFlexAlignmentClasses`) indexes *both* MAIN_AXIS and CROSS_AXIS with the same variable whose type union spans `keyof typeof CROSS_AXIS_ALIGN_MAPPINGS | keyof typeof MAIN_AXIS_ALIGN_MAPPINGS`. Tightening MAIN_AXIS/CROSS_AXIS would make `CROSS_AXIS_ALIGN_MAPPINGS[alignment]` fail type-check for `alignment = "space-between"` (CROSS_AXIS doesn't have that key) — a pre-existing latent mismatch masked by the loose `Record<string, string>` typing. Surfacing and fixing it is out of this ticket's scope. The columns schema keeps `keyof typeof BLOCK_ALIGNMENT_MAPPINGS` / `keyof typeof BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` typing — no schema change needed because the key sets are already correct.
- **Extract `toolbarToAxisKey` pure translator** (post-rework, 2026-04-17). Reason: two active callsites (`ColumnsBlock.tsx` with inline ternary, `mega-menu-columns/edit.tsx` with silent `wide|full → undefined` bug) need the same source-vocab → axis-vocab translation. A single named function closes the bug and removes drift risk. Pure translator beats composed getter — getters calcify, primitives compose. Pattern mirrors the `dtoTo*()` mappers in `frontend/lib/services/` (external vocab → domain vocab at the boundary).
- **Place the helper in `frontend/lib/utilities/block-alignment-utilities.ts`, not `theme.ts`.** Initial placement was `theme.ts` co-located with `MAIN_AXIS_ALIGN_MAPPINGS`, which required relaxing the "no functions in constants files" convention. Hunter's review flagged that `block-spacing-utilities.ts` is the stronger analog — same pattern (block attribute input → class-adjacent output), same naming convention. Utility-file placement keeps `theme.ts` purely declarative, gives future alignment helpers (rename + GroupBlock rework + type tightening in follow-up ticket) a natural landing zone, and avoids the constants-file convention bend. Architect context (`frontend-constants.md` § Source-vocab translation) updated accordingly — the "no functions" rule is restored, and the pattern now prescribes utility-file placement.
- **`BlockToolbarAlignment` is a named literal union in `theme.ts`, modelling WP's upstream contract — not derived via `keyof typeof` from any local mapping.** Reason (Winston 2nd review): the toolbar is the stable upstream source of the vocabulary, and our local mappings are transient. Deriving the type from `BLOCK_ALIGNMENT_MAPPINGS` inverts the dependency (mapping defines vocab) and chains the type to a const that's scheduled for rename/replacement in the follow-up ticket. WP's `@wordpress/block-editor` doesn't ship a TS type for `BlockAlignmentToolbar`'s output, so we declare our own as the canonical representation of the external contract. `BLOCK_ALIGNMENT_MAPPINGS` constrains itself with `satisfies Record<BlockToolbarAlignment, string>` — the mapping conforms to the type, not the reverse. `satisfies` prevents drift at compile time. Perceived literal-union duplication is a TS artifact, not a source-of-truth problem.
- **Utility imports `BlockToolbarAlignment` from constants; does not re-export or re-derive it.** Reason: constants are leaf dependencies; utilities build on them. The type lives with the mapping it describes, which keeps the dependency graph flowing the natural direction (utility → constants) and gives every consumer (current utility, future schema migrations, any new utility) a single canonical import location.
- **Do not extract `BlockVerticalToolbarAlignment` in this pass.** Reason: only one consumer (the vertical mapping's own schema usage). Rule-of-three unmet. The follow-up type-tightening ticket is the natural place to revisit if a second consumer materializes.
- **Encode the distributive child-sizing override into the horizontal alignment class mapping itself** (2026-04-18 reopen). Reason: `isDistributiveAlignment` as a derived boolean sitting alongside the alignment class says the same thing twice — "if horizontalAlignment is left|center|right, emit `justify-*` AND also emit the override." Collapsing both into a single mapping entry keyed by `BlockToolbarAlignment` removes the duplicate predicate, eliminates any risk of editor/frontend drift, and makes the mapping itself the single source of truth for "what classes are needed to make this alignment render correctly in a Columns container." Both call-sites (frontend `ColumnsBlock.tsx`, editor `mega-menu-columns/edit.tsx`) just index into the mapping with no conditional logic.
- **Delete `toolbarToAxisKey` — zero remaining consumers under the encoded-mapping shape.** Reason: the utility existed to bridge toolbar vocab (`wide|full`) to axis vocab (`stretch`) when indexing the axis-agnostic `MAIN_AXIS_ALIGN_MAPPINGS`. The new `HORIZONTAL_ALIGNMENT_FLEX_ROW_CLASSES` mapping speaks toolbar vocab natively — `wide` and `full` are just keys that map to `"justify-stretch"` with no override. Both previous consumers lose their call. Rule of three says delete when zero users remain; preserving it "just in case" fills utility files with phantom helpers nobody's sure are still correct.
- **New mappings live in `frontend/lib/constants/theme.ts` alongside the existing alignment mappings.** Reason (revised after Hunter pushback): `theme.ts` already hosts container-specific mappings (`BLOCK_ALIGNMENT_MAPPINGS`, `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` — both bake flex-col assumptions into their values), so the "keep container-specific concerns out of theme.ts" argument doesn't hold against existing precedent. One shared location for every alignment mapping in the codebase beats scattering them across block-local files: a developer scanning for "what alignment options do we have?" finds them all in one place, and the editor imports via `@frontend/lib/constants` (already an established import path) rather than a new block-file import. The `[&>*]:` selector shape is unusual for `theme.ts` contents — JSDoc above each new mapping calls out why it reaches into children so the file stays legible.
- **Remove inline `style={{ width: "100%", flex: "1" }}` from editor child wrapper; remove inline `flexBasis: "100%"` default from frontend child.** Reason: inline styles always beat class rules regardless of selector specificity, so the parent's `[&>*]:flex-none [&>*]:w-auto` override can never win while the children set their own sizing inline. The `flex-1 w-full` classes (editor) and a new `w-full` class (frontend default) encode the same intent but are overridable. Custom `width` attribute on the frontend child stays inline — that's load-bearing for user-set column widths.
- **Add `h-full` to both editor and frontend child column className.** Reason: for child `verticalAlignment` (`justify-*` on a flex-col child) to have visible effect, the child needs slack on the cross-axis (its height). Defaults collapse the child to content-height once the parent sets `items-*`. `h-full` gives the child explicit row-height regardless of the parent's cross-axis alignment — the mechanical fix for the child-vertical-does-nothing bug.
- **No new prop on `ColumnsBlock` for the override.** Reason: the override is a consequence of which `horizontalAlignment` was picked, not an independent dimension. Exposing it as a prop would split the contract (callers would have to keep two fields in sync) and invite drift (legal-but-wrong combinations like `horizontalAlignment="center", distributiveChildOverride={false}`). Deriving locally from the attribute — or better, encoding into the class mapping — keeps the contract single-sourced.
- **Do not rename `BLOCK_ALIGNMENT_MAPPINGS` / `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` in this ticket.** Reason: names are source-coupled (known anti-pattern per the 2026-04-17 lesson) but rename has broad import reach for zero behavior change. Grandfathered in the architect context with a note. Sequence the rename after `GroupBlock.getFlexAlignmentClasses` rework and axis-mapping type-tightening — all three land in the same follow-up ticket.
- **`GroupBlock.getFlexAlignmentClasses` dual-indexing flagged as follow-up ticket.** Reason: it indexes both `MAIN_AXIS_ALIGN_MAPPINGS` and `CROSS_AXIS_ALIGN_MAPPINGS` with the same variable, violating the "consumer picks which axis" rule. Not in scope for THR-1610; needs its own ticket before axis mappings can be tightened. Tracked as a Cleanup Item below.
- **Relocate Columns-specific alignment mappings out of `theme.ts` into `frontend/blocks/columns/alignment-classes.ts`** (2026-04-18, Winston review on PR #1800, Hunter confirmed). Reason: `HORIZONTAL_ALIGNMENT_FLEX_ROW_CLASSES`, `HORIZONTAL_ALIGNMENT_STACKED_CLASSES`, and the inline `VERTICAL_ALIGNMENT_STACKED_CLASSES` conform to a cross-block contract type (`BlockToolbarAlignment`) but their *values* bake in three Columns-specific commitments — hard-coded `md:` breakpoint, `[&>*]:flex-none [&>*]:w-auto` override tuned for `ColumnBlock` / `mega-menu-column` `flex-1 w-full` child defaults, and vocab closure on `BlockToolbarAlignment` exactly. Verified `GroupBlock` (the other stacking block) cannot reuse them: it stacks at `lg:`, its children don't carry `flex-1 w-full` (so the override would strip widths rather than neutralize them), and its alignment input comes from `layout.justifyContent` which includes keys outside `BlockToolbarAlignment` (`space-between`, `stretch`). CCP framing: these mappings change when Columns changes, not when other blocks change. CRP framing: the axis primitives (`MAIN_AXIS_ALIGN_MAPPINGS`, `CROSS_AXIS_ALIGN_MAPPINGS`) and `BlockToolbarAlignment` itself *are* reused across blocks — they keep `theme.ts` membership; these mappings don't, so they don't. Reverses the 2026-04-18 decision "New mappings live in `frontend/lib/constants/theme.ts`" — superseded by the CCP/CRP distinction that wasn't articulated at the time.
- **`BlockToolbarAlignment` stays in `theme.ts`.** Reason: the contract type is cross-cutting — any block consuming `BlockAlignmentToolbar` needs it, and keeping it in one canonical location prevents re-derivation in downstream code. Only the block-specific mapping *values* move; the contract type is unaffected.

## Structural Concerns

### 1. Parent-block axis mapping is semantically inverted (pre-existing)

**What.** On the shared `ColumnsBlock` view, `verticalAlignment` is mapped via `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` (`justify-*`). In a flex-row container (desktop), `justify-*` is the main axis = horizontal distribution, not vertical. The existing "Vertical Alignment" toolbar on `mega-menu-columns` is therefore controlling horizontal layout on desktop.

**When it hurts.** After this ticket ships, a content editor will see two toolbars — "Vertical Alignment" and "Horizontal Alignment" — that both affect horizontal layout on desktop, one via `justify-*` and one via `items-*` (cross-axis = vertical on flex-row). The effect will be confusing: "Horizontal Alignment" will actually shift columns vertically on desktop, while "Vertical Alignment" shifts them horizontally.

**Why not fix it here.** Fixing the axis mapping (swap `BLOCK_ALIGNMENT_MAPPINGS` ↔ `BLOCK_VERTICAL_ALIGNMENT_MAPPINGS` usage in `ColumnsBlock.tsx` only) would change the visible behavior of existing `verticalAlignment` settings on every site using the mega menu. That's a visual regression that needs its own ticket, QA pass, and migration plan.

**Recommendation.** File a follow-up ticket to rationalize parent-block axis mappings. Tagging: mega menu, technical debt, medium priority. Flagging in the plan's Decisions section is sufficient for now.

### 2. `BLOCK_ALIGNMENT_MAPPINGS` key set includes `wide | full` that don't map cleanly to item alignment

**What.** `BLOCK_ALIGNMENT_MAPPINGS` maps `wide` and `full` to `items-stretch`. WordPress's `BlockAlignmentToolbar` typically uses `wide` / `full` to indicate the block should occupy extra viewport width — not stretch its children. This is pre-existing from THR-1410.

**When it hurts.** If a content editor picks "Wide" on the Columns block thinking it will make the menu wider, they'll actually get `items-stretch` applied to the column row. Confusing UI but not broken.

**Why not fix it here.** Pre-existing behavior from THR-1410. Out of scope.

### Parent horizontalAlignment has no visible effect; child verticalAlignment has no visible effect (post-THR-1610)
- **Status:** `fixed`
- **Severity:** Medium
- **Environment:** WordPress block editor (primary report); frontend likely affected for same reasons
- **File:** `backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/edit.tsx:94-97` (primary), `backend/plugins/gravity-platform-core/src/blocks/mega-menu-columns/edit.tsx:68` (related), `frontend/blocks/columns/ColumnsBlock.tsx` (frontend missing equivalent override)

- **Root cause:** Not a mapping bug — the axis-to-class mappings are correct (parent flex-row: vertical→`items-*`, horizontal→`justify-*`; child flex-col: vertical→`justify-*`, horizontal→`items-*`). Two distinct layout bugs defeat the visible effect:

  **(a) Parent horizontal:** the child `mega-menu-column/edit.tsx` wrapper at [edit.tsx:96](backend/plugins/gravity-platform-core/src/blocks/mega-menu-column/edit.tsx#L96) carries an inline `style={{ width: "100%", flex: "1" }}`. That reasoning was class-vs-class only — **inline `style` attributes beat any stylesheet rule regardless of specificity**, so children still fill the main axis and `justify-*` has no slack to distribute. Eric's third-pass review explicitly verified class specificity but did not consider the inline style. The frontend child `ColumnBlock.tsx:69` has the analogous problem via `style={{ flexBasis: width || "100%" }}` — when no custom `width` is set, the default `100%` inline flex-basis beats any class-level override the parent could emit.

  **(b) Child vertical:** mapping emits `justify-*` on the child's `flex-col` container correctly. But `justify-content` only has visible effect when there is leftover main-axis (vertical) space. In practice the column's height is (i) stretched by the parent to tallest-sibling height under default `align-items: stretch`, or (ii) collapsed to content height when the parent has any non-stretch `items-*` set. Unless one column is visibly shorter than its siblings and a vertical alignment is set on it, the class renders invisibly. No slack → no effect.

- **Steps to Reproduce:**
  1. Insert a Mega Menu Columns (parent) block with 2+ Mega Menu Column (child) blocks inside.
  2. Select the parent Columns block; pick any Horizontal alignment (Left/Center/Right). Observe: the `[&>*]:flex-none [&>*]:w-auto` class is present on the wrapper in devtools, but child columns still render full-width because their inline `flex: 1; width: 100%` wins. No horizontal shift.
  3. With Horizontal still invisible, set Vertical alignment. Observe: works as expected (cross-axis `items-*` on flex-row works regardless of main-axis slack).
  4. Select a child Column; pick Horizontal alignment. Observe: works (cross-axis `items-*` on flex-col positions inner blocks horizontally within their line).
  5. On the same child, pick Vertical alignment (Top/Center/Bottom). Observe: usually no visible change, because the column height equals its content or its tallest sibling — no vertical slack for `justify-*` to distribute.
- **Expected behavior:** Parent Horizontal alignment shifts columns along the horizontal axis; child Vertical alignment shifts the child's inner blocks along the vertical axis.
