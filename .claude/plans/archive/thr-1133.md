# Plan: THR-1133

## Ticket

https://linear.app/tractru/issue/THR-1133/our-mission-block-for-mega-menu

## Goal

Add the `mega-menu-about-us` block ‚Äî a headline + description + alignment block that renders inside Mega Menu V2 column panes.

---

## User Stories

- As a content editor, I want to add an "About Us" or "Our Mission" block inside a Mega Menu pane, so that I can introduce context or brand messaging alongside navigation links.
- As a content editor, I want to set alignment (left / center / right) on the block, so that it matches the visual layout of the pane.
- As a site visitor using a screen reader, I want the section to be labelled by its visible heading, so that the content is navigable and meaningful.

---

## Decisions

- Component lives in `frontend/components/mega-menu-about-us/MegaMenuAboutUs.tsx` ‚Äî decoupled from schema, defines own `MegaMenuAboutUsProps` type
- Resolver lives in `frontend/blocks/mega-menu-about-us/resolver.ts` ‚Äî merges `DEFAULT_ATTRIBUTES` with CMS attrs using `||` for text fields, `??` for alignment enum. No view wrapper ‚Äî `block-views.tsx` imports the component directly from `frontend/components/`
- Heading ID derived from `BaseBlockProps.id` (the block instance ID from `BlocksRenderer`), not a hardcoded constant ‚Äî ensures unique IDs when multiple About Us blocks appear in the same page
- `<section aria-labelledby={id}>` + `<Heading id={id}>` pattern matches `MegaMenuNavigationList` and `MegaMenuFeaturedLinks` ‚Äî consistent WCAG landmark approach across all mega menu content blocks
- `isAlignment` type guard defined at module level in `edit.tsx`, not inside the `Edit` component ‚Äî pure function with no render-scope dependencies
- `MegaMenuAboutUsProps` fields (`headline`, `description`, `alignment`) are required ‚Äî resolver/wrapper always provides them; no component-level defaults needed
- Editor `edit.tsx` lists component props explicitly ‚Äî no `{...attributes}` spread. `||` for clearable text fields (catches empty strings), `??` for numbers/booleans/objects (preserves falsy-but-valid values), direct pass for constrained enums
- `php-blocks.md` renamed to `backend-editor-blocks.md` ‚Äî file documents TSX editor patterns, not PHP. Follows `backend-editor-components.md` / `backend-editor-utils.md` naming convention. (Intermediate name `backend-blocks.md` was used on the feature branch; renamed again for consistency.)
- No `resolveEditorPreview` utility ‚Äî the `||` vs `??` choice is per-field, not per-block. A utility would need to know which fields are clearable text vs numbers vs enums, making the config more complex than just writing the props. Manual explicit listing is 3-5 lines per block; a utility saves those lines but adds indirection and a field-type classification problem. Also avoids confusion with `usePreviewProps`, which remains the single tool for resolver-based blocks.

## Acceptance Criteria

### Behavioral

**WordPress Editor**

- [ ] Given a Mega Menu Column block in the WordPress editor, When I add an "About Us" block inside it, Then the block appears with placeholder text for the headline ("Headline..") and description ("Description..").
- [ ] Given an About Us block is selected in the editor sidebar, When I update the Headline field, Then the editor preview updates immediately.
- [ ] Given an About Us block is selected in the editor sidebar, When I update the Description field, Then the editor preview updates immediately.
- [ ] Given an About Us block with a headline, When I clear the Headline field completely, Then the editor preview does not go blank ‚Äî it shows the default "Headline.." placeholder.
- [ ] Given an About Us block with a description, When I clear the Description field completely, Then the editor preview does not go blank ‚Äî it shows the default "Description.." placeholder.
- [ ] Given an About Us block in the editor sidebar, When I set Block Alignment to "Center", Then the content is centered in the editor preview.
- [ ] Given an About Us block in the editor sidebar, When I set Block Alignment to "Right", Then the content is right-aligned in the editor preview.
- [ ] Given an About Us block in the editor sidebar, When I set Block Alignment to "Left", Then the content is left-aligned in the editor preview.

**Frontend**

- [ ] Given a live page with a Mega Menu that includes an About Us block, When I open the dropdown, Then the headline and description appear correctly.
- [ ] Given two About Us blocks placed in different panes of the same Mega Menu, When I view the page, Then each block renders its own content with no duplicate element IDs.
- [ ] Given an About Us block on the frontend, When I inspect it in browser dev tools, Then `aria-labelledby` on the `<section>` matches the `id` on the heading.
- [ ] Given a viewport narrower than 768px, When I view a page with an About Us block, Then the text scales down appropriately.
