17. **Create `.claude/spec/adrs/0019-block-links-use-ilink.md`** — first codebase-architecture ADR in the corpus. Status `accepted`, date 2026-04-19.

    Required content:
    - `## Context`: block link attributes historically shipped as bare `url: string` or split `url` + `urlTarget` pairs. No guarantee a "link" in one block carried the same fields as another. Migration tickets already in flight — THR-1443, 1445, 1449, 1588, 1619, 1621 — are converting existing blocks to the `ILink` shape one at a time.
    - `## Decision`: all block link attributes take the `ILink` shape (`{ href, target?, rel? }`) defined in `frontend/lib/interfaces/link.ts`. `url: string` and split `url`/`urlTarget` attrs are phased out. Editor controls for links use `LinkControl.tsx` in the sidebar and `ToolbarLink.tsx` in the block toolbar; both wrap `ControlledUrlInput`, which wraps WordPress's `UrlInput` to conform its output to `ILink`.
    - `## Consequences`:
      - Positive: consistent shape across blocks; `target` and `rel` carry predictably; `isLinkObject` enforces at boundaries; `ensureLinkObject` wraps legacy string values at block-attribute write time; editor UX consistent — two components cover every link-editing need.
      - Negative: existing blocks with `url: string` need migration; reviewers must flag `url: string` on new attrs as a regression.
      - Neutral: WordPress's own `UrlInput` is still the underlying control — we wrap, not replace.
    - `## Encounter Protocol` (sub-section within Consequences or its own sibling section):
      - When a block using `url: string` or split `url`/`urlTarget` attributes is discovered during unrelated work, the author judges scope + blast radius + relation to current work before deciding.
      - Migrate in the current ticket when: the block is already being touched by the current work, the migration is contained and doesn't ripple to shared types or multiple consumers, and the PR narrative still reads cleanly.
      - File a Linear ticket when: the block is unrelated, migration touches shared types / resolvers / multiple consumers, or folding in would bloat the PR, broaden review surface, or delay ship.
      - Silent workarounds are the failure mode this ADR prevents — routing data around a `url: string` attribute to avoid touching it is not an option. Flag every time: migrate or ticket, never ignore.
      - Applies the general "do not refactor unrelated code" principle from `.claude/rules/code-standards.md` to this specific migration; not a special-case exception.
    - `## References`: `frontend/lib/interfaces/link.ts`, THR-1443, THR-1445, THR-1449, THR-1588, THR-1619, THR-1621, `.claude/architect/frontend-interfaces.md`, `.claude/architect/backend-editor-components.md`, `.claude/architect/backend-editor-blocks.md`, `.claude/rules/code-standards.md`.

## Decisions

- **Single ticket covers Phase 1 + Phase 2a.** Phase 2a requires Phase 1 (ADRs cite tier names); splitting would create a brittle dependency between two open PRs.
- **New folder `.claude/spec/` rather than `.claude/adrs/` at the top level.** Leaves room for future phases (contracts, schemas, index) without a second reorganization.

- **Three-state git config check, strict string matching.** States: `true` / `false` / unset. Strict matching means only exact `true` or `false` trigger their paths; anything else is treated as unset. Reason: git config values are user-editable strings; lenient matching (`yes` / `1` / `True`) hides behavior behind magic and makes bug reports harder to interpret. Git config wins on universality, discoverability (`git config --list --global`), and fit with existing pattern (`gh.*`, `lfs.*`, etc. already use this shape).
