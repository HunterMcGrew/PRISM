# Plan: THR-1460

## Ticket

THR-1460 â€” Update generated robots.txt to block internal site search and parameterized crawl paths.

## Goal

Stop search engines from wasting crawl budget on internal WordPress search URLs and duplicate URLs produced by common tracking parameters by adding `Disallow` rules to the Next.js-generated `/robots.txt`. Canonical handling already lives in route metadata and is out of scope here.

---

## Decisions

- Only the indexed-homepage branch of `frontend/app/robots.ts` was extended â€” the `noindex` branch already disallows `/` globally so adding rules there would be redundant.
- Search-URL patterns follow Thrive's Next.js conventions, not upstream WordPress conventions: `/*?q=` (Thrive's search param, not WP's `?s=`) and `/search` (no trailing slash â€” Next.js convention). Legacy WP paths like `/page/*/?s=` were removed after Jonathan confirmed Thrive doesn't use `/page` as a route. Nine tracking-parameter shapes (UTM suite, `gclid`, `fbclid`, Mailchimp `mc_cid`/`mc_eid`) stay unchanged since those aren't router-specific.
- Tracking-parameter rules are treated as universal â€” they are not a Yoast-governed concern and are always applied.
- The `unstable_cache`/ISR-style revalidation (`ROBOTS_REVALIDATE_INTERVAL = 60`) and existing `CONTENT_TYPES.ROBOTS` cache tag were left unchanged so operators keep the stale-while-revalidate window.
- No comments were added to the disallow list. Per `.claude/rules/code-comments.md` the patterns are self-describing (Reader Question Test does not demand explanation).
