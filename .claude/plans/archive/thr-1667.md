## Decisions

- THR-1627 (umbrella audit) and THR-1626 (migration/API edits) are already merged to `main` — umbrella docs are current and safe to build on.
- `mega-menu-renderer` and `mega-menu-editor` get dev docs only (no wp-admin user UI). Confirm during drafting.
- Follow-up wiring from Winston's 2026-04-23 audit stays in this branch rather than a new ticket — scope-fit check: same thread (wiring the new per-block docs into the spec system), file overlap is 100%, size is small (7 edits), PR hasn't opened. Creating a separate ticket would be pure overhead.
- Drift prevention lives in Eli's `SKILL.md § After writing` rather than a new rule file — the drift pattern is Eli-specific (template authoring + doc collections), not cross-cutting. The agents who'd benefit are the ones invoking thrive-documentation, so the guidance belongs in the skill that runs it.
- Spec content describes open-ended collections with a membership rule, not a hard count — counts drift silently every time the collection grows. Applies to `.claude/architect/`, `.claude/rules/`, skills, and durable plan sections. Codified in `.claude/rules/writing-voice.md § Count rules, not numbers`. Reason: Eric's PR #1845 finding is the latest of a recurring pattern; `(N files)` parentheticals rot and erode reader trust in the whole spec.
- Mega Menu block-file globs use `mega-menu*.md` (no mandatory hyphen) — the legacy root `mega-menu.md` and umbrella user `mega-menu.md` are part of the per-block doc set but have no hyphen in the filename. Widening the glob is smaller and more accurate than carving those two files out as separate rows.
- Follow-up count-drift cleanup stays in this branch rather than a new ticket — scope-fit check: Eric's minor is open on the active PR, edits touch files already in the diff (plus one new rule entry), size is small (4 tasks, ~15 lines across 3 files), and PR hasn't shipped. The `frontend-blocks.md:13` / `frontend-tests.md:246` offenders stay out of scope — the new rule will surface them when those files are next touched, per Hunter's instruction "we will get it when we see it again."
- Block docs follow a three-way placement rule keyed off `block.json` `"parent"`: (1) systems with an umbrella doc stay flat under the umbrella regardless of individual parent declarations; (2) isolated simple pairs — child has a hard parent, parent is standalone, no family umbrella — nest the child under the parent (`blocks/{parent}/{child}.md`); (3) chains without an umbrella stay flat. Reason: isolated-pair nesting mirrors the frontend source shape (`frontend/blocks/columns/` already holds both `ColumnsBlock.tsx` and `ColumnBlock.tsx` together), umbrella-scoped systems keep the chain readable under one reader entry point, and the `block.json` `"parent"` field + umbrella presence make the test decidable from source. Codified in `.claude/architect/documentation.md § Block Documentation Rule` + `§ Docs-to-Code Mapping`. Mega Menu stays flat per Rule 1 — no doc file changes in this branch. _Superseded 2026-04-24 — see next bullet._
- Rule 1 flipped: umbrella-scoped systems nest into a system folder (`blocks/{system}/` with the umbrella as the folder's entry page), not flat at the top level of `blocks/`. Reason: sidebar scale. A flat layout meant 13 `mega-menu-*` entries at the top level of `user/blocks/` (and 15 on the dev side) scrolled past every unrelated block on every sidebar open. One collapsible system folder fixes that without losing the umbrella's jump-table. Rules 2 (isolated pair) and 3 (chain) are unchanged. Codified in `.claude/architect/documentation.md § Block Documentation Rule` + `§ The decidable test` + `§ Docs-to-Code Mapping`. Mega Menu docs moved in this branch: 14 user pages → `docs/content/user/blocks/mega-menu/`, 15 dev pages → `docs/content/dev/blocks/mega-menu/`. Dev-side system-architecture narrative (`docs/content/dev/architecture/mega-menu.md`) does not move — not a block doc.
