# Plan: THR-1457

## Ticket

https://linear.app/tractru/issue/THR-1457/client-side-code-splitting-for-block-and-template-registries

## Goal

Reduce JS chunk transfer on every page load by moving block and template view `dynamic()` calls into `"use client"` sidecar modules, enabling proper webpack code splitting. Target: ~321kB transferred (down from ~578kB baseline).

---

## Decisions

- `dynamic()` from `next/dynamic` only performs code splitting when called inside a `"use client"` module. In a server module, webpack eagerly includes all referenced chunks in every page's manifest. This is the core architectural insight driving the entire change.
- All block views live in `block-views.tsx` (`"use client"`), both server and client component blocks. Server component blocks wrapped in `dynamic()` from a client module still get code splitting, even though they ship zero JS â€” the chunks are simply not eagerly pre-fetched.
- `block-registry.ts` stays a server module (resolvers import `"server-only"`). It imports pre-built dynamic components from `block-views.tsx` and wires them to resolvers.
- Product template: `template-registry.tsx` static-imports `Product` (async Server Component). `ProductTemplateProvider` (`"use client"`, `ProductTemplateContext.tsx`) wraps the template body inside `Product.tsx` so composite state stays client-side without turning the whole template into a Client Component. (Older `WrappedProduct.tsx` client + `dynamic()` wrapper was removed â€” it broke async RSC.)
- `searchClient` / `createSearchClient` removed from `AppContext` â€” each consumer (`KronosSearchBlock`, `EquipmentListBlock`, `AdvancedSearch`) creates its own instance via `useRef`. This removes the typesense/algoliasearch dependency from the layout chunk.
- `BlockRegistration.view` changed from `() => Promise<{ default: ComponentType }>` to `ComponentType` directly. `BlockService.loadView()` simplified â€” no async, no view cache.
- `TemplateService` simplified to store `ComponentType` directly. `load()` is synchronous.
- PHP `TemplateParts::getTemplatePartsBlocks` returns `'[]'` instead of null for empty template parts. GraphQL fields use `non_null` String since Thrive frontend is the only consumer.
- `KronosSearchFallback.tsx` wraps KronosSearchBlock with `next/dynamic({ ssr: false })` for the search page â€” defers the heavy block from the search route's initial bundle.
- Documented `next/dynamic` code splitting rule in `.claude/rules/headless-architecture.md` and bundle diagnosis workflow in `.claude/rules/bundle-diagnosis.md`.

## Debugged Issues

### `parseBlocks` crash on null template parts
- **Status:** fixed
- **File:** `backend/.../TemplateParts.php`
- **Root cause:** PHP returned null for empty template parts; frontend `parseBlocks` called `.trim()` on null.
- **Recommended fix:** Return `'[]'` from PHP, use `non_null` GraphQL type.
