# Plan: THR-1805

## Ticket

https://linear.app/tractru/issue/THR-1805/re-evaluate-do-not-refactor-unrelated-code-rule-define-related-or

## Goal

Replace the terse "Do not refactor unrelated code" umbrella in `code-standards.md` § General with a defined `### Refactor scope` subsection that states the affirmative permission for local-frame reshape, prohibits drive-by refactor outside the frame, and concretely defines "local frame" — and add the matching affirmative-permission sentence to Clove's § Scope discipline so the discipline composes cleanly across umbrella and skill.

---

## Decisions

- **Keep the umbrella rule and define it; do not remove.**
  - **Root cause:** The umbrella `Do not refactor unrelated code` line in `code-standards.md` § General is more terse than the downstream guidance in Clove's skill (`thrive-code-dev/SKILL.md` § Scope discipline + § Anti-pattern: Drive-by refactoring). When both are in context, the more restrictive umbrella wins on ambiguous calls and overrides Clove's "leave the code better" boy-scout framing — that's the minimum-diff fixation Jonathan hit. Removing the umbrella loses universal coverage; the skill-specific rules don't fire for non-Clove personas (present or future).
  - **Alternatives considered:** (a) Remove the rule entirely, relying on Clove's anti-pattern + Winston's evaluation + Briar's review. (b) Add a "Local frame fitness" bullet to Winston's SKILL.md per Jonathan's narrow suggestion.
  - **Chosen approach:** Restructure the umbrella into a `### Refactor scope` subsection that states the affirmative permission first, defines "local frame" concretely, and prohibits drive-by refactor outside the frame. Beats removal because it preserves universal coverage; beats the Winston bullet because the failure mode is implementation-mode (Clove), not planning-mode (Winston) — fixing it in Clove's existing § Scope discipline addresses the right fire-time.
  - **Implementation guidance:** Two file edits — `code-standards.md` § General gets the new subsection; `thrive-code-dev/SKILL.md` § Scope discipline gets one new sentence affirming local-frame reshape. Both edits are Winston-scope (`.claude/` files).

- **Affirmative-first wording, not prohibition-first.**
  - **Root cause:** Prohibition-first wording ("Do not refactor unrelated code") primes minimum-diff fixation — the reader treats any reshape as risky by default. Affirmative-first wording ("Refactor code you're already modifying, including small local-frame reshape, when…") primes thoughtful local-frame work and lets the prohibition narrow the scope rather than define the whole stance.
  - **Implementation guidance:** Open the new subsection with what's permitted, then narrow with what's prohibited and the "local frame" definition. Same shape as `code-comments` § "Prefer self-documenting code" — leads with the goal, then the boundary.

- **Define "local frame" with concrete IN/OUT examples.**
  - **Root cause:** Without a definition, "related" is a Rorschach test — every reader projects their own threshold. The rule's job is to make the boundary decidable without judgment calls.
  - **Chosen approach:** IN — lines you're already modifying; the function or method containing those lines; helpers extracted from that code; files already in the diff for the ticket. OUT — nearby unmodified code in the same file; sibling files; "while I'm here" cleanup.
  - **Implementation guidance:** Lead each side with a one-line summary, then 2–3 concrete examples. Keep it short — the umbrella's job is to define the rule, not enumerate every edge case.

- **No companion edit to Winston's SKILL.md, no new ADR, no downstream rule edits.**
  - **Root cause:** The minimum-diff fixation is an implementation-mode failure. Winston's evaluation pass already covers local-frame reasoning via "Fit with existing patterns" and "Why does the existing pattern exist?" — a separate "Local frame fitness" bullet would duplicate ground and miss the actual fire-time. An ADR would over-formalize a rule clarification. Downstream rule edits (`code-comments.md`, `component-props-decoupling.md`) that have similar wording are out of scope — they can be updated in a follow-up to cite the new § Refactor scope subsection by name once the umbrella is in place.

   - One affirmative sentence stating what's permitted: refactoring code you're already modifying, including small local-frame reshape (variable initialization, extracting a helper, collapsing branches) when it makes the fix compose more cleanly.
   - One prohibition sentence stating what's not permitted: drive-by refactor of code outside the local frame goes in a follow-up ticket.
   - A short **What "local frame" means** block with IN and OUT examples (2–3 each, one line each).
   - One closing line cross-linking to `.claude/skills/thrive-code-dev/SKILL.md` § Anti-pattern: Drive-by refactoring for the implementation-mode framing.
   - Target length: 5–8 short lines total. Keep the surrounding § General bullets unchanged.

2. **Add affirmative local-frame permission to `.claude/skills/thrive-code-dev/SKILL.md` § Scope discipline.** The existing two paragraphs (line 76-80) cover the boy-scout framing and the flip-side ("when something is clearly wrong, flag/fix"). Insert one new paragraph between them (or extend the first) with the affirmative permission for small local-frame reshape during a fix:
   - State that small reshape inside the local frame (variable init, helper extraction, collapsing branches) is permitted and often correct when the existing frame is making the right answer harder than it needs to be.
   - State the trigger explicitly: when you find yourself bolting fallbacks because the existing code shape doesn't fit the fix, the frame is the problem, not the missing fallback.
   - Cross-link back to `.claude/rules/code-standards.md` § Refactor scope so the skill cites the umbrella as the source of truth.
   - Keep the surrounding paragraphs ("Refactor what you're touching…" and the "flip side: when you're inside a file…") unchanged.

3. **Document the decision in the commit message.** Use the body of the commit (per `.claude/rules/git-conventions.md` heredoc style) to note: kept the rule, defined it instead of removing, why this beats removal (universal coverage), and why no Winston SKILL.md companion edit (right fire-time is implementation-mode). One short paragraph — the next person reading `git log` should see the reasoning without re-litigating.

4. **Self-review pass before ship.** Re-read both edits side by side. Verify: (a) the affirmative permission in `code-standards.md` and the affirmative permission in `thrive-code-dev/SKILL.md` use compatible language and don't drift; (b) the "local frame" definition in the umbrella is the source of truth and the skill cites back to it rather than redefining; (c) Clove's existing `### Anti-pattern: Drive-by refactoring` (line 90-92) still composes cleanly with the new umbrella — no contradiction.
