# Branch Plan: feature/hmcgrew/thr-1131-links-with-icons-block-for-mega-menu-2

## Goal

Add two new Gutenberg blocks — `mega-menu-link-with-icon` (child) and `mega-menu-links-with-icon` (parent) — for the mega menu, with corresponding frontend components, configs, and editor UI.

---

## Decisions

- `MegaMenuLinkWithIcon` is `"use client"` because it uses `useMegaMenuContext` hook
- `MegaMenuLinksWithIcon` is an RSC — no hooks or browser APIs needed
- `IconSelectionGrid` is a new reusable editor-sidebar component for icon picker popover
- `DynamicIcon` was widened to accept `string | null` and includes `SOCIAL_ICONS` (not just `CTA_ICONS`)
- Removed `esty` typo from SOCIAL_ICONS — the correct `etsy: FaEtsy` key already existed separately
- `ensureLinkObject` and `LinkValue` come from PR #1613 (link types and helpers), merged into this branch
- `MegaMenuContext` stub added at `frontend/components/mega-menu-editor/MegaMenuContext.tsx` — remove once mega-menu-editor PR #1543 is merged
- `constants.ts` extracted from `config.ts` in `mega-menu-link-with-icon` block — breaks circular dependency between config.ts and MegaMenuLinkWithIcon.tsx
- `id` kept in `MegaMenuLinkWithIconProps` — it's a framework prop from `BaseBlockProps`/`BlockProcessingContext`, passed by the block renderer even if the component doesn't render it
- `MEGA_MENU_LINK_ICONS` lives in the component (`MegaMenuLinkWithIcon.tsx`), not in schema.ts — the component is the authority on which icons it supports, and per component-props-decoupling the component must not import from `frontend/blocks/`
- Backend edit.tsx rebuilds the same icon map from `@frontend/lib/constants` (CTA_ICONS + SOCIAL_ICONS) instead of importing from the component — avoids webpack pulling `"use client"` module deps (next/link etc.) into the editor bundle
