# Plan: THR-1737

## Ticket

https://linear.app/tractru/issue/THR-1737/third-party-dom-mutating-scripts-audioeye-crash-client-side-navigation

## Goal

Stop third-party DOM-mutating scripts (AudioEye, cookie overlays, translate widgets) from crashing client-side navigation by replacing the `nextjs-toploader` + `nprogress` singleton with a React-controlled progress bar whose state can't be invalidated by external DOM mutations.

## Decisions

- **Replace `nextjs-toploader` + `nprogress`, don't patch it.** The library's design — a module-level singleton that caches DOM references at render time — is fundamentally incompatible with third-party scripts that re-parent or clone the document body. We tried two patch-based approaches (adding AudioEye opt-out attributes on a custom parent container, then monkey-patching `NProgress.set`/`trickle`/`inc`/`done` with orphan-detection recovery). Both left the progress bar in a visibly broken state — stuck partial fill, duplicate bars — even though the crash stopped. Replacing the library eliminates the class of bug rather than whack-a-moling each offending script.
- **React reconciliation is the durability mechanism.** Position and visibility live in component state, so React writes style updates to the fiber's node regardless of where third-party scripts have moved it in the DOM. This is the architectural property that makes the replacement resilient where the patched NProgress was not.
- **Click interception runs in capture phase.** Bubble phase is too late — Next.js `<Link>`'s onClick calls `preventDefault` synchronously, so a bubble-phase listener always sees `defaultPrevented === true` and bails. Capture phase lets us observe the raw click and start the bar before Next.js takes over.
- **`history.pushState` / `replaceState` are patched to serve as nav signals.** Next.js App Router calls `pushState` when navigation finishes, and `react-instantsearch-nextjs`'s `SEOFriendlyRouter` calls `pushState` directly on filter clicks (e.g. EquipmentListBlock, KronosSearchBlock). Without patching, query-only nav (filter changes that don't change pathname) never reaches a completion signal.
- **Completion is scheduled via `queueMicrotask`.** Both Next.js and react-instantsearch call these history methods from inside `useInsertionEffect`, where React forbids scheduling state updates. Deferring to a microtask lets the insertion phase finish before we call `setProgress`.
- **Minimum active duration of 450ms before `complete()` can run.** InstantSearch's routing middleware fires two `pushState` calls in rapid succession on filter clicks (intermediate URL, then final URL). Without this, the bar visibly jumps to 100%, restarts, and re-completes. The gate collapses rapid signals into a single animated pass.
- **URL diffing in `signalUrlChange`.** Next.js and InstantSearch both fire no-op `replaceState` calls on hydration. Skipping same-URL signals prevents a spurious bar flash on every page load.
- **`href="#"` and `href="#!"` are treated as JS-placeholder links, not fragment nav.** InstantSearch's `HierarchicalMenu` renders filter options as `<a href="#">` with an onClick that calls `refine()` and then `pushState`. We start the bar on these clicks so feedback appears immediately instead of waiting for the search query to return. Real in-page anchors (`#section-id`) are still skipped.
- **`popstate` triggers a brief flash via `signalUrlChange`.** Back/forward navigation is instantaneous from the browser's perspective; a full-duration trickle would be misleading. A brief flash matches user expectation for history traversal.
- **30s safety timeout.** If a nav is cancelled (preventDefault with no pushState follow-through — e.g., a link that opens a modal), the bar would otherwise hang at ~90%. The safety timeout force-hides it so the bar is self-healing.
- **Storybook story intentionally waived.** The component is non-visual by default (only visible during navigation) and hard to exercise meaningfully in isolation. Documented in the ticket Notes so reviewers know this is a conscious skip, not an oversight.
- **Jest tests cover the load-bearing behaviors instead of Storybook.** The component is a state machine with timer-driven transitions (start → trickle → complete → hide, gated by `MIN_ACTIVE_MS`). The `450ms` constant is tuned to InstantSearch's empirically observed double-pushState pattern — without a test locking that behavior in, a future "let's make the bar faster" change silently re-introduces the flicker on Equipment pages. See Clove task 4.
- **Single `<ProgressBar />` instance is a hard assumption.** History method patching is per-window. The effect captures the pre-patch originals in closure and restores them on cleanup; a second concurrent instance would corrupt that chain on unmount. Documented in component JSDoc (Clove task 1) rather than enforced by a module-level guard — the documentation is cheaper and catches the scenario that matters (a dev adding a second mount without reading the file first).
- **`SAFETY_TIMEOUT_MS = 5000`, not 30000.** If no nav signal arrives within 5s, no signal is coming; parking a 90%-filled bar for 30s is worse UX than hiding early. 5s covers slow network conditions without dominating the screen on the unusual cancellation paths. See Clove task 3.
- **Third-party script opt-out hints are not a durability strategy.** AudioEye ignored `data-audioeye-ignore` / `data-ae-ignore` hints during the debugging cycle; those attributes are per-vendor and inconsistent across vendors. Future components that need to survive third-party DOM mutation must rely on React reconciliation, not vendor attributes. Promoted to `.claude/architect/frontend-components.md` — see Clove task 5.
- **`shouldInterceptClick` extracts to module scope; the rest of the state machine stays inside the effect.** The eligibility check is closure-free — it produces a boolean from a `MouseEvent` alone — so module-scoping unlocks unit testing and gives JSDoc a place to live. `start`, `complete`, `signalUrlChange`, `clearTimers`, and `scheduleDeferredComplete` share `active`, `startedAt`, and `lastSeenUrl` — three pieces of mutable state that aren't React state (would burn renders) and shouldn't be refs (forces `.current` on every read). Closure over `let` is the right tool for this state machine; pulling the whole shape to module scope or into a hook would either force ref threading or relocate complexity without reducing it. See Clove tasks 11-12.
- **`patchHistoryMethods` is module-scoped because it takes a callback.** The patching block doesn't access closure state directly — it just needs a function to call after each history method fires. Module-scoping shrinks the effect body, gives the 10-line "why" comment a home as JSDoc, and lets the helper be tested in isolation if needed.
- **No `useNavigationProgress` hook extraction.** The component is one consumer at one mount point in the layout. A custom hook would relocate the same closure-bound code without reducing its complexity. Revisit only if a second consumer ever appears (per "don't abstract until you have 2-3 concrete cases").
- **No `createProgressController` factory.** Adds class-like ceremony without earning anything — the closure-bound state machine is already coherent inside the effect, and a factory would still need to live inside the closure to access the same state.

## Review Issues

### prefers-reduced-motion not respected

- **Severity:** `major`
- **Status:** `fixed`
- **File:** `frontend/components/progress-bar/ProgressBar.tsx:262`
- **Problem:** The bar animates `width` (250ms ease-out) and `opacity` (300ms ease-out) unconditionally. Per `.claude/architect/ux-patterns.md` (Motion & Animation: "**`prefers-reduced-motion`:** mandatory") and `.claude/rules/accessibility.md`, all animation must respect this preference. For users with vestibular disorders this is a real harm, not stylistic preference.
- **Suggested fix:** Add a `useReducedMotion` check (or read `window.matchMedia("(prefers-reduced-motion: reduce)").matches` at render). When true, set `transition: "opacity ${HIDE_DURATION_MS}ms ease-out"` only — drop the width transition so progress jumps in discrete steps without the smooth motion. Opacity fade can stay since fades are the minimum essential animation.
- **Fixed in:** Inline `<style>` block with `[data-progress-bar-track]` selector and `@media (prefers-reduced-motion: reduce)` rule that drops the width transition while keeping opacity fade. Follows the codebase precedent in `ScrollingBannerBlock.tsx`. CSS-native — no JS detection, no test mock, no SSR/CSR mismatch surface.

### `replace` callback calls `router.push` instead of `router.replace`

### Orphan `setTimeout` calls not stored in refs

- **Severity:** `minor`
- **Status:** `fixed`
- **File:** `frontend/components/progress-bar/ProgressBar.tsx:120,159`
- **Problem:** `setTimeout(complete, MIN_ACTIVE_MS)` in `signalUrlChange` (line 159) and `setTimeout(complete, MIN_ACTIVE_MS - elapsed)` in `complete()`'s defer branch (line 120) are not stored in refs and not cleared on unmount.
