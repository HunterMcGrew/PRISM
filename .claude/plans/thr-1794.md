# Plan: thr-1794

## Ticket

[THR-1794: Footer-menu resolver crashes when a menu has no location assigned](https://linear.app/tractru/issue/THR-1794/footer-menu-resolver-crashes-when-a-menu-has-no-location-assigned)

## Goal

Eliminate the runtime crash class where `menu.locations` is null (because no menu locations are checked in WP admin), and harden the boundary so the type system tells the truth about what WPGraphQL returns.

---

## Decisions

- **Root cause is a contract violation, not a missing null check.**
  - **Root cause:** `IMenu.locations` is typed `string[]` but WPGraphQL returns `null` for the `locations` field when a menu has no assigned location. The site-globals fetcher passes the GraphQL response through to consumers verbatim (`menus,` at `frontend/lib/wp/functions/get-site-globals.ts:76`) with no DTO→domain normalization, so the lie reaches every consumer.
  - **Implementation guidance:** see the next decision for the chosen seam shape.

- **Hotfix shape introduces a minimal `data-access/menu/gql/` seam, not an inline normalization.**
  - **Root cause:** `IMenu.locations: string[]` claims a non-nullable contract that WPGraphQL doesn't honor. The fix has to choose between hiding the contract violation, exposing it to every consumer, or honoring it at a real seam.
  - **Alternatives considered:**
    - **(A)** Widen `IMenu.locations: string[] | null` and add `?.` at six call sites. Type system enforces every consumer to defend, but regresses domain models toward DTO shapes — every nullable WP field permanently becomes every consumer's problem. Inverts the data-layer architecture from `.claude/architect/data-layer.md`.
    - **(C — original plan)** Inline `node.locations ?? []` in `getSiteGlobals`, keep `IMenu.locations: string[]`, no DTO/mapper. Smallest diff, but creates a silent precedent: types are post-normalization "by convention," with no enforcement that future nullable WP fields get the same treatment. Sasha's first-pass recommendation; sanity-checked at ticket time and walked back after Jonathan's pushback.
  - **Chosen approach:** **(B)** introduce minimum DAL seam. New `GQLMenuDTO` with honest GraphQL nullability, new `mapGQLMenu(dto): IMenu` that normalizes `locations: dto.locations ?? []`, `getSiteGlobals` calls the mapper before assigning to `globals.menus`. Beats C — C's silent precedent reproduces the bug class invisibly for the next nullable WP field. Beats A — A regresses the domain model toward the DTO shape and puts the burden on every consumer in perpetuity. B walks toward the target shape in `.claude/architect/data-layer.md` and survives THR-1795 verbatim.
  - **Implementation guidance:**
    - Scope is the type seam only: DTO type + mapper function + mapper test. Repo class, cached fetcher, and query extraction belong to [THR-1795](https://linear.app/tractru/issue/THR-1795/migrate-menus-and-site-settings-into-the-data-access-layer).
    - The Apollo query in `get-wp-globals.ts` stays composite. `getSiteGlobals` keeps owning the network call.
    - The menu portion of the response is typed as `GQLMenuDTO[]` (in `IWordPressGlobalSettings`) and runs through the mapper before reaching `globals.menus`.
    - `IMenu.locations: string[]` stays exactly as it is — domain model is post-normalization, consumers stay clean.
    - DTO shape reference: `frontend/lib/data-access/gql-event/GQLEventDTO.ts` — every nullable GraphQL field carries `| null`. The mapper is a pure function with no `site` dependency.
    - Watch for friction: if isolating the menu portion of `IWordPressGlobalSettings` resists splitting cleanly (it shouldn't — line 51 of `get-site-globals.ts` types the response loosely), stop and reassess. Fall back is Option A, not Option C.

- **Codified rule: domain model arrays are non-nullable.**
  - **Root cause:** No project rule existed forbidding `T[] | null` on domain models. The `IMenu.locations` violation was the third documented incident in the same shape — `frontend-services.md` § ACF select fields covers a related case but only as an example, not a generally-enforceable rule. Without a rule, the next nullable WP array field reproduces the bug class invisibly.
  - **Alternatives considered:**
    - Skip codification — let the new mapper stand as the example. Rejected: examples don't get enforced at PR review the way named rules do.
    - Standalone `domain-models.md` rule file. Rejected: 25-line constraint doesn't earn its own file and would orphan the rule from the surrounding type-system guidance.
    - Broaden the rule to "domain models eliminate all source-coupled shape" (`.nodes`, `__typename`, etc.). Rejected as out-of-scope: that principle is real but doesn't earn a strict rule until THR-1795 actually applies it across multiple entities — codifying a constraint before the practice is putting the rule before the work.
  - **Chosen approach:** Add § Domain model arrays are non-nullable to `code-standards-ts.md` § Type System (placed after § Modeling external contracts since both are about type-layer contracts). Rule states the constraint, cites THR-1794 as the proof case, prescribes the DTO/mapper boundary as enforcement, and tells Eric and Briar to flag any new domain `T[] | null` at PR review. One-line cross-ref added to `data-layer.md`'s Type Contracts table so the architect doc points at the rule.
  - **Implementation guidance:** Already written by Winston into `code-standards-ts.md` and `data-layer.md`. Lessons entry added. No source-code work for Clove beyond this ticket's hotfix; future PR reviews enforce the rule.

- **Resolver integration tests removed; mapper test + rule are the canonical evidence.**
  - **Root cause:** The 3 resolver integration tests (each runs `locations: null` DTO through `mapGQLMenu` and asserts `menu: null` on the resolver) verify a property the array-nullability rule already guarantees. The mapper test directly proves the boundary contract; the existing "returns null when no X menu exists" tests in each resolver file already cover the no-match path through the standard fixture. The integration tests add ~45 lines for zero unique coverage.
  - **Alternatives considered:** Keep them as defense-in-depth. Rejected: every redundant test is a maintenance load future devs absorb when patterns change. Three layers of defense already in place — type system rejects null at compile time, mapper test proves the contract, rule + reviewers catch any new domain `T[] | null`. A fourth layer doesn't catch anything the first three miss.
  - **Chosen approach:** Remove the 3 resolver integration tests added earlier in this ticket. Keep the mapper test (canonical contract evidence) and the existing resolver tests (cover resolver behavior the mapper doesn't, e.g. `hideOnMobile` defaults, "no match → null").
  - **Implementation guidance:** Clove removes the `it("returns null and does not throw when an upstream menu has locations: null", ...)` test from each of:
    - `frontend/blocks/footer-menu/__tests__/resolver.props.test.ts`
    - `frontend/blocks/colophon-menu/__tests__/resolver.props.test.ts`
    - `frontend/blocks/colophon/__tests__/resolver.props.test.ts`
    Also remove the now-unused `GQLMenuDTO` and `mapGQLMenu` imports from those three test files.

- **`IMenus` named alias eliminated; consumers inline `{ nodes: IMenu[] }`.**
  - **Root cause:** Removing the `IMenus` type from `IWordPressGlobalSettings.menus.nodes` left the alias as a half-orphan — deleted in one place, still in 9 others (`site-globals.ts` + 7 test files + the declaration). Two patterns coexisting (`IMenus` vs the inline shape) is a paper-cut for readers who'd otherwise see a single shape consistently.
  - **Alternatives considered:**
    - Defer to THR-1795. Rejected: the domain reshape in 1795 is a structural change (drop `.nodes` accesses, flatten to `IMenu[]`); this cleanup is just alias removal — same shape, same call sites, ~12 line changes. Different scope, different ticket would be over-correction.
    - Keep `IMenus` and re-add it to `IWordPressGlobalSettings`. Rejected: the new code prefers honest inline `{ nodes: GQLMenuDTO[] }` (DTO at the boundary). Re-adding the named alias would re-introduce a transport-shape wrapper that the seam architecture wants gone.
  - **Chosen approach:** Replace `IMenus` with `{ nodes: IMenu[] }` inline at every call site. Drop the `: IMenus` type annotation from each test's `mockMenus` declaration — TypeScript infers the identical shape from the literal, no need to widen test imports with a named `IMenu`. Delete the `export default interface IMenus` declaration. Same shape, same call sites, no behavior change.
  - **Implementation guidance:** 9 files touched — `menus.ts` (declaration deleted), `site-globals.ts` (inline annotation, swap `IMenus` import for `IMenu`), 7 resolver test files (drop the `IMenus` import + the type annotation, leave the literal). All 43 menu-related tests pass, types clean.

- **Menu domain reshape (drop GraphQL connection structure) deferred to THR-1795.**
  - **Root cause:** Jonathan flagged that `IMenu.menuItems: { nodes: IMenuItem[] }`, `IMenuItem.childItems: { nodes: IMenuItem[] }`, and `connectedObject.__typename` are GraphQL transport artifacts leaking into the domain. The right shape is app-shaped — flat arrays, no `.nodes` wrapper, `__typename` either dropped or replaced with a domain discriminator.
  - **Alternatives considered:** Bundle into THR-1794. Rejected: ~10 menu consumers + ~10 story files of ripple, ~250–300 net lines vs the hotfix's ~70. Production crash exposure favors shipping the surgical fix today; the broader reshape is a natural fit for THR-1795 because every consumer routes through the new `MenuService` interface anyway.
  - **Chosen approach:** Hold the menu domain shape as-is in THR-1794. Update THR-1795's scope to explicitly include the `IMenu`/`IMenuItem` reshape (drop `.nodes`, drop or replace `__typename`, update every menu consumer). THR-1795's description was updated to call this out explicitly.
  - **Implementation guidance:** No source-code work for Clove on this in THR-1794. THR-1795 picks up the reshape when its planning pass starts.

## Debugged Issues

### Footer menu resolver crashes when no menu has a location assigned

- **Status:** `fixed`
- **Fixed in:** Boundary mapper at `frontend/lib/data-access/menu/gql/mappers/map-gql-menu.ts` called from `frontend/lib/wp/functions/get-site-globals.ts`. `IMenu.locations: string[]` is now post-mapper guaranteed.
- **Severity:** Critical
- **Environment:** Production frontend (Next.js RSC). Reproduces any time a WP menu exists with no location checkboxes selected.
- **File:** `frontend/blocks/footer-menu/resolver.props.ts:20`
- **Root cause:** `menu.locations` is `null` (not `[]`) when WPGraphQL returns a menu that has no assigned locations. `IMenu.locations` is typed `string[]`, so the resolver calls `.includes()` on null and throws. The site-globals fetcher passes the GraphQL response through without normalizing the field.
- **Steps to Reproduce:**
  1. In WP Admin → Appearance → Menus, edit any menu that has the `Footer` location currently checked.
  2. Uncheck all "Display location" checkboxes and save.
  3. Load any page with a `footer-menu` block. Page renders 500 / app crashes.
- **Expected behavior:** No menu matches the footer location, `menu` resolves to `null`, the block renders nothing (or its empty state).
- **Actual behavior:** `TypeError: Cannot read properties of null (reading 'includes')` thrown from `resolver.props.ts:20`, BlocksRenderer surfaces a 500 / digest error to the user.
- **Recommended fix:** Normalize `menus[].locations` at the boundary in `frontend/lib/wp/functions/get-site-globals.ts` — map each node to `{ ...node, locations: node.locations ?? [] }` before storing in `globals.menus`. Keep `IMenu.locations: string[]`. See `## Decisions` for full rationale.
- **Suggested tests:**
  1. `getSiteGlobals` unit test with a mocked GraphQL response containing `locations: null` on a menu node — assert the returned `menus[].locations` is `[]`.
  2. Resolver tests for `footer-menu`, `colophon-menu`, `colophon` extended with a fixture menu where `locations: null` — assert no throw and `menu` resolves to `null` when no other menu matches the wanted location.
- **Linear:** `not synced`

### Latent: same crash exists in colophon-menu and colophon resolvers

- **Status:** `fixed`
- **Fixed in:** Same boundary mapper as the footer-menu issue — single seam fixes all three resolvers simultaneously, no per-resolver edits.
- **Severity:** High
- **Environment:** Production frontend. Will crash any site that uses the `colophon-menu` or `colophon` block once any menu in WP has no location set.
- **File:** `frontend/blocks/colophon-menu/resolver.props.ts:20`, `frontend/blocks/colophon/resolver.props.ts:23`
- **Root cause:** Same as above — both resolvers use `menu.locations.includes(...)` with no `?.` guard. Three other resolvers (`navigation-menu`, `site-header`, `site-footer`, `header`) defended with `m.locations?.includes(...)`, so the inconsistency is the surface symptom of the underlying contract bug.
- **Steps to Reproduce:** Any WP menu with no location checkboxes selected, on a page using `colophon-menu` or `colophon` block.
- **Expected behavior:** Block renders empty state.
- **Actual behavior:** Same TypeError as the footer-menu crash.
- **Recommended fix:** Boundary normalization (see footer-menu issue) fixes all three sites simultaneously. No per-resolver edits needed.
- **Suggested tests:** Covered by the resolver-test additions in the recommended fix above.
- **Linear:** `synced` — covered by the THR-1794 ticket description.
