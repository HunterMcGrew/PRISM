# Plan: Mega Menu Sidebar Transitions

## Ticket

https://linear.app/tractru/issue/THR-1559/restore-slide-transitions-for-mega-menu-editor-sidebars

## Goal

Restore slide-in/slide-out transitions for both the right sidebar (block inspector) and left sidebar (block inserter/list view) in the Mega Menu Editor admin page.

---

## User Stories

Not applicable — this is a bug fix / improvement.

---

## Decisions

- Both sidebars must stay in the DOM at all times — toggle visibility via CSS transitions, not conditional rendering (mount/unmount).
- Use `aria-expanded` attribute on the skeleton wrappers to drive the transition state — this is both the CSS hook and an accessibility improvement (screen readers announce panel open/close state).
- Width-based transition (not transform) — matches the original WordPress InterfaceSkeleton pattern. The content area reclaims space smoothly as the sidebar width animates to 0.
- `overflow-x: hidden; overflow-y: auto` resolves the tension between THR-1553's scroll fix and the transition's need to clip content horizontally.
- `visibility: hidden` with a delayed transition (0s delay matching the width duration) hides collapsed content from the accessibility tree and tab order after the animation completes.
- The existing `aria-expanded` transition CSS in `sidebar/styles.scss:46-68` is dead code and should be removed — transitions are now handled at the skeleton wrapper level.
- Respect `prefers-reduced-motion` — disable sidebar transitions for users who prefer reduced motion.
- Update `$sidebar-transition-duration` from 300ms to 650ms in `_variables.scss` — 300ms feels too abrupt for a 320px panel.

Key changes:
- Remove `{isSettingsOpen && ...}` and `{leftPanel !== null && ...}` wrappers
- Add `aria-expanded` driven by the existing state
- `tabIndex` is conditional — only focusable when expanded

### Task 2: `styles.scss` — Add transition CSS to skeleton sidebar wrappers

**File:** `src/apps/mega-menu-editor/styles.scss:128-145`

Replace both `&__sidebar` and `&__secondary-sidebar` blocks:

```scss
&__sidebar {
  display: block;
  width: $sidebar-width;
  max-width: $sidebar-width;
  flex-shrink: 0;
  min-height: 0;
  overflow-x: hidden;
  overflow-y: auto;
  border-left: 1px solid $gray-300;
  transition: width $sidebar-transition-duration (650ms) ease,
              border-left-width 0s 0s,
              visibility 0s 0s;

  &[aria-expanded="false"] {
    width: 0;
    border-left-width: 0;
    visibility: hidden;
    transition: width $sidebar-transition-duration (650ms) ease,
                border-left-width 0s $sidebar-transition-duration (650ms),
                visibility 0s $sidebar-transition-duration (650ms);
  }
}

&__secondary-sidebar {
  display: block;
  width: 350px;
  flex-shrink: 0;
  min-height: 0;
  overflow-x: hidden;
  overflow-y: auto;
  border-right: 1px solid $gray-300;
  transition: width $sidebar-transition-duration (650ms) ease,
              border-right-width 0s 0s,
              visibility 0s 0s;

  &[aria-expanded="false"] {
    width: 0;
    border-right-width: 0;
    visibility: hidden;
    transition: width $sidebar-transition-duration (650ms) ease,
                border-right-width 0s $sidebar-transition-duration (650ms),
                visibility 0s $sidebar-transition-duration (650ms);
  }
}
```

Transition logic:
- **Opening:** `width` animates over 300ms, `border-width` and `visibility` apply immediately (0s delay)
- **Closing:** `width` animates over 300ms, `border-width` and `visibility` are delayed by 300ms (apply after slide completes)

## Debugged Issues

### Sidebar transitions broken — conditional rendering prevents CSS animation
- **Status:** `fixed`
- **Fixed in:** `editor.tsx` (always render wrappers + aria-expanded), `styles.scss` (width transitions + prefers-reduced-motion), `sidebar/styles.scss` (removed dead CSS), `_variables.scss` (650ms duration)
- **Severity:** Medium
- **Environment:** WordPress admin — Mega Menu Editor custom app (`/wp-admin/admin.php?page=mega-menu-editor`)
- **Files:**
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/editor.tsx:741,763` — conditional rendering of sidebar wrappers
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/components/sidebar/Sidebar.tsx` — never sets `aria-expanded`
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/components/sidebar/styles.scss:46-68` — dead transition CSS (aria-expanded never set)
  - `backend/plugins/gravity-platform-core/src/apps/mega-menu-editor/components/left-sidebar/styles.scss` — no panel-level transition CSS
- **Root cause:** Two issues: (1) Both sidebars use conditional rendering (`{condition && <div>}`) in editor.tsx — React unmounts the DOM node on close, so CSS transitions have nothing to animate. (2) The right sidebar's transition CSS is keyed on `[aria-expanded]` but Sidebar.tsx never sets that attribute. Left sidebar has no transition CSS at all.
- **Steps to Reproduce:**
  1. Navigate to the Mega Menu Editor admin page
  2. Click the Settings gear icon to toggle the right sidebar
  3. Observe it appears/disappears instantly with no slide animation
  4. Toggle the block inserter or list view via the left sidebar buttons
  5. Observe same instant appear/disappear
- **Expected behavior:** Both sidebars slide in and out with a 300ms ease transition
- **Actual behavior:** Both sidebars appear/disappear instantly (mount/unmount)
- **Recommended fix:**
  1. Always render both sidebar wrapper divs in editor.tsx — remove the `{condition && ...}` conditional rendering
  2. Pass `aria-expanded={isSettingsOpen}` to the right sidebar wrapper and `aria-expanded={leftPanel !== null}` to the left sidebar wrapper
  3. The right sidebar's existing transition CSS in sidebar/styles.scss will activate once the attribute is set
  4. Add matching transition CSS for the left sidebar (transform + width toggle keyed on aria-expanded)
  5. Update the `__sidebar` and `__secondary-sidebar` skeleton CSS to transition width when aria-expanded changes
- **Suggested tests:** Toggle each sidebar — verify 300ms slide animation in both directions. Verify sidebar content is not interactable when collapsed. Verify screen readers announce expanded/collapsed state.

## Acceptance Criteria

### Behavioral
- [ ] Given the Mega Menu Editor admin page is open, When the Settings gear icon is clicked to open the right sidebar, Then the sidebar slides in from the right over approximately 650 milliseconds
- [ ] Given the right sidebar is open, When the Settings gear icon is clicked to close it, Then the sidebar slides out to the right over approximately 650 milliseconds
- [ ] Given the Mega Menu Editor admin page is open, When the block inserter or list view button is clicked, Then the left sidebar slides in from the left over approximately 650 milliseconds
- [ ] Given the left sidebar is open, When the close button is clicked, Then the left sidebar slides out to the left over approximately 650 milliseconds
- [ ] Given either sidebar is collapsed, When navigating with the Tab key, Then no elements inside the collapsed sidebar receive focus
- [ ] Given a screen reader is active, When a sidebar is toggled, Then the screen reader announces the expanded or collapsed state

### Non-behavioral
- [ ] The sidebar transitions respect the user's `prefers-reduced-motion` system setting — when reduced motion is preferred, sidebars toggle instantly without animation
- [ ] The content area smoothly reclaims the full width when a sidebar closes
- [ ] The fix does not affect sidebar behavior in the standard WordPress block editor — only the Mega Menu Editor admin page
