## Debugged Issues

### Info banner button focus indicator invisible against white panel

- **Status:** `fixed`
- **Fixed in:** `frontend/components/info-banner/MegaMenuInfoBanner.tsx:73` — replaced `focus-visible:outline-current` with `focus-visible:outline-primary` so the focus outline uses an explicit brand color instead of `currentColor` (which collapsed to white-on-white). Tailwind plugin reordered the class to its sort position; no other classes changed.
- **Severity:** High
- **Environment:** Frontend, desktop viewport (lg+), keyboard navigation
- **File:** `frontend/components/info-banner/MegaMenuInfoBanner.tsx:70-79`
- **Root cause:** The button **is in the tab order** — rendered HTML confirms it as a standard `<a href="/...">` element with no `tabindex="-1"`, `aria-hidden`, or `inert`. The reported "Tab does not reach the button" is a perceptual artifact: focus does land on the element, but the focus indicator is invisible, so the tester concludes Tab didn't reach it. Both anchors of the focus indicator collapse against the panel: (1) `focus-visible:outline-current` sets the outline color to `currentColor`, which equals `text-button` = `var(--wp--elements--button--text)`. On primary buttons this WordPress token is typically white. White outline on the `PopoverPanel`'s `bg-white` background is invisible. (2) `ring-offset-current` sets the ring offset color to the same white. The `focus:ring-2` blue ring (default `--tw-ring-color: rgb(59 130 246 / 0.5)`) should appear at 4px outside the button, but its low opacity and the 300ms `transition` on `box-shadow` make it subtle — easy to miss during fast keyboard navigation, especially against a colored button background.
- **Steps to Reproduce:**
  1. Open any dealer site with a mega menu info banner configured (and `--wp--elements--button--text` set to white)
  2. Open the mega menu pane via keyboard
  3. Tab through the pane — focus passes through the info banner button without producing a visible indicator
- **Expected behavior:** A clearly visible focus ring appears on the info banner button when reached via Tab, with at least 3:1 contrast against the white panel background.
- **Actual behavior:** Focus reaches the button, but no visible indicator appears (white outline against white background, faint blue ring).
- **Recommended fix:** In [MegaMenuInfoBanner.tsx:73](frontend/components/info-banner/MegaMenuInfoBanner.tsx#L73), replace `focus-visible:outline-current` with `focus-visible:outline-primary`. The indicator must contrast against the `bg-white` panel, not the button. Per `.claude/rules/accessibility.md`, the focus indicator needs ≥3:1 contrast against adjacent colors (WCAG 1.4.11). Reference: `MegaMenuLinkWithIcon.tsx:55` solves the same problem by omitting `outline-current` entirely and letting the browser default outline color carry, but `outline-primary` is preferred here for brand consistency. The wider question — whether `Button.tsx`'s base styles should keep removing the native outline at all — is a follow-up worth raising with Winston, since other call sites of `Button` on light backgrounds may have the same latent issue.
- **Suggested tests:** Keyboard test in Storybook against a white background — Tab to the button, capture the focus state, verify the indicator is visible. Contrast check on the focus indicator against `#fff`.
- **Linear:** `not synced`

### Tab routing gap in portalled mega menu panel (separate, lower priority)

- **Status:** `open`
- **Severity:** Medium
- **Environment:** Frontend, desktop viewport (lg+), keyboard navigation
- **File:** `frontend/components/navigation-menu-item/NavigationMenuPanel.tsx:39`
- **Root cause:** `NavigationMenuPanel` portals the `PopoverPanel` to `document.body` via React's `createPortal()`, bypassing Headless UI v2's `isPortalled` flag. Headless UI's sentinel focus-guard elements (which would route Tab focus from `PopoverButton` directly into the panel) never render. Tab from the trigger goes to the next element in natural DOM order. This does not prevent Tab from eventually reaching panel content — other elements inside the same panel are reachable — but it does mean keyboard users navigate through unexpected elements before landing in the panel.
- **Recommended fix:** Pass `portal={true}` (or `anchor`) to `PopoverPanel` and remove the manual `createPortal()` wrapper. Headless UI then handles both portaling and sentinel routing. The `buttonBottom` calculation in `MegaMenuNavigationItem.tsx` may become removable if `anchor` handles positioning.
