# Plan: THR-1577

## Ticket

https://linear.app/tractru/issue/THR-1577/header-block-disappears-temporarily-after-expanding-an-initial-mega

## Goal

Fix the temporary disappearance of the header block when a mega menu pane is first expanded on the frontend.

---

## Decisions

- Root cause is a systemic `useId()` hydration mismatch caused by `next/dynamic` (react-loadable) rendering `null` during hydration while chunks load. This shifts the React component tree, causing every subsequent `useId()` call to produce different IDs on server vs client. Headless UI's `Popover` and `Disclosure`, plus 6 of our own components, all use `useId()` — the mismatch triggers React hydration recovery, which destroys and re-renders the affected Suspense boundary's content.
- `React.lazy()` was the architecturally correct fix but is incompatible with Next.js App Router — RSC module serialization requires `next/dynamic` for client component references that cross the server→client boundary. `React.lazy()` resolves to `[object Object]` instead of a component at runtime. This is a Next.js platform constraint.
- Immediate fix scoped to mitigating the header symptom: `<Suspense>` boundary inside `NavigationMenuPanel` isolates portal content so hydration recovery doesn't take out the entire header. The underlying `useId()` mismatch still exists globally — tracked in follow-up ticket.
- Add `relative z-10` to `<header>` elements across all variants — the header should have its own stacking context. Independently correct regardless of the hydration issue.
- `block-views.tsx` remains unchanged — `next/dynamic` stays as the only viable code-splitting mechanism for block views crossing the RSC boundary.

---

## User Stories

- As a site visitor, I want the header to remain visible throughout page load and mega menu interaction, so that navigation is never disrupted.
- As a developer, I want zero React hydration errors in the dev console from Headless UI components, so that `useId()` tree stability is guaranteed.

## Debugged Issues

### Header block disappears temporarily after expanding an initial mega menu pane
- **Status:** `fixed`
- **Severity:** High
- **Environment:** Frontend (staging.thrivewebsiteplatform.com), John Deere HeaderVariantFour with menuType="mega-menu"
- **File:** `frontend/lib/services/block/block-views.tsx`
- **Root cause:** `next/dynamic` uses react-loadable, which renders `null` during hydration for unloaded chunks. This changes the React component tree shape vs SSR, shifting all `useId()` values. Headless UI's `Popover` (mega menu) and `Disclosure` (accordion), plus 6 project components (`MegaMenuFeaturedLinks`, `MegaMenuFeaturedBanner`, `MegaMenuInfoBanner`, `MegaMenuNavigationList`, `EquipmentAddonList`, `EquipmentAddonItem`), all use `useId()` and are affected. The mismatch triggers React hydration recovery, destroying and re-rendering the Suspense boundary content.
- **Steps to Reproduce:**
  1. Navigate to a page with a John Deere header using mega-menu menuType
  2. On initial page load, observe the header block briefly disappears and reappears
  3. Also reproducible on mega-menu-editor-preview page (only the navbar section disappears)
  4. In dev mode, open console — React hydration error from Accordion `DisclosureButton` is visible on every page
- **Expected behavior:** Header remains visible throughout page load and mega menu interaction
- **Actual behavior:** The header block temporarily disappears during hydration recovery, then reappears
- **Recommended fix:** Replace `next/dynamic` with `React.lazy()` in `block-views.tsx`. `React.lazy()` suspends cleanly, preserving server HTML during hydration instead of rendering `null`.
- **Suggested tests:** Verify zero hydration errors in dev console. Verify header doesn't flash. Compare production bundle sizes.
- **Fixed in:** Added `<Suspense fallback={null}>` inside `NavigationMenuPanel` portal to isolate hydration recovery. Added `relative z-10` to all 6 header variant `<header>` elements. `React.lazy()` approach attempted but reverted — incompatible with Next.js RSC. Systemic `useId()` hydration mismatch deferred to follow-up ticket.
