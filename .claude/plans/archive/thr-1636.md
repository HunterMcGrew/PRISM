## Decisions

- Seven explicit manifest routes instead of a single `.claude/**` — excludes `.claude/lessons.md`, `.claude/docs/`, `.claude/design/mocks/` which are working notes and artifacts, not durable spec. Verbosity is intentional and documents what "spec content" means.
- `.claude/references/**` included alongside the ticket's original six — reference docs (plan-lookup.md, architect-context.md, user-doc-template.md, doc-callouts.md) are spec content by the same definition; including now costs nothing and avoids a follow-up drift ticket.
- `spec-editing.md` opens with "include, but aren't limited to" preamble — closure-interpretation (treating an enumerated list as exhaustive) is the known LLM failure mode that produced the THR-1634 voice drift.
- `spec-editing.md` loads alongside `skills-ecosystem.md` (via `**`), not instead of it — the manifest collects all matches; the two files cover different surfaces (ecosystem/lifecycle vs voice/reasoning).
- AC generated despite ticket saying "no AC" — the ticket is labeled Improvement (not DX), the four criteria are verifiable without QA, and ADR-0009 requires AC on every ticket.
- `AGENTS.md` §0 and §9 serve different moments — §0 is pre-persona (detection → invoke), §9 is in-persona (ownership → handoff). Kept as two sections with one-line purpose statements at the top of each; redirect language single-sourced to §9 so the two sections can't drift.
- §9 renamed `Role & Routing` → `Ownership & Handoff` — drops "routing" from §9 so §0 owns that word. Sharpens the distinction between detection and handoff without merging the sections.
- All three skill-membership surfaces in `AGENTS.md` (§0 auto-routing table, §9 ownership table, §9 handoff-language block) must agree on roster — 11 skills, no silent exclusions. Same closure-interpretation lesson that motivated the ticket.
- §9 internal naming made consistent with the rename — block label `**Routing language**` → `**Handoff language**`, intro paragraph "route it to the right person" → "hand it off to the right person," and closing sentence "routing table above" → "ownership table above". Without these, the word "routing" would still appear three times in the renamed §9. In-scope extension of Task 7's explicit intent to purge "routing" from §9.
- PR body describes current scope, not PR-open scope — Thrive squash-merges, so the body becomes `main` history at merge. Documented as ADR-0020.
- Two enforcement points for PR body sync — Winston at plan-change, Clove at push. Same precedent as AC sync to Linear (ADR-0009). Skills fire at the moment of drift; rules and templates are passive and don't catch drift alone.
- Section-based ownership — agent owns templated sections it generates; user owns anything they add (screenshots, reviewer notes, any agent-unrecognized section). Auto-sync rewrites agent-owned sections and leaves user-owned sections verbatim. Avoids clobbering manual user edits while still keeping generated scope current.
- Auto-sync is silent, not prompt-based — the precedent (AC sync to Linear, plan `## History` appends) treats confident triggers as action, not confirmation. Prompting every push is friction disguised as safety. Per-session opt-out via "don't touch the PR body" is the escape hatch.
- Scope fit before new tickets — Winston runs a four-signal check (file overlap, subject adjacency, addition size, PR-shipped status) before recommending a new ticket. New tickets have real overhead; coherent in-scope extensions default to the active ticket. Originating incident: this same session, Winston reached for "new ticket" on ~30-line ADR + four small spec edits without the check.
- Brief PR-body-sync surfaces use principle-only inline phrasing where the parenthetical earns its place; procedural surfaces drop the inline definition entirely and rely on the rule 15 link. Reason: listing "Screenshots, Notes" as user-owned without the seed caveat is misleading (agents seed Notes), AND procedural surfaces in `shipping-flow.md` and Winston's SKILL.md are read by agents that already loaded rule 15 via architect context — the inline parenthetical is redundant. Long-form canonical surfaces (rule:66, ADR-0020:25) keep the example list because the seed caveat sits inline for context. This is Path B from Winston's round-4 evaluation; Path A (verbatim wording match per jmotes) was rejected as keeping a redundant inline definition the agent doesn't need.
- Future audit candidate: other procedural surfaces across `.claude/skills/**` and `.claude/references/**` may carry redundant inline definitions of concepts already covered by rule 15-style cross-cutting summaries. Worth a follow-up ticket — "audit procedural surfaces for inline definitions duplicating skills-ecosystem rules" — once we have a second example. Don't generalize the pattern from a single concept.
