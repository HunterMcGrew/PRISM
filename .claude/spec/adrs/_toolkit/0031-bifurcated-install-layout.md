Number: 0031
Title: Bifurcated Install Layout — `.prism/` Canonical with Platform-Dir Build Copies
Status: accepted
Date: 2026-05-03
---

## Context

PRISM ships into a consumer repo as a multi-platform toolkit — Claude Code, Codex, and Cursor are first-class targets. Phase 1's install layout was asymmetric: platform-specific outputs (skills, native configs) went into `.claude/skills/`, `.agents/skills/`, `.codex/agents/`, and `.generated/cursor-skills/`, but the platform-agnostic content (rules, ADRs, architect docs, templates, references, plans, lessons) only landed in `.claude/`. Codex and Cursor consumers never saw rules or architect docs because their platforms don't auto-load `.claude/`.

The asymmetry was structural. Adding a fourth platform would have copied the rules, architect, and ADRs into yet another platform dir — or skipped that platform entirely, repeating the second-class problem.

Five alternatives were considered:

- **(a) `.prism/` only with no platform copies.** Move all platform-agnostic content to `.prism/<area>/`. Drop the platform-dir copies entirely; agents Read from canonical at session start. Rejected: breaks Claude Code's auto-load — every conversation start would need 5–10 Read calls to pull rules, architect docs, manifest, etc. into context. Slower, more expensive, and easy for an agent to skip a Read and miss a rule.
- **(b) Symlinks from platform dirs to `.prism/`.** Windows doesn't support symlinks reliably without elevated privileges or developer mode. Rejected on portability grounds.
- **(c) `.claude/` as canonical with copies into `.codex/` and `.cursor/`.** Privileges Claude over the other platforms, violates the multi-platform symmetry that motivates PRISM.
- **(d) Top-level `rules/` and `spec/` (no leading dot).** Pollutes the consumer's repo root with PRISM-shaped directories that conflict with project-level conventions.
- **(e) Status quo (everything in `.claude/`).** Codex and Cursor stay second-class. Defeats the multi-platform value proposition.

## Decision

Platform-agnostic canonical content lives at `.prism/<area>/`. Platform dirs (`.claude/`, `.codex/`, `.cursor/`) hold build-time copies of the canonical read-only content, regenerated by `pnpm prism:build`, plus their own platform-specific outputs (skills, native configs).

Agent-written content (plans under `.prism/plans/`, `.prism/lessons.md`) lives only at canonical and is not mirrored — propagating these would create write conflicts when an agent edits the canonical file and the next build overwrites the copy.

A build-time path guard fails the build when any markdown file under a copied area in `.prism/` references `.claude/<area>/`, `.codex/<area>/`, or `.cursor/<area>/` paths. Citing one platform's path inside content that propagates to all three is incorrect — the canonical convention is to cite `.prism/<area>/<file>` everywhere, since every platform's copy of the file will then resolve correctly via its own auto-load.

Every copied area carries a managed marker (`.ai-skill-generated`) at its root inside each platform dir. `pnpm prism:check` detects out-of-band edits to managed copies and reports drift.

The distribution surface (consumer install template) renamed from `templates/claude/` to `templates/install/`. The new sub-layout mirrors the bifurcation: `templates/install/.prism/` for canonical content, `templates/install/.claude/` for Claude-specific behavioral guidance, `templates/install/.codex/` and `templates/install/.cursor/` as anchors for future platform-specific content. `AGENTS.md.tmpl` stays at the install root — it's the cross-platform constitution.

## Consequences

- Positive: every platform sees the same rules and architect docs through its own auto-load mechanism, with a single canonical source. Claude Code keeps its zero-Read session start.
- Positive: adding a fourth platform is a one-line addition to `paths.json` plus a new entry in `platformContentCopies` — the bifurcation pattern absorbs it.
- Positive: agent-written content has a single source. A plan edit lands at `.prism/plans/<id>.md`; no platform copy can drift.
- Positive: the path guard makes the cross-reference convention enforceable rather than aspirational. Reviewers don't have to scan for accidental `.claude/rules/` mentions in canonical content — the build catches them.
- Negative: every read-only canonical edit triggers a `pnpm prism:build` to refresh platform copies. Forgotten builds appear as drift in `prism:check`. This is the same cost-benefit as the existing skill-build flow, just with more files.
- Negative: the build script's `paths.json`, `utils.ts`, and `build.ts` get more surface area. The content-copy logic plus the path guard add ~150 lines.
- Neutral: the `.prism/` namespace is reserved at the consumer's repo root. Consumer teams whose project happens to use `.prism/` for unrelated reasons would conflict — this matches the existing reservation of `.claude/`, `.codex/`, `.cursor/`.
